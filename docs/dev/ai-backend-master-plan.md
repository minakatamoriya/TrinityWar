# TrinityWar AI 后端总计划 v0.1

## 一、文档目的

这份文档不是技术总览，而是一份给 AI 连续执行的后端主计划。

目标很明确：

1. 把当前 demo 的内存态后端，逐步改造成真实数据库驱动后端。
2. 保证后续前端仍然可以持续联调，不因为后端重构而停摆。
3. 把大任务拆成 AI 可以稳定完成的中小任务，降低返工和失控概率。
4. 让不懂后端的人也能据此验收进度，而不是只能看代码猜完成度。

这份文档默认的执行前提是：

1. 使用 Node.js。
2. 使用 TypeScript。
3. 使用 NestJS 单体应用。
4. 使用 Prisma。
5. 使用 PostgreSQL。
6. 使用 Redis。
7. 使用 BullMQ。
8. 使用 Swagger/OpenAPI。

## 二、当前项目现状

当前项目已经具备以下基础：

1. 已有可运行的 Web 验证前端。
2. 已有可运行的 Node 服务端。
3. 已有一批围绕首页、建筑、农场、掠夺、战报、阵营的 demo 接口。
4. 已有一批共享 DTO 和页面展示结构。
5. 已有较完整的设计文档和数值文档。

当前最关键的限制是：

1. 服务端核心状态仍以进程内内存态为主。
2. 数据不会像正式游戏那样稳定持久化。
3. 多玩家隔离、并发安全、审计日志、后台查询、定时任务都还不是正式形态。
4. 前端仍保留 mock/fallback 路径，适合验证玩法，不适合长期真实联调。

因此，真正的后端目标不是“把接口再多写几个”，而是把以下能力补齐：

1. 账号和玩家隔离。
2. 数据库存档。
3. 事务结算。
4. 幂等和并发控制。
5. 定时任务。
6. 日志和后台。
7. 稳定的接口契约。

## 三、总目标定义

本计划的阶段性终点是：

1. 当前六大核心页面全部接真实后端。
2. 关键玩家状态存于 PostgreSQL，而不是内存。
3. 关键延迟结算逻辑由 Redis + BullMQ 承接。
4. 前端停止依赖本地 mock 写操作。
5. 服务端具备最小后台、日志和排错能力。

这一阶段不追求的内容：

1. 微服务拆分。
2. 高并发实时同步战斗。
3. 完整商业化系统。
4. 完整活动运营系统。
5. 完整反作弊体系。
6. 复杂社交系统。

## 四、AI 执行总原则

AI 参与后端开发时，必须遵守以下原则：

1. 先定边界，再写代码。
2. 先定数据模型，再写接口。
3. 先定接口契约，再写前端适配。
4. 先做可回滚的小步改动，不直接大重构。
5. 每次只处理一个可验收的任务切片。
6. 每次改动都要带上最小验证步骤。
7. 每次新增真实写链路时，必须明确：数据表、事务、日志、错误处理、幂等策略。
8. 任何涉及资产、成长、奖励、掠夺的逻辑，必须以后端为准，不能让前端本地推导最终结果。

## 五、推荐总架构

### 1. 技术栈固定

后端固定采用：

1. Node.js
2. TypeScript
3. NestJS
4. Prisma
5. PostgreSQL
6. Redis
7. BullMQ
8. Swagger
9. Pino

原因：

1. 这套技术栈对状态类游戏后端非常常见。
2. AI 生成相关脚手架、模块、Prisma schema、NestJS service/controller 的成功率高。
3. 与现有 TypeScript 前端和 DTO 体系兼容。
4. 对你当前阶段足够稳，不需要为未来过度设计。

### 2. 后端模块建议

按下面模块拆分，便于 AI 分段生成：

1. auth 模块：登录、建档、会话。
2. player 模块：玩家基础信息与总览。
3. wallet 模块：金库、余额、待领取收益。
4. building 模块：主城、金库、灵宠上限、防守、扩展分支。
5. field 模块：农场地块、播种、阶段状态、收取。
6. seed 模块：种子库存、解锁状态、图鉴基础数据。
7. army 模块：灵宠/兵力、训练队列、容量。
8. raid 模块：目标池、发起掠夺、结算、保护期、复仇。
9. report 模块：战报读取与摘要。
10. faction 模块：贡献、分红、上缴、公库。
11. task 模块：每日任务、奖励领取、跨天刷新。
12. leaderboard 模块：排行榜快照。
13. job 模块：定时任务与队列。
14. admin 模块：后台查询和修正。
15. audit 模块：资源变动、修正日志、结算日志。

### 3. 环境建议

至少准备三套环境：

1. 本地开发环境。
2. 测试环境。
3. 线上环境。

本地环境必须具备：

1. PostgreSQL。
2. Redis。
3. NestJS API。
4. BullMQ worker。
5. Prisma migration。

## 六、数据库总拆分思路

这里不直接给最终 schema，而是先给 AI 一个不会跑偏的实体边界。

### 1. 玩家身份层

至少包含：

1. player
2. player_auth_identity
3. player_session 或 token 黑名单/缓存

职责：

1. 建档。
2. 玩家唯一身份。
3. 阵营归属。
4. 最近登录时间。

### 2. 玩家主状态层

至少包含：

1. player_wallet
2. player_building
3. player_army
4. player_field_slot
5. player_seed_inventory
6. player_task_state

职责：

1. 保存玩家当前可变状态。
2. 支撑主界面和各页面汇总读取。
3. 作为命令接口写入的主要目标。

### 3. 战斗与结算层

至少包含：

1. raid_target_pool
2. raid_order
3. raid_settlement
4. battle_report
5. revenge_link 或 revenge_state

职责：

1. 记录掠夺生命周期。
2. 保存结算结果。
3. 供战报查询与后台排错。

### 4. 阵营与排行层

至少包含：

1. faction
2. faction_member
3. faction_contribution_log
4. faction_dividend_log
5. leaderboard_snapshot

职责：

1. 阵营信息。
2. 贡献流水。
3. 分红流水。
4. 排行快照。

### 5. 审计与日志层

至少包含：

1. wallet_change_log
2. building_upgrade_log
3. field_harvest_log
4. admin_adjustment_log
5. system_job_log

职责：

1. 查钱去哪了。
2. 查升级何时发生。
3. 查任务是否执行。
4. 查后台是否改过数据。

## 七、后端阶段计划

## 阶段 A：基础底座

### 目标

让后端从 demo 服务变成正式可扩展工程。

### 输出物

1. NestJS 工程骨架。
2. 环境变量模板。
3. Prisma 初始化。
4. PostgreSQL 接入。
5. Redis 接入。
6. BullMQ 基础接入。
7. Swagger 文档。
8. 基础日志中间件。

### 子任务

1. 建立 apps/api 或 services/game-server-v2 工程骨架。
2. 设计模块目录和命名规范。
3. 配置 prisma/schema.prisma。
4. 配置 Docker Compose 本地依赖。
5. 配置健康检查接口。
6. 配置统一错误结构。
7. 配置请求日志与 trace id。

### 验收标准

1. 本地一条命令能启动 API + PostgreSQL + Redis。
2. Swagger 可访问。
3. Prisma migration 能执行。
4. 健康检查接口返回数据库和 Redis 可用状态。
5. 新人拿仓库可以在文档指导下 30 分钟内跑起来。

## 阶段 B：账号与玩家建档

### 目标

让系统从单玩家 demo 变成“至少支持多玩家隔离”的后端。

### 输出物

1. 微信登录最小闭环。
2. 玩家建档接口。
3. 玩家会话与身份中间件。
4. 当前玩家上下文注入机制。

### 子任务

1. 设计 player 与 auth_identity 表。
2. 设计登录流程：code -> openid -> player。
3. 提供开发环境假登录或测试账号机制。
4. 实现登录接口和玩家初始化逻辑。
5. 实现鉴权 guard。

### 验收标准

1. 两个不同账号登录后，看到的是不同玩家状态。
2. 服务重启后玩家数据不丢。
3. 接口层能拿到当前 playerId。
4. 测试环境支持真实微信 code 登录或等价联调方案。

## 阶段 C：主状态落库

### 目标

把当前最关键的玩家状态从内存搬到 PostgreSQL。

### 输出物

1. 钱包状态表。
2. 建筑状态表。
3. 农场地块状态表。
4. 兵力与训练状态表。
5. 种子库存表。
6. 每日任务状态表。

### 子任务

1. 逐个定义 Prisma model。
2. 写 migration。
3. 写 repository/service 层读取逻辑。
4. 建主界面聚合读取服务。
5. 建页面聚合读取服务。

### 验收标准

1. 主界面读取来自数据库，不来自内存。
2. 建筑页、农场页、部队页的基础数据来自数据库。
3. 刷新页面后状态稳定不丢。
4. 数据库可直接查到对应玩家当前状态。

## 阶段 D：核心写链路真实化

### 目标

把现在最常用的动作改成真实事务结算。

### 优先顺序

1. 建筑升级。
2. 农场播种。
3. 农场收取。
4. 造兵/灵宠训练。
5. 领取税收。
6. 领取分红。
7. 任务领取。

### 每条写链路都必须包含

1. 请求 DTO。
2. 参数校验。
3. 当前状态查询。
4. 规则校验。
5. 数据库事务。
6. 日志记录。
7. 返回最新页面或最新摘要。
8. 错误码与错误文案。

### 验收标准

1. 每条链路都能在数据库里看到结果。
2. 连点两次不会重复扣钱或重复发钱。
3. 接口失败时不会出现部分成功、部分失败的脏数据。
4. 前端刷新后状态仍与刚才操作一致。

## 阶段 E：掠夺系统正式化

### 目标

把当前 demo 中最复杂的一条链路变成可持久化、可审计、可重放的真实系统。

### 输出物

1. 目标池持久化结构。
2. 掠夺订单表。
3. 掠夺结算记录。
4. 战报表。
5. 保护期状态。
6. 复仇关联。

### 子任务

1. 定义 raid_target_pool。
2. 定义 raid_order。
3. 定义 battle_report。
4. 实现发起掠夺接口。
5. 实现结算逻辑。
6. 实现保护期逻辑。
7. 实现复仇资格逻辑。
8. 实现战报读取接口。

### 特别要求

1. 发起时必须冻结可用兵力。
2. 结算时必须明确读取哪一时刻的对手状态或快照。
3. 任何一次掠夺都必须留下可查询日志。
4. 掠夺结算不可因重试而重复执行。

### 验收标准

1. 成功掠夺后，双方状态都能在数据库中正确变化。
2. 战报可查询。
3. 保护期可查询。
4. 同一订单不会重复结算。
5. 后台可按玩家和时间检索最近掠夺。

## 阶段 F：异步任务与定时结算

### 目标

把系统里不适合在请求当下完成的逻辑迁到队列系统。

### 首批任务

1. 小时分红结算。
2. 保护期过期。
3. 训练完成。
4. 目标池刷新。
5. 排行榜快照刷新。
6. 每日任务跨天刷新。

### 每个任务必须包含

1. 入队条件。
2. 幂等标识。
3. 重试策略。
4. 失败日志。
5. 手动补偿思路。

### 验收标准

1. Worker 单独运行可处理任务。
2. 任务失败可记录日志。
3. 同一任务不会无限重复执行。
4. 后台能看到任务执行状态或至少看到 job log。

## 阶段 G：后台与审计

### 目标

让系统具备最小排错和人工修正能力。

### 后台第一批功能

1. 玩家检索。
2. 玩家当前资产查看。
3. 建筑等级查看。
4. 农场地块查看。
5. 最近资源流水。
6. 最近战报。
7. 分红日志。
8. 后台修正金币/等级的最小能力。

### 审计日志必须覆盖

1. 钱包资产变动。
2. 建筑升级。
3. 农场收取。
4. 掠夺结算。
5. 分红结算。
6. 后台修正。

### 验收标准

1. 指定玩家最近 24 小时的关键行为能被查到。
2. 后台修正必须留下人、时间、原因、前值、后值。
3. 至少能排查“钱去哪了”“这次升级何时发生”“为什么有这条战报”。

## 九、AI 任务切片规则

AI 不应该接收“把后端做完”这种任务，而应该接收下面这种格式。

### 标准任务模板

任务名称：

1. 例如：接入 player_wallet Prisma model 与 migration。

任务目标：

1. 新增 player_wallet 表。
2. 支持 vault_gold、vault_capacity、wallet_gold、pending_income_gold 等字段。
3. migration 可执行。

输入约束：

1. 使用 Prisma。
2. 不破坏现有 shared DTO。
3. 不修改无关页面逻辑。

输出要求：

1. schema 变更。
2. migration 文件。
3. 对应 repository/service。
4. 简短验证步骤。

验收标准：

1. migration 成功。
2. 数据库可查表结构。
3. 单元测试或最小联调通过。

### 禁止给 AI 的模糊任务

1. 把掠夺做完。
2. 把数据库接上。
3. 重构整个后端。
4. 把 NestJS 都搭好。

这些任务太大，容易让 AI 一次改太多，最后无法验证。

## 十、推荐 AI 提示词模板

### 1. 基础工程类

你是 TrinityWar 的后端开发代理。现在只处理一个小任务：在 NestJS + Prisma + PostgreSQL 工程里新增 [模块名/表名]。不要改动无关模块。输出必须包括：1. 需要修改的文件；2. schema 或代码改动；3. 最小验证步骤；4. 风险说明。任何涉及资产和结算的字段都不要凭空猜测，优先沿用现有文档与 shared DTO。

### 2. 数据模型类

你现在只负责把 [实体名] 从设计文档落成数据库模型。先根据已有文档列出字段、状态、约束和索引建议，再生成 Prisma model。不要直接扩散到 controller 和前端。完成标准是 migration 可执行、字段命名一致、后续可支撑 [页面/动作]。

### 3. 写链路类

你现在只实现 [动作名] 的真实后端写链路，例如建筑升级或农场收取。必须包含：请求 DTO 校验、当前状态查询、规则校验、数据库事务、日志记录、返回最新状态。禁止保留本地成功但服务端失败的假状态。完成后给出最小联调步骤。

### 4. 并发与幂等类

你现在只修复 [接口名] 的重复点击和并发安全问题。先说明风险场景，再实现幂等或版本控制方案。不要扩大到其他模块。输出必须说明：为什么之前可能重复扣费或重复发奖，现在如何避免。

### 5. 队列与任务类

你现在只实现 [任务名] 的 BullMQ worker。要求说明入队时机、job 数据结构、重试策略、失败日志与幂等处理。不要把其他业务一起改掉。

## 十一、验收清单

项目推进中，每周至少检查一次下面这些问题：

1. 服务重启后，数据是否仍然保留。
2. 两个玩家账号是否完全隔离。
3. 当前已有多少动作真正经过数据库事务。
4. 是否仍然存在本地 mock 写操作。
5. 是否存在重复点击会重复结算的接口。
6. 是否已经有最小资源流水日志。
7. 是否已经有 worker 处理延迟任务。
8. 是否已经能从后台或日志中查某个玩家的问题。
9. Swagger 是否仍与真实接口一致。
10. 前端是否还能持续联调，而不是被后端重构阻塞。

## 十二、风险清单

### 1. 规则未冻结

风险：

1. 数据模型反复改。
2. migration 连续重写。
3. AI 生成的代码不断返工。

处理：

1. 第 1 周冻结首发规则口径。
2. 后续规则变动走明确变更单，不临时口头改。

### 2. 表边界不清

风险：

1. 钱包、建筑、地块、战报互相串字段。
2. 后续查询和事务混乱。

处理：

1. 先画实体边界。
2. 先落 schema，再写 controller。

### 3. 一次改太大

风险：

1. AI 一次改十几个文件。
2. 结果无法验证。
3. 出错后难回滚。

处理：

1. 每次只让 AI 做一个最小任务切片。
2. 每次都要求最小验证。

### 4. 前后端同时大改

风险：

1. 接口字段不稳定。
2. 联调长期中断。

处理：

1. 先稳后端读取与写链路。
2. 再逐步去掉前端 fallback。

### 5. 没有日志和后台

风险：

1. 出问题后只能猜。
2. 无法追溯资产异常。

处理：

1. 第 3 周起就补资源日志。
2. 第 5 周必须做后台最小版。

## 十三、最终执行建议

如果你决定主要用 AI 推后端，这份计划最重要的不是“写得全”，而是“始终按顺序执行”。

最稳的推进顺序只有这一条：

1. 先把技术栈和目录结构固定。
2. 先把核心数据表固定。
3. 先把读取从内存改成数据库。
4. 再把写链路一条条事务化。
5. 再补掠夺和异步任务。
6. 再补后台、日志、监控。
7. 最后配合客户端接入。

不要反过来做，也不要让 AI 在 schema 不稳时提前大规模写 controller 和页面适配。

如果后续需要继续细化，应优先补以下三份文档：

1. Prisma 首批 schema 草案。
2. 首批 API 契约清单。
3. 后台最小功能清单。
