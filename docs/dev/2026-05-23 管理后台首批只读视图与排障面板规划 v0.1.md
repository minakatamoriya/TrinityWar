## 一、文档目的

这份文档是给 AI 和开发执行用的管理后台首批只读视图与排障面板规划。

当前仓库里已经有：

1. 后台总体方向说明：[docs/design/2026-05-18 微信小游戏首发技术方案 v0.1.md](docs/design/2026-05-18 微信小游戏首发技术方案 v0.1.md)

3. 一个非常薄的 admin overview 路由：[services/game-server/src/routes/admin.ts](services/game-server/src/routes/admin.ts)

但要真正支撑后续真实数据库联调，仅有 overview 远远不够。

你很快会需要一个最小后台，至少能回答这些问题：

1. 这个玩家当前到底是什么状态。
2. 为什么他领不了奖励。
3. 为什么这块田不能收。
4. 为什么这次掠夺没有到账。
5. 为什么 worker 没把训练结算完。

这份文档的目标，就是把首批后台只读视图、排障面板和最小 admin 只读 API 范围定清楚，让 AI 后面先做“能查清问题”的后台，而不是一上来做复杂运营系统。

## 二、先给结论：首批后台先只做只读排障，不先做修正操作

首批后台建议明确分两层：

1. 只读排障层
2. 受控修正层

而当前阶段先做第 1 层就够了。

### 为什么

1. 没有稳定的只读视图，就算给了修正按钮，也很容易瞎改。
2. 真实数据库刚起步时，最先需要的是“看清楚”，不是“先改数据”。
3. 只读面板更安全，也更适合 AI 先落基础结构。

## 三、首批后台的核心目标

首批后台不追求“运营系统”，只追求下面 4 个目标：

1. 按玩家查当前聚合状态。
2. 查关键订单与日志。
3. 查 worker/job 执行状态。
4. 快速定位问题落在哪条链路。

## 四、首批后台菜单建议

建议首批后台只做 6 个一级菜单。

## 1. 玩家检索

用途：

1. 按 playerId / nickname / dev login key 找玩家。
2. 打开玩家总览页。

## 2. 玩家总览

用途：

1. 聚合展示 wallet、building、army、field、task、recent reports。

## 3. 订单与结算

用途：

1. 查看训练队列。
2. 查看 raid order。
3. 查看 raid settlement。

## 4. 日志与流水

用途：

1. 查看 wallet change log。
2. 查看 building upgrade log。
3. 查看 field harvest log。
4. 查看 faction contribution log。

## 5. 任务与定时器

用途：

1. 查看 daily task state。
2. 查看 system job log。
3. 查看训练完成、分红、跨天刷新等 worker 状态。

## 6. 系统与联调

用途：

1. 查看服务版本。
2. 查看环境。
3. 查看当前启用的 mock / quick-settle / async-settle 等关键开关。

## 六、排障页首批必须能看的订单

## 1. 训练队列

至少显示：

1. queue id
2. playerId
3. queuedCount
4. totalCostGold
5. startedAt
6. finishAt
7. status

### 主要用于排查

1. 为什么训练没完成。
2. 为什么兵力被冻结或没回到 available。

## 2. RaidOrder

至少显示：

1. orderId
2. attackerPlayerId
3. defenderPlayerId
4. sourceTargetPoolId
5. mode
6. status
7. dispatchedUnitCount
8. requestIdempotencyKey
9. dispatchedAt
10. settleAt
11. settledAt

### 主要用于排查

1. 掠夺到底有没有创建订单。
2. 为什么订单没进入结算。
3. 为什么重试后出现冲突。

## 3. RaidSettlement

至少显示：

1. raidOrderId
2. result
3. lootGold
4. depositedGold
5. overflowGold
6. temporaryClaimExpiresAt
7. attackerLoss
8. defenderLoss
9. createdAt

### 主要用于排查

1. 为什么有订单但没到账。
2. 为什么 overflow 进入了临时待领。

## 4. RaidAssetLock 或等价锁定状态

如果后续落了独立锁表，后台必须能看：

1. lockMode
2. status
3. lockedGold
4. expiresAt

### 主要用于排查

1. 为什么目标收益被占住。
2. 为什么通缉超时后没释放。

1. jobName

## 九、系统与联调面板建议展示的全局状态

首批至少建议有一个“系统状态”页，显示：

1. app version
2. 环境名
3. 数据库连接状态
4. Redis 连接状态
5. worker 在线状态
6. QUICK_SETTLE / ASYNC_SETTLE 模式
7. 是否允许 mock reads / mock commands

### 原因

这样排障时能立刻知道：

1. 是数据问题。
2. 是 worker 没跑。
3. 还是环境开关问题。

## 十、首批 admin 只读 API 建议

当前 [services/game-server/src/routes/admin.ts](services/game-server/src/routes/admin.ts) 只有 overview。

首批只读 API 建议至少补下面这些。

## 1. GET /api/admin/players/search

用途：

1. 按 playerId / nickname / devKey 搜玩家。

## 2. GET /api/admin/players/:playerId/overview

用途：

1. 返回玩家聚合总览。

## 3. GET /api/admin/players/:playerId/wallet-logs

用途：

1. 看资源流水。

## 4. GET /api/admin/players/:playerId/building-logs

用途：

1. 看建筑升级流水。

## 5. GET /api/admin/players/:playerId/field-logs

用途：

1. 看收菜与地块操作。

## 6. GET /api/admin/players/:playerId/orders

用途：

1. 看训练队列与 raid order。

## 7. GET /api/admin/raid/orders/:orderId

用途：

1. 看单条 raid order + settlement + reports。

## 8. GET /api/admin/jobs

用途：

1. 看最近 jobs。

## 9. GET /api/admin/system/status

用途：

1. 看服务与环境状态。

## 十一、首批后台前端页面建议

如果后面要做一个最小后台前端，建议先只做 4 页：

1. 玩家搜索页
2. 玩家总览页
3. 订单与日志页
4. 系统状态页

先不要做：

1. 复杂运营配置页
2. 大量可写修正表单
3. 复杂权限系统

## 十二、AI 执行顺序建议

建议后续 AI 按下面顺序实现后台只读能力：

1. 先补 admin overview 与 system status
2. 再补 player search + player overview
3. 再补 wallet/building/field logs
4. 再补 raid order / settlement / report 排障页
5. 最后补 jobs 面板

## 十三、AI 执行模板

### 1. 生成后台只读 overview

你现在只负责生成后台只读总览接口和页面骨架。范围只包括 system status、模块入口、环境信息。不要顺手生成写操作接口。

### 2. 生成玩家排障页

你现在只负责生成玩家总览只读接口与页面。必须展示 wallet、building、army、field、task 的聚合状态。不要顺手生成资源修正按钮。

### 3. 生成 raid 排障页

你现在只负责生成 raid 订单与结算只读排障接口/页面。目标是能按 orderId 看清 order、settlement、report、lock 状态。不要顺手生成通缉修正功能。

## 十四、验收清单

当首批后台只读视图被认为“足够支撑初始开发排障”时，至少要满足：

1. 能按玩家检索到聚合状态。
2. 能看到 wallet/building/field/army/task 核心状态。
3. 能看到训练队列与 raid order。
4. 能看到关键资源流水。
5. 能看到 worker/job 最近执行状态。
6. 不需要写操作也能定位大部分主循环问题。
7. admin 只读 API 与 client API 权限边界清晰。

## 十五、最自然的下一份规划文档

如果还继续只写规划，不落代码，那么下一份最自然的是：

1. 首批 admin API 清单。
2. 运营修正与审计留痕规则。
3. 系统配置与环境开关矩阵。

如果仍以 AI 实操优先，我更建议先写第 1 个。

因为只读视图与排障面板定好后，下一步最容易直接进入实现的，就是把 admin 只读接口清单和返回结构逐条落成契约。
