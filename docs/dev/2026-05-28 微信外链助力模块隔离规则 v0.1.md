# 2026-05-28 微信外链助力模块隔离规则 v0.1

## 1. 硬规则

微信外链助力、站内社交、微信登录建档、正式经济结算必须分成清晰模块，不允许在同一个页面、接口或服务里混写。

这条规则用于后续 AI 开发交接：任何涉及微信小游戏分享入口、助力确认、新老用户识别、奖励发放的任务，都必须先说明自己改的是哪个边界，不能把测试入口直接扩成正式流程。

## 2. 模块边界

### 2.1 测试入口

当前前端首页的助力按钮只用于本地和开发环境模拟。

约束：

1. 只能打开模拟页。
2. 只能展示流程和确认状态。
3. 不能直接接微信 JS-SDK。
4. 不能直接写正式奖励、正式战斗结算或正式建档逻辑。
5. 文案必须明确标注新用户测试或老用户测试。

### 2.2 外链助力页

外链助力页只承载从微信分享链接进入后的轻量确认体验。

约束：

1. 读取一个助力活动。
2. 展示发起者、助力类型、当前状态。
3. 提交一次助力确认。
4. 展示成功、已助力、过期、次数已满等结果。
5. 不直接承载站内好友列表、社交动态、完整掠夺流程或完整新手教学。

### 2.3 新老用户识别

新老用户识别属于认证与建档边界。

正式微信小游戏接入时，识别顺序建议为：

1. 如果本地已有有效登录态，按老用户处理。
2. 如果有微信 code，则通过服务端换取 openid/unionid。
3. 如果 openid/unionid 已绑定玩家，按老用户处理。
4. 如果没有绑定玩家，先记录助力身份，再进入新手建档流程。

外链助力模块只能消费识别结果，不应该自己实现完整登录建档。

### 2.4 正式助力结算

正式助力结算属于独立 `share-assist` 后端模块。

约束：

1. 幂等确认同一助力者对同一活动只能成功一次。
2. 奖励先写入待领取或事件记录，再由玩家登录后的客户端领取。
3. 发起者通知由通知模块发送，不由前端猜测。
4. 浇水、掠夺助力可以共用活动表，但效果执行必须按 assist type 分流。
5. 掠夺助力不得绕过正式战斗结算服务。

### 2.5 站内社交

站内 `social` 模块负责好友、关注、动态和站内互助。

约束：

1. 站内互助可以调用同一套业务效果，但不能复用微信外链页面状态。
2. 社交动态可以记录助力结果，但不能成为奖励发放的唯一凭据。
3. 站内好友关系沉淀是助力后的后续行为，不是外链助力确认的前置条件。

## 3. 文件命名建议

后续正式开发时建议使用以下目录边界：

```text
apps/game-client/src/ui/share/
```

只放外链助力页和分享承载页。

```text
services/game-server/src/share-assist/
```

只放分享活动、助力记录、幂等确认、奖励事件。

```text
services/game-server/src/social/
```

只放站内社交关系、站内动态、站内互助。

```text
services/game-server/src/auth/
```

只放微信登录、openid/unionid 绑定、玩家建档入口。

## 4. AI 开发检查清单

每次改微信外链助力相关功能前，先检查：

1. 这是测试入口、外链页、认证建档、正式结算、站内社交，还是通知模块？
2. 是否把测试按钮写进了正式流程？
3. 是否让前端直接决定奖励或战斗结果？
4. 是否有幂等键或唯一约束防止重复助力？
5. 是否为新用户和老用户分别定义了后续入口？
6. 是否给发起者写入了服务端通知或待读事件？

只要答案不清楚，就先补设计文档再写代码。
