## 3. 本轮目标

本轮目标不是把整套微信小游戏正式版一次性做完，而是做出一个可持续迭代的 canvas2D 战斗回放基础层。

本轮应达成的最终状态：

1. 可以用 canvas2D 播放 10 回合灵宠对撞
2. 可以消费现有 replay，而不是前端自行重算
3. 有稳定的回合节奏、对撞停顿和回合间隔
4. 能区分普通命中、暴击、克制、闪避、燃血
5. 结构上已经拆出播放控制、场景状态、渲染器和特效层
6. 代码形态适合后续迁移到微信小游戏，而不是继续绑死在 HTML 组件里

## 4. 非目标

本轮明确不做：

1. 正式版微信小游戏页面总集成
2. 复杂技能动画
3. 角色骨骼动画
4. 新战斗规则设计
5. 客户端重新计算战斗伤害
6. 完整美术资产定稿
7. 音效系统完整接入
8. 复杂镜头切换和大招表现

## 5. 实施原则

### 5.1 先做播放骨架，再做视觉打磨

优先级顺序：

1. 播放控制
2. 场景状态
3. canvas 基础绘制
4. 血条和飘字
5. notice 和燃血
6. 卡牌化气质
7. 细节特效

### 5.2 replay 是唯一真相

AI 开发必须遵守：

1. 不在 canvas 层重新随机
2. 不在 canvas 层重新计算伤害
3. 不因为动画补间改变最终胜负和最终血量
4. 只允许对“显示中的血量数值”做 tween

### 5.3 首版只验证 3 件事

本轮真正要验证的只有：

1. 节奏是否对
2. 信息是否读得懂
3. 对撞是否有分量

不要在这一轮同时追求“全都很炫”。

### 5.4 代码层必须解耦

AI 开发时必须拆分：

1. `playback controller`
2. `scene model`
3. `renderer`
4. `effects`
5. `assets`

不要写成单文件巨型战斗组件。

### 5.5 默认面向微信小游戏嵌入

本轮 canvas demo 不是一次性 Web 特化方案，必须按“未来要嵌入微信小游戏正式体系”来写。

AI 开发必须遵守：

1. 核心战斗模块优先纯 TS，不把 React 当作规则层依赖。
2. Web 挂载层和战斗核心层分离。
3. 资源加载、时间推进、画布尺寸适配预留 adapter 层。
4. 不在核心模块中写死浏览器专属 API 依赖。
5. 不把小游戏专属逻辑污染到通用播放层。

建议抽象的 adapter：

1. `timeSource`
2. `assetLoader`
3. `viewportProvider`
4. `platformLifecycle`

### 5.6 这一轮禁止的耦合方式

不要做：

1. controller 直接操作 React state 作为唯一数据源
2. renderer 直接读取页面 DOM 位置做布局
3. scene model 直接发起资源加载
4. Web 事件监听直接写进 battle core
5. 把 canvas demo 写成只能运行在某个页面里的一次性试验代码

### 5.7 棋盘适配的强制约束

本轮必须把“真实屏幕尺寸”和“逻辑战斗棋盘”拆开。

AI 开发必须遵守：

1. 先定义固定逻辑棋盘尺寸
2. 所有关键站位和 UI 坐标优先基于逻辑棋盘计算
3. 棋盘按 `contain` 缩放进真实视口
4. 大屏设备允许留白，不允许把对撞距离无限拉长
5. 单位体积首版保持克制，不要把大部分棋盘空间挤满

## 7. 推荐实施阶段

## 7.1 阶段一：建立 canvas 战斗骨架

目标：

先建立最小可运行的 canvas 播放入口，不追求好看。

建议新建或关注文件：

- `apps/game-client/src/battle-canvas/BattleCanvasPlayer.tsx`
- `apps/game-client/src/battle-canvas/battleCanvasTypes.ts`
- `apps/game-client/src/battle-canvas/battleCanvasTiming.ts`

必须完成：

1. 创建 canvas 容器组件
2. 支持传入 replay
3. 支持基础尺寸和缩放策略
4. 定义 canvas 播放所需的本地类型
5. 定义首版统一时序参数
6. 预留 Web 宿主适配边界，不把挂载组件写成唯一入口
7. 明确逻辑棋盘尺寸与真实 viewport 的映射关系

建议首版参数：

1. `battleStartDelayMs = 160`
2. `roundGapMs = 80`
3. `readyMs = 90`
4. `dashMs = 140`
5. `impactFreezeMs = 70`
6. `hpTweenMs = 260`
7. `returnMs = 130`

验收标准：

1. 页面上能出现 canvas 区域
2. 组件能接收 replay 参数
3. 后续阶段不需要推翻这层结构

## 7.2 阶段二：实现播放控制器

目标：

把 replay step 转成统一播放时间线。

建议新建或关注文件：

- `apps/game-client/src/battle-canvas/battleCanvasController.ts`
- `apps/game-client/src/battle-canvas/battleCanvasTiming.ts`
- 参考：`[apps/game-client/src/battle/battleTimeline.ts](D:/TrinityWar/apps/game-client/src/battle/battleTimeline.ts)`

必须完成：

1. 消费现有 `enter / notice / clash / floatingText / hpChange / return / result`
2. 将 step 映射为内部 phase
3. 根据 step 生成一条可逐帧推进的播放时间线
4. 支持 `requestAnimationFrame` 驱动
5. 支持播放开始、播放结束和重播
6. 让 controller 对外只暴露通用接口，不直接依赖 React hook

内部 phase 建议至少包含：

1. `enter`
2. `round_ready`
3. `round_dash`
4. `round_impact`
5. `round_damage`
6. `round_return`
7. `notice`
8. `blood_damage`
9. `result`

验收标准：

1. 即使还没画正式内容，也能在日志里跑出完整 phase 序列
2. 10 回合普通局总时长稳定
3. 燃血局时长会正确变长

## 7.4 阶段四：做纯占位块对撞模拟

验收标准：

1. 即使没有美术，也能一眼看出是在对撞
2. 撞击停顿存在且有重量感

## 9. AI 开发时最容易犯的错误

### 9.1 误把渲染器写成规则层

不要做：

1. 在 renderer 里判断谁赢谁输
2. 在 renderer 里重新算暴击
3. 在 renderer 里改最终血量

### 9.2 误把时间控制写散

不要做：

1. 到处堆 `setTimeout`
2. phase 没有统一枚举
3. 飘字、血条、位移各走各的时间线

### 9.3 误把首版做太重

不要做：

1. 先做复杂粒子系统
2. 先做曲线路径随机化
3. 先做复杂待机动画
4. 先做大面板特写

### 9.4 误拉长单回合

不要做：

1. 为了炫把每回合做到 1.5 秒以上
2. 每次暴击都加很长停顿
3. 每次 notice 都挡住画面太久

### 9.5 误做成 Web 专用实现

不要做：

1. 核心模块里大量使用 `window`、`document`
2. 依赖 CSS 动画驱动主要战斗表现
3. 资源路径和页面结构强绑定
4. 逻辑只能靠 React 组件挂载后才能运行

## 10. 建议开发顺序

建议按以下顺序提交，便于稳定推进：

1. 搭 canvas 入口和类型
2. 做 controller
3. 做 scene
4. 做占位块对撞
5. 接血条与飘字
6. 接 notice 与燃血
7. 接卡牌化表现
8. 接真实 replay
9. 做性能和节奏调参

原因：

1. 节奏没对，后面越做越返工
2. 可读性没立住，爽感判断会失真
3. 规则对不上，后续微信小游戏接入风险最高

## 11. 最小验收清单

这一轮最小可接受完成态应满足：

1. 有独立 canvas 战斗播放器骨架
2. 能播放真实 replay
3. 有前摇、冲刺、撞击停顿、回位
4. 有血条、飘字、结果
5. 有燃血 notice 和燃血扣血表现
6. 普通局 10 回合播放不拖
7. 暴击、克制、燃血三类反馈可区分
8. 结构上已经为微信小游戏迁移留出边界
9. 核心播放模块不依赖 DOM/CSS 才能成立

## 12. 一句话结论

这不是“把 HTML 页面画到 canvas 上”的简单翻译任务，而是一次战斗播放层重构。AI 开发必须按“先播放骨架、再可读性、再气质、最后接真实 replay”的顺序执行，才能稳。
