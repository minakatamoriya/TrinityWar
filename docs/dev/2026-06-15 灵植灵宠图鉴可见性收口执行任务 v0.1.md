# 2026-06-15 灵植灵宠图鉴可见性收口执行任务 v0.1

## 一、文档目的

本文用于承接“灵植 / 灵宠图鉴可见性与解锁规则”专项的落地执行。

当前目标不是继续讨论规则，而是按已经确认的口径，把前后端、回放、窥视和提示文案逐步收口到一致状态，方便 AI 和人工协作开发。

规则依据：

- [2026-06-15 灵植灵宠图鉴可见性与解锁规则 v0.1](</d:/TrinityWar/docs/design/2026-06-15 灵植灵宠图鉴可见性与解锁规则 v0.1.md>)

## 二、执行原则

### 1. 分步骤修改

本专项必须分步骤推进，不建议一次同时重写前端、后端和回放。

原因：

1. 可见性同时影响图鉴、掠夺、窥视、回放和提示反馈，联动范围大。
2. 若同时改动过多，容易出现“后端已脱敏、前端仍用旧字段”或“前端按新规则、后端仍泄露真名”的半成品状态。
3. 分步骤可以保证每一轮都有明确可验证结果。

### 2. 后端先行，前端跟进

后续一律按以下职责推进：

1. 后端负责可见性判断与脱敏数据输出。
2. 前端负责根据后端状态渲染图标、文案、角标、提示与动画。
3. 禁止新增“前端读取真实字段后自行判断是否隐藏”的实现。

### 3. 每步必须有验收点

每一步都要满足：

1. 改动范围清晰。
2. 可通过构建或接口验证。
3. 不依赖后续步骤才能看见本步是否正确。

## 三、暂不处理范围

本轮先不做：

1. 新增复杂情报系统。
2. 窥视后临时显示灵宠真名。
3. 新增图鉴奖励链。
4. 大范围重构灵植或灵宠的其他成长系统。

## 四、推荐执行顺序

### P0. 文档与口径锁定

目标：

确认设计文档与执行文档可作为后续开发依据。

完成标准：

1. 设计文档已明确后端 / 前端职责。
2. 设计文档已明确窥视不识别灵宠种类。
3. 设计文档已明确状态升级必须提示。

当前状态：

- 已完成。

### P1. 后端灵宠可见性裁决收口

目标：

让灵宠的“是否显示真名”由后端统一判断，先堵住真实名称泄露。

范围：

1. 掠夺目标列表。
2. 掠夺目标详情。
3. 战斗结果返回。
4. 战斗回放数据。

核心改动：

1. 未通过精魄 / 拥有永久解锁的灵宠，不返回真实名称。
2. 回放只返回战斗发生当时玩家有权看到的名称。
3. 窥视只返回属性情报，不返回物种识别结果。

验收点：

1. 未解锁灵宠在列表、详情、战斗、回放中都显示 `未知 / ？？`。
2. 若战斗后未掉到该灵宠精魄，重新查看回放仍不显示真名。
3. 若战斗后首次获得该灵宠精魄，刷新后图鉴可见。

建议优先文件：

1. `services/game-server/src/client-read/scene-content.assembler.ts`
2. `services/game-server/src/raid/raid-target.service.ts`
3. `services/game-server/src/raid/raid-settlement.service.ts`
4. `packages/shared/src/index.ts`

### P2. 前端灵宠图鉴与回放改为消费后端状态

目标：

让前端不再自行猜测真名是否可见，全面消费后端裁决后的字段。

范围：

1. 顶部灵宠图鉴弹窗。
2. 灵宠页图鉴弹窗。
3. 战斗回放页。
4. 与灵宠显示相关的背包、奖励、提示入口。

核心改动：

1. 删除前端基于真实 `definition.label` 的可见性推断。
2. 图鉴状态统一显示 `未可见 / 收集中 / 待合成 / 已拥有` 等明确状态。
3. 进度使用真实 `shardUnlockRequired`，不再写死 100。

验收点：

1. 前端不依赖真实灵宠名自行脱敏。
2. 普通 / 稀有 / 传说进度分别显示 `20 / 60 / 100` 门槛。
3. 战斗回放中的名称与后端返回保持一致。

建议优先文件：

1. `apps/game-client/src/ui/common/SpiritCodexModal.tsx`
2. `apps/game-client/src/modules/spirit/spiritCodexPresentation.ts`
3. `apps/game-client/src/ui/scenes/ArmyScene.tsx`
4. `apps/game-client/src/battle/RaidBattleScreen.tsx`
5. `apps/game-client/src/battle/BattleCard.tsx`

### P3. 灵植显示口径与好友田规则补齐

目标：

保证灵植遵循同一原则，但保留好友田公开名称的例外规则。

范围：

1. 灵植图鉴。
2. 好友灵田预览。
3. 灵植发现与正式解锁提示。

核心改动：

1. 好友田仍直接显示作物名称。
2. 好友田显示不推进访问者自己的图鉴可见性。
3. 灵植图鉴只按玩家自己的研究 / 解锁链路变化。

验收点：

1. 拜访好友田看到陌生灵植时，可见真名。
2. 返回自己的图鉴后，该灵植若未研究到，仍不点亮。
3. 灵植正式解锁时有明确提示。

建议优先文件：

1. `services/game-server/src/social/social.service.ts`
2. `apps/game-client/src/ui/common/PlantCodexModal.tsx`
3. `apps/game-client/src/modules/farm/seedPresentation.ts`

### P4. 状态升级提示补齐

目标：

任何“可见性升级”或“正式解锁”都必须让玩家明确感知。

范围：

1. 灵宠首次获得精魄。
2. 灵宠达到待合成。
3. 灵宠正式入栏。
4. 灵植发现。
5. 灵植正式解锁。

核心改动：

1. 补齐 toast / 解锁弹框 / 奖励浮层中的状态升级提示。
2. 区分“图鉴已可见”“可合成”“已拥有”“已解锁”，避免混用。

验收点：

1. 首次获得某灵宠精魄时，玩家立刻收到“图鉴已可见”提示。
2. 某灵宠达到门槛时，玩家立刻收到“已达到合成条件”提示。
3. 灵植正式解锁时，玩家立刻收到“灵植 XXX 已解锁”提示。

建议优先文件：

1. `apps/game-client/src/App.tsx`
2. `apps/game-client/src/modules/spirit/spiritPresentation.ts`
3. `apps/game-client/src/ui/common/GlobalUnlockModal.tsx`
4. 相关命令返回 summary 的服务端文件

### P5. 旧文案与旧口径清理

目标：

清掉所有仍残留的“统一 100 精魄”“窥视解锁真名”“前端自行判断真名”的旧口径。

范围：

1. 前端文案。
2. 旧设计文档。
3. 默认 schema 文案或注释。

验收点：

1. 玩家可见文案中不再出现固定“满 100 后”。
2. 核心设计文档不再与最新规则冲突。

## 五、推荐开发节奏

建议按 3 轮提交推进：

1. 第一轮：`P1 + P2`
   先堵住后端泄露和前端误判。
2. 第二轮：`P3 + P4`
   再补齐灵植口径和所有提示反馈。
3. 第三轮：`P5`
   最后统一扫尾旧文案与旧文档。

## 六、最小验收清单

完成本专项时，至少需要人工或脚本确认以下结果：

1. 未解锁灵宠在列表、详情、战斗、回放中不显示真名。
2. 未掉精魄时，打完后回放仍不显示真名。
3. 首次获得灵宠精魄后，图鉴立刻变为可见并显示进度。
4. 灵宠门槛按普通 20、稀有 60、传说 100 显示。
5. 窥视只显示属性信息，不识别宠物种类。
6. 好友田中的灵植名称可以显示，但不会点亮访问者图鉴。
7. 灵植正式解锁、灵宠首次获得精魄、灵宠达到合成门槛时都有即时提示。

## 七、结论

现在适合按步骤开始改，而且建议保留这份执行任务文档作为 AI 排任务依据。

后续如由 AI 接手开发，默认按以下规则执行：

1. 先改后端裁决，再改前端展示。
2. 先收灵宠，再补灵植与提示。
3. 每一步都必须能独立构建和独立验收。
