# 2026-06-06 轻量机器人交互测试设计 v0.1

## 一、目标

机器人系统第一版用于模拟玩家今后的日常操作，帮助发现真实链路中的状态错误、接口错误、并发风险和后台排障盲区。

第一版不追求智能决策，不做完整 AI 玩家，只做可重复、可记录、可复现的脚本玩家。

## 二、第一版范围

只覆盖三条日常行为：

- 种田收菜。
- 养宠。
- 掠夺。

第一版固定从 3 个机器人开始：

- `robot-farmer-001`：种田收菜。
- `robot-spirit-001`：养宠。
- `robot-raid-001`：掠夺。

不在第一版做：

- 每阵营 100 个机器人。
- 后台创建复杂循环任务。
- 复杂社交、聊天、阵营战争或智能策略。
- 绕过业务逻辑直接改数据库的压测。

## 三、执行方式

第一版用命令行启动：

```powershell
npm run robot:smoke --workspace @trinitywar/game-server
```

命令职责：

- 创建或复用 3 个开发机器人玩家。
- 每个机器人执行一条固定日常路径。
- 每一步写入机器人动作日志。
- 输出本次任务结果。

后台第一版只做观察：

- 查看最近机器人任务。
- 查看机器人玩家列表。
- 查看最近动作日志。
- 查看最近错误。

后台暂不负责创建、启动、暂停复杂机器人任务。

## 四、真实链路原则

机器人应尽量复用真实玩家业务服务：

- 种田收菜复用 `ClientCommandService`。
- 养宠复用 `SpiritService`。
- 掠夺复用 `RaidTargetService` 和 raid settlement。
- 玩家创建复用开发登录和初始化逻辑。

允许为了第一版稳定性做少量测试准备，例如补足金币、种子、灵宠资源和目标池，但实际动作必须走业务 service。

## 五、日志记录

新增机器人任务和动作日志。

任务记录关注：

- 任务名称。
- 模式。
- 计划机器人数量。
- 成功动作数。
- 失败动作数。
- 状态。
- 开始时间、结束时间。

动作记录关注：

- 机器人 ID。
- 绑定玩家 ID。
- 行为类型。
- 动作名称。
- 是否成功。
- 耗时。
- 错误码和错误信息。
- 关键请求摘要。
- 关键结果摘要。

## 六、第一版通过标准

命令执行后：

- 3 个机器人均能创建或复用成功。
- 至少完成一次种田或收菜动作。
- 至少完成一次灵宠购买、升级、恢复或设置主宠动作。
- 至少完成一次掠夺发起和结算。
- 后台能看到机器人任务和动作日志。
- 失败动作必须有错误码或错误信息，不允许静默失败。

## 七、后续扩展

第一版稳定后再扩展：

- 机器人数量从 3 增加到 10、30、100。
- 后台增加启动、暂停、停止按钮。
- 增加循环执行间隔和最长运行时长。
- 增加按阵营分配机器人。
- 增加统计图表：成功率、平均耗时、接口错误 Top、资源变化趋势。
