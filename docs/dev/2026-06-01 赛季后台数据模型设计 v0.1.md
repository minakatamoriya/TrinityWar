# 2026-06-01 赛季后台数据模型设计 v0.1

## 目标

赛季后台需要支持查询、排障、结算追溯和奖励补发。数据模型必须先保证三件事：

1. 当前赛季可以被准确识别。
2. 玩家跨赛季重置不会重复执行。
3. 赛季结束后的关键数据可以保留为历史快照。

## 已有基础表

### game_season

用途：记录赛季周期。

当前字段：

- `season_number`
- `starts_at`
- `ends_at`
- `created_at`
- `updated_at`

后续建议补充：

- `status`：planned / active / settling / ended / archived
- `settlement_started_at`
- `settlement_finished_at`
- `settlement_version`

### player_season_state

用途：记录玩家当前赛季处理状态。

当前字段：

- `player_id`
- `current_season_number`
- `last_reset_season_number`
- `created_at`
- `updated_at`

后续建议补充：

- `last_settled_season_number`
- `pending_reset`
- `pending_settlement`
- `last_reset_at`
- `last_settled_at`

## 建议新增快照表

### player_season_snapshot

用途：赛季结束后冻结玩家本赛季表现，不再依赖会被清零的实时表。

建议字段：

- `id`
- `player_id`
- `season_number`
- `faction_id`
- `contribution_score`
- `sign_in_days`
- `login_days`
- `harvest_count`
- `raid_count`
- `final_rank`
- `reward_tier`
- `snapshot_json`
- `created_at`

唯一约束：

- `player_id + season_number`

### faction_season_snapshot

用途：冻结阵营赛季表现。

建议字段：

- `id`
- `faction_id`
- `season_number`
- `contribution_score`
- `member_count`
- `final_rank`
- `snapshot_json`
- `created_at`

唯一约束：

- `faction_id + season_number`

### season_reward_grant

用途：记录赛季奖励单。奖励先生成记录，再由玩家领取或后台补发。

建议字段：

- `id`
- `player_id`
- `season_number`
- `reward_type`
- `reward_tier`
- `reward_json`
- `status`
- `claimed_at`
- `created_at`
- `updated_at`
- `idempotency_key`

唯一约束：

- `player_id + season_number + reward_type`
- `idempotency_key`

## 后台查询最小字段

P0 后台只读查询需要：

- 当前赛季信息。
- 历史赛季列表。
- 玩家赛季状态。
- 某赛季玩家重置进度。
- 某赛季阵营贡献概览。

P0 不提供任意字段编辑。
