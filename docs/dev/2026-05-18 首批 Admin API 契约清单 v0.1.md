# TrinityWar 首批 Admin API 契约清单 v0.1

## 一、文档目的

这份文档是给 AI 和开发执行用的首批 admin API 契约清单。

前面的文档已经分别定了：

1. 后台首批菜单与只读排障范围：[dev/2026-05-23 管理后台首批只读视图与排障面板规划 v0.1.md](dev/2026-05-23 管理后台首批只读视图与排障面板规划 v0.1.md)
2. 后端模块边界：[dev/2026-05-18 首批 NestJS 模块拆分与目录骨架规划 v0.1.md](dev/2026-05-18 首批 NestJS 模块拆分与目录骨架规划 v0.1.md)
3. service 分层与复用边界：[dev/2026-05-18 后端 Service 分层设计 v0.1.md](dev/2026-05-18 后端 Service 分层设计 v0.1.md)
4. migration 与日志模型边界：[dev/2026-05-18 首批数据库 Migration 切分计划 v0.1.md](dev/2026-05-18 首批数据库 Migration 切分计划 v0.1.md)

但这些文档还没有把 admin API 逐条落成可联调契约。

真正开始做后台接口时，会立刻遇到下面几个问题：

1. 先做哪些 endpoint，顺序是什么。
2. 每个 endpoint 的 query/path/body 最小字段是什么。
3. 返回结构是偏原始表，还是偏聚合视图。
4. 哪些接口首批只读，哪些明确延后到写操作阶段。

这份文档的目标，就是把首批 admin 只读 API 的路径、参数、返回结构、用途和实现优先级一次写清楚，让 AI 可以直接按接口切片落代码，而不是继续从页面描述里反推接口。

## 二、先给结论：首批 admin API 全部只读，且优先返回聚合视图

首批 admin API 只做两类：

1. 系统状态与入口类
2. 玩家排障与日志查询类

不做下面这些：

1. 资源修正
2. 建筑等级修正
3. 批量发奖
4. 配置发布
5. 公告运营

### 返回风格建议

1. 面向后台页面首批使用场景，优先返回聚合 view model。
2. 只有日志列表这类天然表格数据，才返回接近原始记录的结构。
3. 不要把后台首批接口做成“数据库表直出”。

## 三、统一约定

## 1. 路由前缀

统一使用：

1. `/api/admin/*`

与当前已有路由保持一致，参考 [services/game-server/src/routes/admin.ts](services/game-server/src/routes/admin.ts)。

## 2. 鉴权约定

首批文档先默认：

1. 只有内部后台用户可访问。
2. client token 不可直接访问 admin API。
3. 本地 development 可先走简化 dev-admin 鉴权方案，但不能和 client 身份混用。

## 3. 分页约定

所有列表接口统一支持：

1. `page`
2. `pageSize`

首批默认：

1. `page` 从 1 开始
2. `pageSize` 默认 20
3. `pageSize` 最大 100

## 4. 时间筛选约定

所有日志与订单列表接口统一支持：

1. `from`
2. `to`

格式建议：

1. ISO 8601 字符串

## 5. 错误语义约定

首批建议：

1. 400：参数非法
2. 401：未登录后台身份
3. 403：无后台权限
4. 404：对象不存在
5. 409：查询对象状态冲突
6. 500：服务端异常

## 四、首批接口总览

建议首批按下面顺序落：

1. `GET /api/admin/overview`
2. `GET /api/admin/system/status`
3. `GET /api/admin/players/search`
4. `GET /api/admin/players/:playerId/overview`
5. `GET /api/admin/players/:playerId/wallet-logs`
6. `GET /api/admin/players/:playerId/building-logs`
7. `GET /api/admin/players/:playerId/field-logs`
8. `GET /api/admin/players/:playerId/orders`
9. `GET /api/admin/raid/orders/:orderId`
10. `GET /api/admin/jobs`

### 实现顺序建议

1. overview + system status 先做，帮助确认后台活着。
2. player search + player overview 第二批，帮助查玩家当前状态。
3. 日志与订单接口第三批，帮助定位具体链路。
4. jobs 最后补，帮助定位异步问题。

## 五、接口 1：GET /api/admin/overview

## 用途

1. 后台首页入口。
2. 展示应用名、文档入口、已开放模块。

## 现状

当前已经存在一个最小版实现，见 [services/game-server/src/routes/admin.ts](services/game-server/src/routes/admin.ts#L4)。

## Query 参数

无。

## 200 返回建议

```json
{
  "app": "TrinityWar",
  "docs": "/docs",
  "modules": [
    "system-status",
    "player-search",
    "player-overview",
    "logs",
    "raid-orders",
    "jobs"
  ]
}
```

## 首批要求

1. 允许先保留静态模块列表。
2. 但模块名应开始向真实后台菜单靠拢，不要继续用 announcements、events 这类尚未进入本阶段的占位模块。

## 六、接口 2：GET /api/admin/system/status

## 用途

1. 查看服务与环境状态。
2. 快速判断问题是环境、依赖、worker 还是数据。

## Query 参数

无。

## 200 返回结构建议

```json
{
  "app": "TrinityWar",
  "environment": "development",
  "version": "0.1.0",
  "time": "2026-05-14T10:00:00.000Z",
  "database": {
    "status": "up"
  },
  "redis": {
    "status": "up"
  },
  "workers": [
    {
      "name": "raid-settlement",
      "status": "up"
    }
  ],
  "featureFlags": {
    "quickSettle": true,
    "asyncSettle": false,
    "mockReadsEnabled": false,
    "mockCommandsEnabled": false
  }
}
```

## 字段说明

1. `database.status`：至少有 `up`、`degraded`、`down`。
2. `redis.status`：至少有 `up`、`degraded`、`down`。
3. `workers[]`：首批可以只是静态注册状态，不要求复杂监控指标。
4. `featureFlags`：必须能帮助联调快速看出当前运行模式。

## 七、接口 3：GET /api/admin/players/search

## 用途

1. 结果列表不要返回过于沉重的聚合信息。

## 十五、明确延后到第二阶段的 admin API

下面这些接口不要混进首批：

1. `POST /api/admin/players/:playerId/adjust-gold`
2. `POST /api/admin/players/:playerId/adjust-building`
3. `POST /api/admin/players/:playerId/grant-items`
4. `POST /api/admin/config/*`
5. `POST /api/admin/announcements`

### 原因

1. 这些接口都要求更清晰的权限、审计和回滚策略。
2. 现在先解决“查问题”，不要先做“改问题”。

## 十六、DTO 与 shared 边界建议

首批 admin DTO 建议：

1. 先不强行塞进 client shared DTO。
2. 可以在 `packages/shared` 里新增 admin 专属 response type。
3. client 与 admin 共享的才真正复用。

### 原因

1. admin 页面关心的是排障字段，不是客户端表现字段。
2. 强行共用只会让 DTO 语义混乱。

## 十七、AI 实现顺序建议

建议后续 AI 按下面顺序落接口：

1. `overview`
2. `system/status`
3. `players/search`
4. `players/:playerId/overview`
5. `players/:playerId/wallet-logs`
6. `players/:playerId/building-logs`
7. `players/:playerId/field-logs`
8. `players/:playerId/orders`
9. `raid/orders/:orderId`
10. `jobs`

## 十八、AI 执行模板

### 1. 生成 system status 接口

你现在只负责生成 `GET /api/admin/system/status`。请严格按 [dev/2026-05-18 首批 Admin API 契约清单 v0.1.md](dev/2026-05-18 首批 Admin API 契约清单 v0.1.md) 的结构提供环境、依赖、worker 和 featureFlags 字段。不要顺手生成玩家查询接口。

### 2. 生成 player overview 接口

你现在只负责生成 `GET /api/admin/players/:playerId/overview`。目标是提供后台排障用聚合视图，而不是客户端 DTO。不要顺手生成日志接口。

### 3. 生成 raid order 详情接口

你现在只负责生成 `GET /api/admin/raid/orders/:orderId`。必须能同时看到 order、settlement 和 reports。不要顺手生成 admin 写操作接口。

## 十九、验收清单

当首批 admin API 被认为“足够支撑后台联调与排障”时，至少要满足：

1. overview 和 system status 能快速判断环境是否正常。
2. 能按关键词搜到玩家。
3. 能打开单个玩家的聚合排障页。
4. 能按玩家查看钱包、建筑、田地日志。
5. 能按玩家或 orderId 查看订单与 raid 结算信息。
6. 能看到最近 job 执行状态。
7. 首批范围内没有混入高风险写操作。

## 二十、最自然的下一份规划文档

如果还继续只写规划，不落代码，那么下一份最自然的是：

1. 运营修正与审计留痕规则。
2. 系统配置与环境开关矩阵。
3. admin 页面信息架构与线框清单。

如果仍以 AI 实操优先，我更建议下一步直接开始落第一批接口实现。

因为现在后台菜单、只读排障范围、API 清单三层已经接上了，继续补规划的边际收益已经明显下降。
