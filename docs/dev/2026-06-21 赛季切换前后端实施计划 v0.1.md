# 2026-06-21 赛季切换前后端实施计划 v0.1

## 1. 计划目标

本计划用于把 `docs/design/2026-06-01 赛季重置、继承与追赶完整规则 v0.1.md` 中的 P0 赛季切换流程落地成可开发、可验收的前后端闭环。

本轮目标不是做完整赛季奖励、追赶、复杂继承，而是完成：

1. 玩家在线跨过赛季边界后，不需要退出登录也能进入新赛季状态。
2. 前端不用 WebSocket，也能在赛季结束后自动刷新并提示新赛季开始。
3. 后端所有关键写接口不能继续消费旧赛季上下文。
4. 赛季重置只执行一次，旧赛季快照不丢。
5. 常规阵营页不出现转阵营按钮；阵营调整只在新赛季开始窗口开放。

## 2. 设计依据

开发前必须以以下文档为准：

1. `docs/design/2026-06-01 赛季重置、继承与追赶完整规则 v0.1.md`
2. `docs/design/2026-06-18 关系系统与阵营权限重整理 v0.1.md`
3. `docs/dev/2026-06-01 当前赛季重置边界 v0.1.md`
4. `docs/dev/2026-06-01 赛季重置与结算流程设计 v0.1.md`

本计划只补开发落点，不重新定义重置资产边界。

## 3. 当前代码现状

已有基础：

1. `services/game-server/src/season/season.service.ts`
   - `getCurrentSeason()` 已能计算当前赛季号、周数、开始和结束时间。
   - `ensurePlayerSeason()` 已能发现 `lastResetSeasonNumber < currentSeasonNumber` 并触发重置。
   - `resetPlayerForNewSeason()` 已能重置金币、田地、贡献、俸禄、每日任务、种子库存、掠夺目标、资产锁、未完成订单、灵宠等级/经验/突破/血量等。
2. `services/game-server/src/client-read/client-read.service.ts`
   - `getBootstrap()` 当前会调用 `ensurePlayerSeason()`。
3. `services/game-server/src/verification/verify-season-rollover.ts`
   - 已覆盖基础 rollover 验证：贡献清零、田地清空、灵宠等级重置、词条保留、旧赛季快照存在。

主要缺口：

1. `ClientBootstrapResponse.season` 目前只有 `seasonNumber/currentWeek/totalWeeks`，缺少 `startsAt/endsAt` 和 `seasonTransition`。
2. `ensurePlayerSeason()` 当前只返回 `CurrentSeasonState`，无法告诉调用方本次是否刚刚执行 reset。
3. 写接口没有统一赛季前置守卫；跨赛季瞬间可能继续执行旧操作。
4. 前端没有根据 `season.endsAt` 设置自动刷新。
5. 前端没有 `SEASON_ROLLED_OVER` 错误处理流程。
6. 阵营调整窗口和状态字段还没有实现。

## 4. 开发原则

1. 后端是赛季状态唯一权威。
2. 前端只根据后端返回的 `seasonTransition` 刷新和展示，不自行判定是否已经 reset。
3. P0 不接 WebSocket，不接 SSE。
4. 读接口可以在发现跨赛季后执行 reset 并返回新状态。
5. 写接口发现跨赛季后应先执行 reset，但不继续执行本次旧上下文操作。
6. 所有 reset 逻辑必须幂等；同一玩家同一赛季只能执行一次。
7. 新增字段必须先放入 `packages/shared/src/index.ts`，前后端共用。
8. 不把阵营调整入口重新放回常规阵营页。

export interface ClientSeasonStatus {

如果当前 `ClientSeasonStatus` 已在别处定义，直接扩展原定义，不要重复声明。

## 6. 阶段二：让 ensurePlayerSeason 返回切换摘要

目标：

调用方能知道本次请求是否刚刚触发了赛季重置。

涉及文件：

1. `services/game-server/src/season/season.service.ts`
2. `services/game-server/src/client-read/client-read.service.ts`
3. `services/game-server/src/verification/verify-season-rollover.ts`

建议类型：

```ts
export interface EnsurePlayerSeasonResult {
  season: CurrentSeasonState;
  transition: ClientSeasonTransition;
}
```

开发内容：

1. 不建议直接破坏所有调用方。可以新增方法：

```ts
async ensurePlayerSeasonWithTransition(...): Promise<EnsurePlayerSeasonResult>
```

2. 旧的 `ensurePlayerSeason()` 可以继续保留，并内部调用新方法后只返回 `result.season`，避免一次性改动所有调用方。

3. 新方法逻辑：

```ts
const season = this.getCurrentSeason(now);
await this.ensureCurrentSeasonRecord(client, season);
const state = await client.playerSeasonState.findUnique(...);

let previousSeasonNumber: number | null = null;
let resetApplied = false;

if (!state) {
  create playerSeasonState for current season;
} else if (state.lastResetSeasonNumber < season.seasonNumber) {
  previousSeasonNumber = state.lastResetSeasonNumber;
  await generateSeasonSnapshotsBeforeReset(client, previousSeasonNumber);
  await resetPlayerForNewSeason(client, playerId, season.seasonNumber, now);
  resetApplied = true;
} else {
  update currentSeasonNumber;
}

return {
  season,
  transition: buildSeasonTransition({
    season,
    previousSeasonNumber,
    resetApplied,
    factionChoiceStatus,
  }),
};
```

4. `seasonTransition` 字段建议默认：

```ts
refreshRequired = resetApplied;
factionChoiceStatus = 'not_needed';
factionChoiceDeadlineAt = null;
```

阵营调整窗口在后续阶段再接入；但字段先返回，前端可以先渲染新赛季提示。

5. `getBootstrap()` 改用 `ensurePlayerSeasonWithTransition()`，返回：

```ts
season: {
  seasonNumber,
  currentWeek,
  totalWeeks,
  startsAt: season.startsAt.toISOString(),
  endsAt: season.endsAt.toISOString(),
  transition,
}
```

验收标准：

1. 正常登录 bootstrap 返回 `startsAt/endsAt/transition`。
2. 人工把 `player_season_state.last_reset_season_number` 改成旧赛季后，再请求 bootstrap：
   - 玩家状态被 reset。
   - `season.transition.resetApplied === true`。
   - `previousSeasonNumber` 是旧赛季号。
3. `npm run verify:season-rollover --workspace @trinitywar/game-server` 通过，并补断言 `bootstrap.season.transition?.resetApplied === true`。

## 7. 阶段三：写接口赛季前置守卫

目标：

玩家跨赛季瞬间点击旧操作时，后端不会继续执行旧上下文业务。

涉及文件：

1. `services/game-server/src/client-command/client-command.service.ts`
2. `services/game-server/src/common/errors/business-error.ts`
3. `services/game-server/src/common/errors/global-exception.filter.ts`
4. 所有调用 `ClientCommandService` 的 action controller

开发内容：

1. 在 `ClientCommandService` 内新增私有方法：

```ts
private async ensureNoSeasonRolloverForCommand(
  client: Prisma.TransactionClient,
  playerId: string,
): Promise<void>
```

2. 方法内部调用 `seasonService.ensurePlayerSeasonWithTransition(client, playerId)`。

3. 如果 `transition.resetApplied === true`：

```ts
throw new BusinessError({
  code: ErrorCode.SeasonRolledOver,
  message: '新赛季已经开始，请刷新后继续。',
  statusCode: 409,
  details: { seasonTransition: transition },
});
```

如果 `BusinessError` 当前不支持 `details`，需要扩展它和 `global-exception.filter.ts`，让响应体保留：

```json
{
  "code": "SEASON_ROLLED_OVER",
  "message": "新赛季已经开始，请刷新后继续。",
  "seasonTransition": { ... }
}
```

4. 在每个写操作事务最前面调用该守卫。优先覆盖这些高风险操作：

1. `collectField`
2. `startCultivation`
3. `upgradeBuilding`
4. `claimPending`
5. `claimDailyTask`
6. `claimFactionStipend`
7. `submitFactionTask`
8. `factionDonate`
9. `raidClientTarget`
10. `refreshRaidTargets`
11. `claimSeasonSignIn`
12. `claimStarterSeed`
13. 灵宠喂养、突破、合成、洗词条、设置主宠、分解

5. 注意：如果某个写接口本身就是“确认新赛季提示”或“赛季开始调整阵营”，不能被这个守卫挡住；这类接口要单独白名单。

验收标准：

1. 构造旧赛季玩家后调用任意写接口，返回 `409 SEASON_ROLLED_OVER`。
2. 同一次请求不会继续执行旧操作，例如旧田地不会被收获入账。
3. 玩家状态已被 reset，下一次 bootstrap 返回新赛季状态。
4. 正常赛季内写接口行为不变。

## 8. 阶段四：前端接入 seasonTransition 与自动刷新

目标：

不使用 WebSocket，在线玩家也能在赛季结束后进入新赛季流程。

涉及文件：

1. `apps/game-client/src/App.tsx`
2. `apps/game-client/src/api.ts`
3. `apps/game-client/src/shell/appStateTypes.ts`
4. `apps/game-client/src/shell/GlobalFeatureModalHost.tsx`
5. `apps/game-client/src/ui/common/GlobalFeatureModalContent.tsx`

开发内容：

1. `ClientViewModel` 或 bootstrap 状态中保存 `season.transition`。

2. 在 `App.tsx` 中根据 `bootstrap.season.endsAt` 或 view model 内的 season 数据设置 timer：

```ts
useEffect(() => {
  const endsAt = login/bootstrap/viewModel season endsAt;
  if (!endsAt) return;

  const delay = Math.max(new Date(endsAt).getTime() - Date.now() + 2000, 1000);
  const timer = window.setTimeout(() => {
    void refreshAfterPossibleSeasonRollover();
  }, delay);

  return () => window.clearTimeout(timer);
}, [seasonNumber, seasonEndsAt]);
```

3. 增加页面回前台检查：

```ts
useEffect(() => {
  const handleVisibility = () => {
    if (document.visibilityState === 'visible') {
      void refreshAfterPossibleSeasonRollover();
    }
  };
  document.addEventListener('visibilitychange', handleVisibility);
  return () => document.removeEventListener('visibilitychange', handleVisibility);
}, []);
```

4. `refreshAfterPossibleSeasonRollover()`：

1. 调用 `loadClientViewModel()` 或 bootstrap + view model。
2. 如果赛季号变化或 `season.transition.resetApplied` 为 true：
   - 清理本地 optimistic state。
   - 关闭种田选择、资源面板、战斗弹窗等旧上下文。
   - 设置 `globalFeatureModal` 为新赛季提示。
   - 应用最新 view model。

5. `ApiError` 捕获 `SEASON_ROLLED_OVER`：

在所有统一 action catch 中增加：

```ts
if (error instanceof ApiError && error.code === 'SEASON_ROLLED_OVER') {
  await handleSeasonRolledOver(error.seasonTransition);
  return;
}
```

`handleSeasonRolledOver()` 做：

1. show toast：`新赛季已经开始，已为你刷新状态。`
2. 关闭当前操作弹窗。
3. 重新加载 view model。
4. 展示新赛季提示 modal。

6. 新赛季提示 modal 文案：

标题：`新赛季开始`

正文：

```text
旧赛季竞争状态已经结算并重置。你的身份、图鉴、灵宠本体和已解锁资格会保留，本赛季的金币、田地、贡献、军队和临时战斗状态已重新开始。
```

按钮：

1. `进入新赛季`
2. 如果 `factionChoiceStatus === 'available'`，显示 `调整阵营`

验收标准：

1. 玩家停留页面到赛季结束后，前端能自动刷新，不需要退出登录。
2. 页面后台挂起后再回来，能检查并刷新新赛季。
3. 写接口返回 `SEASON_ROLLED_OVER` 时，前端不会继续展示旧操作成功。
4. 新赛季提示只弹一次，不在每次刷新后重复打扰。

## 9. 阶段五：赛季开始阵营调整窗口

目标：

支持“赛季开始可以调整阵营，平时不能调整阵营”的规则。

涉及文件：

1. `services/game-server/prisma/schema.prisma`
2. `services/game-server/prisma/migrations/*`
3. `services/game-server/src/season/season.service.ts`
4. `services/game-server/src/client-command/client-command.service.ts`
5. `services/game-server/src/client-command/client-command.controller.ts`
6. `services/game-server/src/auth/auth.service.ts`
7. `apps/game-client/src/shell/AuthEntryScreen.tsx`
8. 新增或复用前端阵营选择组件

建议数据模型：

在 `player_season_state` 上增加：

```prisma
factionChoiceUsedSeasonNumber Int? @map("faction_choice_used_season_number")
factionChoiceUsedAt DateTime? @map("faction_choice_used_at")
```

如果想保留历史，也可以新增 `player_faction_choice_log`，P0 可先不做。

阵营调整窗口：

1. 推荐窗口：新赛季开始后 24 小时。
2. 配置常量放在 `season.service.ts` 或 game balance 配置。
3. `getFactionChoiceStatus(playerState, season, now)`：
   - 新用户刚建档并已选阵营：`not_needed`
   - 已经调整过当前赛季：`used`
   - 当前时间 <= `season.startsAt + windowMs`：`available`
   - 其他：`expired`

后端接口：

```text
POST /api/client/actions/change-season-faction
body: { factionCode: 'human' | 'immortal' | 'demon' }
```

规则：

1. 先执行 `ensurePlayerSeasonWithTransition()`，但这个接口不能因 `resetApplied` 被挡住。
2. 校验 `factionChoiceStatus === 'available'`。
3. 校验目标阵营存在。
4. 如果目标阵营等于当前阵营，可以直接标记 used，也可以返回 no-op；推荐 no-op 不消耗机会。
5. 如果目标阵营不同：
   - 更新 `player.factionId`。
   - 更新或重建 `factionMember`。
   - 当前赛季个人贡献保持 0。
   - 写入 `factionChoiceUsedSeasonNumber/currentSeasonNumber` 和 `factionChoiceUsedAt`。
   - 重新加载 bootstrap/read model 返回。
6. 不删除好友、关注、历史战报。

前端入口：

1. 不在常规阵营页显示。
2. 只在“新赛季开始”提示 modal 或个人赛季状态入口显示。
3. 复用新用户阵营选择卡片样式，但文案必须是“本赛季阵营调整”。
4. 确认弹窗说明：

```text
本次调整只影响你本赛季的阵营立场和玩法权限。好友、关注和历史记录不会删除。确认后本赛季不可再次更改。
```

验收标准：

1. 新赛季窗口内可以调整一次。
2. 调整后本赛季再次请求返回拒绝。
3. 窗口过期后返回拒绝。
4. 常规阵营页没有调整按钮。
5. 好友、关注关系不被删除。

## 12. 风险与注意事项

1. `generateSeasonSnapshotsBeforeReset()` 必须在 reset 前执行。
2. 多个请求同时跨赛季进入时，必须避免重复快照或重复 reset。必要时使用事务、唯一索引和 upsert 幂等兜底。
3. 写接口触发 reset 后不要继续执行业务，否则会出现“新赛季刚清零又立刻收获旧田地”的问题。
4. 前端 timer 只是体验优化，不能替代后端守卫。
5. 客户端本地 mock 数据也要补 `startsAt/endsAt/transition`，否则 dev 模式会类型失败。
6. 阵营调整窗口的状态由后端返回，前端不要用本地时间自行开放按钮。
7. 如果服务端时间和客户端时间有差异，以 `serverTime` 和后端返回的 `endsAt` 为准。
8. 当前已有大量工作区改动，开发时必须只改赛季切换相关文件，不回滚其它功能。
