# 2026-05-25 掠夺战斗回放模块方案 v0.1

## 1. 目标

在现有掠夺结算链路中插入一段独立战斗表现：

1. 玩家点击发起掠夺。
2. 服务端按发起瞬间的双方状态创建订单并结算。
3. 服务端返回结算结果和 `battleReplay`。
4. 客户端打开独立战斗界面：敌方在上，我方在下。
5. 双方卡片入场、对撞一次、显示伤害/闪避/暴击/克制等关键事件。
6. 双方回到板边，血条下降，显示结果。
7. 玩家确认后承接现有掠夺奖励弹框。

本阶段只做一次对撞，不做多回合、不做技能动画、不引入复杂动画库。

## 2. 快照结算原则

本次掠夺只查询并锁定发起时双方状态。

- 订单创建时写入 `attackerSnapshotJson` 和 `defenderSnapshotJson`。
- 结算规则优先读取订单快照，不再读取对战表现期间的新等级、新血量或新词条。
- 如果玩家在战斗动画播放期间升级、治疗、换主宠，本次战斗结果不变。
- 后续战报、奖励、扣血均以该订单快照对应的结算结果为准。

这保证了微信小游戏或其他客户端只负责播放 `battleReplay`，不会参与结算。

## 3. 模块边界

前端新增独立模块：

```text
apps/game-client/src/battle/
  battleTypes.ts
  battleTimeline.ts
  BattleCard.tsx
  BattleHealthBar.tsx
  BattleFloatingText.tsx
  RaidBattleScreen.tsx
```

模块约束：

- 只接收纯 JSON `RaidBattleReplay`。
- 不调用 API。
- 不读取 `App.tsx` 的全局状态。
- 不直接修改背包、任务、战报。
- 播放结束只触发 `onComplete`。

## 4. 数据契约

`ClientRaidActionResponse.result.battleReplay`：

```ts
interface RaidBattleReplay {
  orderId: string;
  result: 'WIN' | 'LOSS' | 'DRAW';
  title: string;
  summary: string;
  attacker: BattleUnitSnapshot;
  defender: BattleUnitSnapshot;
  steps: BattleStep[];
  rewardsPreview: {
    goldLoot: number;
    items: Array<{ label: string; quantity: number }>;
  };
}
```

`BattleStep` 第一版只表达关键节点：

```ts
type BattleStep =
  | { type: 'enter'; durationMs: number }
  | { type: 'clash'; durationMs: number }
  | { type: 'floatingText'; side: 'attacker' | 'defender'; text: string; tone: 'damage' | 'miss' | 'crit' | 'buff' }
  | { type: 'hpChange'; side: 'attacker' | 'defender'; from: number; to: number; max: number }
  | { type: 'return'; durationMs: number }
  | { type: 'result'; title: string; summary: string };
```

## 5. 今晚 MVP

1. 后端补 `battleReplay` 返回。
2. 前端新增 `battle/` 模块。
3. 掠夺成功后先播放战斗，再弹现有奖励。
4. 保持现有结算、奖励、战报逻辑不变。

## 6. 持久化与复盘

战斗回放属于战斗资产，不只属于发起掠夺当次响应。

第一版在 `RaidSettlement` 上新增 `battleReplayJson`：

```prisma
model RaidSettlement {
  ...
  battleReplayJson Json? @map("battle_replay_json")
}
```

原因：

1. 一场订单只会有一个最终结算，回放和结算一一对应。
2. 攻击方、被掠夺方、后台排障都应该看到同一份回放。
3. 后续战斗规则或词条变化后，老战斗仍按当时保存的 replay 播放。
4. 前端和微信小游戏只播放 JSON，不重新计算战斗。

不要只保存双方配置快照。配置快照适合排查，但 replay 必须保存最终事件序列和血量变化，否则暴击、闪避、掉落等随机结果无法稳定复现。

## 7. 首次播放与战报回放差异

同一份 `battleReplay` 支持两种播放模式：

1. 首次掠夺：接口返回后自动播放，节奏更快，直接入场对撞。
2. 战报回放：先展示双方配置和词条，玩家点击“开始回放”后再入场对撞，便于分析。

两种模式只影响播放器状态机，不影响 replay 数据结构。

流程边界：

- 首次掠夺播放是掠夺命令流程的中间环节。接口已经完成订单创建和结算，客户端播放结束后继续展示本次结算奖励。
- 战报回放是只读流程。它只读取 `battleReplayJson` 并播放，不创建订单、不重新结算、不发奖励、不扣血、不改保护期。
- 因此战报回放接口必须是 GET，只返回 replay，不返回 mutation 后的 home/scenes。

## 8. 词条扩展原则

单位快照不要只绑定攻防血。后续应逐步扩展为：

```ts
interface BattleUnitSnapshot {
  coreStats: {
    attack: number;
    defense: number;
    hpBefore: number;
    hpAfter: number;
    maxHp: number;
  };
  traits: Array<{
    code: string;
    label: string;
    value: number;
    valueType: 'percent' | 'flat' | 'ratio';
    source: 'spirit' | 'equipment' | 'buff' | 'faction' | 'temporary';
    visible: boolean;
  }>;
}
```

第一版 replay 先保留 `attack/defense/hp` 和事件序列；新增词条时通过 `traits[]` 扩展，而不是给每个词条增加顶层字段。
