# TrinityWar 微信小游戏首发整备流程表 v0.1

## 日期

2026-05-25

## 一、文档目的

这份文档不是新增功能计划，而是首发整备流程表。

当前 TrinityWar 的首发核心功能已经基本齐备，后续工作重点不再是继续堆模块，而是把现有系统收口、减负、统一表达，并完成首发体验整备。

当前确认保留的三条核心主线只有：

1. 种田
2. 养宠
3. 战斗

其他系统，例如签到、关注、通知、商店、阵营上缴与领取、法术升级，统一视为支撑系统，不再作为新的主功能方向继续发散。

这份流程表的目标是明确：

1. 接下来应该按什么顺序推进。
2. 每一轮优化要产出什么。
3. 哪些问题先处理，哪些问题后处理。
4. 如何在不减少首发功能的前提下，把首发体验整理清楚。

## 二、当前阶段判断

当前项目不再处于“功能缺失阶段”，而是进入“首发整备阶段”。

当前最重要的问题，不是有没有系统，而是：

1. 玩家第一次进入时是否能快速理解主循环。
2. 玩家每天回来时是否知道先做什么。
3. 玩家被掠夺后是否能看懂结果并继续留存。
4. 现有系统之间的信息密度是否过高，导致理解成本过重。

所以后续推进原则统一为：

1. 不新增大功能。
2. 优先完善当前系统。
3. 必要时做适度简化，但不主动砍首发核心内容。
4. 所有优化都围绕“更易懂、更顺手、更有回访动力”展开。

## 三、首发整备总原则

### 1. 主循环优先于支撑系统

所有页面、文案、反馈、提醒，都优先服务种田、养宠、战斗三条主线。

### 2. 玩家感知优先于后台规则完整暴露

能放后台计算的规则，不在前台堆解释。

### 3. 先解决理解问题，再解决数值问题

如果玩家都没看懂当前系统，数值调优结论很容易失真。

### 4. 每轮只做一种性质的工作

后续迭代严格拆分为：

1. 体验收口轮
2. 链路修复轮
3. 数值校准轮

不要在同一轮里同时大改体验、链路和数值。

## 四、首发整备流程

## 阶段 1：先定首发目标锚点

### 阶段目标

先把首发体验的判断标准定住，后面所有改动都围绕同一套锚点，不再边做边发散。

### 需要回答的 4 个问题

1. 新玩家前 10 分钟要完成什么。
2. 玩家每天上线 5 分钟先做什么。
3. 玩家一周内最强的成长爽点是什么。
4. 玩家被掠夺后为什么还愿意继续玩。

### 输出物

一份极短的首发目标说明，每题只保留一句话答案。

### 验收标准

团队后续讨论任何调整时，都可以直接回到这 4 句目标，不再靠临场判断。

## 阶段 2：给现有系统分层

### 阶段目标

不删除功能，但明确哪些是主循环系统，哪些只是支撑系统，哪些属于后台规则。

### 分层方式

#### A. 主循环必感知

1. 收菜
2. 种子价值
3. 灵宠成长
4. 自动挂机
5. 突破
6. 掠夺目标选择
7. 战斗结果与收益
8. 灵宠碎片/图鉴推进

#### B. 支撑系统

1. 签到
2. 商店
3. 通知
4. 关注
5. 阵营上缴
6. 阵营领取
7. 法术升级

#### C. 后台规则

1. 掉率与概率
2. 匹配保护
3. 资源上限或阈值
4. 成长公式
5. 结算细则
6. 自动托管细节

### 输出物

一份页面与系统分层表，标明每个系统当前属于 A、B、C 哪一层。

### 验收标准

后续任何页面优化时，都能明确哪些信息必须前置展示，哪些只做弱提示，哪些只留给后台处理。

## 阶段 3：逐系统做减负收口

### 阶段目标

不砍功能，只减少理解负担。

### 逐系统统一检查 4 个问题

1. 玩家是否一眼知道这个系统是干嘛的。
2. 玩家是否知道点完按钮会发生什么。
3. 玩家是否知道什么时候应该回来。
4. 玩家是否知道这个系统和主循环哪一环有关。

### 本阶段优先级

#### 1. 种田页优先

必须解决：

1. 哪块田该收。
2. 哪块田快成熟。
3. 哪块田价值更高。
4. 哪块田存在损失风险。

#### 2. 灵宠页优先

必须解决：

1. 当前主宠强度。
2. 当前经验条。
3. 当前经验速度。
4. 自动加速状态。
5. 剩余加速时间。
6. 何时粮尽。
7. 是否能突破。

#### 3. 掠夺入口优先

必须解决：

1. 这个目标能不能打。
2. 这个目标值不值得打。
3. 打输了会损失什么。
4. 打赢了会推进哪条成长线。

#### 4. 奖励入口统一

必须解决：

1. 奖励如何预览。
2. 奖励何时入账。
3. 奖励到账后的反馈是否统一。

### 输出物

按页面整理的“减负改造清单”，不写大方案，只写待处理项。

### 验收标准

任何首次接触页面的人，都能在较短时间内说出“这个页面主要让我干什么”。

## 阶段 4：改成链路验收，不再做孤立功能验收

### 阶段目标

后续检查不再问“这个功能做了没”，而是问“这条玩家路径通不通”。

### 必跑的 5 条链路

#### 1. 新手成长链

拿到种子 -> 种田 -> 收菜 -> 获得养宠材料 -> 主宠成长 -> 第一次突破

#### 2. 主宠养成链

挂机增长 -> 续粮加速 -> 自动成长 -> 升级 -> 突破 -> 继续挂机

#### 3. 掠夺收益链

找目标 -> 开战 -> 看懂结果 -> 获得材料/碎片 -> 回来继续养宠

#### 4. 被打恢复链

收到通知 -> 看懂损失 -> 明确保护/补救/复仇 -> 恢复主循环

#### 5. 日常回访链

签到 -> 领奖 -> 收菜 -> 看主宠 -> 决定是否掠夺 -> 下线

### 每条链都要记录的问题

1. 卡在哪一步。
2. 哪一步信息太多。
3. 哪一步反馈太弱。
4. 哪一步奖励不明确。
5. 哪一步不知道接下来该干什么。

### 输出物

链路回放记录表。

### 验收标准

每条链都能顺畅走通，且主要问题能被定位到具体步骤，而不是泛泛地说“体验不顺”。

## 阶段 5：先补表现层与引导层，再碰数值平衡

### 阶段目标

先让玩家看得懂，再开始调收益、掉率和战斗平衡。

### 优先处理内容

1. 文案统一
2. 信息层级统一
3. 动效反馈统一
4. 提示语统一
5. 回访提醒统一
6. 新手指引与阶段目标统一

### 暂时后置内容

1. 掠夺收益极限平衡
2. 灵宠成长成本细调
3. 碎片掉率深度平衡
4. 洗练材料后期压力
5. 战斗公式精修

### 原因

如果表现层和引导层没收口，玩家对系统的误解会直接干扰数值判断。

### 输出物

首发表现层统一清单。

### 验收标准

玩家在不看额外说明的情况下，也能较自然理解当前行为和结果。

## 阶段 6：做首发封板与露出控制

### 阶段目标

不删功能，但控制露出层级，避免首发信息量过载。

### 逐项判断 3 个问题

1. 这个系统会不会让新手第一次玩就困惑。
2. 这个系统在首发前 3 天是否真的重要。
3. 这个系统是否增加了理解成本，却没有提供明显爽点。

### 可采用的处理方式

1. 不删除，只降低露出频率。
2. 不删除，只后置到二级入口。
3. 不删除，只在特定阶段解锁。
4. 不删除，只保留弱提示，不前置引导。

### 输出物

首发露出优先级表。

### 验收标准

玩家首发前 3 天看到的信息，不会明显超过主循环理解所需。

## 五、建议的实际推进节奏

## 第 1 天

完成：

1. 首发 4 个目标锚点
2. 系统分层表

## 第 2 天

只看种田页与灵宠页。

完成：

1. 信息层级收口
2. 关键反馈补强
3. 无效或重复表达清理

## 第 3 天

只看掠夺入口、战报、被打反馈。

完成：

1. 风险感与收益感表达
2. 掠夺结果可读性
3. 被打后的继续游戏理由

## 第 4 天

只看奖励入口与回访提醒。

完成：

1. 奖励确认口径统一
2. 到账反馈统一
3. 回访优先级统一

## 第 5 天

使用新手账号完整试玩 3 轮。

要求：

1. 不边玩边改。
2. 只记录问题。
3. 问题按链路归类。

## 第 6 天

只修前 10 个最影响首发理解的问题。

## 第 7 天

再完整试玩一轮，开始记录真正的数值和平衡问题。

## 六、后续每轮工作的固定模板

后续每轮优化，都建议按下面模板推进：

### 1. 本轮类型

只能三选一：

1. 体验收口轮
2. 链路修复轮
3. 数值校准轮

### 2. 本轮只处理一个主题

例如：

1. 只处理种田页
2. 只处理灵宠页
3. 只处理掠夺结果页
4. 只处理回访提醒

### 3. 本轮必须给出 3 项结果

1. 改了什么
2. 解决了什么误解
3. 还剩什么暂不处理

### 4. 本轮结束必须回答

1. 是否让主循环更清楚了。
2. 是否减少了玩家理解负担。
3. 是否没有引入新的主路径复杂度。

## 七、当前阶段的优先级结论

按当前项目状态，接下来最优先做的不是新增功能，而是下面 4 件事：

1. 把种田页和灵宠页的信息表达彻底收口。
2. 把掠夺入口、战报和被打反馈做成能看懂的闭环。
3. 把所有领奖和回访提示统一为同一套玩家心智。
4. 跑完整玩家链路，再开始真正做数值校准。

## 八、当前不建议做的事情

1. 不建议继续加新模块。
2. 不建议同时大改体验和数值。
3. 不建议在没有跑完整链路前就下结论说“数值不对”。
4. 不建议因为功能多就直接砍首发核心内容。

## 九、文档使用方式

这份文档用于后续每一轮首发整备时对照执行。

推荐用法：

1. 每次开始前，先标记当前处于哪个阶段。
2. 每次只取一个主题推进。
3. 每次结束后，把问题和结论补回对应阶段的输出物中。

如果后续需要，可以基于本文件继续拆：

1. 种田页专项整备清单
2. 灵宠页专项整备清单
3. 掠夺体验专项整备清单
4. 首发表现层统一清单
