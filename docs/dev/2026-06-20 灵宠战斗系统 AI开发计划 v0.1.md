# 2026-06-20 灵宠战斗系统 AI开发计划 v0.1

## 1. 计划目标

这份计划用于把 `docs/design/2026-06-17 灵宠特质与词条收口稿 v0.1.md` 落地为可运行、可回放、可压测的灵宠战斗系统。

本轮开发目标不是一次性接入所有正式玩法，而是按安全顺序推进：

1. 先让共享战斗核心按新规则跑通。
2. 再让测试 10 回合对撞入口可验证 6 套套路。
3. 再把录像和前端表现对齐。
4. 最后再接入真实掠夺结算。

## 2. 文档复核结论

开发前已复核以下文档：

1. `docs/design/2026-06-17 灵宠特质与词条收口稿 v0.1.md`
2. `docs/design/2026-06-18 灵宠燃血模式重策划 v0.1.md`
3. `docs/design/2026-06-16 灵宠固定特质与稀有度差异设计稿 v0.1.md`
4. `docs/design/2026-05-18 灵宠等级养成与五行战斗设计草案 v0.1.md`

复核结果：

1. 当前正式灵宠名已统一为 `岩龟`、`玄蛇`、`云鹰`、`讹兽`、`狡`、`凤皇`。
2. 历史旧名已从核心设计稿和燃血设计稿中清理。
3. 五行克制正式采用最终伤害修正：克制目标 `+50%`，被目标克制 `-20%`，闪避时不触发。
4. 15 条首发词条池已明确，词条允许重复。
5. `断续`、`裂伤`、`炽燃` 的边界已补齐。
6. 单回合结算顺序已写入设计稿，可以作为开发依据。

当前没有阻塞开发的文档问题。

## 3. 当前代码风险

现有代码还没有完全匹配新设计，重点风险如下：

1. `packages/shared/src/spiritCollisionBattle.ts` 当前只实现了部分旧词条，例如 `claw`、`thick_skin`、`crit`、`crit_damage`、`dodge`、`lifesteal`。
2. `services/game-server/src/raid/raid-settlement-rule.service.ts` 仍存在五行克制把词条属性放大的旧逻辑，不能直接沿用到新战斗。
3. 当前回放事件类型已有 `element`、`critical`、`dodge`、`lifesteal`、`blood` 等基础，但还缺少足够清晰的词条触发说明。
4. 当前测试入口已经可以演示 10 回合和燃血，但还不是完整的 15 词条与 18 固定特质体系。

开发时必须先以 `packages/shared/src/spiritCollisionBattle.ts` 作为新规则源头，真实掠夺结算后接入，避免旧规则污染新战斗。

## 4. 开发原则

1. 战斗规则只实现设计稿中确认的 15 条词条和 18 个固定特质草案。
2. 不新增反击、破甲、韧性、复活、护盾、额外攻击段。
3. 结算顺序必须和设计稿 `2.3 单回合结算顺序` 一致。
4. 所有可重复词条按总值叠加，再按规则统一结算。
5. 五行只影响命中后的最终伤害，不放大词条属性。
6. 所有关键事件必须能被录像还原。
7. 每一步先用测试对撞入口和模拟脚本验证，再接真实掠夺。

## 5. 阶段一：统一词条与特质配置

目标：

把 15 条词条和 18 个固定特质变成代码可读的配置层。

涉及文件：

1. `packages/shared/src/index.ts`
2. `packages/shared/src/spiritCollisionBattle.ts`
3. `services/game-server/src/raid/raid-battle-replay.ts`
4. `services/game-server/src/seed/seed-data/spirits.ts`

开发内容：

1. 定义首发 15 条词条 code。
2. 明确旧 code 到新 code 的保留关系，例如 `thick_skin` 是否继续代表 `厚皮`。
3. 移除或冻结首发不使用的旧词条：`counter`、`tenacity` 等。
4. 为 18 个灵宠建立固定特质 code 映射。
5. 让回放快照能显示固定特质和 5 个可洗词条。

验收标准：

1. 所有 15 条词条都有唯一 code、名称、描述和数值。
2. 所有 18 只灵宠都有固定特质配置。
3. 前端战斗卡片可以展示 5 个词条。
4. 不再出现首发未采用词条的默认展示。

## 6. 阶段二：重写共享战斗结算核心

目标：

让 `buildSpiritCollisionBattleReplay` 按新设计完整计算 10 回合对撞和燃血。

涉及文件：

1. `packages/shared/src/spiritCollisionBattle.ts`
2. `packages/shared/src/index.ts`

开发内容：

1. 实现战斗开始时最大生命和攻击修正。
2. 实现每回合条件判断：`背水`、`收割`、`破血`、固定特质。
3. 实现干扰词条：`断续`、`压制`、`裂伤`、`炽燃`。
4. 实现高风险词条：`利刃`、`铁骨`。
5. 实现五行最终伤害修正。
6. 实现燃血递增、燃血减免和 `炽燃` 增伤。
7. 移除共享战斗里的旧反击链路，首发不使用 `counter`。

关键口径：

1. 闪避成功时伤害为 0，不触发暴击、五行、裂伤、吸血。
2. 最终伤害修正采用加总后统一乘一次。
3. `断续 x5` 最多把吸血和回血压到 0。
4. `裂伤` 不影响燃血扣血。
5. `炽燃` 不影响前 10 回合普通对撞。

验收标准：

1. 15 条词条在模拟中都能产生效果。
2. 5 个相同词条可叠加。
3. 五行克制不再翻倍词条属性。
4. 燃血模式能在双方存活 10 回合后进入，并按 8%、10%、12% 递增。

## 7. 阶段三：补充录像事件

目标：

让玩家和开发者都能从回放里看懂战斗为什么这样结算。

涉及文件：

1. `packages/shared/src/index.ts`
2. `packages/shared/src/spiritCollisionBattle.ts`
3. `apps/game-client/src/battle/battleTimeline.ts`
4. `apps/game-client/src/battle/RaidBattleScreen.tsx`

开发内容：

1. 为五行克制追加事件和飘字 tone。
2. 为暴击保留放大数字表现。
3. 为燃血扣血、`炽燃` 加压、`断续` 压回复保留事件说明。
4. 让每回合至少能解释：闪避、暴击、五行、主要词条触发、吸血、燃血。

验收标准：

1. 回放中能看出哪一方五行克制。
2. 玩家能看出暴击和普通伤害区别。
3. 玩家能看出燃血伤害来自燃血，而不是普通攻击。
4. 不增加遮挡主战斗画面的中心结算面板。

## 8. 阶段四：测试入口做成六套路试验场

目标：

让当前 10 回合测试对撞入口可以快速切换和观察 6 套套路。

涉及文件：

1. `apps/game-client/src/App.tsx`
2. `apps/game-client/src/battle/RaidBattleScreen.tsx`
3. `apps/game-client/src/styles.css`

开发内容：

1. 准备 6 套默认测试构筑：快攻爆杀、血牛拖燃、反血牛、残血反打、干扰压制、闪避赌命。
2. 每套构筑都给 5 个词条，允许重复。
3. 至少准备一组五行克制、一组五行被克、一组无克制对照。
4. 保留当前血条飘字、碰撞、火花、暴击放大表现。

验收标准：

1. 用户能从测试入口看出不同套路差异。
2. 快攻能在前 10 回合制造明显压迫。
3. 血牛能更稳定进入燃血。
4. 反血牛打高血目标明显更疼。
5. 闪避赌命有波动，但不会稳定无敌。

## 9. 阶段五：批量模拟和平衡检查

目标：

用脚本批量跑对撞，先发现明显失衡，再接真实掠夺。

涉及文件：

1. `services/game-server/src/verification/simulate-spirit-collision-battle.ts`
2. `packages/shared/src/spiritCollisionBattle.ts`

必须压测的组合：

1. `厚皮 x5`
2. `铁骨 x5`
3. `吸血 x5`
4. `断续 x5`
5. `炽燃 x5`
6. `破血 x5`
7. `闪避 x5`
8. 五行克制快攻
9. 五行被克快攻
10. 血牛对反血牛

重点观察：

1. `炽燃 +3%` 是否让燃血过快结束。
2. `铁骨 x5` 是否让血牛过于稳定。
3. `吸血 x5` 是否会被 `断续` 和燃血递增合理克制。
4. `闪避 x5` 是否只是高波动，而不是最优解。
5. 五行 `+50% / -20%` 是否足够强，但没有压扁构筑。

验收标准：

1. 每套套路都有能赢的对象。
2. 没有单一词条 x5 成为明显无脑最优。
3. 五行克制有强感知。
4. 普通宠在正确构筑和五行优势下可以击败更高稀有宠。

## 10. 阶段六：接入真实掠夺结算

目标：

把新共享战斗核心接到真实掠夺结算，替换旧五行和旧词条规则。

涉及文件：

1. `services/game-server/src/raid/raid-settlement-rule.service.ts`
2. `services/game-server/src/raid/raid-settlement.service.ts`
3. `services/game-server/src/raid/raid-battle-replay.ts`
4. `services/game-server/src/raid/raid-target.service.ts`
5. `packages/shared/src/spiritCollisionBattle.ts`

开发内容：

1. 真实结算调用共享战斗核心，避免两套规则分叉。
2. 移除真实结算中的旧五行 `traitMultiplier: 2`。
3. 结算结果继续产出战报、奖励和战斗回放。
4. 保证旧订单回放仍能读取，不因新事件类型崩溃。

验收标准：

1. 新订单使用新战斗规则。
2. 旧订单战报仍能打开。
3. 后端 typecheck 通过。
4. 掠夺结算验证脚本通过。

2. 五行克制命中时增加短促特效和颜色提示。

4. 不出现中心大面板遮挡宠物对撞。

## 12. 阶段八：回归验证命令

每个阶段至少运行对应验证。

基础验证：

```bash
npm run build --workspace @trinitywar/shared
npm run typecheck --workspace @trinitywar/game-server
npm run build --workspace @trinitywar/game-client
```

战斗验证：

```bash
npm run verify:spirit-battle-rules --workspace @trinitywar/game-server
npm run simulate:spirit-battle --workspace @trinitywar/game-server -- --runs=100
npm run simulate:spirit-battle --workspace @trinitywar/game-server -- --runs=100 --matrix
```

首轮玩法打磨时不做过重压测。

原则：

1. 默认跑 100 次小样本，看趋势，不追求统计学定论。
2. 只有准备接真实掠夺结算或改动核心数值时，才临时提高到 500/1000 次。
3. 每次最多调整 1 到 2 个玩法点，避免不知道胜率变化来自哪里。
4. 当前优先观察 `干扰压制` 克制面是否过宽、`极限燃血` 主动进攻是否过弱。

数据库验证：

```bash
npm run prisma:validate --workspace @trinitywar/game-server
```

真实掠夺接入后追加：

```bash
npm run verify:test-flow --workspace @trinitywar/game-server
npm run verify:raid-shard-drops --workspace @trinitywar/game-server
```

## 13. 推荐开发顺序

推荐按下面顺序交给 AI 开发：

1. 先做 `packages/shared/src/spiritCollisionBattle.ts` 的新规则核心。
2. 再补 15 词条配置和 18 固定特质映射。
3. 再扩展回放事件。
4. 再做测试入口六套路。
5. 再跑批量模拟，调 `炽燃`、`铁骨`、`闪避`。
6. 最后接真实掠夺结算。

不要一开始就改真实掠夺结算。

原因：

1. 战斗规则还需要看实际对撞表现。
2. 真实掠夺涉及奖励、战报、订单状态，回滚成本更高。
3. 共享战斗核心稳定后，真实接入会简单很多。

## 14. 首轮开发交付标准

第一轮 AI 开发完成时，应满足：

1. 测试入口可以跑新 15 词条。
2. 6 套套路都有可观察样例。
3. 五行克制按最终伤害生效。
4. 燃血递增和 `炽燃` 生效。
5. 暴击、闪避、燃血、五行都有录像反馈。
6. shared build、game-server typecheck、game-client build 通过。

真实掠夺结算可以作为第二轮交付，不强行塞进第一轮。
