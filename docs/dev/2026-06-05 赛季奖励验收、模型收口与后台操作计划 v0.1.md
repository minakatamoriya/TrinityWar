# 2026-06-05 赛季奖励验收、模型收口与后台操作计划 v0.1

## 一、文档目的

本文只收口前一轮功能缺失分析中的 3、4、5 三项：

3. 赛季奖励发放验收。
4. 奖励数据模型收口。
5. 后台赛季操作最小计划。

本轮不做数值平衡，不扩展新赛季玩法，不引入复杂排行榜动画、追赶机制或奖励规则后台配置化。

## 二、当前判断

### 1. 赛季奖励发放不是从零实现

当前已经落地：

- 根据赛季快照生成玩家赛季奖励单。
- 奖励单通过通知附件发放。
- 玩家领取通知后资源入账。
- 通知领取后奖励单状态更新为 `claimed`。
- 领域奖励会写入赛季奖章/成就记录。
- 后台已有奖励统计、奖励单列表、奖章成就列表。
- 客户端已有奖章入口和按赛季分组的奖章陈列柜数据。

因此第 3 点后续重点是验收和展示口径收口，不是重做奖励系统。

### 2. 奖励数据模型已经有基础

当前核心表：

- `player_season_reward_grant`
- `player_season_achievement`
- `player_notification`

当前核心状态：

- `generated`
- `notified`
- `claimed`
- `voided`

因此第 4 点后续重点是确认模型能否支撑后台补发、撤销和审计，不先新建复杂批次表。

### 3. 后台赛季操作需要先做最小闭环

后台写操作会影响奖励和玩家资产，不能只按按钮补接口。第 5 点应先实现受控的最小操作，不做泛化编辑。

## 三、3. 赛季奖励发放验收计划

### 1. 发放链路验收

必须通过：

- 同一玩家、同一赛季、同一奖励类型只生成一条奖励单。
- 重复执行奖励生成不会重复创建奖励单。
- 奖励单生成后会创建可领取通知。
- 通知附件内容与奖励单 `rewardJson` 一致。
- 玩家领取通知后，对应资源真实入账。
- 领取成功后，奖励单状态变为 `claimed`，并写入 `claimedAt`。
- 已领取通知不能重复领取。
- 赛季奖励不再写入植物精华流水。

### 2. 奖章/成就验收

必须通过：

- 每个领域奖励生成对应奖章/成就记录。
- 同一玩家、同一赛季、同一 `achievementKey` 不重复写入。
- 玩家可见奖章只展示中文标题和中文描述。
- 如接口或历史数据中仍有英文兼容字段，不作为玩家端展示内容。
- 奖章能通过 `rewardGrantId` 追溯到奖励单。
- 奖励单已领取后，奖章仍保留展示。
- 奖励单被撤销后，奖章展示规则需要在后台操作计划中明确。

### 3. 奖章陈列柜展示验收

当前用户期望的陈列柜口径如下：

- 按赛季分组展示，例如 `第1赛季`、`第2赛季`。
- 平时只展示奖章 icon，不在列表里铺开长文本。
- 每排最多 6 个 icon。
- 点击某个 icon 后，再查看奖章详情。
- 详情中展示：
  - 中文奖章名。
  - 所属赛季，例如 `第2赛季`。
  - 领域。
  - 中文描述。
  - 对应奖励档位或来源。
- `第2赛季`奖章详情也必须保持全中文展示。

本项属于现有奖章陈列柜的展示方式调整，不新增奖章类型，不新增奖励来源。

### 4. 当前可接受的暂不处理项

暂不处理：

- 排行榜排名奖励。
- 阵营排名奖励。
- 复杂战斗成就。
- 赛季奖励规则后台配置化。
- 数值平衡和奖励数量调整。

## 四、4. 奖励数据模型收口计划

### 1. 当前模型继续保留

首版继续使用 `player_season_reward_grant` 作为奖励单主表。

当前字段能支撑：

- 玩家 ID。
- 赛季号。
- 奖励类型。
- 奖励档位。
- 奖励状态。
- 通知 ID。
- 关键赛季快照数值。
- 奖励 JSON。
- 领取时间。

### 2. 状态流转口径

建议收口为：

| 状态 | 含义 | 是否可领取 | 是否可后台撤销 |
| --- | --- | --- | --- |
| `generated` | 奖励单已生成，但尚未绑定通知 | 否 | 是 |
| `notified` | 已创建通知，等待玩家领取 | 是 | 是 |
| `claimed` | 玩家已领取 | 否 | 否 |
| `voided` | 后台撤销或作废 | 否 | 否 |

原则：

- `claimed` 不做后台撤销。
- 如未来必须补偿已领取错误奖励，应走补偿/修正流程，不直接回滚已领取奖励。
- `voided` 只表示奖励单不再可领取，不删除历史记录。

### 3. 奖章与奖励单关系

当前奖章记录可通过 `rewardGrantId` 关联奖励单。

建议：

- 已领取奖励对应的奖章永久展示。
- `notified` 状态下已生成的奖章可以展示，但详情中应能看到奖励尚未领取。
- `voided` 奖励对应奖章默认不在客户端陈列柜展示，后台仍可查历史。
- 如果撤销时奖章已经展示过，不删除成就记录，只通过奖励状态控制客户端展示。

### 4. 兼容字段边界

旧 `seedId`、`essenceType` 相关字段只保留兼容：

- 不作为新的赛季奖励附件类型。
- 不作为新的植物解锁来源。
- 不作为后台新建奖励时的可选项。
- 历史通知如果存在旧记录，只做只读兼容或历史展示。

### 5. 暂不新增批次表

当前不急于新增奖励批次表。

原因：

- 当前奖励生成可以通过赛季号、玩家、奖励类型保证幂等。
- 后台只读统计已经能按赛季汇总。
- 首版后台写操作以单玩家、单赛季、单奖励单为主。

如果后续需要全服批量重算、批量补发、批量撤销，再单独设计 `season_reward_batch`。

## 五、5. 后台赛季操作最小计划

### 1. 首版允许的后台操作

首版只做以下操作：

1. 手动生成指定赛季奖励。
2. 预览指定玩家的赛季奖励结果。
3. 补发某个玩家缺失的赛季奖励。
4. 撤销某个未领取的赛季奖励单。
5. 查看玩家赛季奖励历史。

### 2. 首版不做的后台操作

暂不做：

- 修改进行中赛季时间。
- 修改已结束赛季时间。
- 批量重算全服奖励。
- 直接编辑奖励 JSON。
- 直接编辑玩家赛季快照。
- 直接扣回已领取奖励。
- 后台配置奖励规则。

### 3. 操作 1：手动生成指定赛季奖励

入口：

- 后台赛季页面。

前置条件：

- 指定赛季存在。
- 指定赛季已有玩家赛季快照。
- 操作只针对指定赛季。

行为：

- 调用现有奖励生成逻辑。
- 已存在的奖励单不重复创建。
- 返回本次新建或确认存在的奖励单数量。

保护：

- 幂等。
- 写管理员操作日志。
- 二次确认。

### 4. 操作 2：预览指定玩家的赛季奖励结果

入口：

- 玩家赛季详情。

行为：

- 根据玩家指定赛季快照计算可获得的奖励。
- 不写奖励单。
- 不发通知。
- 不写奖章。

用途：

- 后台补发前核对。
- 排障玩家为什么有或没有某个奖励。

### 5. 操作 3：补发某个玩家缺失的赛季奖励

入口：

- 玩家赛季详情或奖励单列表。

前置条件：

- 玩家指定赛季有快照。
- 奖励单不存在，或奖励单存在但仍为可修复状态。

行为：

- 生成缺失奖励单。
- 创建通知附件。
- 写奖章/成就记录。

保护：

- 同一玩家、同一赛季、同一奖励类型不能重复补发。
- 已领取奖励不能重复补发同类型奖励。
- 写管理员操作日志。
- 二次确认。

### 6. 操作 4：撤销未领取赛季奖励

入口：

- 奖励单列表。

允许撤销：

- `generated`
- `notified`

禁止撤销：

- `claimed`
- `voided`

行为：

- 奖励单状态改为 `voided`。
- 如果有关联通知且通知未领取，将通知置为不可领取或过期/删除状态，具体实现前再按现有通知状态选择最小改动。
- 不删除奖励单。
- 不删除奖章记录，但客户端陈列柜不展示 `voided` 奖励关联的奖章。

保护：

- 写管理员操作日志。
- 二次确认。
- 返回撤销前后的状态。

### 7. 操作 5：查看玩家赛季奖励历史

入口：

- 玩家详情。

展示：

- 赛季号。
- 奖励类型。
- 奖励档位。
- 状态。
- 通知 ID。
- 创建时间。
- 领取时间。
- 奖章/成就关联。

本项只读，可以先于其他写操作实现。

## 六、接口草案

后台接口建议：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/admin/seasons/:seasonNumber/rewards/generate` | 手动生成指定赛季奖励 |
| `GET` | `/admin/seasons/:seasonNumber/rewards/preview?playerId=...` | 预览玩家赛季奖励 |
| `POST` | `/admin/seasons/:seasonNumber/rewards/grants/:grantId/reissue` | 补发或修复单个奖励 |
| `POST` | `/admin/seasons/:seasonNumber/rewards/grants/:grantId/void` | 撤销未领取奖励 |
| `GET` | `/admin/players/:playerId/season-rewards` | 查看玩家赛季奖励历史 |

实际实现时可以根据现有 controller 命名微调，但行为边界不应扩大。

   - 覆盖奖励生成、通知领取、奖章记录、无植物精华流水。

   - 每排最多 6 个 icon。
   - 点击 icon 能看到详情。
   - `第2赛季`奖章详情保持全中文展示。

1. 补强 `verify-season-rewards`，先把第 3 点验收跑稳。
