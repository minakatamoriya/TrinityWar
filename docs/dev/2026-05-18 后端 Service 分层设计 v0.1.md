# TrinityWar 后端 Service 分层设计 v0.1

## 一、文档目的

这份文档是给 AI 和开发执行用的后端分层设计文档。

它的目标不是解释某个框架，而是提前把后端代码应该落在哪一层说清楚，避免后续出现下面这些问题：

1. controller 里写满业务逻辑。
2. service 里直接拼 SQL 和 Prisma 查询细节。
3. worker 又复制一套和 API 不同的结算逻辑。
4. DTO 映射、数据库写入、规则判断、日志记录混在一个文件里。
5. 同一个动作在多个地方各写一套，后续一改规则全线漂移。

这份文档默认基于以下前提：

1. 后端使用 NestJS。
2. 数据访问使用 Prisma。
3. 数据模型参考 [dev/2026-05-18 首批数据库表与 Prisma Schema 草案 v0.1.md](dev/2026-05-18 首批数据库表与 Prisma Schema 草案 v0.1.md)。
4. 接口契约参考 [dev/2026-05-18 首批真实接口契约清单 v0.1.md](dev/2026-05-18 首批真实接口契约清单 v0.1.md)。
5. 总体节奏参考 [dev/2026-05-18 AI 后端总计划 v0.1.md](dev/2026-05-18 AI 后端总计划 v0.1.md)。

## 二、分层总原则

### 1. 一层只做一类事

每一层都应该有明确职责，不要图省事把所有事情塞到一起。

### 2. 规则只保留一份

任何会影响金币、升级、收取、训练、掠夺、分红、任务奖励的规则，都只能有一份主实现。

不能出现：

1. API service 一套。
2. Worker 再写一套。
3. Admin 修正接口再写一套。

### 3. Controller 不做结算

Controller 只负责：

1. 接请求。
2. 调应用服务。
3. 返回响应。

Controller 不负责：

1. 算金币。
2. 扣资源。
3. 决定升级是否成功。
4. 决定掠夺是否成功。
5. 写数据库事务。

### 4. Repository 不做规则判断

Repository 只负责：

1. 读数据库。
2. 写数据库。
3. 封装 Prisma 细节。

Repository 不负责：

1. 判断主城等级够不够。
2. 判断金库是否足够。
3. 判断任务是否可领取。
4. 判断收取是否允许。

### 5. 应用服务负责一个“用例”

应用服务应该围绕具体业务动作来设计。

例如：

1. UpgradeBuildingUseCase
2. CollectFieldUseCase
3. ClaimPendingGoldUseCase
4. RaidTargetUseCase

而不是写成一个巨大的 `GameService` 把所有东西都塞进去。

## 三、推荐分层结构

建议后端按下面 7 层组织。

### 1. Controller 层

职责：

1. 接 HTTP 请求。
2. 调用应用服务。
3. 处理响应映射。
4. 处理 guard、pipe、decorator。

不要做：

1. 业务规则计算。
2. Prisma 读写。
3. 跨模块事务。

### 2. DTO / Contract 层

职责：

1. 保存请求 DTO。
2. 保存响应 DTO。
3. 做 class-validator 校验或等价输入校验。
4. 对齐 shared 契约。

不要做：

1. 业务逻辑。
2. 数据库模型定义。

### 3. Application Service / UseCase 层

职责：

1. 编排单个业务动作。
2. 读取当前状态。
3. 调用领域规则服务。
4. 启动事务。
5. 写入数据库。
6. 写日志。
7. 返回结果对象。

这是整个系统最关键的一层。

### 4. Domain Service / Rule 层

职责：

1. 保存纯规则判断。
2. 保存纯计算逻辑。
3. 保证相同规则可复用给 API、Worker、Admin。

例如：

1. 建筑升级成本规则。
2. 灵宠上限门槛规则。
3. 农场阶段收益计算。
4. 掠夺结果计算。
5. 分红计算。

### 5. Repository 层

职责：

1. 封装 Prisma 查询。
2. 提供稳定的读写接口。
3. 隐藏数据库实现细节。

### 6. Assembler / Mapper 层

职责：

1. 把数据库模型转成客户端 DTO。
2. 把多个表聚合成 HomeSummary 或 SceneContent。
3. 把战报、目标、任务状态转成前端可用格式。

### 7. Worker / Job Handler 层

职责：

1. 接收 BullMQ job。
2. 调用和 API 相同的应用服务或领域规则。
3. 记录 job log。

Worker 不应该直接自己重新实现规则。

## 五、每层职责细化

## 1. Controller 层

### 应该做什么

1. 定义路由。
2. 声明请求 DTO。
3. 声明响应类型。
4. 从 Request Context 拿 playerId。
5. 调用对应 use case。
6. 返回 use case 的结果。

### 不应该做什么

1. `if (vaultGold < cost) throw ...`
2. `await prisma.playerWallet.update(...)`
3. 手动拼 `home` 和 `scenes` 聚合。

### 示例职责

以 `POST /api/client/actions/upgrade-building` 为例：

Controller 只做：

1. 校验 `targetType`、`buildingId`、`extensionId`。
2. 获取当前玩家身份。
3. 调 `upgradeBuildingUseCase.execute(...)`。
4. 返回结果。

## 2. Application Service / UseCase 层

### 应该做什么

1. 读取当前玩家聚合状态。
2. 调用规则服务判断是否允许执行。
3. 开启事务。
4. 扣金币、升等级、写日志。
5. 重新聚合返回前端需要的数据。

### 不应该做什么

1. 直接依赖 HTTP request 对象。
2. 把所有 Prisma 查询散落在 use case 内部。
3. 复制大量 DTO 映射逻辑。

### 推荐命名

1. `UpgradeBuildingUseCase`
2. `CollectFieldUseCase`
3. `StartCultivationUseCase`
4. `RecruitArmyUseCase`
5. `ClaimPendingGoldUseCase`
6. `ClaimDailyTaskRewardUseCase`
7. `RaidTargetUseCase`
8. `DonateFactionGoldUseCase`

## 3. Domain Service / Rule 层

### 应该做什么

1. 输入当前状态和配置。
2. 输出判断结果和计算结果。
3. 不依赖 NestJS。
4. 尽量不直接依赖数据库。

### 示例

1. `BuildingUpgradeRuleService`
2. `PopulationGateRuleService`
3. `FieldLifecycleRuleService`
4. `RaidSettlementRuleService`
5. `FactionDividendRuleService`
6. `TaskRewardRuleService`

### 典型返回结构

例如升级建筑规则服务可以返回：

```ts
{
  allowed: true,
  costGold: 700,
  nextLevel: 5,
  reasonCode: null
}
```

而不是直接在规则服务里抛 HTTP 异常。

## 4. Repository 层

### 应该做什么

1. 读取单表或有限关联表。
2. 封装 Prisma `findUnique`、`findMany`、`update`、`create`。
3. 在必要时提供 `findForUpdate` 等价事务读取模式。

### 不应该做什么

1. 计算升级成本。
2. 决定掠夺是否成功。
3. 决定是否允许领取奖励。

### 推荐拆分

不要做一个大 `GameRepository`。

建议按实体拆：

1. `PlayerRepository`
2. `PlayerWalletRepository`
3. `PlayerBuildingRepository`
4. `PlayerArmyRepository`
5. `PlayerFieldSlotRepository`
6. `PlayerSeedInventoryRepository`
7. `PlayerDailyTaskRepository`
8. `RaidOrderRepository`
9. `RaidSettlementRepository`
10. `BattleReportRepository`
11. `FactionRepository`
12. `FactionMemberRepository`
13. `WalletChangeLogRepository`
14. `SystemJobLogRepository`

## 5. Assembler / Mapper 层

### 应该做什么

1. 把数据库模型组装成 `HomeSummaryResponse`。
2. 把数据库模型组装成 `ClientSceneContentResponse`。
3. 把战报模型转成 `ClientReportEntry`。
4. 把农场地块转成 `ClientFarmField`。

### 为什么需要单独一层

因为当前前端展示结构和数据库模型不是一一对应的。

例如：

1. `HomeSummaryResponse` 需要同时来自 player、wallet、task、report、claim。
2. `ClientSceneContentResponse` 需要同时来自 building、field、army、raid、faction。

如果不单独拆 assembler，这些 DTO 拼装代码会扩散到所有 use case 里。

## 六、推荐共享组件层

除了业务模块，还建议有一层 shared 组件。

### 1. Error Factory

职责：

1. 统一业务错误对象。
2. 统一 error code 和 message。

建议：

1. `BusinessError`
2. `ErrorCode` enum
3. `errorFactory` helpers

### 2. Request Context

职责：

1. 提供当前 playerId。
2. 提供 traceId。
3. 提供 requestTime。

### 3. Transaction Runner

职责：

1. 统一 Prisma 事务执行封装。
2. 减少 use case 里重复写 transaction 样板。

### 4. Idempotency Service

职责：

1. 存储或校验幂等键。
2. 防止关键命令重复执行。

### 5. Version Check Service

职责：

1. 校验 walletVersion、fieldVersion、armyVersion、buildingVersion。
2. 统一抛出 `STATE_VERSION_CONFLICT`。

## 七、建议的读取服务设计

读取服务和命令 use case 不要混在一起。

### 1. HomeSummaryQueryService

职责：

1. 聚合首页所需数据。
2. 返回 `HomeSummaryResponse`。

依赖：

1. PlayerRepository
2. PlayerWalletRepository
3. PlayerDailyTaskRepository
4. PendingClaimAssembler
5. HomeSummaryAssembler

### 2. SceneContentQueryService

职责：

1. 聚合建筑、农场、部队、掠夺、战报、阵营页数据。
2. 返回 `ClientSceneContentResponse`。

### 3. RaidTargetDetailQueryService

职责：

1. 读取目标详情。
2. 校验目标池可见性。
3. 返回 `ClientRaidTargetDetailResponse`。

这些读取服务应该在：

1. 页面初始化接口复用。
2. 命令执行后的刷新响应复用。
3. 后台调试读取时复用。

## 九、Worker 设计原则

Worker 只负责处理异步入口，不负责拥有自己的业务规则版本。

### 1. DividendSettlementWorker

不应该：

1. 自己实现一套分红计算。

应该：

1. 调用 `FactionDividendSettlementUseCase` 或同类应用服务。

### 2. ArmyTrainingReadyWorker

不应该：

1. 直接手写 SQL 把训练数量加回 army。

应该：

1. 调用 `CompleteArmyTrainingUseCase`。

### 3. ProtectionExpireWorker

应该：

1. 调用 `ExpireProtectionUseCase`。
2. 记录 `SystemJobLog`。

### 4. DailyTaskRefreshWorker

应该：

1. 调用 `RefreshDailyTaskUseCase`。
2. 不直接修改随机任务逻辑实现。

## 十、推荐事务边界

事务通常应该由 use case 层控制，而不是 repository 控制。

### 1. 为什么事务不要放到 repository 层

因为一个业务动作往往涉及多张表。

例如收菜：

1. 更新地块。
2. 更新钱包。
3. 写收取日志。
4. 写资产日志。
5. 可能推进任务。

这些不应该分散由每个 repository 各自决定事务。

### 2. 推荐做法

UseCase：

1. 开启事务。
2. 在事务上下文里调用多个 repository。
3. 提交后再做响应聚合。

### 3. 注意点

1. 查询聚合响应可以在事务后做。
2. 关键变更和日志写入要在同一事务里。
3. 外部调用如消息推送不要塞进主事务。

## 十一、推荐返回对象模式

UseCase 不建议直接返回 HTTP Response，而应返回结果对象。

例如：

```ts
interface UpgradeBuildingResult {
  summary: string;
  refreshHome: boolean;
  refreshScenes: boolean;
  result?: {
    buildingId: string;
    oldLevel: number;
    newLevel: number;
  };
}
```

然后由 controller 或 response mapper 转成真正的 `ClientStateMutationResponse`。

这样好处是：

1. 同一个 use case 可被 worker、后台、测试复用。
2. 不会把 HTTP 细节污染到核心业务层。

## 十二、日志与审计接入位置

### 1. 业务日志

写在 use case 成功完成后。

例如：

1. `logger.info({ playerId, buildingId, oldLevel, newLevel }, 'upgrade building success')`

### 2. 审计日志

写在事务内部。

例如：

1. WalletChangeLog
2. BuildingUpgradeLog
3. FieldHarvestLog

### 3. 错误日志

写在 controller filter 或 use case catch 边界。

不要在每个 repository 里各写一份重复 error log。

## 十三、AI 执行边界模板

### 1. 生成 controller

你现在只负责生成 [模块名] controller。不要把业务规则、Prisma 查询、事务写进 controller。controller 只允许做 DTO 校验、鉴权上下文提取、调用 use case、返回结果。

### 2. 生成 use case

你现在只负责生成 [动作名] use case。必须包含：状态读取、规则校验、事务写入、日志记录、返回结果对象。不要把 DTO 校验细节和 HTTP request 对象带进 use case。

### 3. 生成 repository

你现在只负责生成 [实体名] repository。只封装 Prisma 查询和写入，不做业务规则判断，不拼接前端 DTO。

### 4. 生成 assembler

你现在只负责生成 [页面/响应名] assembler。目标是把数据库模型转换为 shared DTO。不要在 assembler 中写数据库查询或规则判断。

### 5. 生成 worker

你现在只负责生成 [任务名] worker。worker 不能自带独立规则实现，必须调用已有 use case 或 domain service。

## 十四、验收清单

当某个模块开始进入实现阶段时，至少检查下面这些点：

1. controller 是否足够薄。
2. 业务规则是否集中在 domain service 或 use case 中。
3. repository 是否只做数据访问。
4. assembler 是否只做 DTO 映射。
5. worker 是否复用了主规则。
6. 是否存在同一条规则写在三个地方。
7. 事务边界是否由 use case 控制。
8. 关键命令是否写了审计日志。
9. query service 是否可复用。
10. 代码结构是否支持 AI 继续逐模块追加，而不是越写越糊。

## 十五、最重要的执行建议

如果只记一条，记下面这条：

任何会改玩家钱、等级、地块、兵力、战报的动作，都应该遵循同一条链路：

1. Controller 接收请求。
2. UseCase 编排动作。
3. RuleService 给出规则判断和计算结果。
4. Repository 执行事务内读写。
5. LogRepository 写审计日志。
6. QueryService + Assembler 返回前端刷新数据。

不要跳层。

一旦开始跳层，后面 AI 继续接手时，就会越来越难保持一致。
