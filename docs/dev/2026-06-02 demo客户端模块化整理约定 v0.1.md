# 2026-06-02 demo客户端模块化整理约定 v0.1

## 目标

当前 Web demo 继续作为玩法、页面结构和接口联调的主验证入口，不做运行方式迁移。整理目标是控制主文件膨胀，让后续几个月的新增功能能落在清晰模块里，便于以后重组为微信小游戏客户端。

## 目录职责

- `config/`：静态配置、导航配置、种子表、阵营配置、背景图映射。
- `utils/`：平台无关的格式化、时间、解析等纯工具函数。
- `modules/<feature>/`：按玩法模块放 selector、局部展示类型、低副作用业务辅助函数。
- `shell/`：以后承接应用壳、全局导航、全局弹窗、toast 等跨模块 UI 状态。
- `api/`：以后承接真实请求、mock 请求、session 与错误处理拆分。当前 `api.ts` 暂时保留，避免一次性大拆。

## 新增功能流程

新增玩法或页面时，优先按下面顺序落地：

1. 在 `packages/shared` 补客户端 DTO 或命令类型。
2. 在 API 层补接口函数。
3. 在 `modules/<feature>/` 写 selector，把后端 DTO 转成页面展示数据。
4. 在 `modules/<feature>/` 写 action/helper，封装该模块操作流程。
5. React 组件只消费展示数据并派发明确操作。
6. `App.tsx` 只负责接入模块、全局状态、导航和弹窗。

## 禁止继续加重的点

- 不继续往 `App.tsx` 塞大段静态配置、展示计算和模块细节流程。
- 不在 React 组件里用中文按钮文案判断业务行为；新增逻辑应使用稳定 action key 或 command type。
- 不把可复用业务规则写进 JSX。
- 不让新增 mock 逻辑继续混在真实请求函数里。
- 不把 `window`、`localStorage`、`fetch`、分享等平台能力散落到模块业务逻辑里。

## 渐进整理顺序

1. 先抽配置和纯工具函数。
2. 再抽 selector，保持页面行为不变。
3. 再抽模块 action/helper。
4. 最后拆 API 与 mock。

## 验收标准

- 每轮整理后 Web demo 能构建。
- 核心流程行为不变。
- 新文件职责单一，命名能看出归属模块。
- `App.tsx` 行数和职责逐步下降，而不是继续增长。
