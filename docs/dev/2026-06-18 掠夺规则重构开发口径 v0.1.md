# 2026-06-18 掠夺规则重构开发口径 v0.1

日期：2026-06-18

## 1. 文档目的

本文用于沉淀 2026-06-18 当日关于掠夺玩法的集中讨论结果，作为后续 AI 与人工开发的统一口径。

本文不是脑暴记录，而是开发口径文档。后续实现、改表、改接口、改前端文案时，应优先以本文为准。

本文覆盖以下主题：

1. 保护期取消与保护次数替代方案
2. 被掠夺机制从“扣对方资源”改为“系统发放奖励”
3. 目标匹配与刷新机制
4. 胜平负奖励矩阵
5. 前端展示口径
6. 并发失败提示与原子性要求
7. 后端、Shared、前端的改造边界

## 2. 本轮要解决的问题

当前实现存在以下体验与实现问题：

1. 时间保护期不直观。用户很难理解自己何时受保护、他人何时受保护。
2. 目标列表与保护期强耦合。列表会因为目标进入保护期而变化，用户会觉得“人怎么突然消失了”。
3. 目标可见不代表可打。玩家 A 刷到玩家 B，但在点击时玩家 C 可能已先一步攻击了 B。
4. 当前掠夺收益依赖真实可扣资源。若某目标已被多次掠夺，后续玩家即使打赢，也可能拿不到合理收益。
5. 当前刷新目标没有明确等级段约束，容易出现低级玩家刷到远高于自身等级的大号。
6. 真实扣对方资源会显著增加挫败感，并放大并发结算复杂度。

## 3. 本轮已定决策

以下内容视为本轮已定规则，后续开发默认按此执行：

1. 取消常规掠夺中的时间保护期。
2. 不再使用 `protectedUntil` 决定普通目标池展示与可攻击性。
3. 常规掠夺改为“次数保护”。
4. 被掠夺者不再因普通掠夺损失金币。
5. 进攻方收益改为系统奖励，而不是从防守方资产中扣除。
6. 普通目标匹配改为优先相似等级段，不允许 1 级直接刷到 50 级的大号。
7. 并发冲突仍然存在，但冲突失败不应额外惩罚玩家，且应提供明确文案。
8. 每日重置时区统一采用 `Asia/Shanghai (UTC+8)` 自然日，即每日 `00:00:00` 重置。
9. 攻击者每日普通掠夺总挑战次数定为 `10` 次。
10. 后续允许使用商业货币“天机符”增加每日挑战次数。
11. 刷新目标列表有每日次数限制，后续允许使用“天机符”增加刷新次数。
12. 完败不得为 `0` 收益，必须给极小安慰奖励。
13. 防守方不增加额外系统奖励，默认只获得复仇资格。
14. 复仇目标可绕过普通目标池的同攻同守限制与普通匹配限制，但仍消耗攻击者每日总挑战次数。
15. 普通目标刷新允许向上扩展到更高两个档次，用于给愿意搏高风险收益的玩家保留挑战空间。
16. 探索与战斗主链路只针对对手灵宠，不再展示或依赖对手田地信息。
17. 掠夺详情不再展示田地预览，也不再展示预估可掠夺资金。
18. 深度窥视功能继续保留。
19. 未知灵宠、图鉴未解锁时的遮罩展示继续保留。

## 3.1 组件化与解耦原则

本轮设计默认遵守当前项目的组件化与解耦方式，后续实现必须继续遵守：

1. 数值参数进 `services/game-server/src/lib/game-balance.js`
2. 规则编排留在 service 层，不把大段业务判断写进 controller、assembler 或前端
3. Shared 只定义契约，不承载服务端结算逻辑
4. 前端只消费结果与标签，不自行推导真实结算规则

### 3.1.1 哪些内容必须放进 `game-balance.js`

以下内容属于可调平衡参数，必须配置化，不得散落硬编码到业务文件：

1. 每日普通掠夺总挑战次数
2. 普通目标每日成功被掠夺上限
3. 免费刷新次数
4. 胜平负奖励系数矩阵
5. 各等级段基础奖励 `B`
6. 普通目标列表中“稳妥目标 / 挑战目标”的投放比例
7. 更高一档、两档目标的混入比例
8. 天机符增加挑战次数和刷新次数的价格梯度

### 3.1.2 哪些内容不属于平衡参数

以下内容不应塞进 `game-balance.js`，而应放在规则服务、仓储或独立状态结构中：

1. 某个玩家今天已经挑战了谁
2. 某个防守者今天已经被成功掠夺几次
3. 某个目标当前是否还能出现在普通池
4. 某张卡本次刷新是否已被别人抢先结算

这些属于运行态状态，不属于数值配置。

### 3.1.3 推荐的模块边界

后续实现时建议继续保持以下职责边界：

1. `game-balance.js`：只放数值与规则参数
2. `RaidSettlementRuleService`：根据输入快照和数值配置计算奖励与结果
3. `RaidTargetService`：负责下单前规则校验、并发控制和订单创建
4. `ClientReadService` / 目标池相关服务：负责目标生成与列表展示数据
5. `SceneContentAssembler`：只负责把后端读模型组装成前端 DTO，不做关键规则判断

如果后续普通目标池逻辑继续变复杂，建议从 `ClientReadService` 中再抽出独立 `RaidTargetPoolService`，避免目标生成逻辑继续堆在读服务里。

## 4. 术语定义

后续实现中必须使用以下统一定义：

### 4.1 普通目标池

指玩家在掠夺页刷新、浏览和发起挑战时看到的匿名系统分发目标列表。

不包括：

- 复仇目标
- 通缉目标
- 教学机器人目标

### 4.2 一次结算

指一次普通掠夺挑战从创建订单到产生最终胜、平、负结果的完整过程。

无论最终胜平负，只要进入正式结算并写入最终结果，都记为“一次结算”。

### 4.3 成功掠夺

指进攻方在一次普通掠夺结算中取得胜利，并获得本次掠夺的主奖励。

本文中“成功掠夺”不再表示“从对方身上扣到了资源”，因为普通掠夺不再扣防守方金币。

### 4.4 同攻同守限制

指同一攻击者对同一防守者，在同一自然日内最多允许完成一次普通掠夺结算。

注意：这里是“最多结算一次”，不是“最多成功一次”。

### 4.5 防守方成功被掠夺次数

指同一防守者在同一自然日内，被不同或相同攻击者成功掠夺的累计次数。

仅“进攻方获胜”的结算计入该次数。

## 5. V1 规则总览

### 5.1 保护规则

普通掠夺取消时间保护期，改为次数保护：

1. 同一攻击者对同一防守者，每日最多结算 1 次。
2. 同一防守者每日最多被成功掠夺 5 次。
3. 当防守者当日被成功掠夺达到 5 次后，应退出普通目标池，直至次日重置。
4. 平局和失败不计入“防守者被成功掠夺 5 次”，但仍占用“同攻同守每日 1 次”。

### 5.2 收益规则

普通掠夺改为系统奖励制：

1. 防守方不因普通掠夺损失金币。
2. 防守方不因普通掠夺扣除田地当前可收金币。
3. 进攻方奖励由系统发放。
4. 胜、平、负都可以配置不同档位奖励。
5. 奖励总通胀由“攻击者每日挑战总次数上限”和奖励表共同控制。

### 5.3 匹配规则

普通目标刷新必须优先匹配相似等级段目标：

1. 优先同等级段。
2. 同段不足时，只放宽到相邻等级段。
3. 不允许无限放宽到差距极大的目标。
4. 仍优先跨阵营。

### 5.3.1 信息架构调整

普通探索与战斗链路从本轮开始彻底改为“灵宠对抗”视角：

1. 选择目标时只关注对手灵宠
2. 目标详情只展示灵宠与战斗相关信息
3. 不再通过田地状态判断是否值得打
4. 不再在掠夺详情中展示对手田地
5. 不再在掠夺详情中展示预估可掠夺资金

设计原因：

1. 普通掠夺奖励已改为系统发放，和对手田地不再直接相关
2. 若继续展示田地和可掠夺资金，会误导用户以为掠夺仍是“抢田地收益”
3. 田地信息应回归农场/社交访问链路，而非探索战斗链路

### 5.4 并发规则

列表可见与点击成功仍不是同一原子时刻，因此必须保留最终原子校验：

1. 目标在列表中可见，不代表点击时仍可结算。
2. 最终是否允许创建订单，以后端事务内校验为准。
3. 若因他人抢先结算、目标次数打满、同攻同守次数已用完而失败，不应额外扣挑战次数。
4. 前端应给出明确失败原因，并优先自动补入新目标，而不是强依赖整页手动刷新。

### 5.5 每日重置规则

V1 每日重置统一采用服务器配置时区 `Asia/Shanghai (UTC+8)`：

1. 每日 `00:00:00` 重置攻击者总挑战次数
2. 每日 `00:00:00` 重置普通目标刷新次数
3. 每日 `00:00:00` 重置同攻同守限制
4. 每日 `00:00:00` 重置防守者当日成功被掠夺次数

说明：

1. 当前项目整体文案和运营语境均以中文区服为主，V1 直接采用北京时间最稳。
2. 若未来做多区服，需改为“区服时区配置”，但当前不得提前抽象过度。

## 6. 普通掠夺详细规则

### 6.1 攻击者每日总挑战次数

本文明确：系统奖励制的通胀控制依赖攻击者每日总挑战次数上限。

V1 固定为：

1. 每位玩家每日普通掠夺总挑战次数为 `10` 次。
2. 普通目标挑战与复仇目标挑战都消耗该总次数。
3. 后续允许使用商业货币“天机符”购买额外挑战次数。
4. 奖励发放逻辑不得绕过该总次数限制。

实现要求：

1. `10` 次应做成可配置常量，不要写死在前端文案中。
2. 天机符扩容逻辑本轮只预留能力，不要求本次一并完成购买入口。

### 6.2 同攻同守限制

后端在创建普通掠夺订单前，必须原子校验：

1. 当日是否已有 `attackerPlayerId + defenderPlayerId` 的已结算普通掠夺记录。
2. 若存在，则拒绝再次创建订单。
3. 该限制对胜、平、负全部生效。

设计原因：

1. 防止玩家对同一目标反复试到赢。
2. 保持单个目标的人际纠缠频率可控。
3. 规则更容易向用户解释。

### 6.3 防守者每日成功被掠夺上限

后端在创建普通掠夺订单前，必须原子校验：

1. 防守者当日已成功被掠夺次数是否小于 5。
2. 若已达到 5，则拒绝继续创建普通掠夺订单。
3. 仅胜利结算计入该次数。
4. 平局与失败不计入。

### 6.4 防守者资源不再被扣

后端在普通掠夺结算中，必须停止以下行为：

1. 不再扣减防守方 `player_wallet.vault_gold`。
2. 不再扣减防守方 `player_field_slot.current_claimable_gold`。
3. 不再依赖 `lockedGold` 计算本次普通掠夺实际可得金币。
4. 不再因为目标已被别人打空而使后续获胜者拿到接近 0 的收益。

注意：

1. 本条仅针对普通目标池中的常规匿名掠夺。
2. 若后续复仇、通缉、特殊活动要采用不同资源口径，应单独写文档，不得默认复用本文例外。

## 7. 匹配与刷新规则

### 7.1 V1 等级段定义

V1 先采用简单等级段，不直接上复杂综合战力分：

1. `1-10`
2. `11-20`
3. `21-30`
4. `31-40`
5. `41+`

当前字段口径优先使用 `castleLevelCache` 作为等级段基准。

### 7.2 刷新目标候选逻辑

普通目标刷新时，候选池筛选顺序建议为：

1. 跨阵营
2. 不在同攻同守当日已结算名单内
3. 防守者当日成功被掠夺次数 < 5
4. 优先同等级段
5. 同等级段不足时，放宽到相邻等级段
6. 在保留主体公平性的前提下，可额外投放更高一档与更高两档的挑战型目标
7. 仍不足时，可有限放宽，但必须有硬限制

V1 建议硬限制：

1. 优先按段匹配
2. 默认候选主体应来自同段与相邻段
3. 允许少量“挑战型目标”来自更高一档与更高两档
4. 不允许出现 1 级直接刷到 50 级的情况

### 7.3 列表构成原则

普通目标列表不应全部由同一难度构成，而应同时包含：

1. 稳妥目标：同段或相邻段，适合大多数玩家稳定消耗次数
2. 进取目标：更高一档，适合想利用五行克制搏收益的玩家
3. 越级目标：更高两档中的少量卡位，作为高风险选项

实现要求：

1. 列表中必须保留“能打的目标”和“可搏的目标”两类，不应全部偏稳妥或全部偏越级。
2. “可搏的目标”应作为少量补充，不应反客为主。
3. 前端展示上要通过风险标签和奖励预估区分稳妥与挑战，不要让用户误以为所有目标都同难度。

### 7.4 刷新目标的用户心智

前端和产品口径必须统一：

1. 刷新目标是在“合理候选池中重抽”
2. 刷新不保证更肥
3. 刷新不保证更弱
4. 刷新只保证“更符合匹配规则”

### 7.5 刷新次数与商业化预留

普通目标刷新应有每日次数限制：

1. 刷新次数与挑战次数分开统计
2. 每日免费刷新次数用完后，后续允许使用“天机符”增加刷新次数
3. 本轮先把数据结构与接口口径预留好，不要求同时完成商业化扣费入口

注意：

1. 本轮未锁死“每日免费刷新次数”的具体数值。
2. AI 开发不得擅自编造免费刷新次数或天机符价格。

## 8. 胜平负奖励矩阵

### 8.1 奖励模型

V1 采用“目标基础奖励 B × 结果系数”的系统奖励模型。

其中：

1. `B` 由匹配段位、风险标签、目标档位决定
2. 结果系数由胜平负结果决定
3. 可附带金币、兽魂、贡献等次级奖励

### 8.2 建议矩阵

V1 建议使用以下默认结果系数：

1. 完胜：`1.00B`
2. 大胜：`0.80B`
3. 小胜：`0.60B`
4. 相持：`0.25B`
5. 小败：`0.15B`
6. 大败：`0.05B`
7. 完败：`0.02B`

说明：

1. 完败必须保留极小安慰收益，最终结算不得因为向下取整而变为 `0`。
2. 若后续数值希望改为固定保底，也不得高于大败档。

### 8.3 奖励类型建议

V1 建议：

1. 胜利档给主要金币奖励
2. 胜利档可附带少量兽魂或贡献
3. 平局档给少量保底
4. 失败档给极少量保底或仅给复仇势能

### 8.4 防守方收益

本轮已决定不为防守方增加系统奖励，因此当前开发口径为：

1. 防守方默认不掉金币
2. 防守方默认获得复仇资格
3. 防守方不增加“守住奖励”

若后续要加“守住奖励”，必须单独确认经济来源与 UI 文案。

## 9. 复仇目标例外规则

复仇目标不属于普通目标池，允许绕过部分普通限制。

### 9.1 可绕过的限制

1. 不受普通目标池等级段匹配限制
2. 不受普通目标池“同攻同守每日最多结算 1 次”限制
3. 不受普通目标池“防守者每日最多被成功掠夺 5 次”限制

### 9.2 仍需遵守的限制

1. 仍消耗攻击者每日总挑战次数
2. 仍需通过后端最终原子校验
3. 仍需校验攻击者当前出战状态是否有效

设计原因：

1. 复仇的核心情绪价值高于普通公平匹配。
2. 玩家可能因为换宠、换五行、恢复血量后重新尝试，因此不能用普通目标池的一次性限制完全锁死。

## 10. 前端展示口径

### 9.1 列表展示

普通目标卡面不再强调“对方当前有多少可抢金币”，而应强调：

1. 相近等级段
2. 风险标签
3. 主灵宠轮廓或简化情报
4. 当前是否已挑战过该目标
5. 是否属于稳妥目标或挑战目标

普通目标卡面不应继续向用户传达以下旧心智：

1. 这是一只等待被抢空的钱包
2. 这张卡的收益直接等于对方当前余额
3. 只要现在看得到，点击就一定能打

### 10.1 掠夺详情信息边界

掠夺详情页应只展示：

1. 对手主灵宠外观与等级
2. 深度窥视结果
3. 风险与战斗提示
4. 挑战规则说明

掠夺详情页不再展示：

1. 对手田地卡片
2. 对手田地成熟情况
3. 预计可掠夺金币
4. 任何暗示“本次能从对方田地拿多少钱”的信息

### 9.2 页内状态展示

普通掠夺页建议明确展示：

1. 今日剩余挑战次数
2. 当前刷新按钮的用途说明
3. 目标是否已因次数限制不可再挑战

### 9.3 结果展示

结算结果弹窗与战报文案必须明确：

1. 奖励由系统发放
2. 本次挑战不会扣除对方金币
3. 胜、平、负分别对应不同奖励档位

### 9.4 不再展示的口径

普通目标池与普通掠夺页中，不再展示：

1. “保护中，还剩 xx 分钟”
2. “目标刚结束保护期”
3. 依赖 `protectedUntil` 的时间型文案

## 11. 并发失败提示文案

以下文案作为 V1 建议口径，后端错误码与前端 toast/弹窗应尽量对齐：

1. `该目标刚被其他玩家捷足先登，已为你补入新目标。`
2. `你今天已经挑战过这个目标了，换一个试试。`
3. `该目标今日已达到挑战上限，已为你刷新候选。`
4. `你今日挑战次数已用完，明日再来。`
5. `你的出战状态已变化，请刷新后重试。`

实现要求：

1. 这类失败默认不扣挑战次数
2. 可自动补 1 张卡时，优先自动补卡
3. 若无法局部补卡，再提示整页刷新

## 12. 原子性与后端实现要求

### 11.1 必须保留最终原子校验

即使前端列表已展示目标，后端在创建普通掠夺订单时仍必须在事务内校验：

1. 目标仍在普通目标池可见集合中
2. 攻击者当日总挑战次数未超限
3. 同攻同守当日未结算过
4. 防守者当日成功被掠夺次数未达 5
5. 攻击者出战状态与版本号仍有效

### 11.2 失败不应额外惩罚

若后端在创建订单时因上述校验失败而拒绝：

1. 不应扣攻击者挑战次数
2. 不应写入成功结算记录
3. 不应写入防守方成功被掠夺次数

### 11.3 计数口径建议

建议新增或沉淀以下统计口径：

1. 攻击者每日总挑战已结算次数
2. `attackerPlayerId + defenderPlayerId + dateKey` 的唯一结算记录
3. 防守者每日成功被掠夺次数

实现上可采用：

1. 独立计数表
2. 在 `raid_order` / `raid_settlement` 基础上按日聚合

V1 更推荐独立或半独立计数结构，避免高频聚合放大并发复杂度。

## 14. 对 AI 开发的明确要求

后续 AI 开发必须遵守以下约束：

1. 不要把时间保护期继续作为普通目标池的筛选条件。
2. 不要在普通掠夺中继续扣减防守方金币或田地收益。
3. 不要把“同攻同守每日 1 次”误实现成“同攻同守每日成功 1 次”。
4. 不要把平局或失败计入“防守者每日成功被掠夺 5 次”。
5. 不要默认刷新目标可以跨很大等级差。
6. 不要把并发失败当成扣次数的正常失败。
7. 不要在前端继续沿用“保护中 xx 分钟后解除”的旧文案。
8. 不要给防守方自动发放任何额外系统奖励。
9. 不要把复仇目标错误地套进普通目标池的匹配和次数限制。
10. 不要把完败奖励结算成 `0`。

## 15. 本轮未定事项

以下内容本轮未完全定死，开发时不得擅自补成最终产品规则：

1. 每日免费刷新次数具体数值
2. 天机符增加挑战次数的价格与梯度
3. 天机符增加刷新次数的价格与梯度
4. 各等级段基础奖励 `B` 的精确数值表
5. 复仇目标是否沿用普通目标池的完整奖励矩阵，还是单独做倍率

若后续开发必须依赖这些值，应先补一轮规则确认，再落代码。

## 16. 推荐实施顺序

建议实现顺序如下：

1. 先改后端规则，移除时间保护期与真实扣资源逻辑
2. 再补同攻同守与防守者成功被掠夺次数校验
3. 再改目标池匹配逻辑，加入等级段筛选
4. 再改 Shared DTO
5. 最后改前端文案、展示与失败提示

原因：

1. 后端口径先统一，前端才不会反复返工
2. 收益与次数规则是最核心的真规则
3. 匹配和 UI 文案是围绕真规则构建的外层表现

## 17. 一句话结论

普通掠夺 V1 的统一方向是：

取消时间保护期，改为次数保护；取消真实抢钱，改为系统奖励；每日 10 次挑战、北京时间重置；刷新以同段为主并混入少量高两档挑战目标；复仇作为普通池例外处理；并发失败不额外惩罚玩家，且必须给出可理解提示。
