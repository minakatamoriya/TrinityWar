# TrinityWar 掠夺订单与异步结算专项设计 v0.1

## 一、文档目的

这份文档是给 AI 和开发执行用的掠夺订单与异步结算专项设计。

前面的文档已经分别定了：

1. 掠夺玩法方向与首发范围。
2. raid-target 接口契约。
3. 掠夺链路的事务、幂等、日志要求。
4. 掠夺相关数据表的大致 schema。

但真正开始写 raid 模块时，最容易写乱的不是某一个 controller，而是下面这几个核心问题：

1. RaidOrder 到底什么时候创建。
2. 兵力是发起时冻结，还是结算时再扣。
3. 目标池快照、真实目标状态、暴露资产锁定三者如何衔接。
4. 首版如果想先做“看起来即时出结果”，代码结构要不要直接写死成同步黑盒。
5. 异步 settlement job 失败后，如何补偿和重试。
6. 同一个订单、同一个目标、同一批暴露资产，怎样避免重复结算和脏数据。

这份文档的目标，就是把这些点一次定清楚，让 AI 后续能按订单模型、状态机、事务边界和 worker 边界分步落地，而不是把 raid 写成一段巨大的 if-else。

## 二、先给结论：首版不要把 raid 写成纯同步黑盒

首版可以让玩家体验上接近“很快出结果”，但代码结构不应该写成单个接口里一次做完所有事情、完全不保留订单层。

建议首版采用：

1. 发起订单事务。
2. 结算逻辑服务。
3. 可选的异步 settlement worker。

也就是说：

1. 玩家点击掠夺时，先创建 RaidOrder。
2. 发起时立刻冻结兵力、锁定结算输入。
3. 首版如果要表现成“快速出结果”，可以由 API 在订单创建后立即调用 settlement service 完成结算。
4. 但 settlement service 本身必须是可被 worker 单独调用的。

这样做的好处是：

1. 首版联调体验仍然快。
2. 后续迁到真实异步结算，不需要重写业务骨架。
3. 订单、结算、战报、重试、补偿的边界都更清楚。

## 三、首发范围下的 raid 模式划分

首发掠夺建议明确区分两类订单。

## 1. 单人抢夺订单

特点：

1. 单个 attacker 发起。
2. 目标可见且可立即判断。
3. 从用户体验上接近即时结算。

### 建议处理方式

1. 发起成功后立刻创建 RaidOrder。
2. 同一个请求内可继续完成 settlement。
3. 但 settlement 必须仍然落 RaidSettlement 和 BattleReport。

## 2. 双人通缉订单

特点：

1. 由发起人创建通缉令。
2. 等待协力者接受。
3. 接受后才进入正式结算。

### 建议处理方式

1. 发起通缉时先创建 BOUNTY_PENDING 类订单或等价状态。
2. 接受时再冻结第二份兵力和锁定最终份额。
3. 正式结算建议走异步 worker，更符合玩法本身。

### 结论

首发虽只做两种模式，但订单状态机不要混成一个状态。

至少要让系统能区分：

1. 单人即时抢夺订单。
2. 双人通缉待接受订单。
3. 双人通缉已接受待结算订单。

## 四、掠夺链路必须锁定的 4 类东西

任何一次 raid，只要发起成功，都必须明确锁住下面 4 类输入中的至少 3 类：

## 1. 进攻方兵力输入

至少包含：

1. attackerPlayerId
2. dispatchedUnitCount
3. 发起瞬间 availableCount / frozenCount 快照
4. transportCapacitySnapshot

### 原则

1. 兵力必须发起时或接受时冻结。
2. 不能等结算时再扣，否则期间用户还能去训练、去升级、去重复出征。

## 2. 防守目标输入

至少包含：

1. defenderPlayerId
2. defenderFieldSlotId 或目标对象定位键
3. defender 当前相关地块/暴露资产快照
4. 防守方风险快照

### 原则

1. 不能只保留一个 targetId 然后结算时再临时查最新状态。
2. 否则会出现“看到很肥，结算时被别人打空”的体验断裂。

## 3. 暴露资产锁定份额

这部分是 raid 最关键的一层。

建议明确区分：

1. 软锁
2. 硬锁

### 软锁

含义：

1. 目标池生成或查看详情时，为了展示稳定性，临时记录一份可抢快照。
2. 软锁不阻止别人查看，也不真正占用资产。

### 硬锁

含义：

1. raid order 发起成功后，正式锁住本次可结算份额。
2. 从硬锁开始，这一份额不能再被防守方收回、也不能被其他 raid order 重复占用。

### 原则

1. 单人抢夺建议在订单创建时直接进入硬锁。
2. 双人通缉建议发起时先软锁，接受时转硬锁。

这和仓库记忆里的结论一致：

1. 通缉令发起时软锁暴露资产。
2. 通缉令接受时硬锁份额。

## 4. 幂等与结算资格输入

至少包含：

1. requestIdempotencyKey
2. protection 校验结果快照
3. revenge eligibility 快照
4. visibility snapshot

### 原则

1. raid 不是普通扣钱接口，重试带来的重复结算风险更大。
2. 因此 request 幂等和业务资格都必须落单据，不应只在内存里判断一次。

## 五、推荐订单状态机

首版建议不要只用 `PENDING / SETTLED / FAILED` 三个状态。

建议至少细化到下面这些状态。

## 1. 单人抢夺订单状态

1. `CREATED`
2. `LOCKED`
3. `SETTLING`
4. `SETTLED`
5. `SETTLEMENT_FAILED`
6. `CANCELLED`

### 含义

1. `CREATED`：订单已生成，但还未完成关键锁定写入。
2. `LOCKED`：兵力和可结算份额都已锁定。
3. `SETTLING`：已进入结算过程。
4. `SETTLED`：结算完成，结果已落库。
5. `SETTLEMENT_FAILED`：结算异常，需要补偿或重试。
6. `CANCELLED`：发起后因目标失效或资格失效被取消。

## 2. 双人通缉订单状态

1. `BOUNTY_CREATED`
2. `BOUNTY_WAITING_PARTNER`
3. `BOUNTY_ACCEPTED`
4. `LOCKED`
5. `SETTLING`
6. `SETTLED`
7. `SETTLEMENT_FAILED`
8. `BOUNTY_EXPIRED`
9. `CANCELLED`

### 说明

1. 如果首版不想把状态码做得太多，可以在数据库 enum 上先收口，业务层再做二级语义映射。
2. 但不建议把“待协力”和“已锁定待结算”混成同一个状态。

## 六、推荐订单生命周期

## 1. 单人抢夺完整生命周期

步骤如下：

1. 玩家点击目标。
2. API 校验目标仍在自己可见目标池中。
3. API 校验 protection、冷却、兵力、次数。
4. 事务内创建 RaidOrder。
5. 事务内冻结兵力。
6. 事务内锁定本次暴露资产份额。
7. 提交订单事务。
8. 调用 settlement service。
9. settlement service 写 RaidSettlement、BattleReport、WalletChangeLog。
10. 解冻或扣减兵力到最终状态。
11. 返回结果。

### 关键点

1. 第 4 到 6 步必须在同一个事务里。
2. 第 8 到 10 步可以首版同步执行，但逻辑必须可独立搬进 worker。

## 2. 双人通缉完整生命周期

步骤如下：

1. 发起人选择明显更强目标，点击通缉。
2. API 校验目标资格和通缉资格。
3. 事务内创建通缉订单。
4. 发起人兵力先冻结发起份额或记录预占。
5. 对目标收益做软锁快照。
6. 订单进入 `BOUNTY_WAITING_PARTNER`。
7. 协力者接受通缉。
8. API 校验协力者资格、兵力、通缉是否过期。
9. 事务内将订单从软锁升级为硬锁。
10. 冻结协力者兵力。
11. 锁定最终可结算份额。
12. 提交后投递 settlement job。
13. worker 执行结算。
14. 落结算、战报、奖励和固定协力奖励。

### 关键点

1. 通缉单的发起和接受是两个独立事务。
2. 接受成功前，不应占死全部收益份额。
3. 首版如暂不开放外部微信好友真实入战，也应保留“协力者接受 -> 进入结算”的结构。

## 七、软锁与硬锁的推荐落地方式

首版不建议只靠 `targetSnapshotJson` 来表达锁定。

建议最少有下面两层结构：

## 1. RaidTargetPool

职责：

1. 用于展示当前玩家可见目标。
2. 保存列表和详情展示所需快照。

### 不负责

1. 真实锁定资产份额。
2. 真实结算唯一来源。

## 2. RaidAssetLock 或等价结构

建议新增一个独立模型，或在 RaidOrder 内明确等价字段。

### 建议字段

1. id
2. raidOrderId
3. defenderPlayerId
4. assetType
5. sourceEntityId
6. lockedGold
7. lockedItemJson
8. lockMode
9. status
10. expiresAt
11. createdAt
12. updatedAt

### lockMode 建议

1. `SOFT`
2. `HARD`

### status 建议

1. `ACTIVE`
2. `RELEASED`
3. `CONSUMED`
4. `EXPIRED`

### 作用

1. 让暴露资产锁定变成可查、可补偿、可释放的正式实体。
2. 避免把所有锁定语义都塞进 RaidOrder 的 JSON 快照里。

### 如果你想压缩首版表数量

也可以先不单独建表，但至少要在 RaidOrder 中显式存：

1. lockMode
2. lockedGold
3. lockExpiresAt
4. lockStatus

不建议完全不落结构化锁定字段。

## 八、兵力冻结与训练队列的关系

仓库已有明确结论：

1. 兵力训练必须走耗时队列，不能瞬发。
2. 否则会破坏战力展示、事前准备和通缉判断的可信度。

因此 raid 对 army 的约束必须明确：

## 1. 可出征兵力只能来自 availableCount

不能来自：

1. training 中单位
2. 已冻结单位
3. 未来将完成的单位

## 2. 发起成功后必须立即反映到 frozenCount

这样才能保证：

1. 首页资源条正确。
2. 下一次 raid 资格正确。
3. 训练、升级、通缉判断不会读到假兵力。

## 3. settlement 结束后再转最终状态

建议：

1. 成功结算后，从 frozenCount 转为消耗后的 total/available 状态。
2. 如需保留伤兵，可写入 woundedCount。
3. 如首版不做伤兵，可直接扣 totalCount。

## 九、settlement service 和 worker 的职责边界

## 1. RaidSettlementService

职责：

1. 接收一个已锁定的 RaidOrder。
2. 读取结算输入快照。
3. 调用 raid rule service 计算胜负、收益、战损。
4. 写 RaidSettlement。
5. 写 BattleReport。
6. 写 WalletChangeLog。
7. 释放或消费 RaidAssetLock。
8. 回写 RaidOrder 最终状态。

### 不应该做

1. 创建订单。
2. 负责通缉接受流程。
3. 直接处理 HTTP request。

## 2. RaidSettlementWorker

职责：

1. 从 job queue 取到 `raidOrderId`。
2. 调用 RaidSettlementService。
3. 记录 SystemJobLog。
4. 处理重试与失败转移。

### 不应该做

1. 自己重写一套结算公式。
2. 自己直接拼 battle report 文案。

## 3. RaidOrderUseCase

职责：

1. 校验当前玩家是否可发起 raid。
2. 创建订单。
3. 锁定输入。
4. 同步模式下调用 settlement service，或异步模式下投递 worker。

### 结论

raid 至少要拆成三层：

1. order use case
2. settlement service
3. settlement worker

## 十、首版同步体验与后续异步能力如何兼容

这是最关键的工程策略之一。

建议首版支持两种运行模式：

## 1. QUICK_SETTLE 模式

特点：

1. 订单创建成功后，API 直接调用 RaidSettlementService。
2. 用户几乎立即拿到结果。
3. 主要用于单人抢夺的首版验证。

### 适用场景

1. 单人抢夺。
2. 本地和测试环境。
3. 首批 UI 联调。

## 2. ASYNC_SETTLE 模式

特点：

1. 订单创建后只返回 `orderCreated` 和预估状态。
2. 由 worker 异步结算。
3. 前端通过刷新 scene-content 或 reports 获取最新结果。

### 适用场景

1. 双人通缉。
2. 复杂结算。
3. 线上正式模式。

### 建议

首版代码结构先按这两种模式都能承载来设计，即使初期只启用 QUICK_SETTLE。

## 十一、失败与补偿策略

raid 最大的问题不是“打赢打输”，而是“订单半成功”。

## 1. 订单已创建，但 settlement service 报错

处理建议：

1. RaidOrder 标记为 `SETTLEMENT_FAILED`。
2. 冻结兵力暂不直接丢失。
3. 按补偿策略释放兵力或进入人工修复队列。
4. SystemJobLog 记录失败原因。

## 2. 结算结果已写一半

要求：

1. settlement 核心写入必须放单事务。
2. 包括 RaidSettlement、BattleReport、WalletChangeLog、RaidAssetLock 状态更新、RaidOrder 最终状态。

这样可避免：

1. 有结算无战报。
2. 有战报无到账。
3. 有到账无订单结束状态。

## 3. settlement worker 重试

要求：

1. 通过 RaidSettlement 的 `raidOrderId` 唯一约束防止重复落结算。
2. worker 每次执行前先检查订单状态是否已 `SETTLED`。
3. 如果已完成，直接幂等返回。

## 4. 目标锁定过期

要求：

1. 软锁和硬锁都要有过期时间。
2. 通缉单超时未接受，软锁自动释放。
3. 硬锁订单长时间未结算，必须由补偿任务扫描并处理。

## 十二、推荐数据库约束与索引补充

除了 schema 文档里已有的 RaidOrder、RaidSettlement、BattleReport 索引外，建议额外补下面几类约束。

## 1. RaidOrder

建议补充：

1. `sourceTargetPoolId + status` 查询索引
2. `requestIdempotencyKey` 唯一约束保持不变
3. `attackerPlayerId + createdAt` 索引

## 2. RaidAssetLock

如果独立建表，建议有：

1. `defenderPlayerId + status + expiresAt` 索引
2. `raidOrderId` 唯一或索引
3. `sourceEntityId + lockMode + status` 索引

## 3. BattleReport

建议补充：

1. `ownerPlayerId + reportType + createdAt` 索引
2. `opponentPlayerId + createdAt` 可选索引

## 十三、接口返回建议

## 1. 单人抢夺返回

首版建议返回：

1. orderId
2. settlementStatus
3. result 摘要
4. home
5. scenes

### 如果 QUICK_SETTLE

可返回：

1. 最终结果
2. 是否进入保护期
3. 获得金币
4. 溢出待领取信息

## 2. 双人通缉返回

建议分两种：

### 发起通缉返回

1. orderId
2. status=`BOUNTY_WAITING_PARTNER`
3. bountyExpiresAt
4. inviterSnapshot

### 接受通缉返回

1. orderId
2. status=`LOCKED` 或 `SETTLING`
3. settleMode=`async`
4. estimatedReadyAt

## 十四、AI 实现边界建议

### 1. 第一轮 AI 任务

只生成：

1. RaidOrder 状态机常量
2. RaidOrderUseCase 骨架
3. RaidSettlementService 骨架
4. RaidSettlementWorker 骨架

不要第一轮就把 battle formula、报文文案、前端联调一次写完。

## 2. 第二轮 AI 任务

只生成：

1. 单人抢夺订单创建事务
2. 兵力冻结
3. 资产硬锁
4. QUICK_SETTLE 模式下的 settlement 调用

## 3. 第三轮 AI 任务

只生成：

1. RaidSettlement 事务
2. BattleReport 落库
3. WalletChangeLog 落库
4. 错误补偿路径

## 4. 第四轮 AI 任务

只生成：

1. 双人通缉创建
2. 通缉接受
3. 软锁升级为硬锁
4. ASYNC_SETTLE 模式

## 十五、验收清单

一套 raid 订单与结算设计进入“可执行”状态前，至少应满足：

1. 订单创建和订单结算是两个清晰阶段。
2. 兵力冻结时点明确。
3. 暴露资产锁定机制明确。
4. settlement service 和 worker 边界明确。
5. 支持 QUICK_SETTLE 与 ASYNC_SETTLE 两种模式。
6. settlement 核心写入可放在单事务里。
7. 重试不会造成重复结算。
8. 失败后存在释放锁定和恢复兵力的补偿策略。
9. 双人通缉不是在单人订单上硬套 if-else。
10. AI 可以按订单创建、结算、worker、通缉接受四块分步实现。

## 十六、最自然的下一份规划文档

如果还继续只写规划，不落代码，那么下一份最自然的是：

1. Prisma seed 与初始静态数据装载计划。
2. 后端初始建档与首批测试账号方案。
3. battle report 与 raid page 前端联调字段清单。

如果仍以 AI 实操优先，我更建议先写第 1 个。

因为 schema、migration、读写链路、mock 退场、raid 订单都已经比较完整了，下一步最容易卡住 AI 的，就是静态定义和初始化数据到底怎么装载、哪些走 seed、哪些走配置、哪些必须首批预置。
