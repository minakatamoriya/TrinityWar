# 2026-06-06 后台危险操作审计与二次确认 v0.1

## 一、当前结论

P1-5「后台危险操作审计与二次确认」已经在本轮首发收口中完成。

本轮覆盖的危险操作：

- 删除玩家。
- 删除灵植定义。
- 删除灵宠定义。
- 后台发送带资源附件通知。

仍然暂缓的赛季后台写操作：

- 手动生成指定赛季奖励。
- 补发缺失赛季奖励。
- 撤销未领取赛季奖励。

这些赛季写操作不纳入本轮 P1-5 完成标准，后续如果要做，应单独设计权限、审计和回滚规则。

## 二、数据库变更

新增审计表：

```prisma
model AdminOperationAuditLog
```

对应迁移：

```powershell
services/game-server/prisma/migrations/052_admin_operation_audit_log/migration.sql
```

审计表字段：

- `action`：操作类型。
- `target_type`：操作目标类型。
- `target_id`：操作目标 ID。
- `admin_actor`：后台操作者，当前固定为 `admin-console`。
- `reason`：操作原因。
- `confirm_text`：二次确认文本。
- `metadata_json`：操作附加信息。
- `created_at`：审计记录创建时间。

## 三、后端确认规则

删除类操作要求请求体包含：

- `reason`：4 到 200 个字符。
- `confirmText`：必须与目标 ID 完全一致。

带附件通知要求请求体包含：

- `reason`：4 到 200 个字符。
- `confirmText`：必须等于 `SEND_ATTACHMENT_NOTIFICATION`。

已落审计的后端动作：

- `delete-player`
- `delete-seed-definition`
- `delete-spirit-definition`
- `create-global-notification-with-attachments`
- `create-player-notification-with-attachments`

## 四、后台前端变更

后台前端已经为危险操作提交审计字段。

保留浏览器二次确认弹窗的操作：

- 删除玩家。
- 删除灵植定义。
- 删除灵宠定义。

新增二次确认弹窗的操作：

- 发送带资源附件的全服通知。
- 发送带资源附件的单玩家通知。

## 五、验收记录

已执行命令：

```powershell
npm run prisma:generate --workspace @trinitywar/game-server
npm run build --workspace @trinitywar/shared
npm run prisma:validate --workspace @trinitywar/game-server
npx prisma migrate deploy --schema prisma/schema.prisma
npm run verify:admin-dangerous-audit --workspace @trinitywar/game-server
npm run build --workspace @trinitywar/game-admin
npm run build --workspace @trinitywar/game-server
```

说明：

- `npx prisma migrate deploy --schema prisma/schema.prisma` 需要在 `services/game-server` 目录下执行，才能读取本地 `.env`。
- 验收脚本会创建一次性 dev 玩家和通知，不会删除真实玩家或真实配置定义。

验收脚本结果：

```json
{
  "ok": true,
  "auditedActions": [
    "create-player-notification-with-attachments",
    "delete-player"
  ]
}
```

脚本覆盖内容：

- 带附件通知缺少二次确认时返回 `400`。
- 带附件通知二次确认正确时写入审计。
- 删除玩家确认文本错误时返回 `400`。
- 删除玩家确认文本正确时写入审计。
- 删除灵植、灵宠定义时，确认文本不匹配会在实际删除前被拒绝。

## 六、遗留风险

当前 `admin_actor` 仍固定为 `admin-console`，因为项目现阶段使用 debug header 作为后台访问控制，还没有正式后台账号和角色体系。

后续进入正式生产后台前，应补充：

- 后台操作者账号。
- 角色与权限。
- 操作人 ID 写入审计。
- 高风险操作的更严格审批或回滚机制。
