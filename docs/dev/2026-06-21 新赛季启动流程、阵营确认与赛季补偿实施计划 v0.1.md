# 2026-06-21 新赛季启动流程、阵营确认与赛季补偿实施计划 v0.1

## 1. 目标

本文件用于替代“单个新赛季提示弹窗”的临时方案，定义一个可持续扩展的 **新赛季启动流程**：

1. 赛季切换仍以后端为唯一权威。
2. 在线玩家跨赛季后，不需要重新登录，但必须进入一个阻塞式启动流程。
3. 当前启动流程至少包含：
   - 新赛季开启说明
   - 阵营确认（保持当前阵营 / 调整阵营）
4. 启动流程要为未来扩展赛季玩法、赛季规则说明、赛季创意步骤预留结构。
5. 赛季补偿机制必须从本次开始纳入设计，避免以后继续在业务逻辑里临时打补丁。

本文件不直接实现赛季追赶。赛季追赶继续后置。

## 2. 当前结论

### 2.1 新赛季启动流程采用阻塞式

新赛季开始后，玩家不能直接继续停留在旧页面上下文，也不能立即进入新赛季主界面。

必须先完成一次 **新赛季启动流程**，然后才进入首页。

当前阶段的启动流程最少包含两个步骤：

1. `season-intro`
2. `faction-confirm`

### 2.2 阵营确认不是“可稍后处理”的普通入口

如果本赛季允许阵营调整，则玩家必须在启动流程中完成一次阵营确认：

1. 保持当前阵营
2. 调整为其他阵营

“保持不变”也算完成了一次确认。

每赛季最多确认一次。

### 2.3 新手阵营选择 UI 可以复用，但页面语义不同

当前已有新手阵营选择页面：

- `apps/game-client/src/shell/AuthEntryScreen.tsx`
- `apps/game-client/src/config/sceneConfig.ts`

可复用的内容：

1. 阵营卡片视觉样式
2. 阵营标题、特性文案、阵营 code

不可直接复用的内容：

1. 页面标题与说明
2. 按钮文案
3. 进入方式
4. 提交逻辑
5. 确认规则

建议做法：

1. 提取一个共享阵营卡片组件
2. 新手建档与赛季启动分别包自己的页面壳与提交流程

### 2.4 赛季补偿机制需要分成“正常切赛季”与“异常补偿”两类

不能把所有赛季切换损失都直接折算成补偿，否则会形成利用窗口。

必须区分：

1. **正常定时赛季结束**
2. **异常提前结束 / 故障性中断 / 人工紧急切赛季**

当前建议：

1. 正常定时赛季结束：默认不对“未完成中的赛季内竞争过程”做全价补偿。
2. 异常切赛季：通过统一补偿机制发补偿通知。

## 3. 用户流程

### 3.1 常态阶段

前端常态展示赛季剩余时间：

1. 首页顶部
2. 赛季入口
3. 必要时在首页主卡片显示“本赛季剩余 X”

注意：

1. 倒计时只是体验提示。
2. 真正切赛季以后端为准。

### 3.2 临近结束阶段

建议增加一个临近结束阈值，先不阻塞：

1. 默认 `24 小时`
2. 如果要更轻，也可以先做 `1 小时`

临近结束时，前端强化提示：

1. 首页顶部加强调文案
2. 灵田、战斗、俸禄等强季节性感知模块可出现轻提示

例如：

`本赛季即将结束，未完成的赛季内进度将在新赛季重置。`

### 3.3 跨赛季瞬间

以服务端状态为准：

1. 前端 timer 到点后触发刷新
2. 页面从后台切回前台时也会刷新
3. 写接口被 `SEASON_ROLLED_OVER` 拦截时也会刷新

刷新后如果发现：

1. `seasonNumber` 变化
2. 或 `season.transition.resetApplied === true`

则进入启动流程。

### 3.4 启动流程

建议做成一个阻塞式的全屏层或大 modal，不允许直接关闭回游戏：

步骤一：`season-intro`

1. 轻过场动画
2. 标题：`第 X 赛季开启`
3. 展示“保留 / 重置 / 奖励 / 补偿”摘要
4. 按钮：`继续`

步骤二：`faction-confirm`

如果 `factionChoiceStatus === 'available'`：

1. 展示当前阵营
2. 提供两个主操作：
   - `保持当前阵营`
   - `调整阵营`

如果 `factionChoiceStatus !== 'available'`：

1. 该步骤自动跳过
2. 或仅展示“本赛季阵营已锁定”

步骤三：`faction-select`

只有在玩家选择“调整阵营”时进入：

1. 复用新手阵营卡片组件
2. 选择后弹二次确认框
3. 提交成功后结束启动流程

步骤结束后：

1. 清理旧页面上下文
2. 返回首页
3. 标记本赛季启动流程已完成

## 4. 当前建议的补偿机制

## 4.1 为什么要独立设计

赛季补偿不是简单的“把所有未完成进度折价发钱”。

如果规则过宽，会直接形成利用：

1. 玩家在赛季结束前故意把资源压到“半完成状态”
2. 期待系统按高于真实风险的方式自动折算
3. 变相把赛季末行为变成套利动作

所以补偿机制必须遵守：

1. 只补“不可归责于玩家的损失”
2. 不补“玩家本来就知道赛季会结束的正常时点风险”
3. 不把赛季补偿做成新赛季开局优势来源

## 4.2 当前首发建议

### A. 正常定时赛季结束

正常赛季结束时：

1. 不对灵田成长进度做全价补偿
2. 不对战斗次数未用完做补偿
3. 不对俸禄未领取做额外补偿
4. 不对训练、成长中过程进度按最终产出折现

原因：

1. 赛季结束时间是公开的
2. 玩家有机会自行安排收尾
3. 全价补偿会产生季末套利

### B. 异常切赛季 / 提前切赛季 / 服务端故障

只有在异常情况下，才走补偿机制。

补偿发放方式建议统一为：

1. `player_notification`
2. `category = compensation`
3. 通过通知附件领取
4. 需要审计与幂等

## 4.3 灵田补偿建议

针对你提出的例子：

`灵田种植了一半，是否全价折算补偿？`

当前建议：

### 正常定时赛季结束

不做全价折算。

理由：

1. 半成长不等于成熟产出
2. 全价补偿会鼓励玩家在赛季末把田全部种满来套补偿
3. 灵田是季节性竞争过程，不是长期资产

### 异常提前结束

建议只补 **已投入且不可回收的确定成本**，不补成熟后的理想收益。

当前系统更适合的口径是：

1. 如果灵田有明确 `investedGold`
   - 可以考虑按 `investedGold` 的一定比例补
2. 不按 `currentClaimableGold + 预估成熟收益` 补
3. 不直接按“如果长完能收多少”补

首发建议：

1. 正常切赛季：`无灵田补偿`
2. 异常切赛季：`按投入成本补，不按未来收益补`

## 4.4 未来可扩展的补偿类型

建议把赛季补偿先抽象成一套枚举，不要以后每种都临时写 if：

```ts
type SeasonCompensationReason =
  | 'early-rollover-field'
  | 'early-rollover-training'
  | 'early-rollover-stipend'
  | 'manual-fix'
  | 'ops-compensation';
```

首发不一定全部做，但文档要先定口径。

## 5. 后端状态设计建议

当前已有：

```ts
type ClientSeasonTransition = {
  currentSeasonNumber: number;
  previousSeasonNumber: number | null;
  resetApplied: boolean;
  refreshRequired: boolean;
  seasonStartsAt: string;
  seasonEndsAt: string;
  factionChoiceStatus: 'not_needed' | 'available' | 'used' | 'expired';
  factionChoiceDeadlineAt: string | null;
};
```

为了支撑启动流程，建议新增独立结构，不继续把所有步骤都塞进 `seasonTransition`：

```ts
type ClientSeasonStartupStepType =
  | 'season-intro'
  | 'faction-confirm';

interface ClientSeasonStartupState {
  seasonNumber: number;
  blocking: boolean;
  completed: boolean;
  currentStep: ClientSeasonStartupStepType | null;
  availableSteps: ClientSeasonStartupStepType[];
  factionChoiceStatus: 'not_needed' | 'available' | 'used' | 'expired';
  factionChoiceDeadlineAt: string | null;
  compensationNotificationIds?: string[];
}
```

建议作为 `bootstrap.season.startUp` 或 `bootstrap.season.startup` 返回。

不建议把未来的步骤继续堆在 `transition` 里。

## 6. 启动流程持久化建议

为了支持“在线玩家刷新后仍知道自己走到哪一步”，建议给玩家赛季状态补最小持久化字段。

建议在 `player_season_state` 或单独新表中记录：

```prisma
startupCompletedSeasonNumber Int? @map("startup_completed_season_number")
startupCompletedAt DateTime? @map("startup_completed_at")
factionChoiceUsedSeasonNumber Int? @map("faction_choice_used_season_number")
factionChoiceUsedAt DateTime? @map("faction_choice_used_at")
```

当前阶段如果只做最小版本，至少要有：

1. 本赛季启动流程是否已完成
2. 本赛季阵营确认是否已完成

## 7. 阵营确认规则建议

### 7.1 规则

1. 新赛季开始后，玩家必须完成一次阵营确认；未确认前启动流程持续阻塞，不设置过期窗口。
2. 阵营确认包括：
   - 保持当前阵营
   - 更换为其他阵营
3. 保持不变也会消耗本赛季这一次确认机会。
4. 一旦确认，当前赛季不能再改。
5. 好友、关注、历史记录不删除。
6. 只影响本赛季阵营立场与玩法权限。

### 7.2 确认时限建议

当前建议：

1. 阵营确认没有 24 小时过期窗口。
2. 对于阻塞式启动流程，玩家第一次进入新赛季时必须先做确认。
3. 如果玩家异常中断后重新进入，后端继续返回未完成的启动流程。
4. `factionChoiceDeadlineAt` 当前返回 `null`，字段保留给未来活动型赛季规则使用。

### 7.3 为什么“保持当前阵营”也要算一次确认

因为你的产品意图不是“给一次免费反悔机会”，而是：

`新赛季开始时，必须完成一次站队确认。`

这和非阻塞的“稍后再说”是完全不同的。

## 8. 前端实现建议

## 8.1 不建议继续在现有 `globalFeatureModal` 上叠补丁

当前 `globalFeatureModal` 适合：

1. 说明弹窗
2. 商店弹窗
3. 赛季签到

不适合承担：

1. 阻塞式多步骤启动流程
2. 全屏过场
3. 阵营确认工作流

建议新建：

1. `SeasonStartupFlowHost`
2. `SeasonStartupIntroStep`
3. `SeasonStartupFactionConfirmStep`
4. `SeasonStartupFactionSelectStep`

## 8.2 新手阵营选择复用建议

建议抽出：

1. `FactionChoiceCardGrid`
2. `FactionChoiceCard`

由两套壳层复用：

1. `AuthEntryScreen` 中的新手建档流程
2. `SeasonStartupFactionSelectStep` 中的赛季调整流程

## 8.3 返回首页策略

本轮已确认：

1. 赛季启动确认完成后返回首页
2. 同时重置 tab 与临时交互上下文

启动流程完成后不建议回原页面，因为原页面上下文已经失效。

## 9. 后端接口建议

建议新增这些接口，而不是继续把所有逻辑塞进已有通用接口：

### 9.1 确认赛季启动说明

```text
POST /api/client/actions/confirm-season-startup-intro
```

作用：

1. 记录玩家已完成 `season-intro`
2. 进入下一步

### 9.2 确认保持当前阵营

```text
POST /api/client/actions/confirm-season-faction
body: { mode: 'keep-current' }
```

### 9.3 调整赛季阵营

```text
POST /api/client/actions/change-season-faction
body: { factionCode: 'human' | 'immortal' | 'demon' }
```

注意：

1. 这两类接口不能被 `SEASON_ROLLED_OVER` 通用守卫拦掉
2. 它们本身属于新赛季启动流程白名单

## 10. 文案重写建议

当前赛季规则说明文案必须按已确认规则重写，不再沿用旧表述。

建议至少包含：

### 保留

1. 天机符保留
2. 灵宠词条保留
3. 旧赛季快照、奖励、荣誉保留

### 重置

1. 金币清零
2. 灵田清空
3. 阵营贡献与排行重置
4. 灵宠等级与培养进度重置
5. 赛季签到重置
6. 每日俸禄状态重置
7. 今日战斗次数重置

### 补偿

1. 正常赛季结束不做全价灵田成长补偿
2. 异常提前结束时才按补偿机制处理

## 12. 验收标准

### P0.1 启动流程

1. 玩家跨赛季后自动进入启动流程。
2. 启动流程未完成前，不能直接进入主界面。
3. 启动流程完成后，返回首页。

### P0.2 阵营确认

1. 如果本赛季阵营确认可用，玩家必须完成一次确认。
2. 保持当前阵营也会把状态变成 `used`。
3. 当前赛季不能重复确认。
4. 常规阵营页没有长期转阵营按钮。

### P0.3 补偿机制

1. 正常定时赛季结束，不自动按灵田成熟收益全价补偿。
2. 异常切赛季可以通过补偿通知附件发放补偿。
3. 补偿通知可审计、可幂等、可查看。

## 13. 当前建议结论

本轮建议确认三条正式口径：

1. 新赛季启动流程采用阻塞式。
2. 阵营确认是启动流程的一部分，不是稍后处理的普通入口。
3. 灵田半成长等过程进度在正常赛季结束时不做全价补偿；异常切赛季才进入补偿机制。

如果这三条确认，本文件即可作为下一轮开发输入。
