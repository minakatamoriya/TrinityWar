# 2026-05-28 下一阶段开发计划 v0.1

## 1. 目标

本阶段目标不是继续堆新玩法，而是把游戏从“主框架可跑”推进到“小规模可测、可调、可运营”的状态。

重点补齐三类能力：

1. 分享、邀请、助力：补齐裂变和早期留存入口，并绑定真实游戏行为。
2. 平衡性调整：建立可跑的数值模拟和调参依据，避免太肝或过快毕业。
3. 后台系统完善：让配置、审计、回滚、用户修正、公告补偿、风控排查具备基本运营能力。

## 2. 开发原则

1. 邀请和助力不做成单纯“拉人领奖”，必须和收菜、养宠、守田、战斗、阵营贡献产生关系。
2. 数值先能模拟，再大规模改配置。没有模拟结果前，不做大范围拍脑袋调参。
3. 后台优先服务排障和调参，先做高频刚需，不追求一次性做成完整 GM 系统。
4. 所有影响资源、贡献、等级、任务奖励的后台操作，都必须有操作记录。
5. 每个阶段完成后必须有 smoke test，保证登录、首页、农场、灵宠、战斗、阵营、后台任务配置主链路可用。

## 3. 阶段排期

### P0：稳定性基线

目标：保证后续开发不会在已有硬 bug 上叠加风险。

开发项：

1. 固化后端 typecheck/build 为日常验证命令。
2. 补一轮核心接口 smoke test：
   - 登录
   - 首页读取
   - 场景读取
   - 播种
   - 收获
   - 灵宠恢复
   - 灵宠突破
   - 掠夺发起
   - 战报读取
   - 后台任务配置读取
3. 确认任务配置表 migration 可在新环境自动执行。
4. 清理 dev-login、admin debug key 的上线风险边界。

验收标准：

1. `game-server` typecheck/build 通过。
2. 核心 smoke test 可重复执行。
3. 新库执行 migration 后能登录并读取首页、场景和后台任务配置。

### P1：分享、邀请、助力 MVP

目标：做出第一版可用裂变闭环，让分享能带来游戏内真实收益，但不破坏平衡。

当前代码现状：

1. 前端已有新用户/老用户微信助力模拟入口。
2. 前端已有独立助力页，并能在新用户成功后进入新手流程。
3. 后端 `social` 模块已有站内浇水接口。
4. 数据库已有 `player_assist_record`。
5. 浇水接口已经能记录助力、缩短田地时间、写入社交动态。

当前缺口：

1. 还没有正式分享活动或分享 token。
2. 还没有新玩家邀请绑定关系。
3. 还没有外链公开读取和确认助力接口。
4. 还没有后台邀请/助力记录查询。
5. 前端仍有战斗助力模拟入口残留，需要在当前 MVP 隐藏。

核心设计：

1. 邀请新玩家：
   - 邀请者获得轻量资源或一次性助力额度。
   - 被邀请者完成新手流程后，邀请者才获得完整奖励。
   - 每日/每周设置邀请有效奖励上限。

2. 助力行为：
   - 首发只做邀请浇水助力。
   - 浇水缩短田地成长时间或提升本轮培育效率。
   - 被助力者不损失收益，助力者获得小额参与奖励。
   - 战斗助力、通缉助力、灵宠恢复助力暂缓。

3. 分享入口：
   - 邀请新玩家。
   - 田地成长中分享请求浇水。
   - 每日收菜完成后分享轻量炫耀或邀请。
   - 被偷后求助、战斗摇人、灵宠恢复分享暂缓。

推荐 MVP 范围：

1. 邀请关系表。
2. 助力记录表。
3. 邀请码/分享 token。
4. 助力领取接口。
5. 浇水助力后的田地缩时或培育效率结算。
6. 前端分享入口和助力反馈。
7. 后台查看邀请和助力记录。

建议数据表：

1. `player_invite_relation`
   - 邀请者
   - 被邀请者
   - 邀请来源
   - 是否完成新手流程
   - 奖励状态

2. `player_assist_record`
   - 发起玩家
   - 助力玩家
   - 助力类型
   - 助力目标
   - 助力数值
   - 每日计数
   - 过期时间

3. `share_token`
   - token
   - 分享玩家
   - 分享场景
   - 目标实体
   - 过期时间
   - 使用次数

接口范围：

1. `POST /api/client/share-token`
2. `GET /api/client/share-token/:token`
3. `POST /api/client/invite/accept`
4. `POST /api/client/assist`
5. `GET /api/client/assist/summary`
6. `GET /api/admin/social/invites`
7. `GET /api/admin/social/assists`

反滥用规则：

1. 同一玩家每天可助力次数有限。
2. 同一目标每天可被助力次数有限。
3. 同设备/同账号体系的重复邀请不发完整奖励。
4. 新玩家必须完成指定新手节点后才算有效邀请。
5. 助力奖励只给轻量加速和资源，不直接给高价值稀缺道具。
6. 首发不开放战斗助力、通缉助力、灵宠恢复助力。

验收标准：

1. 玩家可以生成分享 token。
2. 新玩家通过 token 建立邀请关系。
3. 被邀请者完成新手流程后，邀请者获得奖励。
4. 老玩家可以对好友进行一次助力。
5. 浇水助力效果能在田地玩法里真实生效。
6. 后台能查到邀请和助力记录。

### P2：平衡性模拟与调参体系

目标：建立一套可跑的数值模拟，用数据判断金币、种子、精华、灵宠、征召、掠夺和每日任务是否合理。

模拟对象：

1. 金币：
   - 收获产出
   - 建筑消耗
   - 征召消耗
   - 灵宠购买/养成消耗
   - 掠夺收益

2. 精华/种子：
   - 种植产出
   - 解锁消耗
   - 阵营任务上缴
   - 掠夺获得

3. 灵宠经验：
   - 自然成长
   - 投喂加成
   - 突破节点
   - 50 级封顶

4. 征召/战斗：
   - 兵力恢复速度
   - 掠夺成功率
   - 复仇收益
   - 保护时间

5. 每日任务：
   - 普通每日任务收益
   - 阵营每日任务贡献收益
   - 完成耗时
   - 是否和自然行为重叠过高

玩家模型：

1. 轻度玩家：每天 1-2 次上线，每次 3-5 分钟。
2. 标准玩家：每天 3-4 次上线。
3. 高频玩家：每天 6 次以上上线。
4. 晚入场玩家：落后 3 天/7 天。
5. 社交玩家：每天有 1-3 次助力。

输出指标：

1. 第 1 天、第 3 天、第 7 天、第 14 天玩家等级和资源分布。
2. 每日金币净增长。
3. 每日精华净增长。
4. 灵宠到达 9/19/29/39/49 的预计时间。
5. 建筑升级节奏。
6. 普通玩家和高频玩家差距。
7. 掠夺收益占总收益比例。
8. 邀请助力收益占总收益比例。

开发项：

1. 新建数值模拟脚本。
2. 将关键配置从代码中提取为模拟可读取的数据。
3. 输出 markdown/csv/json 三类结果。
4. 给后台预留“当前配置版本”和“模拟版本”的对应关系。
5. 建立一份首发推荐数值表。

验收标准：

1. 可以一键跑 7 天/14 天模拟。
2. 可以比较两套配置的差异。
3. 能看出轻度、标准、高频玩家的成长差距。
4. 能看出邀请助力是否破坏经济平衡。
5. 每次大调数值前有模拟结果留档。

### P3：后台系统完善

目标：后台从“能看一点配置”升级为“能运营、能排障、能回滚”的基础工具。

#### P3.1 任务配置版本管理

开发项：

1. 任务配置版本表。
2. 当前生效版本标记。
3. 草稿/发布状态。
4. 发布前 diff 预览。
5. 历史版本查看。

验收标准：

1. 管理员可以修改任务文案、目标、奖励。
2. 修改先进入草稿。
3. 发布后新任务刷新使用新配置。
4. 老任务是否受影响有明确规则。
5. 可以查看版本差异。

#### P3.2 修改记录和操作人审计

开发项：

1. 统一 admin audit log。
2. 记录操作人、操作类型、操作前、操作后、备注、IP/UA。
3. 接入任务配置、用户资源修正、公告、补偿、活动配置。

验收标准：

1. 每次后台写操作都有审计记录。
2. 审计记录不可被普通后台操作删除。
3. 后台可以按用户、模块、时间筛选记录。

#### P3.3 一键回滚

开发项：

1. 任务配置版本回滚。
2. 公告/活动配置回滚。
3. 资源修正不做自动回滚，必须走反向修正并留审计。

验收标准：

1. 任务配置可以回滚到历史版本。
2. 回滚行为本身也有审计记录。
3. 回滚后前端读取到正确配置。

#### P3.4 用户查询与修正

开发项：

1. 按玩家 ID/昵称/登录标识查询玩家。
2. 查看钱包、种子、精华、灵宠资源、阵营贡献、建筑等级、任务状态。
3. 支持小范围资源修正。
4. 支持封禁/解封或限制登录预留字段。

验收标准：

1. 后台可以定位单个玩家当前状态。
2. 可以修正金币、精华、灵宠资源、阵营贡献。
3. 每次修正必须填写原因。
4. 修正后玩家前端读取状态正确。

#### P3.5 公告、补偿、活动配置

开发项：

1. 公告配置。
2. 全服补偿。
3. 指定玩家补偿。
4. 限时活动配置预留。

验收标准：

1. 后台可以发布公告。
2. 玩家能看到公告。
3. 后台可以创建补偿邮件/通知。
4. 补偿领取有幂等和审计。

#### P3.6 风控与异常用户查看

开发项：

1. 异常资源增长榜。
2. 高频掠夺/助力/邀请榜。
3. 同设备或同登录来源关联查询预留。
4. 单玩家操作日志聚合页。

验收标准：

1. 能看到异常金币、精华、贡献增长。
2. 能看到异常邀请和助力行为。
3. 能快速跳转到玩家详情页处理。

## 4. 推荐执行顺序

第一周：

1. 完成 P0 smoke test。
2. 完成任务配置 migration/发布流程确认。
3. 设计分享、邀请、助力的表结构和接口契约。
4. 开始数值模拟脚本骨架。

第二周：

1. 完成分享 token、邀请关系、助力记录后端。
2. 完成前端第一版分享和助力入口。
3. 完成 7 天/14 天经济模拟。
4. 后台补任务配置版本草稿/发布。

第三周：

1. 完成助力效果接入灵宠恢复、守田或收获加成中的 1-2 个场景。
2. 完成邀请奖励结算和反滥用上限。
3. 完成用户资源查询与修正。
4. 输出第一版平衡性调整建议。

第四周：

1. 完成公告和补偿。
2. 完成 admin audit log。
3. 完成风控异常用户列表。
4. 做一轮小规模内测前验收。

## 5. 本阶段不做或暂缓

1. 小赛季奖励：建议保留设计空间，但本阶段不优先实现。
2. 大型活动系统：先做公告、补偿、简单配置，活动系统后置。
3. 复杂邀请裂变商业化：先验证助力和留存，再放大奖励。
4. 大规模数值重构：先模拟，再根据结果调整。
5. 完整 GM 系统：先覆盖排障和运营刚需。

## 6. 关键风险

1. 邀请奖励过重会导致刷号。
2. 助力收益过高会破坏轻量日常节奏。
3. 普通每日和阵营每日再次发生目标重叠。
4. 后台配置直接改线上数据，如果没有版本和审计，出错很难回滚。
5. 数值模拟如果不和真实配置同源，后续会失去参考价值。

## 7. 最小上线验收清单

1. 主链路接口稳定。
2. 任务配置可查、可改、可发布、可回滚。
3. 后台所有写操作有审计。
4. 分享邀请可以建立关系并发放受限奖励。
5. 助力至少接入一个核心行为。
6. 7 天/14 天模拟结果可输出。
7. 管理员可以查询和修正单个玩家资源。
8. 公告和补偿可用。
9. 异常邀请/助力/资源增长可查看。
