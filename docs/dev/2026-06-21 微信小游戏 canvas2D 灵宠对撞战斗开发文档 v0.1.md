# 2026-06-21 微信小游戏 canvas2D 灵宠对撞战斗开发文档 v0.1

## 1. 文档目标

本文用于指导 TrinityWar 后续把现有 HTML 战斗原型，逐步迁移为适合微信小游戏的 canvas2D 战斗演出。

这份文档的目标不是设计新的战斗规则，而是定义一套 AI 和开发都能直接执行的实现口径：

1. 明确战斗视觉方向。
2. 明确 canvas2D 首版只做什么，不做什么。
3. 明确 10 回合对撞的节奏和时间间隔。
4. 明确渲染层和结算层的边界。
5. 明确从“简单可跑”到“有爽感”的迭代顺序。

## 2. 战斗视觉定位

### 2.1 视觉方向一句话

战斗表现目标不是复杂动作动画，而是：

`卡牌棋盘式对撞 + 炉石式台面压迫感 + 短促有力的撞击反馈 + 清晰可读的回合节奏`

### 2.2 参考方向

可以参考《炉石传说》的以下气质，但不要照搬：

1. 棋盘中轴明显，双方单位在上下两端对峙。
2. 单位平时静止，真正的爽点集中在“冲刺 -> 对撞 -> 停顿 -> 回位”。
3. 暴击、克制、燃血等反馈短、狠、准。
4. 画面中心始终是战斗主体，不被大面板遮挡。
5. 卡面、边框、底座、光效比复杂骨骼动画更重要。

### 2.3 首版追求的爽感来源

首版爽感主要来自 5 个点：

1. 出场时两边单位有压迫感。
2. 对撞瞬间有明显停顿和冲击。
3. 扣血反馈和飘字反应快。
4. 暴击、克制、燃血三类反馈明显区分。
5. 10 回合播放时节奏稳定，不拖、不乱、不疲劳。

## 3. 首版边界

### 3.1 必须保留

1. 双方上下对峙棋盘布局。
2. 回合制自动播放。
3. 每回合一次对撞。
4. 血条变化。
5. 飘字反馈。
6. 暴击、克制、闪避、燃血提示。
7. 回合数展示。
8. 战斗结果展示。

### 3.2 首版不要做

1. 复杂骨骼动画。
2. 单位自由移动寻路。
3. 技能镜头切换。
4. 全屏大招演出。
5. 过多粒子特效。
6. 多层复杂 UI 面板同时弹出。
7. 在客户端重新计算战斗结果。

### 3.3 核心原则

首版 canvas2D 只负责：

`把后端或 shared 已经算好的 replay，用更有质感的方式播放出来`

不要让 canvas renderer 自己决定胜负、伤害、暴击和燃血规则。

## 4. 现有基础与迁移边界

### 4.1 现有可复用内容

当前项目已经具备以下基础：

1. 共享战斗规则：[packages/shared/src/spiritCollisionBattle.ts](D:/TrinityWar/packages/shared/src/spiritCollisionBattle.ts)
2. HTML 回放节奏层：[apps/game-client/src/battle/battleTimeline.ts](D:/TrinityWar/apps/game-client/src/battle/battleTimeline.ts)
3. HTML 战斗界面原型：[apps/game-client/src/battle/RaidBattleScreen.tsx](D:/TrinityWar/apps/game-client/src/battle/RaidBattleScreen.tsx)

### 4.2 正确的迁移边界

迁移到微信小游戏时，建议严格拆成三层：

1. `Battle Rules`
说明：负责结算、生成 replay、生成 steps/events。

2. `Battle Playback State`
说明：负责按时间推进 step，得到“当前应该显示什么”。

3. `Canvas Renderer`
说明：只根据当前播放状态做绘制和动效。

对应关系：

```text
shared 规则层 -> replay/steps -> 播放状态机 -> canvas2D 渲染
```

### 4.3 不应该做的错误迁移

不要把现有 React 页面直接“翻译”为 canvas 版本的大杂烩。

错误方向：

1. 一边播放一边重新算伤害。
2. 在 draw 里写业务规则判断。
3. 把词条、结算、UI、特效全部耦在一个大文件里。

### 4.4 低耦合与小游戏嵌入原则

这次 canvas demo 必须默认服务于“未来嵌入微信小游戏正式体系”，而不是只服务于当前 Web 验证。

必须遵守：

1. 战斗核心模块优先写成纯 TS 模块，不依赖 React 组件生命周期。
2. 播放控制、场景状态、渲染器分层明确，不允许互相倒灌业务逻辑。
3. 除最外层挂载组件外，核心模块不直接依赖 DOM 查询、CSS 动画和浏览器专属节点结构。
4. 不把资源加载、时间推进、尺寸适配写死在某一个页面组件里。
5. 不把小游戏平台逻辑直接写进通用战斗核心层。

推荐分层：

```text
Battle Core
  - replay 消费
  - playback controller
  - scene model

Runtime Adapter
  - 时间源
  - 资源加载
  - 尺寸与 DPR
  - 平台生命周期

Renderer
  - canvas2D draw
  - 特效
  - UI 绘制

Host Layer
  - React Web 容器 或 微信小游戏页面容器
```

其中：

1. `Battle Core` 应尽量可直接迁入小游戏工程。
2. `Runtime Adapter` 应作为 Web 与微信小游戏的差异隔离层。
3. `Host Layer` 可以替换，但不应该影响战斗规则和播放状态。

### 4.5 当前阶段的技术约束

为保证后续好迁移，当前 demo 阶段建议额外遵守：

1. 不在核心模块里使用 `window`、`document`、`HTMLElement` 作为必要依赖。
2. 不在核心模块里依赖 CSS transition 作为动画主实现。
3. 不把图片资源路径硬编码进多个文件，统一走资产清单或 loader。
4. 不把尺寸、坐标、时间常量散落在渲染代码各处，统一收口到 timing/layout 配置。
5. 不把事件监听和战斗推进逻辑耦在一起。

## 5. 战斗演出结构

### 5.1 画面结构

建议战斗画面分 5 层：

1. `backgroundLayer`
说明：棋盘、台面、阵营氛围底图。

2. `boardLayer`
说明：上下双方卡位底座、中轴碰撞线。

3. `unitLayer`
说明：双方灵宠卡牌或立绘主体。

4. `fxLayer`
说明：撞击火花、克制闪光、暴击爆闪、燃血红雾。

5. `uiLayer`
说明：血条、名字、回合数、飘字、结果、notice。

### 5.2 构图原则

1. 敌方在上，我方在下。
2. 中间留出明确的碰撞区域。
3. 单位主体尽量居中，不贴边。
4. 血条和名字贴近单位，不要离太远。
5. 飘字靠近受击方，避免在画面中心堆叠。

### 5.3 棋盘适配原则

不同手机的屏幕高度不一致，不能让“对撞距离”直接跟着真实屏幕高度一起变长。

正确做法是：

`先定义固定逻辑棋盘，再把棋盘居中适配进真实屏幕`

推荐口径：

1. 先定义一套固定逻辑尺寸，例如 `390 x 760`
2. 所有棋盘坐标、单位站位、notice 区域、结果面板都基于这套逻辑尺寸计算
3. 渲染时再把这套逻辑棋盘按 `contain` 方式缩放进真实视口
4. 大屏设备可以留更多上下氛围空间，但不能让实际对撞路程无限变长
5. 棋盘缩放可以变小，但默认不要放大超过逻辑原尺寸

一句话原则：

`适配的是舞台，不是冲撞距离`

### 5.4 站位原则

灵宠站位不建议直接写成“整屏高度百分比”，否则高屏设备会把上下距离拉得过大。

推荐口径：

1. 棋盘中心点固定
2. 敌我初始站位用“中心点 ± 固定半径距离”定义
3. 冲撞时双方都朝同一个中心点推进
4. 灵宠体积首版应偏克制，不要过大遮挡棋盘和反馈层
5. 上下单位之间要留出足够的碰撞区和 notice 区

推荐理解方式：

```text
centerY = boardCenterY
defenderBaseY = centerY - laneHalfDistance
attackerBaseY = centerY + laneHalfDistance
```

其中 `laneHalfDistance` 应视为棋盘内部参数，而不是直接从整屏高度实时线性推导。

## 6. 单位表现形式

### 6.1 推荐首版形式

灵宠不建议首版做完整角色动画，建议采用：

`卡牌框 + 立绘主体 + 底座光圈 + 少量位移/缩放/震动`

### 6.2 单位的最小动画集

每个单位首版只需要这几种状态：

1. `idle`
轻微呼吸感，可选，不强求。

2. `ready`
对撞前轻微蓄力，向中轴方向位移 4 到 10 像素。

3. `dash`
快速冲向碰撞点。

4. `impact`
撞击瞬间短暂停顿，并伴随缩放/震动。

5. `return`
快速回到原位。

6. `defeated`
透明度下降、缩小、下沉或轻微翻倒。

## 7. 10 回合节奏设计

### 7.1 节奏目标

10 回合战斗最怕两件事：

1. 每回合太慢，玩家看到第 4 回合就疲劳。
2. 每回合太快，玩家看不清为什么掉血。

首版目标是：

`普通局 8 到 12 秒播完，燃血局 12 到 16 秒播完`

### 7.2 普通回合标准节奏

建议每个普通回合采用固定四拍：

1. `前摇`
2. `对撞`
3. `伤害结算`
4. `回位`

推荐首版时长：

```text
前摇: 90ms
冲刺: 140ms
撞击停顿: 70ms
受击/飘字: 220ms
血条变化: 260ms
回位: 130ms
回合间隔: 80ms
```

合计约：

```text
990ms / 回合
```

10 回合普通局总时长约：

```text
9.5s 到 10.5s
```

这是一档比较稳妥的首版速度。

### 7.3 为什么“撞击停顿”很重要

撞击瞬间的 `60ms 到 90ms` 停顿非常关键。

如果没有这一下：

1. 画面会像两张图直接穿过去。
2. 没有重量感。
3. 暴击和普通攻击难以拉开体感差异。

所以首版宁可少做粒子，也要保住这一下停顿。

### 7.4 回合间隔建议

每次对撞结束后的空隙建议保留 `60ms 到 100ms`。

推荐默认值：

```text
roundGapMs = 80
```

原因：

1. 给玩家眼睛一个“结算完成”的喘息点。
2. 给血条和飘字留一点收尾空间。
3. 避免 10 回合连续冲撞看起来像机械连播。

### 7.5 不同局势的节奏微调

可以在同一套结构下做轻微变速：

1. 普通命中
`标准节奏`

2. 暴击
`撞击停顿 +20ms`

3. 五行克制
`在撞击后插入 80ms 的闪光强调`

4. 闪避
`省略血条扣减动画，整体快 120ms`

5. 燃血回合
`在回合开始前插入 180ms 的燃血提示`

核心原则是：

`结构不变，只做短促的局部增减速`

## 8. 燃血模式节奏

### 8.1 燃血模式目标

燃血模式是整场战斗的第二个高潮，不应该只是多扣一段血。

玩家必须立刻感知到：

1. 战斗进入加时。
2. 压力上来了。
3. 这一回合比普通回合更危险。

### 8.2 进入燃血时的推荐演出

进入燃血时插入一个短 notice：

```text
时长: 450ms 到 700ms
标题: 燃血模式
副文案: 8% 起，每回合 +2% 最大生命
```

视觉建议：

1. 屏幕边缘轻微红闪。
2. 棋盘整体压暗 10% 到 15%。
3. 中轴或底座增加红色脉冲。

### 8.3 燃血回合结构

燃血回合建议分成 5 拍：

1. `燃血提示`
2. `燃血扣血`
3. `死亡检查`
4. `若双方存活，再执行普通对撞`
5. `回位/下一回合`

推荐时长：

```text
燃血提示: 180ms
燃血飘字: 180ms
血条扣减: 240ms
死亡停顿: 120ms
若继续互撞，沿用普通回合节奏
```

## 9. 播放器状态机要求

### 9.1 AI 实现时必须遵守

canvas 播放器不要直接依赖 React 组件生命周期来表达所有动画。

建议显式定义播放状态机：

```text
enter
round_ready
round_dash
round_impact
round_damage
round_return
notice
blood_damage
result
```

### 9.2 每帧渲染输入

每一帧渲染器只关心这些数据：

1. 当前回合数
2. 当前 phase
3. 双方单位位置
4. 双方单位缩放
5. 双方血量显示值
6. 当前飘字列表
7. 当前 notice
8. 当前 impact tone
9. 是否进入燃血
10. 是否显示结果

### 9.3 推荐播放方式

推荐由统一 `ticker` 或 `requestAnimationFrame` 驱动：

1. 根据当前 step 的开始时间和结束时间算进度。
2. 根据进度插值位置、缩放、透明度、血量显示值。
3. 在 step 切换点生成一次性特效和飘字。

不建议只靠很多 `setTimeout` 拼完整个 canvas 动画。

## 10. 数据结构约束

### 10.1 replay 仍然是唯一真相

canvas 播放器的输入仍然应该是 replay。

播放器不得：

1. 在前端重新随机。
2. 在前端重新计算伤害。
3. 根据视觉效果修改最终血量。

### 10.2 需要的最小 step 语义

AI 实现 canvas 版时，至少要能消费以下 step：

1. `enter`
2. `notice`
3. `clash`
4. `floatingText`
5. `hpChange`
6. `return`
7. `result`

如果后续要细化，可以把 `clash` 拆成：

1. `ready`
2. `dash`
3. `impact`
4. `settle`
5. `return`

但首版不强制要求在共享协议里立刻拆这么细，也可以先在播放器内部二次拆分。

## 11. canvas2D 首版开发顺序

### 11.1 阶段 1：只做对撞模拟

目标：

先不接完整 UI，只做上下两个占位块在棋盘中线对撞。

验收：

1. 能播 10 回合。
2. 有前摇、冲刺、撞击停顿、回位。
3. 回合间隔可配置。
4. 不依赖复杂素材。

这一阶段只验证：

`节奏是否对`

### 11.2 阶段 2：接血条和飘字

目标：

在对撞模拟上加入：

1. 双方血条
2. 扣血数字
3. miss
4. crit
5. elemental
6. blood

验收：

1. 玩家能看懂每回合谁掉血。
2. 暴击和普通伤害视觉差异明显。
3. 燃血扣血和普通伤害不会混淆。

这一阶段只验证：

`可读性是否够`

### 11.3 阶段 3：接卡牌化单位表现

目标：

把占位块升级成：

1. 卡框
2. 立绘
3. 稀有度边框
4. 元素标识
5. 底座光圈

验收：

1. 不动的时候也像“单位”，不是普通 UI 卡片。
2. 上下对峙时有棋盘压迫感。
3. 对撞时仍然保持清晰，不花。

这一阶段只验证：

`气质是否立住`

### 11.4 阶段 4：接 notice 和燃血模式

目标：

补充：

1. 进入燃血提示
2. 红边脉冲
3. 棋盘压暗
4. 燃血飘字

验收：

1. 玩家一眼知道战斗进入加时。
2. 燃血回合有更紧张的压力感。
3. 不需要大动画也能明显提气氛。

### 11.5 阶段 5：接完整 replay

目标：

让 canvas renderer 能直接消费真实 replay。

验收：

1. 与 HTML 版胜负一致。
2. 与 HTML 版血量变化一致。
3. 与 HTML 版燃血触发一致。
4. 能作为微信小游戏真实战斗回放雏形。

## 12. AI 实现约束

### 12.1 代码结构要求

AI 在实现 canvas2D 战斗时，建议拆为以下模块：

1. `battlePlaybackController`
说明：推进 replay step，产出当前播放状态。

2. `battleSceneModel`
说明：维护单位位置、缩放、血量显示值、飘字、notice。

3. `battleRenderer`
说明：真正调用 canvas2D API 绘制。

4. `battleEffects`
说明：管理撞击闪光、震屏、红边、粒子等短期效果。

5. `battleAssets`
说明：加载卡框、底图、立绘、图标。

### 12.2 AI 不应擅自扩展的内容

1. 不新增复杂战斗规则。
2. 不新增新的胜负判定。
3. 不新增整套技能系统。
4. 不把 UI 面板做成全屏复杂界面。
5. 不为了“更炫”拉长单回合时间。

### 12.3 AI 优先级判断规则

当效果和性能冲突时，优先级如下：

1. 回合节奏正确
2. 血量和结果可读
3. 中轴碰撞有重量感
4. 画面气质统一
5. 额外粒子和细节特效

## 14. 首版验收标准

canvas2D 战斗首版完成时，应满足：

1. 不看说明也能看出上下双方在对撞。
2. 普通 10 回合战斗不会让人觉得拖。
3. 第一次进入燃血时能感到紧张。
4. 暴击、克制、燃血三类反馈能明显区分。
5. 卡牌式单位和棋盘台面能传达出你这款游戏的气质。
6. 不依赖大量角色动画，仍然能做出“撞得有分量”的体感。

## 15. 当前结论

当前最合理的开发方式，不是直接把整套 HTML 战斗页面搬到微信小游戏，而是：

1. 先做一个纯 canvas2D 对撞模拟器。
2. 先把 10 回合节奏和回合间隔调对。
3. 再把血条、飘字、notice、燃血补进去。
4. 再逐步替换为卡牌化灵宠表现。
5. 最后再接入真实 replay 和小游戏页面。

一句话总结：

`先验证节奏，再验证可读性，再验证气质，最后才是完整接入。`
