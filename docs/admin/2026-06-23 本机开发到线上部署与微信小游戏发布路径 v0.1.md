# 2026-06-23 本机开发到线上部署与微信小游戏发布路径 v0.1

## 一、文档目标

本文用于整理 Trinity War 从当前“本机开发 + 本机测试”状态，走到后期“线上服务器 + 线上后端 + 网络数据库 + 微信小游戏发布”的推荐路径。

重点回答四个问题：

1. 当前架构是否在按正式线上模式开发。
2. 从本机到线上，整体复杂度大概如何。
3. 最推荐的分阶段推进顺序是什么。
4. 每个阶段需要补哪些关键事项，避免后期返工。

一句话总结：

> 现在的后端底座已经接近正式线上形态，客户端当前仍是 Web 验证层；最稳妥的路线是先把 HTML demo、前后端、后台和测试收口，再切测试环境、线上环境和微信小游戏客户端壳层。

## 二、当前项目状态判断

### 1. 当前后端状态

当前后端已具备正式线上底座特征：

- `NestJS` 单体服务。
- `Prisma + PostgreSQL`。
- `Redis`。
- `BullMQ worker`。
- `.env` 环境变量。
- `Dockerfile`。
- `Swagger` 接口文档。
- 独立管理后台。
- 已存在 `WECHAT` / `DEV_FAKE` 认证提供者口径。

结论：

- 后端不是只能在本机跑的纯 demo 结构。
- 后端整体是在按“未来可部署线上服务”的模式开发。

### 2. 当前客户端状态

当前客户端是：

- `React + Vite`。
- Web 验证客户端。
- 用于先跑通玩法、接口、状态、后台和测试闭环。

结论：

- 当前客户端不是最终微信小游戏正式客户端。
- 当前客户端是为正式产品提供规则验证、交互验证和联调验证的 Web 验证层。

### 3. 当前整体判断

整体状态可概括为：

- 后端：已接近正式线上架构。
- 后台：已是正式运营/排障方向。
- 客户端：仍处在 Web 验证阶段。
- 项目整体路线是正确的，不需要推翻重来。

## 三、复杂度判断

### 1. 线上部署本身

复杂度：中等。

它不是最难的部分，但会涉及：

- Linux 服务器。
- 环境变量管理。
- Docker / 容器部署。
- Nginx 反向代理。
- HTTPS。
- 应用日志。
- 数据库迁移。
- worker 常驻任务。

结论：

- 不算特别复杂。
- 但事项较多，必须按顺序推进。

### 2. 线上数据库与缓存

复杂度：中等。

主要工作：

- PostgreSQL 上云。
- Redis 上云。
- 环境切换。
- 数据库迁移执行。
- 备份与恢复策略。

建议：

- 首发优先使用托管 PostgreSQL。
- 首发优先使用托管 Redis。
- 先少维护基础设施，把精力放在应用层。

### 3. 微信小游戏发布

复杂度：中等偏麻烦。

难点不在于玩法代码，而在于：

- `AppID`。
- 微信开发者平台配置。
- 开发版 / 体验版 / 正式版流程。
- 合法域名。
- HTTPS。
- 登录与 openid。
- 包体、资源和小游戏环境适配。
- 提审资料、截图、简介、类目、资质。

结论：

- 不属于“写不出来”的难度。
- 但属于“事项碎、容易漏”的流程型工作。

### 4. 综合判断

若只看整体难度，可粗略理解为：

- 不是一晚部署完毕的轻任务。
- 也不是需要推翻现有架构的大工程。
- 只要按阶段推进，完全是可落地路线。

## 四、最推荐的推进顺序

### 阶段 1：继续把本机 HTML demo 收口到 90%

当前最优先事项：

- 把玩法规则收口。
- 把前后端接口口径统一。
- 把后台功能做到能运营、能排障、能补发、能查日志。
- 把关键测试全部跑通。

这个阶段的目标不是画面，而是：

1. 能玩。
2. 能测。
3. 能调。
4. 能查。
5. 能稳定复现问题。

### 阶段 2：建立测试环境

在正式上线上线前，必须有一套独立测试环境。

职责：

- 联调真实后端。
- 联调真实数据库。
- 验证 Redis / worker 异步任务。
- 验证环境变量切换。
- 验证数据库迁移脚本。
- 验证后台是否能对测试服正常操作。

测试环境不要求高并发，但必须稳定、独立、可重复部署。

### 阶段 3：建立线上正式环境

推荐最小结构：

- 1 台 Linux 云服务器。
- Nginx。
- Game Server API 容器。
- BullMQ worker 容器。
- Admin 容器。
- PostgreSQL（优先托管）。
- Redis（优先托管）。

这个阶段要完成：

- 正式域名。
- HTTPS 证书。
- 线上环境变量。
- 正式数据库连接。
- 日志路径和错误监控。

### 阶段 4：小游戏客户端壳层接入

建议在本机 demo、测试环境和正式后端稳定后，再开始正式小游戏客户端壳层接入。

原因：

- 如果现在规则和接口还在大改，太早切小游戏壳层会放大返工成本。
- 先稳住玩法，再切容器和表现层，整体效率更高。

这一阶段重点不是重做玩法，而是：

- 接小游戏登录。
- 接小游戏网络环境。
- 接小游戏资源与页面壳。
- 跑开发版 / 体验版。

### 阶段 5：小游戏提审与正式上线

当前面四个阶段稳定后，再处理：

- 小游戏平台配置。
- 提审资料。
- 截图、图标、简介。
- 类目、资质、审核要求。
- 发布节奏：开发版 -> 体验版 -> 提审 -> 正式版。

## 五、推荐的环境划分

至少保留三套环境：

### 1. 本地开发环境

作用：

- 快速开发。
- 本机调试。
- 本机联调。
- 本机验证单个功能。

特点：

- 可以使用本地数据库和本地 Redis。
- 可以保留 dev 登录和测试开关。

### 2. 测试环境

作用：

- 多端联调。
- 测试小游戏登录链路。
- 验证数据库迁移。
- 验证异步任务。
- 验证补偿、赛季切换、通知附件、后台操作等高风险流程。

特点：

- 独立域名或子域名。
- 独立数据库。
- 独立 Redis。
- 独立后台入口。

### 3. 正式线上环境

作用：

- 面向真实玩家。
- 跑正式数据。
- 承接小范围灰度和正式提审版本。

特点：

- 禁止 dev 假登录。
- 禁止危险测试接口暴露。
- 必须有日志、监控和备份。

## 六、当前项目最适合的最小线上部署方案

推荐结构：

- 应用服务器：1 台 Linux 云服务器。
- Nginx：反向代理与 HTTPS。
- Game Server：NestJS API 容器。
- Worker：BullMQ worker 容器。
- Admin：后台容器。
- PostgreSQL：托管优先。
- Redis：托管优先。

这样做的好处：

- 运维成本低。
- 上手快。
- 足够支撑首发验证。
- 后续如果规模上来，再逐步拆基础设施也不晚。

## 七、上线前必须确认的事项

### 1. 后端侧

- 所有关键写接口以后端为准。
- 幂等逻辑完整。
- 重复点击不会重复扣资源或重复领奖。
- worker 任务可重试、可排查。
- 数据库迁移可重复执行。
- 关键日志完整。

### 2. 数据层

- PostgreSQL 连接串切到线上。
- Redis 连接串切到线上。
- 首次初始化脚本明确。
- 备份策略明确。
- 恢复策略明确。

### 3. 后台侧

- 能查玩家状态。
- 能查资源流水。
- 能查战报。
- 能查任务和赛季奖励状态。
- 能补发通知附件或补偿。
- 危险操作有二次确认和审计。

### 4. 客户端侧

- 真实 API 地址可切换。
- 错误提示明确。
- 本地 mock 已退场或隔离。
- 关键页面都能在真实后端下跑通。

### 5. 微信小游戏侧

- AppID 准备好。
- 开发者账号准备好。
- 合法域名配置好。
- HTTPS 可访问。
- 登录链路可跑通。
- 开发版 / 体验版 / 提审流程验证过一轮。

## 八、当前最可能的风险点

### 1. 规则还没完全冻结就开始切小游戏壳层

风险：

- 一边改后端规则，一边改小游戏壳层，返工会放大。

建议：

- 先把 HTML demo、接口和后台收口。

### 2. 本地 mock 退场不彻底

风险：

- 本地看起来成功，真实后端其实失败。

建议：

- 在转小游戏前尽量让关键写链路完全走真实后端。

### 3. 异步任务上线后暴露问题

风险：

- 本机单流程没问题，线上 worker、分红、保护期、赛季切换开始暴露幂等和并发问题。

建议：

- 测试环境必须先验证这些流程。

### 4. 后台排障能力不足

风险：

- 上线后出了问题，只能查数据库或改代码，不够快。

建议：

- 在正式上线前把后台的查询、补发、审计能力做扎实。

### 5. 微信小游戏流程型事项被低估

风险：

- 代码没问题，但域名、登录、提审、类目、资料准备卡住。

建议：

- 在技术完成度接近 90% 时，就开始平行准备小游戏平台相关事项。

## 九、当前项目是否适合这条路线

结论：

- 适合。
- 而且当前项目结构本身就在往这条路线走。

理由：

- 服务端已是正式线上底座方向。
- 管理后台已独立。
- 共享类型已拆出。
- 环境变量、Dockerfile、Prisma、Redis、BullMQ 都已经在位。
- 当前 Web 客户端只是验证壳，不会阻碍后续切小游戏客户端。

## 十、推荐的实际执行顺序

建议按下面的顺序推进：

1. HTML demo 收口到 90%。
2. 前后端真实接口收口。
3. 后台功能收口。
4. 关键验证脚本与人工测试跑通。
5. 搭测试环境。
6. 跑一轮测试环境联调。
7. 搭正式线上环境。
8. 再开始小游戏客户端壳层接入。
9. 跑开发版 / 体验版。
10. 补小游戏平台配置与提审资料。
11. 小范围灰度。
12. 正式上线。

## 十一、最终结论

- 现在的后端架构已经在按正式线上模式开发。
- 当前前端仍是 Web 验证层，这是合理且正确的阶段性选择。
- 从本机开发走到线上部署和微信小游戏发布，不是特别复杂，但事项很多，顺序必须对。
- 最稳的路线不是现在就急着切小游戏客户端，而是先把 HTML demo、前后端、后台、测试做扎实，再进入部署与发布阶段。

一句话总结：

> 先把产品和服务底座做稳，再把它搬上云、搬进微信，这样后面的部署、提审和小游戏发布才会真正顺手。
