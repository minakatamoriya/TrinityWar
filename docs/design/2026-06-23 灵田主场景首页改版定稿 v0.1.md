# 2026-06-23 灵田主场景首页改版定稿 v0.1

## 一、定稿结论

本次首页改版不再保留独立的卡片式首页。

新的默认进入页为：

**灵田主场景。**

灵田主场景同时承担两类职责：

1. 灵田玩法本身：插卡、等待、成熟、收取。
2. 原首页职责：展示当前最值得处理的机会，承接全局功能入口。

也就是说：

```text
旧结构：进入游戏 -> 首页 -> 农场/灵宠/探索/社交/阵营
新结构：进入游戏 -> 灵田主场景 -> 灵宠/战斗/社交/阵营/运营入口
```

新首页不是 QQ 农场式大背景，也不是卡片信息流，而是**卡牌桌面式灵田**。

## 二、设计目标

### 1. 微信小游戏优先

页面必须适配微信小游戏的使用习惯：

1. 玩家碎片时间进入，第一眼要知道有没有收益。
2. 操作必须直接，少读说明。
3. 右上必须避开微信官方胶囊菜单。
4. 不依赖复杂 hover 或长文本。
5. 常用功能入口要稳定。

### 2. 玩法优先

默认页应优先呈现核心循环：

1. 种田。
2. 收菜。
3. 养宠。
4. 战斗。
5. 社交助力。
6. 阵营贡献。

### 3. 低耦合

灵田主场景可以显示其他系统的轻提示，但不能直接承载其他系统的完整逻辑。

原则：

1. 灵田页只消费各模块提供的摘要状态。
2. 灵田页不直接计算复杂社交、战斗、赛季规则。
3. 点击入口后进入对应模块处理真实业务。
4. 每个模块维护自己的详情页、图鉴、规则说明和操作流。

## 三、非目标

本次改版不做以下事情：

1. 不做 QQ 农场式完整田园背景。
2. 不做建筑主城。
3. 不做大段卡片说明。
4. 不做独立“推荐行动”列表。
5. 不在默认页展示普通社交动态。
6. 不把图鉴、赛季规则、背包详情铺在默认页。
7. 不重排新手教程具体步骤。
8. 不重构现有 dialog 对话系统。

新手教程可以在页面改版后重新编排。

## 四、对话系统约束

现有 dialog 对话系统本次先维持现状，不做架构调整。

当前 dialog 属于较重的底部升起式对话，行为接近带 mask 的 modal。它会阻塞底层界面点击，需要玩家先点掉对话，才能继续操作页面。

这套机制本次不改，但后续不建议继续把日常轻提示都接入重型 dialog。

### 1. 狐狸对话

狐狸对话继续保留。

在旧首页中，狐狸承担首页引导和轻提示角色。改成灵田主场景后，狐狸仍然可以作为默认引导角色，用于承接：

1. 当前最值提示。
2. 新手期提示。
3. 系统轻提醒。

本次不改变狐狸对话的基础机制。

### 2. 土地对话

土地对话资源保留，但灵田主场景首版先不自动出现土地对话。

原因：

1. 旧结构中，首页是狐狸对话，进入灵田后是土地对话。
2. 新结构中，默认进入页就是灵田主场景。
3. 如果继续自动出现土地对话，会和狐狸默认引导职责冲突。
4. 土地后续如何出现，应等整体对话系统规划时再决定。

因此首版规则为：

1. 保留土地角色与已有对话资源。
2. 不删除土地相关代码和素材。
3. 灵田主场景默认不触发土地对话。
4. 后续再决定土地是否作为灵田教学、灵植图鉴、特殊事件或地块说明角色出现。

### 3. 后续轻量对话方向

后续对话系统可以考虑拆成两层：

1. 重型剧情/教学对话。
2. 轻量闲聊/提示气泡。

重型剧情/教学对话可以继续使用当前底部 dialog，用于必须停下来确认的内容。

轻量闲聊/提示气泡应是非阻塞的，不带强 mask，不阻止玩家继续点击主界面。表现形式可以是：

1. 角色头像从屏幕边缘冒出。
2. 一到两句短气泡。
3. 自动淡出或允许玩家忽略。
4. 点击气泡可以展开详情或跳转，但不强制。

轻量气泡适合承接：

1. 灵田卡成熟提醒。
2. 狐狸闲聊。
3. 轻度氛围文案。
4. 弱引导提示。
5. 不需要玩家马上处理的系统提醒。

后续规划时，应避免把这类轻提示做成必须点掉的 modal。

## 五、最终信息架构

### 1. 一级底部入口

底部保留五个一级入口：

```text
灵田 | 灵宠 | 战斗 | 社交 | 阵营
```

说明：

| 入口 | 职责 |
| --- | --- |
| 灵田 | 默认页，插卡种植、成熟收取、灵植选择 |
| 灵宠 | 主宠养成、升级、突破、洗点、灵宠图鉴 |
| 战斗 | 目标、复仇、战报、情报、出战 |
| 社交 | 好友、助力、邀请、关系、社交动态 |
| 阵营 | 贡献、俸禄、排行、阵营信息 |

`探索`改名为`战斗`。

原因：

1. 含义更直接。
2. 更符合对战、复仇、战报心智。
3. 微信小游戏底部入口需要短而明确。

### 2. 全局轻入口

以下功能不进入底部栏：

| 功能 | 放置位置 | 原因 |
| --- | --- | --- |
| 签到 | 右侧运营入口 | 回流福利，不是主循环一级页 |
| 活动/福利 | 右侧运营入口 | 运营入口，允许随活动变化 |
| 赛季奖励 | 右侧赛季入口 | 赛季是独立运营/周期入口，不归入阵营 |
| 赛季说明 | 独立赛季页内 | 规则信息不占默认页 |
| 背包 | 顶部资源旁 | 与资源和道具相关，属于全局工具 |
| 玩家头像 | 左上 | 进入个人资料 |
| 奖章/称号 | 玩家资料页内，首页只展示当前佩戴项 | 避免首页信息过载 |
| 设置 | 头像页或左上次级入口 | 不占主循环入口 |

### 3. 图鉴位置

图鉴不做全局常驻入口。

| 图鉴 | 放置位置 |
| --- | --- |
| 灵植图鉴 | 灵田模块内 |
| 灵宠图鉴 | 灵宠模块内 |
| 奖章图鉴 | 玩家资料或赛季页内 |

理由：

1. 玩家选择灵植卡时才最需要灵植图鉴。
2. 玩家养宠或合成时才最需要灵宠图鉴。
3. 图鉴不是默认页的高频动作。

## 六、灵田主场景布局

### 1. 页面结构

推荐结构：

```text
┌──────────────────────────────┐
│ 头像/奖章  资源条/背包     [微信胶囊留空] │
│                              │
│ 当前目标/轻提示              右侧运营入口 │
│                              │
│      四个灵田卡槽             签到/福利   │
│   [空] [成长] [成熟] [空]      活动/赛季   │
│                              │
│ 社交/战斗待处理轻提示         │
├──────────────────────────────┤
│ 灵田   灵宠   战斗   社交   阵营 │
└──────────────────────────────┘
```

### 2. 顶部区域

顶部左侧：

1. 玩家头像。
2. 当前佩戴奖章或称号。

顶部中间：

1. 金币。
2. 灵宠核心资源，例如兽魂、灵根或天机符。
3. 今日贡献或距下一档。
4. 背包按钮。

顶部右侧：

1. 不放业务按钮。
2. 避开微信官方胶囊菜单。

实现时在微信小游戏环境中应读取官方菜单安全区域。

### 3. 中央灵田卡槽

灵田是四个插卡槽。

每个卡槽是独立组件，状态包括：

| 状态 | 视觉 | 主操作 |
| --- | --- | --- |
| 空槽 | 空卡框 | 插入灵植卡 |
| 生长中 | 插入灵植卡，显示倒计时 | 查看或等待 |
| 成熟 | 卡槽冒光，卡面高亮 | 收取 |
| 枯萎/异常 | 卡面变暗或裂纹 | 回收/处理 |
| 锁定 | 灰色卡槽 | 查看解锁条件 |

成熟提示应优先通过卡槽视觉表达，而不是额外写一张“成熟田地”说明卡。

### 4. 当前目标/轻提示

默认页只允许出现一条当前最值提示。

示例：

1. 2 张灵植卡已成熟。
2. 主战灵宠可突破。
3. 1 个目标可复仇。
4. 2 位好友可助力。
5. 今日贡献距下一档还差 18。

这条提示可以来自灵田主场景聚合层，但点击后必须跳转到对应模块。

不允许出现多条任务列表。

### 5. 右侧运营入口

右侧入口从微信胶囊菜单下方开始排列。

建议初期保留：

1. 签到。
2. 福利。
3. 活动。
4. 赛季。

这些入口应该由运营入口组件管理，不与灵田卡槽逻辑耦合。

### 6. 底部导航

底部导航固定五项：

```text
灵田 | 灵宠 | 战斗 | 社交 | 阵营
```

底部入口可以显示红点或数字，但必须对应真实可处理事项。

允许：

1. 灵田：可收 2。
2. 灵宠：可突破。
3. 战斗：可复仇 1。
4. 社交：申请 1 / 可助力 3。
5. 阵营：可领取。

不建议：

1. 社交：未读 6。
2. 阵营：排行变化 3。
3. 战斗：动态 5。

## 七、社交入口规则

社交仍然是底部一级入口。

原因：

1. 微信小游戏天然依赖好友、邀请、助力。
2. 当前玩法已有好友助力、好友田地、关系和邀请。
3. 社交如果藏在二级页，会削弱助力回流。

### 1. 默认页允许展示的社交提示

只能展示可处理事项：

1. 好友申请。
2. 可助力好友田地。
3. 组队邀请。
4. 可复仇关系。
5. 邀请奖励可领取。

### 2. 默认页不展示的社交信息

不展示普通动态：

1. 某好友帮你回浇。
2. 某玩家关注了你。
3. 普通动态未读数。
4. 好友总数。
5. 关注总数。

这些信息留在社交页。

### 3. 社交页内部结构

社交页可以继续保留：

1. 好友。
2. 关系/关注。
3. 动态。

但首页和底部红点只消费可处理摘要，不直接读取普通动态未读。

## 八、模块边界与低耦合要求

### 1. 灵田模块

职责：

1. 渲染四个灵田卡槽。
2. 处理插卡、收取、查看灵植。
3. 提供灵植图鉴入口。
4. 输出灵田摘要给灵田主场景聚合层。

不负责：

1. 灵宠养成逻辑。
2. 战斗目标选择。
3. 社交助力规则计算。
4. 赛季奖励规则。

建议摘要：

```ts
interface FarmModuleSummary {
  collectableCount: number;
  growingCount: number;
  emptyCount: number;
  lockedCount: number;
  nextReadySeconds: number | null;
  hasWitheredField: boolean;
}
```

### 2. 灵宠模块

职责：

1. 主宠状态。
2. 升级、突破、洗点、合成。
3. 灵宠图鉴。
4. 输出灵宠摘要给灵田主场景聚合层。

不负责：

1. 灵田卡槽渲染。
2. 战斗目标列表。
3. 社交关系。

建议摘要：

```ts
interface SpiritModuleSummary {
  mainSpiritName: string | null;
  mainSpiritLevel: number | null;
  canBreakthrough: boolean;
  readyComposeCount: number;
  hasTraitRollOpportunity: boolean;
}
```

### 3. 战斗模块

职责：

1. 目标列表。
2. 复仇。
3. 战报。
4. 情报。
5. 出战确认。

不负责：

1. 首页机会排序。
2. 社交动态展示。
3. 阵营贡献规则解释。

建议摘要：

```ts
interface BattleModuleSummary {
  revengeableCount: number;
  highValueTargetCount: number;
  unreadBattleReportCount: number;
  remainingAttempts: number | null;
}
```

首页可以使用 `revengeableCount`，但不应展示普通未读战报，除非战报对应可复仇或可领取奖励。

### 4. 社交模块

职责：

1. 好友。
2. 助力。
3. 邀请。
4. 关系。
5. 社交动态。

不负责：

1. 首页主机会排序。
2. 灵田卡槽状态。
3. 战斗结算。

建议摘要：

```ts
interface SocialModuleSummary {
  pendingFriendRequests: number;
  assistableFriendCount: number;
  pendingTeamInvites: number;
  claimableInviteRewards: number;
}
```

注意：

`feedUnread` 不进入首页摘要。

### 5. 阵营模块

职责：

1. 今日贡献。
2. 俸禄。
3. 阵营排行。
4. 阵营信息。

不负责：

1. 灵田插卡。
2. 灵宠养成。
3. 社交动态。
4. 赛季说明和奖励。

建议摘要：

```ts
interface FactionModuleSummary {
  todayContribution: number;
  nextContributionTarget: number;
  contributionRemaining: number;
  stipendClaimable: boolean;
}
```

### 6. 赛季模块

赛季不归入阵营模块。

职责：

1. 赛季说明。
2. 赛季倒计时。
3. 赛季奖励。
4. 赛季结算与历史。

放置位置：

1. 右侧运营入口。
2. 独立赛季页或弹层。
3. 默认页只允许显示可领取或即将结束等强提醒。

建议摘要：

```ts
interface SeasonModuleSummary {
  seasonRewardClaimable: boolean;
  seasonEndingSoon: boolean;
  remainingSeconds: number | null;
}
```

## 九、灵田主场景聚合层

灵田主场景需要一个轻量聚合层，用于把各模块摘要转成：

1. 当前最值提示。
2. 底部入口红点。
3. 顶部轻进度。

聚合层不处理业务写入。

建议结构：

```ts
interface FarmMainSceneSummary {
  progress: {
    todayContribution: number;
    contributionRemaining: number;
    nextReadySeconds: number | null;
  };
  primaryHint: FarmPrimaryHint | null;
  navBadges: FarmMainNavBadge[];
  operationBadges: FarmMainOperationBadge[];
}

interface FarmPrimaryHint {
  source: 'farm' | 'spirit' | 'battle' | 'social' | 'faction';
  title: string;
  description?: string;
  actionLabel: string;
  target: 'farm' | 'spirit' | 'battle' | 'social' | 'faction';
  priority: number;
}

interface FarmMainNavBadge {
  target: 'farm' | 'spirit' | 'battle' | 'social' | 'faction';
  label: string;
  count?: number;
  tone: 'normal' | 'attention' | 'urgent';
}

interface FarmMainOperationBadge {
  key: 'sign-in' | 'welfare' | 'event' | 'season';
  label: string;
  count?: number;
  tone: 'normal' | 'attention' | 'urgent';
}
```

### 当前最值提示优先级

建议优先级：

1. 即将枯萎灵植。
2. 已成熟灵植。
3. 俸禄已经可领。
4. 主战灵宠可突破。
5. 可复仇目标。
6. 好友申请或组队邀请。
7. 可助力好友田地。
8. 高价值战斗目标。
9. 距下一档贡献很近。
10. 下一张灵植卡即将成熟。
11. 赛季奖励可领取或赛季即将结束。

如果没有重要机会，可以不展示提示，让灵田卡槽本身承担状态表达。

## 十、建议前端改造步骤

### 第一步：场景概念收口

本次不要为了省事继续让 `home` 承担灵田职责。

原因：

1. 当前只是 HTML demo，后续还要迁移到 canvas 微信小游戏。
2. 如果新 UI 叫灵田、内部却继续叫 home，后续 AI 和开发者都容易误解。
3. 概念债会随着教程、埋点、红点、对话、canvas 场景迁移一起放大。

因此新结构应直接按业务概念命名：

```ts
type GameMainSceneKey = 'farm' | 'spirit' | 'battle' | 'social' | 'faction';
```

前端新代码、新组件、新 CSS、新文档都应使用：

```text
farm / spirit / battle / social / faction
```

不要继续新增：

```text
home / raid / report
```

旧 key 的处理方式：

1. `home` 只作为旧入口兼容，进入时归一化到 `farm`。
2. `raid` 作为旧入口兼容，进入时归一化到 `spirit`。
3. `report` 作为旧入口兼容，进入时归一化到 `battle`。
4. 新底部导航不再出现 `home`。
5. 新页面组件不再命名为 `HomeScene`。
6. 后端 `HomeSummaryResponse` 可暂时保留为数据摘要，不再代表页面。

建议命名：

| 旧命名 | 新命名 |
| --- | --- |
| HomeScene | FarmMainScene |
| home scene | farm scene |
| raid scene | spirit scene |
| report scene | battle scene |
| home-* CSS | farm-main-* CSS |

### 第二步：路由与导航调整

1. 默认进入场景改为 `farm`。
2. 底部固定五项：`farm`、`spirit`、`battle`、`social`、`faction`。
3. 底部展示文案为：灵田、灵宠、战斗、社交、阵营。
4. `normalizeScene` 保留旧 key 兼容，但所有旧 key 都应归一化到新 key。
5. 旧链接或旧 action 进入 `home` 时，应自动落到 `farm`。

### 第三步：灵田主场景替换卡片首页

1. 删除独立首页中的大段说明卡。
2. 将默认页改为灵田卡槽桌面。
3. 复用已有农场字段数据渲染四个插卡槽。
4. 成熟、空槽、成长中、锁定状态通过卡槽视觉表达。

## 十一、验收标准

### 1. 首屏体验

1. 手机首屏内能看到四个灵田卡槽。
2. 不需要阅读大段文字也知道哪里可点。
3. 右上微信胶囊区域没有被业务按钮遮挡。
4. 底部五个入口稳定可见。

### 2. 信息密度

1. 默认页没有推荐行动列表。
2. 默认页没有普通社交动态。
3. 默认页没有图鉴、赛季规则、奖章列表等详情内容。
4. 默认页最多只有一条当前提示。

### 3. 模块耦合

1. 灵田主场景不直接写社交、战斗、灵宠业务逻辑。
2. 首页提示来自摘要或 view model。
3. 图鉴入口属于对应模块。
4. 红点和徽标都能通过模块摘要独立计算。

### 4. 命名

1. 默认页文案为灵田，不再叫首页。
2. 探索入口文案改为战斗。
3. 社交入口继续保留为一级底部入口。

### 5. 对话

1. 现有 dialog 对话系统没有被重构。
2. 狐狸对话仍可作为默认引导角色出现。
3. 土地对话资源保留，但灵田主场景首版不自动触发土地对话。
4. 当前重型 dialog 不继续扩展为日常轻提示的主要形式。
5. 后续可以规划非阻塞角色气泡，用于闲聊和轻提醒。

## 十二、给 AI 实现者的重点提醒

1. 不要把旧首页卡片换皮成新卡片。
2. 中央核心必须是四个灵田插卡槽。
3. 首页职责通过卡槽状态、少量提示和入口红点表达。
4. 社交保留底部一级入口，但首页只展示可处理社交事项。
5. 右上要留给微信官方菜单。
6. 灵植图鉴在灵田模块，灵宠图鉴在灵宠模块。
7. 背包、签到、赛季、奖章不要挤进底部主导航。
8. 新手引导暂不在本次定稿中重排，只预留主场景点位。
9. 不要改动 dialog 系统架构。
10. 保留狐狸对话，土地对话资源保留但首版不自动触发。
11. 后续轻提示不要默认做成带 mask 的阻塞式 modal，对话系统可以另行规划非阻塞气泡层。
