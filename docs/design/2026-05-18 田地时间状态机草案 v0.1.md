# 2026-05-18 田地时间状态机草案 v0.1

## 一、文档目标

这份文档只解决四个问题：

1. 每块田需要记录哪些时间字段。
2. 作物从播种到收取的状态如何流转。
3. 前端倒计时如何显示，后端如何结算。
4. 被掠夺、收取、过熟衰减时，时间如何统一判定。

这份草案的核心原则只有一句话：

前端负责展示倒计时，后端负责记录开始时间并在关键操作时按服务器时间统一结算。

## 二、首发设计底线

首发版本建议守住下面六条：

1. 每块田不做本地时间权威判定。
2. 每块田不做每秒刷新数据库。
3. 每块田的成熟阶段由服务器时间动态计算。
4. 前端倒计时只用于展示，不决定最终收益。
5. 收取、掠夺、衰减结算时，统一以后端当前时间为准。
6. 被掠夺后扣的是当前可收收益的一部分，不是整块田直接消失。

## 三、每块田建议记录的核心字段

首发每块田不需要记录很多零散倒计时，只需要记录少量关键字段。

建议最少包含：

1. 田地 ID。
2. 玩家 ID。
3. 当前种植物类型，例如仙藤。
4. 播种开始时间。
5. 当前成长配置 ID。
6. 当前状态。
7. 当前剩余可收价值。
8. 已被掠夺扣减值。
9. 最后一次结算时间。

如果后续需要扩展，还可以增加：

1. 收取完成时间。
2. 最后一次被掠夺时间。
3. 当前保护加成快照。
4. 当前成熟曲线修正值。

## 四、建议状态定义

首发建议每块田只保留下面几种核心状态：

1. 空闲：未种植。
2. 播种中：已播种，尚未进入可提前收取阶段。
3. 成长期：已进入可提前收取阶段，但尚未达到标准成熟收益。
4. 成熟中：进入标准可收、可掠夺阶段。
5. 丰熟中：进入高收益、高暴露阶段。
6. 过熟中：收益衰减阶段。
7. 已收取：本轮已结算完成。
8. 已枯竭：本轮已结束，等待重新播种。

这里建议不要再拆太多临时状态，避免首发复杂度失控。

## 五、时间阶段的计算方式

首发推荐采用“记录开始时间 + 按当前服务器时间现算阶段”的方式。

例如一块田的成长曲线配置如下：

1. 播种后 0 到 1 小时：播种中。
2. 1 到 3 小时：成长期，可提前收取。
3. 3 到 4 小时：成熟中。
4. 4 到 6 小时：丰熟中。
5. 6 小时后：过熟中。

那么服务器每次只需要做下面这件事：

1. 读取播种开始时间。
2. 读取当前服务器时间。
3. 计算已生长时长。
4. 按成长曲线判断当前所处阶段。

如果需要知道“超出成熟多久”或“进入过熟多久”，也不建议单独存字段。

更推荐直接按下面方式动态计算：

1. 已生长时长减去成熟起点时间。
2. 得到已成熟时长。
3. 已生长时长减去丰熟起点时间。
4. 得到已丰熟时长。
5. 已生长时长减去过熟起点时间。
6. 得到已过熟时长。

## 六、收益字段建议

首发建议把一块田当前价值拆成两个理解层：

1. 当前理论总价值。
2. 当前实际可收价值。

建议规则：

1. 当前理论总价值由成长阶段决定。
2. 当前实际可收价值等于理论总价值减去已被掠夺扣减值。
3. 收取时按当前实际可收价值结算。
4. 被掠夺时只减少当前实际可收价值，不改变播种开始时间。

这样就能自然表达：

1. 一块丰熟仙藤当前理论价值 1000G。
2. 被掠夺后扣掉 200G。
3. 当前实际可收价值变为 800G。
4. 玩家后续仍可收取剩余的 800G。

## 七、状态流转建议

首发建议采用下面这条主状态流：

1. 空闲 -> 播种成功 -> 播种中。
2. 播种中 -> 到达成长期阈值 -> 成长期。
3. 成长期 -> 到达成熟阈值 -> 成熟中。
4. 成熟中 -> 到达丰熟阈值 -> 丰熟中。
5. 丰熟中 -> 超过丰熟上限 -> 过熟中。
6. 成长期、成熟中、丰熟中、过熟中 -> 玩家收取 -> 已收取。
7. 已收取 -> 结算完成 -> 已枯竭。
8. 已枯竭 -> 重新播种 -> 播种中。

被掠夺不会改变阶段主状态，只会改变当前实际可收价值。

也就是说：

1. 丰熟中被掠夺后，仍然是丰熟中。
2. 成熟中被掠夺后，仍然是成熟中。
3. 被掠夺只影响剩余收益，不重置成长计时。

## 八、前端倒计时显示建议

前端应显示倒计时，但只负责显示，不负责最终结算。

建议前端每次获取田地数据时，同时拿到：

1. 服务器当前时间。
2. 该田的播种开始时间。
3. 当前阶段阈值时间。
4. 当前剩余可收价值。

前端可以据此显示：

1. 距离成熟还剩多久。
2. 距离丰熟还剩多久。
3. 已超过成熟多久。
4. 当前可收价值是多少。
5. 该田的成长时间基于自己的播种时刻独立计算，不按整点刷新。

推荐展示方式：

1. 播种中时显示“距离成长期：00:42:15”。
2. 成长期时显示“距离成熟：01:18:10”。
3. 成熟中时显示“距离丰熟：00:26:40”。
4. 丰熟中时显示“丰熟中，已持续：00:12:35”。
5. 过熟中时显示“已过熟：00:12:35，收益衰减中”。

## 九、前后端时间对齐建议

前后端时间不需要每毫秒绝对一致，但必须做到展示近似一致、结算绝对以后端为准。

推荐做法：

1. 后端每次返回田地数据时，附带一个服务器当前时间。
2. 前端记录本地接收时刻，估算本地与服务器的时间偏移。
3. 倒计时展示时，用本地时间加偏移量模拟服务器时间流逝。
4. 玩家点击收取、掠夺或查看关键状态时，再由后端重新判定。

这样能同时满足两件事：

1. 玩家看到的倒计时基本稳定。
2. 关键收益与阶段结算不会因为本地时间误差出错。

## 十、关键操作时的后端判定顺序

### 1. 播种时

后端应：

1. 校验该田是否为空闲或可重种状态。
2. 校验金币是否足够。
3. 扣除投入。
4. 写入播种开始时间。
5. 写入成长配置和初始状态。

### 2. 收取时

后端应：

1. 读取该田当前状态。
2. 读取当前服务器时间。
3. 重新计算当前所处阶段。
4. 重新计算当前实际可收价值。
5. 执行收取并把收益直接转入金库。
6. 写入已收取状态和结算时间。

### 3. 被掠夺时

后端应：

1. 读取该田当前状态。
2. 读取当前服务器时间。
3. 判断当前是否处于可掠夺阶段。
4. 计算本次理论可掠夺量。
5. 扣减当前实际可收价值。
6. 写入已被掠夺扣减值和最后一次结算时间。

### 4. 过熟衰减时

首发不建议每秒写衰减值。

更好的做法是：

1. 读取当前服务器时间。
2. 计算已过熟时长。
3. 按衰减曲线算出当前理论总价值。
4. 再减去已被掠夺扣减值。
5. 得到当前实际可收价值。

## 十一、首发推荐规则组合

如果首发要做最小可用版本，建议直接采用下面这套：

1. 每块田记录播种开始时间、成长配置、当前状态、当前剩余可收价值。
2. 成熟、丰熟、过熟阶段全部按服务器时间动态计算。
3. 前端只显示倒计时，不决定最终结算。
4. 收取与被掠夺都以后端当前时间重新判定。
5. 被掠夺扣减当前可收价值，不重置成长时间。
6. 收取后直接折算金币进入金库，不进入额外仓储链路。

只要守住这六条，首发的田地时间系统就能同时满足三件事：

1. 倒计时体验清楚。
2. 前后端时间逻辑一致。
3. 服务器实现成本可控。
