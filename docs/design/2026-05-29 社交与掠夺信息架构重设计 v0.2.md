# 2026-05-29 社交与探索信息架构重设计 v0.2

## 1. 本文目的

这份文档用于冻结今天开始实现的首发信息架构。

旧版社交设计已经提出好友、关注、仇敌、阵营、浇水、守田、灵宠恢复和组队挑战等方向，但范围偏大。现在需要先解决更基础的问题：

1. 探索页和社交页功能边界不清。
2. 关注、仇敌、战报、动态之间存在重复。
3. 微信小游戏首发需要更短路径、更少 tab、更直接的操作。
4. 后续实现需要一个可以分阶段落地的前后端切片。

本版原则：

> 探索负责“发现目标、发起战斗、查看战报、发布通缉”；社交负责“认识谁、维护关系、互相帮忙”。

命名约定：前端显示使用“探索 / 战斗”，内部 `raid` key、接口路径和旧数据字段暂不改名。

## 2. 一级页面边界

### 2.1 探索页定位

探索页是外部目标与战斗行为页。

玩家进入探索页时，主要意图是：

1. 找目标。
2. 发起战斗。
3. 查看战报。
4. 复仇。
5. 对打不过的目标发起通缉。

探索页不承担好友经营，不承担完整关系管理，不展示通用社交动态。

### 2.2 社交页定位

社交页是关系经营页。

玩家进入社交页时，主要意图是：

1. 看今天谁和我有关。
2. 找好友互助。
3. 查看和维护关系标签。
4. 从关系对象跳转到访问、浇水、查看或战斗。

社交页不重复完整战报列表，不做独立战斗目标池，不承载通缉接单主流程。

## 3. 探索页最终结构

首发建议保留 3 个 tab。

### 3.1 战斗

用途：

1. 展示系统推荐目标。
2. 刷新目标列表。
3. 发起单人战斗。
4. 从关系标签筛选目标。

筛选建议：

1. 推荐。
2. 仇敌。
3. 关注目标。
4. 可复仇。

注意：

原“关注”tab 不再保留。关注只是目标筛选条件，不是探索页的独立页面。

### 3.2 战报

用途：

1. 查看我挑战别人。
2. 查看我被别人挑战。
3. 回放战斗。
4. 从战报发起复仇。
5. 从战报标记关注或仇敌。

战报是战斗流水，不是社交动态。

战报行建议展示：

1. 攻守双方。
2. 胜负结果。
3. 获得或损失。
4. 是否可复仇。
5. 关系快捷按钮。

### 3.3 通缉

用途：

1. 对自己打不过的大佬发起通缉。
2. 查看自己发布的通缉。
3. 查看可参与的通缉。
4. 作为后续微信助力战斗链接的站内承载入口。

首发可以先做轻：

1. 有 tab。
2. 有空态和说明。
3. 可从目标详情或战报进入“发起通缉”占位。
4. 暂不实现完整接单和奖励分账。

## 4. 社交页最终结构

首发建议保留 3 个 tab。

### 4.1 动态

动态是轻摘要，不是战报复制。

展示内容：

1. 谁帮我浇水。
2. 谁请求帮忙。
3. 谁关注了我或我关注了谁。
4. 好友近期可互动提示。
5. 被挑战、复仇等战斗事件的轻提示。

战斗事件只显示摘要，点击后跳转到探索页战报详情。

首发可以先做：

1. 系统生成的静态或真实轻摘要。
2. 站内通知、浇水、关系变化的占位事件。
3. 不做无限流，不做复杂评论。

### 4.2 好友

好友是互助主入口。

展示内容：

1. 好友列表。
2. 可浇水状态。
3. 最近活跃。
4. 访问田地。
5. 微信分享邀请入口。

首发第一优先级动作：

1. 帮忙浇水。
2. 访问好友田地。
3. 邀请或测试入口。

暂缓动作：

1. 帮忙守田。
2. 灵宠恢复助力。
3. 好友聊天。

### 4.3 关系

关系合并原“关注”和“仇敌”。

关系对象可以带多个标签：

1. 好友。
2. 关注。
3. 仇敌。
4. 同阵营。
5. 不同阵营。
6. 最近互动。

筛选建议：

1. 全部。
2. 好友。
3. 关注。
4. 仇敌。
5. 同阵营。

注意：

仇敌不做单独大 tab。仇敌是一种系统沉淀或玩家标记的关系标签，可在关系页筛选，也可在探索页作为战斗筛选。

## 5. 关系定义

### 5.1 好友

好友是高频互助关系。

阵营限制：

1. 好友只允许同阵营玩家建立。
2. 不同阵营玩家不能成为好友，不能互相浇水。
3. 不同阵营关系承接到探索页：战斗、战报、关注、仇敌。
4. 如果历史数据存在跨阵营好友，客户端不展示为有效好友，后端也不允许继续浇水。

来源：

1. 微信邀请。
2. 手动添加。
3. 双向确认。

首发可简化：

1. 暂不做复杂申请流。
2. 可以先用“测试好友”或单向添加验证 UI。
3. 后续再接微信 openid/unionid。

核心行为：

1. 浇水。
2. 访问田地。
3. 发送互助请求。
4. 删除好友。

删除规则：

1. 删除好友是单方操作。
2. A 删除 B 后，双方好友关系同时解除。
3. 删除后不自动变成仇敌，也不自动取消关注。
4. 之后如需重新成为好友，需要重新申请。

最后阶段保留：

1. 好友局内预设语句。
2. 好友局内轻表情包。
3. 只允许选择官方预设内容，不做自由聊天。
4. 当前 MVP 不实现。

微信邀请首版入口：

1. 社交页好友 tab 顶部放“邀请领好礼”按钮。
2. 好友空态不重复放邀请按钮，只保留顶部第一条主入口。
3. 文案必须明确：邀请新玩家建档并成为好友，双方可领取新友奖励。
4. 游戏登录入口保留两个模拟入口：“新玩家受邀领新友奖励”“老友回归并肩同行”。
5. 当前按钮只验证入口位置和流程感，后续接独立 `friend_invite` 类型的分享活动。

新友奖励方向：

1. 邀请者：新玩家完成建档并成为好友后发放。
2. 新玩家：完成新手流程后通过通知领取。
3. 邀请新玩家浇水和普通邀请新玩家成为好友，使用同一套新友奖励；浇水只是额外多一次即时助力表现。
4. 奖励应明显高于普通日常小奖励，作为拉新动机。
5. 建议首版奖励：邀请者金币、植物精华、灵宠精魄；新玩家金币、植物精华、基础养成材料。
6. 老玩家受邀成为好友不给拉新奖励，避免刷号套利；文案提供重逢、并肩、互相浇水的情绪价值。

受邀新玩家阵营选择：

1. 不强制玩家选择邀请人的阵营。
2. 阵营选择页必须醒目展示邀请人的昵称和阵营。
3. 与邀请人相同阵营的卡片显示推荐提示：可成为好友并领取新友奖励。
4. 不同阵营卡片也可以选择，但必须明确提示：不能与邀请人成为好友，本次邀请不建立好友关系。
5. 后端仍以同阵营为最终校验，前端提示只负责降低误选。

测试入口说明：

1. 好友邀请 URL 只标识邀请活动和邀请发起人，不负责判断对方是新玩家还是老玩家。
2. 测试阶段登录入口提供两个输入框：新玩家邀请 URL、老玩家邀请 URL。
3. 点击“新玩家接受邀请并建档”时，按新玩家流程处理，并进入阵营选择。
4. 点击“老友回归并肩同行”时，按老玩家流程处理，当前固定使用测试用户1。
5. 正式微信环境中，新老用户判断应由登录态、openid/unionid 绑定状态和建档状态共同决定。

### 5.2 关注

关注是单向标记。

用途：

1. 记住强者。
2. 记住想以后挑战的人。
3. 记住同阵营活跃成员。
4. 让探索战斗页可以筛选关注目标。

关注不等于好友，不需要对方确认。

### 5.3 仇敌

仇敌是冲突标签。

来源：

1. 对方挑战过我。
2. 对方偷过我。
3. 我从战报手动标记。
4. 多次交战后系统自动沉淀。

用途：

1. 探索页筛选仇敌。
2. 战报页快速复仇。
3. 社交关系页回看冲突对象。

仇敌可以和关注同时存在。

### 5.4 同阵营

同阵营不是手动关系，而是派生标签。

用途：

1. 推荐互助对象。
2. 好友候选。
3. 后续阵营协作活动。

## 6. 首发前后端建议

### 6.1 数据模型方向

推荐最终使用统一玩家关系表，而不是好友、关注、仇敌各自一张表。

建议模型：

```text
player_social_relation
- id
- owner_player_id
- target_player_id
- relation_type: friend | follow | enemy
- source_type: manual | raid | revenge | invite | system
- strength
- last_interacted_at
- created_at
- updated_at
```

说明：

1. 好友如果需要双向确认，可以用两条 relation 或额外状态字段。
2. 关注和仇敌都是 owner 对 target 的单向标签。
3. 同阵营不需要入表，读接口按 faction 派生。
4. 微信好友关系不直接等于游戏好友，必须先绑定玩家。

首发如果已有旧表，可以先由接口层合并输出，不急着迁移。

### 6.2 接口草案

社交读接口：

```text
GET /api/client/social/summary
GET /api/client/social/feed
GET /api/client/social/relations?filter=all|friend|follow|enemy|same-faction
```

社交写接口：

```text
POST /api/client/social/relations/follow
POST /api/client/social/relations/unfollow
POST /api/client/social/relations/mark-enemy
POST /api/client/social/relations/remove-enemy
POST /api/client/social/water-field
```

探索读接口补充：

```text
GET /api/client/raid/targets?filter=recommended|enemy|follow|revenge
GET /api/client/raid/reports?scope=all|attacker|defender
GET /api/client/raid/warrants
```

探索写接口补充：

```text
POST /api/client/raid/warrants
POST /api/client/raid/warrants/:id/join
```

首发第一阶段可以只接：

1. 社交关系列表。
2. 关注/取消关注。
3. 标记/移除仇敌。
4. 探索目标按关系筛选。

浇水可以作为第二阶段。

## 6.3 好友互助开发规则补充：浇水、采摘、亲密度

本节作为后续开发规则记录。首版目标是让好友关系每天有轻量互动，但不能让好友互助变成偷菜、战斗或强制刷收益。

### 6.3.1 好友关系历史保留

当前好友关系使用 `player_social_relation` 存储。

关键字段：

1. `playerId`：关系拥有者。
2. `targetPlayerId`：关系对象。
3. `relationType`：`FRIEND` / `FOLLOWING` / `ENEMY` / `BLOCKED`。
4. `status`：`ACTIVE` / `PENDING` / `MUTED`。
5. `sourceType`：关系来源，例如 `friend-invite`、`manual-friend-request`、`friend-deleted`。
6. `intimacy`：亲密度。
7. `lastInteractedAt`：最近互动时间。

好友关系采用双向记录：

1. A 和 B 成为好友时写入 A -> B、B -> A 两条 `FRIEND / ACTIVE`。
2. 任意一方删除好友时，不物理删除记录，而是把双方 `FRIEND` 关系改成 `MUTED`。
3. 删除后仍保留 `intimacy`、`createdAt` 和历史来源，用于判断“曾经是好友”。
4. 再次加好友时，`MUTED` 可以恢复为 `PENDING` 或 `ACTIVE`，但不重置亲密度。
5. 新人邀请奖励只看新人建档与邀请关系是否首次有效，不因删除后再次邀请老玩家而重复发奖。

防刷规则：

1. 首次成为好友可以给一次性亲密度奖励。
2. 曾经是好友后重新加回，不再发“首次好友”亲密度奖励。
3. 删除好友不清空亲密度，避免删除再加循环刷亲密度。
4. 删除好友当天，双方不应通过重新添加再次获得首次关系奖励。
5. 以后如加入“重逢奖励”，只能是很小的情绪价值奖励，不能产出稀缺资源。

### 6.3.2 亲密度规则

亲密度定位：

1. 亲密度用于表达好友关系沉淀。
2. 亲密度影响好友列表排序、采摘收益和采摘次数。
3. 亲密度不直接大幅提高核心经济产出，避免玩家被迫社交。

亲密度来源建议：

1. 首次成为好友：+20，仅限双方从未有过 `FRIEND` 历史关系。
2. 浇水一次：+5。
3. 采摘一次：+2。
4. 连续多日互助：每日首次互助额外 +1，可后置实现。
5. 删除后重新成为好友：不加首次好友亲密度。

亲密度增长限制：

1. 每对好友每日亲密度增长上限：10。
2. 每个玩家每日总亲密度增长上限：100。
3. 同一好友同一天重复互动收益递减。
4. 亲密度增长必须写入助力/互动记录，不能只改关系表，方便后台排查。

亲密度档位建议：

1. 0-49：新识。
2. 50-199：熟识。
3. 200-499：亲近。
4. 500-999：挚友。
5. 1000+：知交。

亲密度作用建议：

1. 好友列表排序：高亲密度更靠前。
2. 采摘金币收益加成：低幅度，建议 0%-20%。
3. 每日好友采摘次数加成：达到档位后增加少量次数。
4. 后续组队/通缉助战可用作轻微协同加成。

### 6.3.3 好友浇水规则

定位：

1. 浇水是帮助好友缩短田地成长时间。
2. 浇水不扣好友金币。
3. 浇水不扣好友种子、精华或任何资源。
4. 浇水不触发战斗，不进入探索战报。

当前已有规则应保留：

1. 只能同阵营好友互相浇水。
2. 非好友不能浇水。
3. 不同阵营不能浇水。
4. 已成熟、枯萎、空地不能浇水。
5. 每块田每个生长周期最多被浇水 1 次。
6. 每个玩家每天主动浇水次数有限。
7. 每个玩家每天被浇水次数有限。

待补齐规则：

1. 浇水成功后，双方关系亲密度增加。
2. 每日每对好友浇水增加亲密度最多 1 次。
3. 浇水动态写入双方社交动态。
4. 浇水失败要明确原因：非好友、不同阵营、没有可浇水田地、次数已满。

### 6.3.4 好友采摘规则

定位：

1. 采摘是好友拜访收益，不是偷菜。
2. 采摘不扣好友金币。
3. 采摘不扣好友种子、精华或产物。
4. 采摘不改变好友田地状态。
5. 采摘不触发战斗、防护或探索战报。
6. 采摘收益由系统额外发放，文案可解释为“拜访灵田获得灵气余韵”。

前端流程：

1. 好友列表每个好友行在“浇水”旁边增加“采摘”按钮。
2. 点击“采摘”后打开好友田地预览弹框。
3. 弹框展示好友当前种植了什么、哪些田可采摘、预计可获得什么。
4. 玩家确认采摘后调用后端采摘接口。
5. 成功后弹出通用奖励接收弹框。
6. 奖励确认后入账，并写入社交动态。

可采摘对象：

1. 只允许采摘同阵营好友。
2. 好友必须是 `FRIEND / ACTIVE`。
3. 好友有正在培育、成熟或丰熟的田地时，可以展示采摘预览。
4. 枯萎田、空地、锁定田不提供采摘收益。
5. 首版可以不要求好友田地真实成熟，避免好友上线时间差导致体验太差；但奖励应按作物档位和状态控制。

收益规则建议：

1. 采摘收益来自系统，不从好友实际收益中扣除。
2. 不直接按好友田地真实收益百分比给，避免高级号带飞小号。
3. 使用“作物档位基础奖励 + 亲密度加成 + 每日衰减”计算。
4. 普通灵植：少量金币，低概率普通精华。
5. 稀有灵植：中等金币，低概率稀有精华。
6. 传说灵植：较高金币，极低概率传说精华。
7. 同一好友每天最多被同一玩家采摘 1 次。
8. 每日总采摘收益有上限。

亲密度对采摘的影响：

1. 亲密度越高，金币收益略高。
2. 亲密度越高，每日可采摘次数略多。
3. 亲密度不应大幅提高精华概率，避免强制刷好友。

建议首版档位：

1. 新识：每日好友采摘 3 次，金币加成 0%。
2. 熟识：每日好友采摘 4 次，金币加成 5%。
3. 亲近：每日好友采摘 5 次，金币加成 10%。
4. 挚友：每日好友采摘 6 次，金币加成 15%。
5. 知交：每日好友采摘 7 次，金币加成 20%。

天机符付费扩展：

1. 基础采摘次数用完后，可消耗天机符购买额外采摘次数。
2. 额外次数只增加当天采摘次数，不增加亲密度上限。
3. 每日购买次数必须有上限。

5. 付费扩展应优先增加金币收益机会，不直接稳定产出稀有精华。

推荐首版付费参数：

1. 每日最多购买 3 次额外采摘。
2. 第 1 次：1 天机符。
3. 第 2 次：2 天机符。
4. 第 3 次：3 天机符。
5. 次日重置。

## 7. UI 精简原则

适配微信小游戏，必须控制信息密度。

### 7.1 tab 数量

每个一级页面最多 3 个 tab。

探索：

1. 战斗。
2. 战报。
3. 通缉。

社交：

1. 动态。
2. 好友。
3. 关系。

### 7.2 列表行信息

移动端每行只展示 3 类信息：

1. 对方是谁。
2. 为什么推荐或为什么相关。
3. 一个主操作。

更多操作放进详情页或小菜单。

### 7.3 文案

避免内部术语：

1. `relation` 显示为“关系”。
2. `follow` 显示为“关注”。
3. `enemy` 显示为“仇敌”。
4. `warrant` 显示为“通缉”。
5. `feed` 显示为“动态”。
