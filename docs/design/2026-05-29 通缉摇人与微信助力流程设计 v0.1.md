# 2026-05-29 通缉摇人与微信助力流程设计 v0.1

## 1. 设计目标

这份文档专门解决三个容易混在一起的问题：

1. 探索页里的“通缉”到底是什么。
2. 玩家说的“摇人”到底摇谁、怎么加入。
3. 微信分享进来的新人和老玩家，如何助力但不破坏战斗平衡。

核心结论：

> 通缉是站内战斗订单，摇人是寻找协力者的入口，微信助力是外部确认和转化链路。

三者不能混成一个功能，否则会出现规则不清、结算难做、刷号风险高的问题。

当前开发阶段补充约定：

1. 先做邀请新玩家。
2. 先做邀请浇水助力。
3. 战斗助力、通缉助力、灵宠恢复助力全部暂缓。
4. 本文的通缉和微信助战设计作为后续展开依据，不进入当前 MVP。

## 2. 三个概念拆分

### 2.1 通缉

通缉是探索页的站内玩法。

它解决的问题是：

1. 我看到一个目标。
2. 我单人打不过或不想冒险。
3. 我发布通缉，等待一名协力者加入。
4. 协力者确认后，系统按双人战斗结算。

通缉必须绑定真实游戏对象：

1. 发起玩家。
2. 目标玩家。
3. 目标快照。
4. 过期时间。
5. 结算状态。
6. 奖励分配规则。

### 2.2 摇人

摇人不是一种结算方式，只是找协力者。

摇人可以有三种入口：

1. 站内邀请好友或同阵营成员。
2. 发布到社交动态或阵营动态。
3. 生成微信分享链接。

不管从哪里摇人，最终都要落到同一张通缉单。

### 2.3 微信助力

微信助力是外链落地页。

它解决的问题是：

1. 玩家从微信点进来。
2. 系统判断他是老玩家还是新玩家。
3. 老玩家可以登录后成为真实协力者。
4. 新玩家可以先完成助威和邀请绑定，不能直接提供真实战力。

微信助力本身不应该直接结算战斗。

## 3. 首发原则

### 3.1 通缉只做二人

首发只支持：

1. 一个发起者。
2. 一个协力者。
3. 一个目标。

不做多人围攻，不做队伍房间，不做实时组队。

### 3.2 老玩家才能真实入战

真实协力者必须是已有玩家，并且满足：

1. 已创建角色。
2. 已完成首宠创建。
3. 主位灵宠可出战。
4. 今日助战次数未满。
5. 不是发起者本人。
6. 不是同设备或同账号体系的高风险小号。

### 3.3 新玩家只能先助威

新玩家从微信链接进入时，不能直接提供战力。

原因：

1. 他还没有角色。
2. 没有灵宠、等级和战力快照。
3. 直接给战力会变成刷号工具。

首发推荐把新玩家行为定义为“助威”：

1. 助威可以让发起者获得小额参与奖励或轻量士气加成。
2. 助威可以建立邀请关系。
3. 新玩家完成新手流程后，双方领取邀请奖励。
4. 如果通缉仍未过期，并且新玩家已经有首宠，后续可以选择正式加入，但首发可以先不做。

### 3.4 微信链接不替代站内确认

老玩家从微信链接进入后，必须登录并确认加入。

不能只点外链按钮就直接消耗他的战斗次数或使用他的战力。

## 4. 推荐玩家流程

### 4.1 发起通缉

入口：

1. 探索页目标详情。
2. 战报详情。
3. 复仇失败或判断目标过强时的引导。

流程：

1. 玩家点击“发布通缉令”。
2. 系统展示目标摘要、预计难度、奖励分配、过期时间。
3. 玩家确认发布。
4. 系统创建通缉单，进入“等待协力”。
5. 通缉页显示这张单。
6. 玩家可以选择站内邀请或微信摇人。

首发通缉确认页只需要展示：

1. 目标昵称和阵营。
2. 目标大致等级和主宠外观。
3. 预计可获得资源区间。
4. 协力者收益说明。
5. 过期时间。

不要展示过细资源和精确战力，避免变成蹲点工具。

### 4.2 站内摇人

入口：

1. 通缉详情里的“邀请好友”。
2. 通缉详情里的“邀请同阵营”。
3. 社交关系页对象行里的“邀请助战”。

流程：

1. 发起者选择一个好友或同阵营成员。
2. 系统给对方写一条社交动态。
3. 对方在社交动态或通缉页看到邀请。
4. 对方点击“加入助战”。
5. 系统校验资格。
6. 成功后通缉进入“已接取 / 待结算”。
7. 战斗结算，双方看到战报。

### 4.3 微信摇人给老玩家

入口：

1. 通缉详情点击“微信摇人”。
2. 生成分享链接。
3. 老玩家从微信打开。

流程：

1. 外链页展示“某某邀请你助战通缉”。
2. 老玩家点击“登录并助战”。
3. 如果已有登录态，直接进入确认页。
4. 如果没有登录态，完成微信登录。
5. 登录后读取通缉状态。
6. 如果通缉仍可加入，展示站内确认。
7. 玩家确认后成为协力者。
8. 系统结算通缉。

老玩家外链页不要说“助力成功”后直接结束，应该明确：

```text
登录后确认助战，才会正式加入本次通缉。
```

### 4.4 微信摇人给新玩家

入口和老玩家相同，但系统发现 openid/unionid 未绑定角色。

流程：

1. 外链页展示“好友邀请你为这次通缉助威”。
2. 新玩家点击“帮 TA 助威”。
3. 系统记录 guest 助威记录。
4. 新玩家进入新手流程。
5. 建档后和发起者建立邀请关系。
6. 新玩家完成指定新手节点后，双方领取邀请奖励。

新玩家助威成功文案建议：

```text
助威成功

你的朋友已经收到这次助威。
创建角色后，你们会成为好友，并可领取新友奖励。
```

不要承诺“你已经加入战斗”，除非他已经完成角色和首宠创建。

## 5. 通缉状态机

首发建议状态：

1. `waiting_ally`：等待协力者。
2. `ally_invited`：已指定邀请对象，仍未接受。
3. `accepted`：协力者已接受，准备锁定结算。
4. `settling`：结算中。
5. `settled`：已结算。
6. `expired`：超时未接。
7. `cancelled`：发起者取消或目标失效。
8. `failed`：结算失败，等待补偿或重试。

现有代码里的 `TeamChallengeStatus` 可以先承载首版：

1. `PENDING` 对应等待协力。
2. `ACCEPTED` 对应已接取。
3. `REJECTED` 对应拒绝。
4. `EXPIRED` 对应过期。
5. `SETTLED` 对应已结算。

显示层统一叫“通缉”，内部可以继续沿用 `team_challenge`，后续再决定是否重命名。

## 6. 结算规则

### 6.1 创建通缉时

创建通缉时做软锁：

1. 保存目标快照。
2. 保存目标当时可争夺资源区间。
3. 保存发起者战力快照。
4. 不立即占死全部资源。
5. 不立即结算。

这样可以避免发起者发布后一直卡住目标资源。

### 6.2 协力者接受时

协力者接受时做硬锁：

1. 校验通缉未过期。
2. 校验目标仍可被挑战。
3. 校验发起者和协力者资格。
4. 锁定本次可结算份额。
5. 记录协力者战力快照。
6. 进入结算。

### 6.3 战力合并

首发建议：

1. 发起者战力按 100% 计入。
2. 协力者战力按 60% 计入。
3. 好友关系额外 +5% 协同效率。
4. 同阵营额外 +5% 协同效率。
5. 总协同效率需要上限，例如最高 70%。

这样通缉能帮助弱者挑战强目标，但不会变成无限堆战力。

### 6.4 奖励分配

首发建议：

1. 发起者获得主要收益，约 70%。
2. 协力者获得协力奖励，约 30% 或固定参与奖励。
3. 协力者失败也获得少量安慰奖励。
4. 防守方如果防守成功，获得防守记录和少量防守奖励。
5. 新玩家助威不参与战斗收益分成，只走邀请/助威奖励。

通缉收益不要高于单人战斗太多。它的价值是“能打更强目标”和“社交协作”，不是无脑放大产出。

## 7. 微信助力状态

外链助力建议独立建模，不直接塞进社交或通缉表。

### 7.1 share assist campaign

一条分享活动。

字段建议：

```text
share_assist_campaign
- id
- owner_player_id
- campaign_type: water | warrant_cheer | warrant_ally
- related_entity_type: field | warrant
- related_entity_id
- status: active | full | expired | cancelled
- max_assist_count
- current_assist_count
- expires_at
- created_at
- updated_at
```

### 7.2 share assist record

一次助力记录。

字段建议：

```text
share_assist_record
- id
- campaign_id
- helper_player_id nullable
- helper_openid_hash nullable
- helper_device_hash nullable
- helper_audience: guest | player
- assist_role: cheer | ally_candidate | water_helper
- status: confirmed | bound | converted | rewarded | rejected
- created_at
- bound_at nullable
```

### 7.3 invite relation

邀请关系可以独立表，也可以先通过 share assist record 派生。

如果做表，建议：

```text
player_invite_relation
- id
- inviter_player_id
- invited_player_id nullable
- invited_openid_hash nullable
- source_campaign_id
- status: pending_bind | bound | tutorial_completed | rewarded | invalid
- created_at
- bound_at nullable
- rewarded_at nullable
```

## 8. 新老玩家分流

### 8.1 老玩家

判断条件：

1. 已登录，或微信身份已绑定 player。
2. 玩家已经完成首宠创建。

可做行为：

1. 浇水助力。
2. 通缉真实助战。
3. 领取助力参与奖励。
4. 和发起者建立或强化好友关系。

### 8.2 新玩家

判断条件：

1. 微信身份未绑定 player。
2. 或刚建档但未完成首宠创建。

可做行为：

1. 浇水助力可以立即生效。
2. 通缉只能助威，不能真实入战。
3. 记录邀请来源。
4. 新手流程完成后发放双方奖励。
5. 后续可以引导加入同阵营或添加好友。

### 8.3 为什么浇水和通缉不同

浇水是轻互助，不影响对抗结果，所以新人可以直接帮。

通缉影响战斗胜负和资源转移，所以新人不能直接提供战力。

这条边界要写进实现和文案里。

## 9. 前端信息架构

### 9.1 探索页通缉 tab

建议分三段，不再额外开四级复杂 tab：

1. 我的通缉。
2. 可接通缉。
3. 已完成。

每张通缉卡只展示：

1. 目标是谁。
2. 发起者是谁。
3. 还差什么。
4. 剩余时间。
5. 一个主按钮。

主按钮根据身份变化：

1. 我发起的等待中：`微信摇人` / `邀请好友`。
2. 别人发起的可接：`加入助战`。
3. 已接受：`查看进度`。
4. 已结算：`看战报`。

### 9.2 社交动态

社交动态里展示轻提示：

1. 某好友邀请你助战。
2. 某同阵营成员发布了通缉。
3. 某好友帮你助威。
4. 你发布的通缉已有人加入。
5. 通缉已结算。

点击后跳转探索页通缉详情。

### 9.3 微信外链页

外链页只保留一个主动作。

老玩家通缉页：

```text
好友邀请你助战

TA 正在通缉一个强敌。
登录后确认助战，可以提供一部分战力并获得参与奖励。

[登录并助战]
```

新玩家通缉页：

```text
好友邀请你助威

先帮 TA 打气。创建角色后，你们会成为好友并领取新友奖励。

[帮 TA 助威]
```

## 10. 风控规则

首发必须有的限制：

1. 每个玩家每天最多发布 1 到 2 张通缉。
2. 每个玩家每天最多助战 2 次。
3. 同一目标每天最多被同一发起者通缉 1 次。
4. 通缉链接 2 小时过期。
5. 同一微信身份或设备对同一链接只能助力 1 次。
6. 新玩家助威不直接给战斗收益分成。
7. 同设备、同登录来源、同异常指纹账号不能互相真实助战。
8. 失败通缉也要消耗发起次数，但可以给少量保底或返还部分成本。
9. 通缉不能绕过保护期。
10. 被通缉目标结算后进入更长保护。

## 11. 后续 MVP 建议

下面阶段不属于当前“邀请新玩家 + 浇水助力”MVP。等邀请、分享 token、浇水助力和后台记录稳定后，再从阶段 1 开始推进通缉。

### 阶段 1：站内通缉占位可用

目标：

1. 探索页通缉 tab 有“我的通缉 / 可接通缉 / 已完成”展示。
2. 目标详情可以创建通缉占位。
3. 通缉状态可展示、可过期。
4. 不接微信。
5. 不做真实双人结算，先打通 UI 和数据结构。

### 阶段 2：站内老玩家助战

目标：

1. 好友或同阵营成员可以接受通缉。
2. 使用现有 `team_challenge` 能力扩展为通缉显示。
3. 接受后生成战报或简化结算结果。
4. 社交动态写入邀请和接受事件。

### 阶段 3：微信通缉助威

目标：

1. 通缉详情可以生成微信链接。
2. 新玩家打开后完成助威。
3. 助威记录和邀请关系落库。
4. 新玩家建档后绑定邀请关系。
5. 不让新玩家直接入战。

### 阶段 4：微信老玩家助战

目标：

1. 老玩家从微信链接进入。
2. 登录后看到通缉详情。
3. 满足条件后确认助战。
4. 进入站内通缉结算。

### 阶段 5：完整奖励和后台

目标：

1. 通缉奖励分配。
2. 助威奖励。
3. 邀请奖励。
4. 后台查询通缉、助力、邀请记录。
5. 风控异常列表。

## 12. 需要避免的设计

不建议首发做：

1. 新玩家直接提供战力。
2. 多人围攻同一目标。
3. 微信链接点一下就自动结算战斗。
4. 通缉奖励高于单人战斗太多。
5. 通缉永久挂起。
6. 对目标资源做长时间硬锁。
7. 让助力者承担复杂损耗。
8. 把浇水助力、通缉助战、邀请奖励全塞进同一张表。

## 13. 推荐落点

通缉恢复开发时，最适合先做阶段 1：

1. 把通缉 tab 的空态改成真实结构。
2. 增加通缉卡片的数据契约。
3. 目标详情点击“发布通缉令”后创建一条占位通缉。
4. 社交动态能显示“某某发布了通缉”。
5. 暂不接微信和真实双人结算。

这样能先把玩家理解路径跑通，再逐步把老玩家助战、新玩家助威和微信链接接上。

但当前下一步仍建议优先做：

1. 邀请新玩家。
2. 邀请浇水助力。
3. 浇水助力记录和奖励领取。
4. 后台查看邀请和浇水助力记录。
