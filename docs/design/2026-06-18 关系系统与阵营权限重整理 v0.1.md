# 2026-06-18 关系系统与阵营权限重整理 v0.1

## 1. 目的

本文用于统一当前版本的关系系统口径，重点解决以下几个已变化的前提：

1. 玩家可以转阵营。
2. 当前掠夺不会让防守方产生真实经济损失。
3. 既希望保留冲突感和复仇动机，又不希望“仇敌”语义过重或失真。
4. 旧文档中“好友、关注、仇敌、阵营、互助、掠夺”之间的边界已经不够清晰。

本文作为当前推荐口径，优先级高于旧的零散讨论稿。

## 2. 当前版本的核心判断

当前版本建议采用下面四条总原则：

1. 关系长期保留，权限实时计算。
2. 同阵营不进行掠夺战斗。
3. 掠夺默认不自动生成“仇敌”。
4. 恩怨感主要来自战报、复仇、纪录和目标追踪，而不是经济损失。

一句话总结：

> 先区分“我和你是什么关系”，再区分“我们现在是什么阵营立场”，最后再决定“此刻能做什么”。

### 2.1 微信小游戏首版收口口径

考虑到本项目首发形态是微信小游戏，关系操作必须足够轻。首版前台只暴露 3 种状态：

1. `好友`
2. `关注`
3. `无关系`

其中玩家主动操作只保留两类：

1. `加好友 / 删除好友`
2. `关注 / 取消关注`

`仇敌`、`屏蔽`、`对手`不作为首版玩家主动维护的主关系渠道。历史数据层可以继续兼容旧枚举，但首版产品主路径不展示为核心关系类型。

首版一句话口径：

> 好友用于互助，关注用于找回，无关系按阵营进入普通匹配；转阵营不删除关系，只重算权限。

### 2.2 好友与关注的互斥规则

`好友` 与 `关注` 不叠加，好友优先级高于关注。

具体规则：

1. 无关系玩家可以加好友，也可以关注。
2. 已关注玩家可以取消关注，也可以发起好友申请。
3. 成为好友后，双方已有关注关系自动取消。
4. 已是好友时，不再展示关注按钮。
5. 删除好友后，不自动恢复旧关注关系。

原因：

1. 好友本身已经是可找回关系，不需要再次关注。
2. 关注只负责弱记录，不提供额外权限。
3. 转阵营后权限更清楚：好友永远不可互相掠夺，关注对象是否可打只看当前阵营和战斗规则。

### 2.3 关注人数上限

关注是“找回目标”的轻关系，但不能无限增长。

首版建议：

1. 基础关注上限：30 人。
2. 后续可用天机符购买扩容。
3. 每次扩容增加 10 个关注位。
4. 最终硬上限：100 人。
5. 删除或取消关注后释放关注位。
6. 成为好友后自动取消关注，也会释放关注位。

设计原因：

1. 微信小游戏不适合让玩家维护过大的弱关系列表。
2. 关注列表主要服务“找回目标”，不是完整通讯录。
3. 有上限可以避免关系表无限膨胀，也能给天机符提供一个轻度便利消耗点。
4. 硬上限必须存在，避免付费扩容破坏轻社交定位。

前台展示建议：

> 关注 12/30

达到上限时提示：

> 关注人数已达上限。后续可通过天机符扩容，但最高不超过 100。

### 2.4 好友人数上限

好友是互助关系，比关注更重，也必须限制规模。

首版建议：

1. 基础好友上限：50 人。
2. 后续可用天机符购买扩容。
3. 每次扩容增加 10 个好友位。
4. 最终硬上限：150 人。
5. 删除好友后释放好友位。
6. 发起好友申请和接受好友申请时都要检查双方容量。

设计原因：

1. 好友会影响互助、拜访、采摘、亲密度和社交动态，比关注更消耗系统和玩家管理成本。
2. 好友上限可以高于关注，但不能无限增长。
3. 好友扩容和关注扩容都可以作为天机符的轻便利消耗，但必须保留硬上限。

达到上限时提示：

> 好友人数已达上限。后续可通过天机符扩容，但最高不超过 150。

### 2.5 赛季重置保留规则

好友、关注和已购买的关系上限扩容都属于长期社交资产，不参与赛季重置。

赛季重置时必须保留：

1. 好友关系。
2. 关注关系。
3. 好友上限扩容等级。
4. 关注上限扩容等级。
5. 好友亲密度和最近互动时间，除非后续单独设计“赛季亲密度”。

赛季重置时可以重算：

1. 关注对象当前是否可掠夺。
2. 好友或关注对象当前是否同阵营。
3. 复仇窗口是否因阵营变化进入休战。

一句话口径：

> 赛季清成长，不清关系；赛季清战斗状态，不清社交资产。

## 3. 三层模型

### 3.1 关系

关系是长期存在的社交或冲突标签。

底层可以兼容以下旧类型：

1. `好友`
2. `关注`
3. `仇敌`
4. `屏蔽`

但微信小游戏首版只把 `好友` 和 `关注` 作为玩家可见、可维护的关系类型。

另外可以增加一个展示层概念，但不强制落表：

5. `对手`

`对手` 表示“最近交手过的人”，更偏战斗记录，不等同于强情绪关系。

### 3.2 立场

立场不是关系，而是由当前阵营实时派生：

1. `同阵营`
2. `敌对阵营`

玩家转阵营后，只变化立场，不自动删除好友、关注或仇敌。

### 3.3 权限

权限是最终行为层，建议拆成：

1. `可拜访`
2. `可助力`
3. `可掠夺`
4. `可复仇`
5. `可切磋`

## 4. 关系定义

### 4.1 好友

好友是互助关系，不是阵营绑定关系。

建议定义：

1. 好友可跨阵营保留。
2. 好友之间永远不可互相掠夺。
3. 好友之间可互相拜访、浇水、采摘等轻互助。
4. 若后续需要强调阵营归属，可以把“阵营协作加成”只给同阵营好友，而不是直接禁用跨阵营好友互助。

这样做的原因是：

1. 转阵营不会破坏社交资产。
2. 玩家不会因为换阵营失去既有好友价值。
3. 同时又保留了“同阵营协作更高效”的扩展空间。

### 4.2 关注

关注是单向观察关系。

用途：

1. 记住某个玩家。
2. 关注动态。
3. 后续快速进入目标详情、战报或主页。

关注不代表：

1. 不是好友。
2. 不自动可助力。
3. 不自动可掠夺。
4. 不自动产生仇恨语义。

### 4.3 对手

对手是“最近交战记录”的展示概念，建议作为系统自动派生标签。

来源：

1. 最近主动掠夺过你。
2. 最近被你掠夺过。
3. 最近发生过复仇。

用途：

1. 在战报和关系页快速回看“最近和谁打过”。
2. 作为复仇和追踪的中间层。

对手的语义应保持中性，不直接升级为仇敌。

### 4.4 仇敌

仇敌是强情绪关系，不建议由单次无损掠夺自动生成。

当前版本建议：

1. 仇敌默认由玩家手动标记。
2. 单次普通掠夺只生成“对手”或“交战记录”。
3. 若后续重新引入真实经济损失、连续针对、强纪录破坏，再考虑自动升级为仇敌。

## 5. 权限矩阵

微信小游戏首版权限矩阵收口为：

| 条件 | 可拜访 | 可助力 | 可掠夺 | 可复仇 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 好友 | 可以 | 可以 | 不可 | 不可 | 好友是互助关系，不因阵营变化变成可掠夺对象 |
| 同阵营关注 | 可以或查看资料 | 不可 | 不可 | 暂停 | 关注只负责找回，不绕过同阵营休战 |
| 敌对阵营关注 | 可以或查看资料 | 不可 | 可以 | 满足复仇窗口时可 | 是否能打由阵营和战斗规则决定 |
| 同阵营无关系 | 可以或查看资料 | 不可 | 不可 | 暂停 | 普通同阵营对象不进入掠夺 |
| 敌对阵营无关系 | 可以或查看资料 | 不可 | 可以 | 满足复仇窗口时可 | 普通敌对对象可作为掠夺目标 |

说明：

1. `可拜访` 是轻社交行为，不等于可助力。
2. `关注` 不提供额外权限，只影响找回和列表展示。
3. `可复仇` 依赖单独的复仇窗口，不由关注关系直接赋予。
4. `可切磋` 不是当前必做项，后续如需同阵营对抗再单独开放。

## 6. 同阵营战斗规则

当前建议不改“同阵营不能掠夺”的设定。

原因：

1. 阵营边界仍然是 PvP 的核心分层。
2. 如果任意阵营都能掠夺，阵营差异会明显变弱。
3. 玩家转阵营后，关系变化会更混乱。

因此建议：

1. `同阵营不可掠夺`
2. `敌对阵营可掠夺`
3. 如需同阵营对抗，只开放 `切磋 / 演武`
4. 切磋不产生资源损失，只产生记录、排名或任务进度

## 7. 转阵营规则

转阵营不是随时行为，而是赛季节点行为。建议限制为：每个赛季开始时，玩家只有一次转阵营机会；一旦确认，本赛季内不可再次更换。

这样做的原因是：
1. 避免玩家为了短期收益频繁横跳，削弱阵营边界。
2. 让阵营选择更像赛季承诺，而不是临时开关。
3. 降低关系、复仇、匹配和社交预期反复变化带来的理解成本。

### 7.1 切换时机与次数

建议口径：
1. 仅在每个赛季开始阶段开放转阵营入口。
2. 每名玩家每赛季仅可转阵营一次。
3. 一旦完成转阵营，当赛季内不可撤销、不可再次改选。

### 7.2 保留的内容

1. 好友保留
2. 关注保留
3. 最近对手历史保留
4. 复仇记录保留

### 7.3 实时变化的内容

1. 是否可掠夺
2. 是否可复仇
3. 是否属于同阵营或敌对阵营
4. 是否显示“同阵营休战”

### 7.4 转阵营确认提示

转阵营确认弹窗里，建议明确告诉玩家好友会怎么变化，而不是只提示“是否确认转阵营”。

建议明确写清楚：
1. 你的好友、关注关系不会被删除。
2. 你与好友之间依然保持好友关系。
3. 好友是否属于同阵营/敌对阵营会随本次转阵营实时重算。
4. 好友之间依然不能互相掠夺。
5. 若因本次转阵营变为同阵营，与该好友相关的复仇将进入“同阵营休战”状态。

建议弹窗文案示例：

> 你将在本赛季切换至【XX阵营】。  
> 本次切换后，本赛季内无法再次更换阵营。  
> 你的好友和关注不会被删除；系统只会重算你与他们当前的阵营立场和可执行行为。  
> 若你与部分好友变为同阵营，相关复仇将暂时进入“同阵营休战”。  
> 是否确认切换？

### 7.5 复仇窗口处理

建议口径：

1. 若双方因转阵营变为同阵营，复仇窗口不删除，只暂停。
2. 暂停期间 UI 显示“同阵营休战，暂不可复仇”。
3. 若之后再次变回敌对阵营，且窗口未过期，则恢复可复仇。

这样可以避免转阵营把冲突数据清空，也避免玩家误解为系统 bug。

## 8. 无真实经济损失时的恩怨设计

既然当前掠夺不让防守方真正损失经济，就不应再把“仇敌”作为默认自动结果。

当前版本的恩怨感，建议主要依靠以下四类来源：

1. `战报可见性`
   - 谁打了你
   - 谁打断了你的纪录
   - 谁成为你的最近对手

2. `复仇窗口`
   - 被攻击后可立即复仇
   - 复仇成功可给少量额外名望、徽记或成就进度

3. `纪录破坏`
   - 连胜被终结
   - 连续防守成功被终结
   - 某日安稳状态被打破

4. `手动升级关系`
   - 玩家自己把某人列为仇敌
   - 或持续追踪某个对手

当前版本不建议：

1. 单次无损掠夺自动标记仇敌
2. 因为被普通攻击一次就进入强仇恨关系

## 9. 命名建议

### 9.1 社交页

保留：

1. `好友`
2. `关注`
3. `关系`

在关系页中显示标签：

1. `好友`
2. `关注`
3. `对手`
4. `仇敌`
5. `同阵营`
6. `敌对阵营`
7. `同阵营休战`

### 9.2 战斗与战报页

不建议继续直接使用“关注”作为战斗按钮名。

建议改为：

1. `追踪目标`
2. `取消追踪`
3. `列为仇敌`
4. `移出仇敌`
5. `复仇`

原因：

1. “关注”更适合社交观察关系。
2. “追踪”更适合战术意图。
3. 可以避免社交关注和战斗标记混淆。

## 10. 数据模型建议

当前代码里已存在：

1. `friend`
2. `following`
3. `enemy`
4. `blocked`

建议当前版本继续沿用这四类主关系，不急着增加新的底层枚举。

推荐做法：

1. `好友` 映射 `friend`
2. `关注` 映射 `following`
3. `仇敌` 映射 `enemy`
4. `对手` 先做展示层派生标签，不作为首发强制存储关系类型

`对手` 可由以下数据派生：

1. 最近战报
2. 最近复仇记录
3. 最近掠夺交手记录

这样改动最小，也最符合当前实现状态。

## 11. MVP 建议

当前版本建议先收口到下面这套：

1. 保留 `同阵营不可掠夺`
2. 好友可跨阵营保留
3. 好友可跨阵营互助
4. 转阵营仅在赛季开始时开放，且每赛季仅可更换一次
5. 转阵营确认时明确告知“好友不删除，只重算阵营立场与权限”
6. 掠夺默认只生成 `对手/交战记录`
7. 不自动生成 `仇敌`
8. 仇敌只由手动标记产生
9. 战斗页按钮从“关注”改为“追踪”
10. 复仇窗口独立于关系系统存在
11. 转阵营时不删关系，只重算权限

如果后续要增强阵营味道，再增加：

1. 同阵营好友互助加成
2. 阵营任务协助仅限同阵营
3. 同阵营切磋

## 12. 最终口径

当前版本推荐的产品口径是：

> 好友是互助关系，关注是观察关系，对手是交战记录，仇敌是强情绪标签。  
> 阵营决定能不能打，关系决定愿不愿意来往，权限由两者实时共同计算。  
> 转阵营只在赛季开始时开放且每赛季只能改一次；转阵营不清空关系，只改变当下可执行的行为。
