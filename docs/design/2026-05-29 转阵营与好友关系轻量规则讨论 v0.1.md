# 2026-05-29 转阵营与好友关系轻量规则讨论 v0.1

## 1. 背景问题

玩家未来可能支持转阵营，但现有好友系统与阵营玩法存在冲突：

1. 好友可以采摘、浇水、助力，但通常要求同阵营。
2. 敌对阵营可以掠夺玩家。
3. 玩家转阵营后，原本同阵营好友可能变成异阵营。
4. 如果异阵营好友自动变成可掠夺对象，会破坏社交关系。
5. 如果好友都能免掠夺，又可能出现大量互加好友规避对抗的问题。

本讨论的目标是给出一套初期可落地、规则简单、玩家容易理解的方案。

## 2. 核心原则

推荐原则：

> 好友关系高于私人掠夺，但不高于公共阵营战争。

含义：

1. 好友关系是玩家的社交资产，转阵营不应自动删除好友。
2. 只要双方仍是好友，就不能互相掠夺。
3. 同阵营好友可以进行灵田互助。
4. 异阵营好友保留好友名义和基础社交，但不能灵田互助。
5. 阵营战、宗门战、据点战、世界 Boss 阵营争夺等公共阵营玩法，可以按阵营规则处理，不受好友保护完全覆盖。

## 3. 轻量首发规则

首发建议只做以下规则：

```text
好友不断关系
同阵营好友可以互助
异阵营好友不能互助
好友之间不能互相掠夺
删除好友后 24 小时内不能互相掠夺
转阵营后不删除好友，只重新计算互助权限
```

### 3.1 关系与行为矩阵

| 关系 | 聊天/赠礼/送体力 | 借灵宠助战 | 浇水/助力/采摘 | 掠夺 |
|---|---:|---:|---:|---:|
| 同阵营好友 | 可以 | 可以 | 可以 | 不可以 |
| 异阵营好友 | 可以 | 可以 | 不可以 | 不可以 |
| 敌对非好友 | 不一定 | 不可以 | 不可以 | 可以 |
| 刚解除好友 | 不可以或按普通关系 | 不可以 | 不可以 | 24 小时后才可掠夺 |

说明：

1. 跨阵营好友不是敌人，只是互助权限暂停。
2. 掠夺主要面向敌对非好友玩家。
3. 如果玩家非要掠夺旧好友，必须先解除好友，并等待冷却结束。

## 4. 转阵营后的处理

转阵营时不改好友关系本身，只根据双方当前阵营重新判断权限。

### 4.1 转阵营后仍为同阵营好友

处理：

1. 好友关系保留。
2. 灵田互助继续可用。
3. 双方仍不可互相掠夺。

### 4.2 转阵营后变成异阵营好友

处理：

1. 好友关系保留。
2. 聊天、赠礼、灵宠助战等基础社交保留。
3. 浇水、助力、采摘等灵田互助暂停。
4. 双方仍不可互相掠夺。

建议状态名：

```text
异阵营好友
仙缘封印
互助封印
```

推荐文案：

```text
你与该道友分属不同阵营，灵田互助暂时封印。
聊天、赠礼、灵宠助战不受影响。
若日后同归一阵营，互助关系将自动恢复。
```

### 4.3 原敌对好友转为同阵营

如果玩家转阵营后，某个原本异阵营好友变成同阵营：

1. 好友关系继续保留。
2. 灵田互助自动恢复。
3. 双方仍不可互相掠夺。

这可以作为转阵营去找朋友一起玩的正向收益。

## 5. 掠夺判定优先级

推荐判定：

```text
是否公共阵营玩法？
  是：按阵营规则处理
  否：进入私人关系判断

是否好友？
  是：禁止掠夺
      同阵营：允许灵田互助
      异阵营：禁止灵田互助，保留基础社交
  否：
      敌对阵营：允许掠夺
      同阵营：不可掠夺或按普通同阵营规则处理
```

关键点：

1. 私人资源玩法里，好友保护优先。
2. 公共阵营玩法里，阵营对抗优先。
3. 好友不是免除所有阵营冲突的万能身份，只保护私人灵田掠夺。

## 6. 防止“世界大同”的轻量限制

如果好友上限过高、加好友成本过低、好友又能免掠夺，玩家可能把好友系统当作免战名单。

初期不建议做复杂社交分层，只加两个硬限制：

```text
好友上限：50
每日发送好友申请上限：20
```

可选成长设计：

```text
初始好友上限：30
后续通过境界、月卡、仙盟科技或付费权益提升到 50/80
```

注意：

1. 仙盟成员不自动等于好友。
2. 好友位是一种有限社交资源。
3. 跨阵营好友不能互助，因此大量添加敌对好友的收益有限。

## 7. 加好友入口轻量版

初期只保留 3 个入口。

### 7.1 微信好友 / 邀请码

定位：

1. 用来添加现实认识的人。
2. 可以跨阵营添加。
3. 异阵营好友不能互助，也不能互相掠夺。

原因：

现实社交优先，不能因为阵营选择切断真实好友关系。

### 7.2 同阵营推荐

定位：

1. 系统每天推荐一批同阵营活跃玩家。
2. 玩家可以一键申请好友。
3. 推荐规则首发保持简单。

首发推荐条件：

```text
最近登录
等级接近
灵田等级接近
```

### 7.3 同阵营排行榜

定位：

1. 贡献榜、活跃榜、战力榜等同阵营榜单可点击头像申请好友。
2. 只允许加同阵营玩家。
3. 用于鼓励同阵营抱团。

### 7.4 初期暂不做

首发阶段暂不做以下入口：

```text
不做聊天频道加好友
不做最近互助加好友
不做战报加好友
不做敌对玩家加好友
不做复杂关注系统
不做临时好友/正式好友分层
不做跨阵营搜索加好友
```

尤其注意：

> 敌对关系产生的场景，不提供加好友入口。

例如战报、掠夺记录、复仇列表、敌对排行榜都不应直接导流加好友。玩家都已经准备掠夺对方了，再引导加好友会显得逻辑冲突，也容易被用来规避掠夺。

## 8. 转阵营确认文案

建议在转阵营前明确告知影响，降低玩家心理负担。

```text
确定要转投新的阵营吗？

转阵营后：
· 好友关系不会消失
· 与原阵营好友的灵田互助可能暂停
· 与新阵营好友的灵田互助可能开启
· 跨阵营好友不可互助，也不可互相掠夺
· 再次转阵营需等待 14/30 天

仙途易轨，道友三思。
```

如果首发暂不做转阵营冷却，可先在文案中不展示最后一条。

## 9. 推荐实现口径

不要在转阵营时批量修改好友关系，而是在具体行为时判断权限。

推荐伪代码：

```text
canAssist(actor, target):
  if !isFriend(actor, target):
    return false
  if actor.faction != target.faction:
    return false
  return true

canRaid(actor, target):
  if isPublicFactionMode():
    return isFactionEnemy(actor, target)
  if isFriend(actor, target):
    return false
  if wasFriendRecentlyDeleted(actor, target, within=24h):
    return false
  return isFactionEnemy(actor, target)
```

数据层建议：

1. 好友关系保留历史记录，不物理删除。
2. 删除好友时记录 `deleted_at` 或将状态改为 `MUTED/DELETED`。
3. 掠夺校验读取最近删除时间，用于 24 小时保护。
4. 转阵营只修改玩家阵营字段，不批量改好友表。

## 10. 后续可扩展项

等社交系统变重后，再考虑以下扩展：

1. 转阵营冷却期：14 天或一个赛季。
2. 删除好友后掠夺冷却从 24 小时提高到 72 小时。
3. 正式好友保护：成为好友满 24/48 小时后才触发免掠夺保护。
4. 最近发生掠夺关系的双方，72 小时内不能添加好友。
5. 跨阵营好友可以借灵宠助战，但不参与灵田互助。
6. 阵营公共玩法明确无视好友保护，保持阵营对抗存在感。

这些不是首发必需项，避免初期社交系统过重。

## 11. 最终结论

初期采用最简单的一句话规则：

> 好友不断关系；同阵营能互助；异阵营只保留好友名义；好友不可互抢；删好友后 24 小时才可抢。

这个方案的优点：

1. 玩家容易理解。
2. 转阵营不会造成社交资产损失。
3. 好友不会突然变成可掠夺对象。
4. 阵营对抗仍然存在，目标主要来自敌对非好友。
5. 通过好友上限和每日申请上限，避免全服互加好友导致私人掠夺失效。
