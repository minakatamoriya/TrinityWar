# Trinity War 微信小游戏开发清单 v0.1

## 一、目标定义

这份清单服务于首发玩法验证，不服务于完整版产品一次做完。

当前版本的核心目标只有四个：

1. 验证玩家是否理解“生产慢但稳，掠夺快但有风险”。
2. 验证主城后期外露资源门槛是否能逼玩家进入对抗循环。
3. 验证小时分红和贡献是否能提高回流频率。
4. 验证单兵种、双层资源池、三阵营差异是否足够成立。

一句话目标：

做一个后端驱动、前端轻表现的微信小游戏原型，先把三阵营经营策略战争的核心闭环跑通。

## 二、首发范围边界

### 必做

1. 微信小游戏登录。
2. 玩家建档。
3. 三阵营选择。
4. 双层资源池。
5. 农民与掠夺兵。
6. 主城和住房升级。
7. 单兵种训练。
8. 异步掠夺结算。
9. 战报。
10. 每小时分红。
11. 上缴换贡献。
12. 掠夺榜、发展榜、贡献榜。

### 暂不做

1. 复杂兵种。
2. 大地图争夺。
3. 复杂交易市场。
4. 复杂阵营任务树。
5. 赛季大事件。
6. 重战斗演出。
7. 复杂社交系统。

## 三、技术架构清单

首发建议采用单体后端，而不是一开始做复杂拆服务。

### 1. 客户端

职责：

1. 登录。
2. 展示资源、人口、主城、部队、分红、战报、排行榜。
3. 发起升级、训练、掠夺、上缴、领取分红等操作。

要求：

1. 界面轻量。
2. 信息密度高。
3. 不依赖复杂动画。
4. 所有关键数值以后端返回为准。

### 2. 后端

职责：

1. 玩家状态管理。
2. 资源与人口结算。
3. 掠夺匹配与战斗结算。
4. 分红结算。
5. 贡献结算。
6. 排行榜刷新。
7. 战报生成。

原则：

1. 后端权威结算。
2. 前端不保留真实经济和战斗计算逻辑。
3. 所有增长、损耗、掠夺、分红都以后端状态为准。

### 3. 数据库

职责：

1. 存玩家核心状态。
2. 存资源与人口。
3. 存战报和分红记录。
4. 存贡献和排行榜快照。

### 4. 定时任务模块

这是异步经营游戏必须有的模块。

职责：

1. 每小时分红结算。
2. 离线资源累积。
3. 目标刷新。
4. 护盾到期处理。
5. 排行榜定时刷新。
6. 伤兵恢复或人口恢复。

## 六、数据库表清单

2. player_resource
3. player_population
4. player_building
5. player_army
6. raid_report
7. contribution_log

## 七、定时任务清单

### 每分钟或短周期任务

1. 刷新目标池。
2. 处理恢复中的兵力或人口。
3. 清理过期战报缓存。

### 每小时任务

1. 阵营分红结算。
2. 排行榜刷新。
3. 玩家资源离线累积对账。

### 每日任务

1. 重置部分掠夺次数或保护次数。
2. 生成简单阵营任务。
3. 清理过期日志数据。

## 八、玩法验证指标清单

首发不是为了堆功能，而是为了拿验证数据。

### 核心验证指标

1. 次日留存。
2. 第三日留存。
3. 人均每日登录次数。
4. 小时分红领取率。
5. 人均每日掠夺次数。
6. 被掠夺后次日是否回流。
7. 上缴行为发生率。
8. 三阵营人数分布。

### 需要重点观察的问题

1. 玩家是否只会种田不去抢。
2. 玩家是否被抢一次就流失。
3. 小时分红是否真的能拉起短频登录。
4. 主城后期外露资源门槛是否真的能推动对抗。

## 九、微信小游戏专项清单

### 1. 平台接入

1. 小游戏登录能力接入。
2. 用户信息与 openid 体系接入。
3. 分包和资源加载策略。
4. 微信分享能力接入。
5. 版本提审与灰度发布流程。

### 2. 客户端适配

1. 竖屏优先。
2. 低端机流畅优先。
3. 资源加载尽量轻量。
4. 首页进入速度优先。

### 3. 不要优先投入的东西

1. 重动画战斗演出。
2. 大体量 3D 资源。
3. 复杂实时同屏系统。

## 十、建议开发顺序

### 第一阶段：跑通闭环

1. 登录建档。
2. 资源与人口。
3. 主城和住房。
4. 单兵种训练。
5. 异步掠夺。
6. 战报。
7. 小时分红。

### 第二阶段：增加止损与回流

1. 保护仓。
2. 城防。
3. 新手保护。
4. 免费复仇。
5. 上缴与贡献。
6. 三张榜。

### 第三阶段：准备小规模测试

1. 简单阵营任务。
2. 数据埋点。
3. 分享拉回。
4. 灰度版本。
5. 运营后台最小功能。

## 十一、上线前检查清单

1. 玩家断线重连后资源是否一致。
2. 离线结算是否可重复领取。
3. 掠夺是否可能重复结算。
4. 分红是否可能重复发放。
5. 主城升级是否正确卡住外露资源门槛。
6. 人口和兵力是否可能出现负数。
7. 战报是否完整可读。
8. 新手是否会被过早打穿。

## 十二、结论

如果你是按微信小游戏来验证玩法，这个项目的正确做法不是先堆画面，而是先把下面这条链条做实：

1. 农民稳定产出。
2. 掠夺兵单兵种结算。
3. 主城后期必须吃外露资源。
4. 小时分红拉回登录。
5. 上缴和贡献形成阵营目标。

这条链条成立，你的微信小游戏原型就有测试价值。
