# 2026-05-18 赛季重置、继承与追赶完整规则 v0.1

> 2026-06-01 重写：本文件替代旧版规则。旧版存在编码损坏，并且仍沿用“种子库存”等过时概念；当前系统已经改为“可种植资格 + 植物精华”，因此赛季规则需要按现有开发进度重新核对后再开发。

## 一、文档目标

本文件用于定义首发阶段可执行的赛季重置、继承、结算奖励和追赶边界。

当前阶段的原则是：

1. 先明确资产边界，再开发自动重置。
2. 先做最小可靠闭环，再做奖励和追赶。
3. 不把旧赛季积累直接变成新赛季碾压优势。
4. 不再使用“种子库存”说法，统一改为“可种植资格”和“植物精华”。

一句话原则：

赛季重置的是竞争进度，不是玩家身份、图鉴、收藏和已经解锁的长期资格。

## 二、当前系统资产分类

### 1. 账号与身份资产

包含：

1. 账号、昵称、头像。
2. 当前阵营归属。
3. 历史身份记录。
4. 图鉴、发现记录、收藏记录。

处理原则：

永久保留。

### 2. 植物资产

当前植物系统不再有“种子库存”概念。

需要区分：

1. 可种植资格：玩家是否已经解锁某种植物的种植权限。
2. 植物精华：玩家通过收获、掠夺、奖励等获得的消耗型资源。
3. 当前田地状态：某块田当前是否正在成长、成熟、枯萎、可领取金币。
4. 植物研究/发现记录：玩家是否见过、研究过某种植物。

处理原则：

1. 可种植资格保留。
2. 植物研究/发现记录保留。
3. 当前田地状态重置。
4. 植物精华是否清零或折算，需要单独核对。P0 不先做折算奖励。

### 3. 灵宠资产

需要区分：

1. 灵宠本体：玩家已经拥有的灵宠。
2. 灵宠词条：已经洗练出的词条。
3. 灵宠图鉴和碎片。
4. 灵宠等级、经验、突破阶段。
5. 灵宠当前血量、赛季内战斗状态。
6. 灵宠培养资源：兽魂、天机符、灵根、灵髓、灵玉等。

处理原则：

1. 灵宠本体保留。
2. 灵宠词条保留。
3. 灵宠图鉴和碎片保留。
4. 灵宠等级、经验、突破阶段重置。
5. 灵宠血量回到 1 级基础状态。
6. 灵宠培养资源是否清零或折算，暂不在 P0 自动处理，需核对经济影响后再定。

### 4. 阵营资产

需要区分：

1. 个人赛季贡献。
2. 阵营赛季总贡献。
3. 贡献日志。
4. 每日俸禄领取状态。
5. 阵营归属。
6. 历史贡献、历史职位、历史荣誉。

处理原则：

1. 阵营归属保留。
2. 个人赛季贡献清零。
3. 阵营赛季总贡献清零。
4. 本赛季贡献日志清空或归档。
5. 每日俸禄状态清空。
6. 历史贡献和荣誉展示暂不实现，后续作为永久身份资产设计。

### 5. 战斗与掠夺资产

包含：

1. 当前掠夺目标池。
2. 未完成掠夺订单。
3. 当前资产锁。
4. 保护状态。
5. 战报。
6. 复仇、通缉、组队掠夺等非 MVP 状态。

处理原则：

1. 当前掠夺目标池重置。
2. 未完成掠夺订单取消。
3. 当前资产锁过期。
4. 保护状态清空或按新赛季保护规则重建。
5. 历史战报是否保留需要核对展示价值。P0 可以先保留历史记录，但不参与新赛季结算。
6. 通缉、组队掠夺等非 MVP 内容暂不进入赛季结算设计。

### 6. 经营资源

包含：

1. 金库金币。
2. 外露金币。
3. 待领取税收。
4. 待领取分红。
5. 掠夺溢出待领取金币。
6. 建筑等级与法术/科技等级。
7. 兵力、训练队列。

处理原则：

1. 金币类竞争资源重置或折算，不原样继承。
2. 兵力和训练队列重置。
3. 建筑等级、法术等级是否压缩继承，需要单独核对。P0 不先做复杂压缩。

### 7. 签到与每日状态

包含：

1. 赛季签到记录。
2. 每日任务状态。
3. 每日广告奖励次数。
4. 每日恢复次数、侦查次数等日限状态。

处理原则：

1. 赛季签到按赛季隔离，新赛季重新开始。
2. 每日任务和阵营任务状态清空。
3. 普通每日限制本来按日期自然刷新，不需要作为赛季资产继承。

## 三、P0 赛季重置范围

P0 只做“今天是新赛季时，玩家不会继续沿用旧赛季竞争状态”的闭环。

必须完成：

1. 服务端能计算当前赛季号、当前周、总周数。
2. 固定 28 天一个赛季。
3. 新赛季开始后，玩家首次进入时只执行一次赛季重置。
4. 赛季签到按赛季号隔离。
5. 阵营贡献清零。
6. 当前田地状态清空。
7. 金币、兵力、训练队列清空。
8. 灵宠本体和词条保留，但等级、经验、突破、血量重置。
9. 掠夺目标、资产锁、未完成订单等临时竞争状态清理。

P0 不做：

1. 赛季结算奖励。
2. 排名奖励。
3. 追赶机制。
4. 遗产点数。
5. 建筑压缩继承。
6. 植物精华折算。
7. 灵宠培养资源折算。

## 四、P1 赛季结算奖励方向

P1 需要先设计奖励结构，再开发。

推荐奖励分层：

1. 参与奖励：只要达到最低参与门槛即可获得。
2. 阵营贡献奖励：按个人赛季贡献档位发放。
3. 排名奖励：只给荣誉、外观、少量便利资源，不给大量战力资源。
4. 收藏奖励：称号、头像框、赛季纪念记录。

不建议：

1. 大量金币直发。
2. 大量兵力直发。
3. 大量植物精华直接带入新赛季。
4. 大量灵宠培养资源直接带入新赛季。
5. 让排名奖励变成新赛季开局碾压。

## 五、P1 追赶机制方向

追赶机制只服务晚入场玩家的基础盘，不服务冲榜套利。

可能包含：

1. 新赛季前几天基础保护。
2. 晚入场基础任务链。
3. 少量基础植物精华补给。
4. 前期建筑/田地节奏补齐。
5. 低贡献玩家的基础俸禄补齐。

限制：

1. 每账号每赛季只触发一次。
2. 不直接给高阶灵宠资源。
3. 不直接给排名积分。
4. 不绕过植物可种植资格。
5. 需要基础反滥用规则。

## 六、P0 前后端切换流程

P0 不要求接入 WebSocket。赛季切换采用“后端权威兜底 + 前端轻量检查 + 写接口冲突保护”的方式实现。

### 1. 总原则

1. 后端是赛季状态唯一权威。
2. 前端不能自行判断是否已经重置，只能根据后端返回状态刷新 UI。
3. 玩家在线跨过赛季边界时，允许前端晚几秒感知，但不允许继续用旧赛季状态执行关键操作。
4. 所有赛季重置必须具备幂等性，同一玩家同一赛季只执行一次。
5. 赛季开始时可以开放一次阵营调整机会；如果玩家不主动调整，默认保留原阵营。

### 2. 后端读接口流程

读接口包括登录后 bootstrap、首页/场景模型、赛季签到、赛季奖励等。

推荐流程：

1. 请求进入后端。
2. 后端根据当前时间计算 `currentSeasonNumber`。
3. 查询玩家 `player_season_state`。
4. 如果 `lastResetSeasonNumber < currentSeasonNumber`：
   - 生成旧赛季快照。
   - 执行玩家赛季重置。
   - 写入新的 `currentSeasonNumber` 与 `lastResetSeasonNumber`。
   - 返回读模型时带上 `seasonTransition.resetApplied = true`。
5. 如果已经是当前赛季：
   - 正常返回读模型。
   - `seasonTransition.resetApplied = false`。

前端只要重新拉取读模型，就能拿到清零后的权威状态。

### 3. 后端写接口流程

写接口包括收获、培育、升级、掠夺、领取奖励、提交任务、领取俸禄等。

推荐流程：

1. 写接口处理业务前先执行赛季检查。
2. 如果发现玩家仍停留在旧赛季：
   - 先执行旧赛季快照与新赛季重置。
   - 不继续执行本次旧上下文操作。
   - 返回业务错误 `SEASON_ROLLED_OVER`。
3. 错误响应中带上：
   - `currentSeasonNumber`
   - `previousSeasonNumber`
   - `resetApplied: true`
   - `refreshRequired: true`
   - `factionChoiceStatus`
4. 前端收到 `SEASON_ROLLED_OVER` 后关闭当前确认弹窗或操作面板，重新拉取 bootstrap/read model，并进入新赛季提示流程。

这样可以避免玩家在旧赛季结束瞬间继续提交“旧田地收获”“旧目标掠夺”“旧贡献领取”等操作。

### 4. 前端无 WebSocket 切换流程

P0 客户端不需要 WebSocket。推荐使用三个触发点：

1. 首次进入或刷新页面时调用 bootstrap。
2. 根据后端返回的 `season.endsAt` 设置本地 timer，在 `endsAt + 1 到 3 秒` 后重新拉取读模型。
3. 页面从后台回到前台时，通过 `visibilitychange` 或同等机制重新检查一次。

前端发现 `currentSeasonNumber` 变化或 `seasonTransition.resetApplied = true` 后：

1. 停止当前本地倒计时、战斗预览、种田选择等旧上下文展示。
2. 清理本地 optimistic 状态。
3. 重新拉取最新读模型。
4. 展示“新赛季开始”提示。
5. 如果阵营调整窗口开放，展示阵营调整入口。
6. 玩家确认或跳过后进入新赛季主界面。

### 5. 前端展示状态

建议前端至少支持三个状态：

1. `normal`：正常赛季中。
2. `season-transition`：后端刚完成重置，前端展示新赛季开始提示。
3. `faction-choice-available`：赛季开始调整窗口内，玩家可以选择是否转阵营。

不建议把阵营调整按钮放在常规阵营页里。常规阵营页只展示阵营信息、贡献、俸禄和排行。

### 6. 阵营调整流程

赛季开始时的阵营调整不是平时转阵营。

P0 推荐规则：

1. 新赛季开始后，玩家默认保留原阵营。
2. 后端在赛季状态中返回 `factionChoiceStatus`。
3. 只有 `factionChoiceStatus = available` 时，前端才展示阵营调整入口。
4. 玩家每赛季最多调整一次。
5. 调整完成后，本赛季不可再次更改。
6. 超过调整窗口后，后端拒绝更改。
7. 调整阵营不删除好友、关注、历史记录，只重算阵营立场与玩法权限。

建议状态：

1. `not_needed`：不需要选择，例如新玩家已经在建档时选过。
2. `available`：当前赛季开始窗口内，可以调整一次。
3. `used`：本赛季已经调整过。
4. `expired`：调整窗口已关闭。

### 7. 推荐返回字段

读模型或 bootstrap 中建议加入统一赛季切换摘要：

```ts
type ClientSeasonTransition = {
  currentSeasonNumber: number;
  previousSeasonNumber?: number;
  resetApplied: boolean;
  refreshRequired: boolean;
  seasonStartsAt: string;
  seasonEndsAt: string;
  factionChoiceStatus: 'not_needed' | 'available' | 'used' | 'expired';
  factionChoiceDeadlineAt?: string;
};
```

写接口触发赛季切换时，可以返回业务错误：

```ts
type SeasonRolledOverError = {
  code: 'SEASON_ROLLED_OVER';
  message: '新赛季已经开始，请刷新后继续。';
  seasonTransition: ClientSeasonTransition;
};
```

### 8. 实现顺序

建议按下面顺序推进：

1. 补齐所有读接口的赛季检查与返回字段。
2. 补齐所有写接口的赛季前置检查和 `SEASON_ROLLED_OVER` 错误。
3. 前端 bootstrap/read model 接入 `seasonTransition`。
4. 前端根据 `season.endsAt` 设置轻量 timer。
5. 前端接入页面回前台检查。
6. 前端实现新赛季提示弹窗。
7. 后端实现赛季开始阵营调整接口。
8. 前端实现赛季开始阵营调整入口。

## 七、待核对问题

开发前需要确认：

1. 植物精华跨赛季是清零、保留、还是折算。
2. 天机符是否属于赛季资源，是否清零或保留。
3. 兽魂、灵根、灵髓、灵玉是否跨赛季保留。
4. 建筑等级是否本赛季先全部清零，还是等 P1 做压缩继承。
5. 战报是否需要按历史赛季归档。
6. 阵营历史贡献是否需要展示。
7. 赛季奖励是否在本阶段开发，还是先只做重置。

## 八、外部表达

面向玩家不强调“清档”，而强调：

1. 新赛季会重置竞争格局。
2. 已解锁资格、图鉴、灵宠和词条会保留。
3. 赛季内养成和竞争资源会重新开始。
4. 奖励和追赶机制会在确认经济边界后开放。

推荐表达：

新赛季重置的是竞争，不是你已经建立起来的身份与收藏。
