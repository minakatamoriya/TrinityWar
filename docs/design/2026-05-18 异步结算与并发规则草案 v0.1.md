# 2026-05-18 异步结算与并发规则草案 v0.1

## 一、文档目标

这份文档只解决两个问题：

1. 首发版本如何真正做成异步阵营经营策略战争，而不是在线手速对抗。
2. 当收获、补兵、转账、上缴、掠夺等操作同时发生时，服务器如何稳定结算。

这份草案的核心原则只有一句话：

所有关键结果以服务器在结算时刻看到的真实状态为准，玩家比的是节奏判断和长期准备，不比在线反应速度。

## 二、异步结算与并发规则

### 1. 异步的定义

首发版本里的“异步”不是指玩家永远不会撞到同一个时间窗口，而是指：

1. 玩家不需要在线守着资源成熟。
2. 玩家被攻击时，不要求立刻上线才能避免核心损失。
3. 掠夺胜负主要由出征时机、预先配置和服务器结算决定，不由即时操作抢最终结果。

只要这三条成立，游戏就是异步为主，而不是伪异步。

### 2. 首发必须避免的伪异步机制

下面几种做法会把游戏拖向即时对抗，首发不建议采用：

1. 被攻击后允许玩家立刻秒收全部成熟收益。
2. 被攻击后允许玩家瞬间把战备槽金币全部回存金库。
3. 被攻击后允许玩家临时补兵并立刻形成完整战力。
4. 实时推送“有人正在偷你，快上线抢操作”的强提醒。

这些机制会直接带来四个问题：

1. 在线玩家优势过大。
2. 玩家被迫守时，异步定位被破坏。
3. 客户端请求频率和服务器争抢冲突明显增多。
4. 玩家体验从策略经营退化成拼手速和网络。

### 3. 推荐的异步掠夺结构

首发建议采用“发起后延迟结算”的掠夺结构，而不是“点击即命中”的即时结构。

建议流程如下：

1. 攻击方选择目标并发起掠夺。
2. 服务器校验攻击方是否满足出征条件，并冻结本次出征所需兵力。
3. 服务器写入一条掠夺事件，记录目标、发起时间、预计结算时间和攻击方快照。
4. 到达结算时间后，服务器读取防守方结算时刻的真实状态。
5. 服务器统一计算攻防胜负、双方战损和本次可掠夺资产。
6. 服务器原子写入结果、战报、复仇资格和相关资源变更。

这种结构的核心价值是：

1. 攻击方能赌时机，但不能靠手速截胡瞬时空档。
2. 防守方能靠平时配置降低损失，但不需要收到提醒后拼命上线。
3. 服务器只在关键时刻结算，不需要持续高频对战同步。

### 4. 攻守双方哪些状态应当实时生效

为保证系统简单且符合直觉，首发建议把状态分成两类。

第一类是可在下次结算中生效的长期状态：

1. 城防等级。
2. 保护建筑等级。
3. 驻守兵数量。
4. 战备槽当前余额。
5. 外场当前成熟阶段。

第二类是不应在被攻击瞬间立刻翻盘的状态：

1. 刚提交的秒收请求。
2. 刚提交的秒回存请求。
3. 刚提交的临时补兵请求。
4. 刚提交的临时换防请求。

首发推荐采用下面这条统一规则：

任何会显著改变掠夺结果的操作，都应有明确冷却、耗时或结算边界，不能在被攻击瞬间无成本翻盘。

### 5. 收获规则的异步边界

外场收获是最容易与掠夺撞车的操作，必须先定义优先级。

首发建议采用以下规则：

1. 收获请求到达服务器后，只有在服务器实际校验成功并完成落库时，收益才算进入金库。
2. 还未完成服务器结算的收获请求，不视为已安全到账。
3. 如果外场已被某次掠夺事件成功命中，则该次被命中的份额不能再被收获请求重复领取。
4. 如果收获先完成并写入金库，则这部分收益不再进入日常可掠夺资产。

这条规则的本质是：

先由服务器决定哪一个请求先完成，再由结果决定资产归属。

### 6. 通知规则

通知可以做，但不要把通知做成“在线翻盘入口”。

首发建议只做三类提醒：

1. 即将成熟提醒：提醒玩家回来做收或继续赌的选择。
2. 高风险暴露提醒：提醒玩家当前已有丰熟外场或高额战备暴露。
3. 战报结果提醒：提醒玩家已经被掠夺或成功掠夺，可以查看结果与复仇资格。

首发不建议做下面这类强对抗提醒：

1. 某玩家正在攻击你，立即上线可逆转结果。
2. 你的资源将在几秒内被偷走，快点操作。

原因很直接：

1. 它会鼓励长期挂机守资源。
2. 它会制造强在线优势。
3. 它会让玩家把系统理解成即时 PvP。

### 7. 并发场景下的统一判定规则

首发建议所有资源与战斗相关请求统一遵守下面四条：

1. 客户端数字只用于展示，不作为最终结算依据。
2. 服务器收到请求后必须重新读取最新状态再判断能否执行。
3. 一个请求要么整体成功并落库，要么整体失败，不允许半扣半不扣。
4. 同一份资产在同一时刻只能被一个成功事务消耗或转移。

这四条会直接决定后面的接口、数据库和任务队列怎么设计。

## 三、服务器结算原则与请求并发处理草案

### 1. 服务器权威原则

所有关键状态必须以服务器为准，包括：

1. 金库金币。
2. 战备槽金币。
3. 外场项目状态。
4. 士兵数量。
5. 出征状态。
6. 防守配置。
7. 上缴额度。

客户端只负责展示、提交意图和接收结果，不负责判定最终数值。

### 2. 关键请求的处理模型

首发可把请求分成三类。

第一类是资源类请求：

1. 收获外场。
2. 金库转入战备槽。
3. 战备槽回存金库。
4. 金币上缴阵营。
5. 建筑升级。

第二类是兵力类请求：

1. 训练士兵。
2. 驻守调整。
3. 发起出征。

第三类是结算类请求：

1. 掠夺结算。
2. 小时分红结算。
3. 过熟衰减结算。
4. 活动奖励结算。

这三类请求都应采用“读取最新状态 -> 校验 -> 原子写入 -> 返回结果”的统一模型。

### 3. 原子事务原则

所有会改变金币、兵力或外场归属的请求，都应放在单次原子事务中处理。

单次事务至少需要保证：

1. 校验读取的是同一时刻的一致状态。
2. 扣减、增加、写战报、写日志在同一事务里完成。
3. 失败时完整回滚，不留下半成结果。

如果不这样做，就会出现下面这些首发阶段最危险的问题：

1. 金币被重复花费。
2. 同一份成熟收益被重复收取。
3. 同一支部队被同时用于训练、驻守和出征。
4. 掠夺成功但战报没写，或战报写了但资产没扣。

### 4. 版本号与幂等键

首发建议关键请求都带上版本号或幂等键。

版本号适合解决旧界面数据覆盖新状态的问题。

典型场景：

1. 玩家界面上看到战备槽有 500 金币。
2. 实际上另一条请求已经把其中 300 用于补兵。
3. 旧界面再提交上缴 400 时，服务器应发现版本不匹配并拒绝旧请求。

幂等键适合解决重复点击和网络重试问题。

典型场景：

1. 玩家连点两次收获。
2. 网络抖动导致客户端重复重发。
3. 服务器应识别这是同一操作，只执行一次。

### 5. 锁的粒度建议

首发不需要把所有状态都锁成全局大锁，但必须对关键资产采用足够小且明确的锁粒度。

建议最少对下面这些对象做行级锁或等价控制：

1. 玩家主资产账户。
2. 战备槽账户。
3. 单个外场培育位。
4. 单次出征记录。
5. 单条掠夺结算记录。

这样做的目标是：

1. 避免同一玩家的金币被并发重复扣减。
2. 避免同一块地被同时收获和被重复掠夺。
3. 避免同一场战斗被重复结算。

### 6. 外场成长的服务器实现原则

外场成长不建议做高频实时刷表。

首发建议只记录：

1. 投入时间。
2. 当前配置。
3. 成长公式。
4. 已结算状态。

玩家查看时，由服务器按当前时间临时计算阶段。

只有在下面这些关键时点才真正落库：

1. 收获成功。
2. 被掠夺成功。
3. 项目被取消。
4. 项目进入过熟并触发状态变化。

这套方式能明显降低数据库写压力，因为系统不是每分钟都在修改所有外场记录，而是在关键事件发生时结算一次。

### 7. 掠夺结算草案

首发建议掠夺结算按下面顺序执行：

1. 锁定本次掠夺记录，防止重复结算。
2. 读取攻击方已冻结兵力与运载值。
3. 读取防守方结算时刻的真实防守配置。
4. 读取防守方当前可掠夺资产。
5. 计算胜负、双方战损与最大运载量。
6. 按保护比例和优先级计算实际可掠夺收益。
7. 原子扣除防守方损失，并原子增加攻击方收益。
8. 写入双方战报、复仇标记和统计数据。
9. 将本次掠夺记录改为已结算。

这里必须守住两条：

1. 同一条掠夺记录只能成功结算一次。
2. 结算成功前，不向客户端展示最终已到账结果。

### 8. 训练、上缴、转账的并发处理草案

这三类请求最容易发生“请求前一个数字，请求后又变了”的问题。

首发建议统一做法如下：

1. 请求到达后先锁定玩家主资产和相关子账户。
2. 读取最新金币、人口、战备槽、版本号。
3. 校验本次请求是否仍满足条件。
4. 满足则执行扣减和增加，并更新版本号。
5. 不满足则返回明确失败原因。

建议失败原因至少分成下面几类：

1. 金币不足。
2. 人口不足。
3. 战备槽余额不足。
4. 版本过期，请刷新后重试。
5. 目标状态已变化，请重新确认。

这样做虽然会让部分请求失败，但它是可解释的失败，不是脏数据或错账。

### 9. 队列与定时任务建议

首发建议把下面这些操作放入任务队列或定时任务处理：

1. 掠夺延迟结算。
2. 小时分红发放。
3. 通知投递。
4. 通缉状态更新。
5. 过熟衰减扫描。

不建议把玩家点击后的收获、补兵、转账全部扔到长延迟异步队列里再慢慢处理。

更合适的做法是：

1. 玩家主动请求走实时接口。
2. 会跨时间触发的系统事件走队列。
3. 结算结果仍以数据库事务为最终落点。

### 10. 首发性能控制原则

如果采用上面的结构，服务器压力通常是可控的。

首发主要压力来自高峰期的：

1. 收获请求。
2. 出征请求。
3. 掠夺结算。
4. 分红结算。

控制压力的关键不是弱化玩法，而是限制无意义的高频写操作。

首发建议守住下面五条：

1. 外场阶段按时间计算，不做高频全量刷新。
2. 资源操作只在关键请求时落库。
3. 结算记录带状态位，避免重复扫描和重复执行。
4. 对重复点击、重试请求做幂等处理。
5. 对高频接口做基础限流和排队。

## 四、首发必须写死的结算底线

### 1. 资产底线

1. 金库金币默认不进入日常掠夺目标。
2. 已成功收获并写入金库的收益，不可再次被掠夺。
3. 同一份外场收益不能同时被收获和被掠夺。

### 2. 攻防底线

1. 掠夺胜负由服务器按结算时刻状态统一判定。
2. 防守收益主要来自预先配置，不来自被攻击后的临场秒操作。
3. 攻击方不能通过重复请求刷出多次结算结果。

### 3. 技术底线

1. 所有关键资源请求必须有原子事务保护。
2. 所有关键操作必须支持幂等。
3. 所有客户端显示数值都允许落后，但最终结算必须一致。
4. 宁可请求失败重试，也不能出现重复扣减和错账。

## 五、首发落地建议

如果首发只想做最小可用版本，建议先落下面这套组合：

1. 掠夺采用延迟结算，不做实时在线拦截。
2. 外场收获按服务器先成功落库者为准。
3. 补兵、转账、上缴全部走原子事务与版本校验。
4. 通知只做成熟提醒、高风险提醒和战报提醒。
5. 外场成长按时间公式计算，不做实时刷阶段。

只要这五条先做对，首发版本就能同时满足三件事：

1. 保住异步定位。
2. 保住数值一致性。

1. 如何保证 Trinity War 仍然是异步阵营经营策略战争游戏，而不是谁在线谁赢的即时对抗游戏。

## 二、异步结算与并发设计底线

首发版本必须守住下面六条：

1. 掠夺比的是时机判断和长期准备，不比在线反应速度。
2. 玩家在线不能通过秒收、秒转移、秒补兵完全推翻既有风险暴露。
3. 所有关键资源操作都由服务器最终判定，客户端数字只用于展示。
4. 同一时刻的冲突操作必须有明确先后顺序，不能靠客户端本地状态决定结果。
5. 被攻击方必须保留恢复能力，但不能通过临场操作完全免疫损失。
6. 首发版本优先用事件驱动结算，不做高频实时同步和持续 tick 对抗。

## 三、异步结算规则

### 1. 异步的真实含义

本项目中的异步，不等于所有结果都离线发生，也不等于玩家在线无影响。

异步的正确定义是：

1. 核心博弈来自预先配置、成熟窗口和出征时机。
2. 服务器在关键事件点统一结算，不要求双方同时在线操作。
3. 玩家在线只能影响自己当前可提交的合法操作，不能把战斗改成即时拼手速。

因此，首发版本不应设计成下面这种模式：

1. 攻击方点掠夺后立刻命中。
2. 防守方收到通知后秒收成熟资源。
3. 防守方瞬间把战备槽全部转回金库。
4. 防守方临时秒补兵并立即生效。

这类设计会把异步掠夺退化成半即时对抗，并直接抬高服务器同步压力。

### 2. 掠夺的推荐结算形态

首发建议采用发起后延迟结算，而不是点击即命中。

推荐流程如下：

1. 攻击方发起掠夺请求。
2. 服务器校验攻击资格、目标状态、兵力和体力等基础条件。
3. 校验通过后，生成一条掠夺事件，记录发起时间、预计结算时间、攻击方快照和目标标识。
4. 到达结算时点时，服务器读取防守方当时的真实状态并进行一次最终结算。
5. 结算完成后写入战报、资源变化和复仇资格。

这个结构的核心价值是：

1. 攻击方不能靠看到成熟瞬间就无条件抢走。
2. 防守方也不能靠在线蹲守实现最后一秒无成本规避。
3. 最终结果由服务器统一结算，规则边界清晰。

### 3. 收获、补兵、转账的生效边界

首发版本必须把操作分成三类：

#### 3.1 立即生效类

这类操作提交成功后立刻落库，但只能影响尚未冻结的后续事件。

建议包括：

1. 收取外场物资。
2. 从金库转入战备槽。
3. 金币上缴阵营。
4. 主城、金库、外场等建筑升级。

#### 3.2 延迟生效类

这类操作不能在点击后立即改变战斗结果，用来避免临场秒操作。

建议包括：

1. 训练士兵。
2. 战损恢复。
3. 大额资产回存。
4. 高价值防守强化。

首发建议至少保留短时间生效延迟，例如 30 秒到数分钟不等，具体数值可后调，但机制上必须存在。

#### 3.3 事件冻结类

当某个关键事件进入结算窗口后，部分数据应被冻结，不再允许被同类操作即时改写。

建议冻结的对象包括：

1. 已被掠夺事件锁定的外场格子。
2. 进入战斗结算窗口的参战兵力。
3. 当前掠夺事件要读取的战备槽可掠夺部分。

### 4. 结算窗口与冻结窗口

首发建议明确两个时间概念：

1. 结算窗口：从掠夺发起到最终计算结果的这段时间。
2. 冻结窗口：在最终结算前的一小段时间内，部分关键资产和关键状态不再接受即时改写。

推荐规则：

1. 掠夺发起后先进入飞行或行军阶段。
2. 临近最终结算时进入冻结窗口。
3. 冻结窗口内，目标外场不能被手动收走。
4. 冻结窗口内，战备槽可掠夺部分不能全部瞬间转回金库。
5. 冻结窗口内，新训练的兵不能计入本次防守。

这条规则的目的不是剥夺操作感，而是避免最后几秒的在线反应破坏异步公平性。

### 5. 通知策略

首发版本不建议把“有人正在偷你，快上线”作为核心通知。

更适合的通知分三类：

1. 即将成熟提醒：提醒玩家外场将进入成熟或丰熟阶段。
2. 高风险暴露提醒：提醒玩家当前外场或战备槽处于高暴露状态。
3. 结果战报提醒：提醒玩家已被掠夺、已成功掠夺、已获得复仇资格。

不建议首发做下面两种强实时通知：

1. 某玩家正在攻击你，请立刻上线处理。
2. 你的外场在 10 秒内将被抢走，请立刻收取。

原因如下：

1. 会把游戏从异步回流，做成被通知绑架的守时游戏。
2. 会放大在线玩家对离线玩家的优势。
3. 会刺激高频刷状态与高频同步，增加服务端压力。

### 6. 防守的正确来源

防守收益主要来自预先配置，而不是报警后的秒操作。

首发阶段建议把防守收益来源固定为：

1. 保护建筑提供的收益保留率。
2. 城防提供的掠夺比例减免。
3. 驻守兵提供的战损抬升和胜率修正。
4. 阵营或角色差异带来的轻量保护优势。

玩家在线时可以做经营调整，但不应该依赖临场手速守家。

## 四、服务器结算原则与请求并发处理草案

### 1. 总原则

服务器应遵守下面七条原则：

1. 服务器权威：最终资源、兵力、外场阶段和战斗结果都以服务器状态为准。
2. 原子提交：同一笔关键资源变动必须一次成功或一次失败，不能成功一半。
3. 幂等处理：同一请求重复到达时，只能生效一次。
4. 顺序明确：冲突请求必须有稳定的先后规则。
5. 快照结算：战斗和掠夺使用规则约定的状态快照进行计算。
6. 读写分离思维：展示状态可以宽松，扣减与结算必须严格。
7. 事件驱动：按操作触发结算，不做无意义的高频全量刷新。

### 2. 客户端数字与真实状态的关系

客户端上的金币、兵力、外场成熟度都只能视为最后一次同步时的展示值。

服务器收到请求后，必须重新读取当前状态，再决定：

1. 请求是否合法。
2. 资源是否足够。
3. 状态是否仍满足执行条件。
4. 这次操作是否与其他操作冲突。

因此，前端显示“还有 1000 金币”并不代表本次请求一定能花掉这 1000。

### 3. 请求并发冲突的推荐处理方式

#### 3.1 资源消耗冲突

典型场景：

1. 玩家同时点击补兵和上缴。
2. 玩家同时点击升级和转入战备槽。
3. 玩家在多个端重复提交同一资源消耗操作。

推荐处理规则：

1. 服务器按到达顺序进入同一玩家资源事务队列。
2. 每次只处理一个关键资源写操作。
3. 每个操作处理前重新校验余额与状态。
4. 前一个操作成功扣减后，后一个操作按新余额重新判断。
5. 不满足条件的请求直接失败，并返回最新状态。

#### 3.2 外场收取与被掠夺冲突

典型场景：

1. 玩家正要收取成熟外场。
2. 同时有一条掠夺事件进入结算窗口。

推荐处理规则：

1. 先判断该外场格子是否已经进入冻结窗口。
2. 若未进入冻结窗口，先成功提交的操作生效。
3. 若已进入冻结窗口，收取请求直接失败并提示该外场正在结算中。
4. 最终由掠夺事件按冻结时的状态统一结算。

#### 3.3 补兵与防守冲突

典型场景：

1. 玩家发现风险后立刻补兵。
2. 希望这批兵马上进入本次防守。

推荐处理规则：

1. 训练行为记录开始时间和完成时间。
2. 只有在战斗结算时已经完成训练的兵，才能计入防守。
3. 冻结窗口后新开的训练订单，不计入本次战斗。
4. 训练成功只表示订单成立，不代表战力即时生效。

### 4. 数据模型建议

首发版本至少需要为下面几类对象准备版本控制或状态标记：

1. 玩家资源池：金库、战备槽、贡献待结算值。
2. 外场格子：阶段、开始时间、预计成熟时间、冻结状态、版本号。
3. 部队状态：当前兵力、训练中兵力、恢复中兵力、版本号。
4. 掠夺事件：事件编号、攻击方、目标方、发起时间、结算时间、状态。
5. 幂等键：客户端请求唯一编号，用于去重。

建议规则：

1. 每次关键写操作都带上当前版本号或状态校验条件。
2. 更新成功后版本号递增。
3. 如果版本不一致，则返回状态已变化，请刷新后重试。

### 5. 锁与事务的推荐粒度

首发版本不需要一开始就上特别复杂的分布式事务，但必须明确最小锁粒度。

建议如下：

1. 玩家资源操作：以单玩家为粒度串行化关键扣减。
2. 单个外场格子操作：以格子为粒度加锁或条件更新。
3. 单场掠夺结算：以掠夺事件为事务边界，同时锁定攻击方必要数据、目标方必要数据和目标外场格子。

这样做的目标是：

1. 避免因为全局大锁影响吞吐。
2. 避免同一玩家在多端重复操作导致资源透支。
3. 避免同一外场格子被重复收取或重复掠夺。

### 6. 幂等与重复请求处理

移动端和小游戏环境里，重复点击、网络重试、弱网重发都很常见。

首发必须做幂等处理的请求包括：

1. 收取外场。
2. 发起掠夺。
3. 上缴金币。
4. 训练部队。
5. 建筑升级。

推荐规则：

1. 每个关键请求都带客户端生成的唯一请求号。
2. 服务器先查请求号是否已处理。
3. 如果已处理，直接返回上次结果。
4. 如果未处理，才进入正式事务。

### 7. 外场成长的计算方式

首发不建议每秒刷新所有外场的成长状态。

更合适的方案是：

1. 外场开始培育时，只写入开始时间、配置参数和收益曲线标识。
2. 玩家查看外场时，服务器根据当前时间实时计算其所处阶段。
3. 玩家执行收取、被掠夺结算、系统发提醒时，再按当下时间进行一次正式判定。
4. 只有发生关键结果变动时才落库。

这样能显著降低定时刷新和频繁写库带来的压力。

### 8. 掠夺结算的推荐顺序

首发阶段建议统一使用下面的服务端顺序：

1. 读取掠夺事件。
2. 校验事件是否已结算、是否重复执行。
3. 锁定攻击方关键状态、目标方关键状态和目标外场格子。
4. 重新计算目标当前可掠夺状态。
5. 读取冻结时刻可用的防守配置。
6. 计算攻防胜负、双方战损和可运载量。
7. 计算保护比例与最终可掠夺收益。
8. 扣减目标方外场或战备槽暴露资产。
9. 写入攻击方收益、目标方损失和双方战报。
10. 标记事件完成，并发放复仇资格或通缉状态。

任何一步失败，都应整场回滚，不允许半场落库。

### 9. 错误返回与玩家感知

并发冲突不是异常体验，关键是返回语义必须清楚。

首发建议统一几类错误提示：

1. 资源已变化，请刷新后重试。
2. 该外场正在结算中，暂时无法收取。
3. 兵力训练尚未完成，暂不能参与战斗。
4. 当前余额不足，无法完成本次操作。
5. 该请求已处理，请勿重复提交。

如果提示语清楚，玩家会把这理解为规则边界，而不是系统 bug。

## 五、首发阶段的推荐简化方案

为了尽快落地，首发建议只保留下面这些必要规则：

1. 掠夺采用发起后延迟结算。
2. 存在短冻结窗口，冻结外场收取与部分资产转移。
3. 补兵不是即时生效，而是训练完成后生效。
4. 客户端展示值不作为最终扣减依据。
5. 单玩家关键资源请求串行处理。
6. 单外场格子结算采用版本校验或条件更新。
7. 所有关键请求必须做幂等。

只要先把这七条做扎实，首发版本就能既保住异步体验，也不会把服务端复杂度一下子推到过高水平。

## 六、建议写回主设计文档的结论

后续可以把下面这段话提炼回主设计文档：

掠夺比的是出征时机、成熟窗口判断和长期准备，不比在线反应速度。防守收益主要来自预先配置，而不是报警后的临场操作。所有关键资源与战斗结果均由服务器在规则窗口内统一结算。
