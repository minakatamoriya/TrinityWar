# 勤奋玩家赛季模拟 v1 单日流程与统计口径

日期：2026-06-08

## 目标

本文件定义 `season-sim v1` 的单日模拟契约。

它回答三个问题：

1. 一天里玩家具体做什么。
2. 模拟时钟怎么推进。
3. 每一步的数据怎么记，方便后续汇总 28 天赛季。

本阶段优先级：

1. 先把单日流程跑准。
2. 再把单日流程重复 28 天。
3. 最后做总报、趋势图和调参分析。

## 范围

单日模拟先覆盖以下核心行为：

1. 种田收菜。
2. 播种。
3. 领取阵营俸禄。
4. 灵宠挂机结算。
5. 灵宠喂养。
6. 灵宠突破判断。
7. 好友灵田互助。
8. 掠夺 3 次。
9. 记录当日快照和问题日志。

暂不纳入单日主流程：

1. 洗练。
2. 亲密度长线经营策略。
3. 更复杂的活动日历。

## 单日原则

1. 单日模拟必须尽量走真实业务 service，不手写一套平行数值系统。
2. 允许卡点，但要可观察、可统计。
3. 资源不足优先记为流程状态，不立刻中止整天。
4. 只有接口异常、状态冲突、规则矛盾，才算硬错误。
5. 阵营优势默认不纳入基线单日计算；需要专项验证时再通过规则开关开启。

## 活跃度设定

基线 9 个赛季机器人按阵营平均分布，每个阵营各包含 1 名低活跃、1 名标准活跃、1 名高活跃玩家。

| 档位 | 上线时间 | 好友互助 | 掠夺 | 预期动作 |
| --- | --- | --- | --- | --- |
| 低活跃 | 9:00 / 21:00 | 9:00 / 21:00 | 3 次 | 9 次/日 |
| 标准活跃 | 8:00 / 12:00 / 18:00 / 22:00 | 8:00 / 22:00 | 3 次 | 11 次/日 |
| 高活跃 | 7:00 / 11:00 / 15:00 / 19:00 / 22:00 | 7:00 / 22:00 | 3 次 | 12 次/日 |

高活跃不使用 3 小时准点循环，避免把默认平衡基线调成极限蹲守玩家模型。8 次准点上线只保留为压力测试或专项对照，不进入默认 9 机器人基线。

## 模拟时钟

单日模拟使用独立的 `simNow`。

### 基本约定

1. 一天不是按真实 24 小时等时跑。
2. `simNow` 只服务于 `season-sim v1`。
3. 单日流程中所有“今天是否可领”“是否已冷却”“是否已成熟”的判断，都以 `simNow` 为准。
4. 每个阶段推进后，都要重新结算依赖时间的状态。

### 推荐推进方式

1. 日初：设定 `simDayStartAt`。
2. 农田阶段：按若干个“日内轮次”推进。
3. 灵宠阶段：按日结算经验和饱食时间。
4. 掠夺阶段：每次掠夺前后都可再结算一次相关状态。
5. 日终：记录 `simDayEndAt`。

### 说明

如果后续要压缩 28 天赛季，外层只需要把 `simNow` 往前拨，不需要改单日规则本身。

## 阵营优势

三阵营优势暂时剥离出基线赛季模拟。

基线目标是先验证真实玩家日常循环、活跃度收益曲线、好友互助、俸禄、养宠和掠夺本身是否稳定。阵营优势会作为单独附加条件接入专项测试，避免在基础经济尚未稳定时混入额外变量。

### 人界

1. 成熟农田收益更高。
2. 可收窗口更长。

### 仙界

1. 灵宠挂机经验更高。
2. 灵宠喂养持续时间更长。

### 魔界

1. 战斗攻击更高。
2. 战后恢复更强。

### 实现要求

1. 默认规则集为 `none`，不启用阵营优势。
2. 专项验证可使用 `factionRuleSet=current` 开启当前阵营优势配置。
3. 单日流程只读规则开关和配置，不硬编码阵营结论。
4. 后续如果调参，只改配置，不改 day runner 主体。

## 单日流程

推荐顺序如下。

### 1. 日初结算

先处理所有依赖时间的状态：

1. 农田结算到当前 `simNow`。
2. 灵宠挂机经验结算到当前 `simNow`。
3. 俸禄日期键切换检查。
4. 掠夺保护期检查。

### 2. 收菜

遍历可收的成熟/枯萎田地：

1. 先结算田地状态。
2. 找到当前可收地块。
3. 执行收菜。
4. 记录本次获得金币、植物精华或其他奖励。
5. 重新结算田地状态，直到没有可收田。

### 3. 日初好友互助

每名玩家拜访 1 位好友灵田：

1. 优先选择环形好友关系中的下一名玩家。
2. 先查看好友田地。
3. 若有成熟田，执行采摘。
4. 若无成熟田但有成长中田，执行浇水。
5. 若没有可助力田地，记录 `no-op`，不算卡点。

日初互助主要覆盖“好友已有成熟田时，勤奋玩家先采摘”的真实行为。

### 4. 播种

遍历空地：

1. 找到可播种空地。
2. 播种青灵麦。
3. 记录播种数量和种子消耗。
4. 重新结算田地状态。

### 5. 领取阵营俸禄

1. 读取当日贡献值。
2. 按贡献档位发放俸禄。
3. 记录金币、灵根、灵髓、灵玉、兽魂等发放结果。
4. 同一天重复领取要记为“已领取”，不是硬错误。

### 6. 灵宠挂机结算

1. 结算灵宠从上次到当前 `simNow` 的挂机经验。
2. 计算阵营加成。
3. 记录是否到达突破节点。

### 7. 灵宠喂养

1. 若当前灵根足够，则尝试喂养。
2. 计算饱食持续时间延长。
3. 记录灵根消耗、饱食时间变化和经验加成。
4. 灵根不足不算硬错误，记为卡点。

### 8. 灵宠突破判断

1. 到达突破节点时，检查材料。
2. 材料足够则突破。
3. 材料不足则记为成长卡点。
4. 不足时不要中止整天，只记录当前卡住的原因。

### 9. 日末好友互助

每名玩家再次拜访 1 位好友灵田：

1. 仍使用环形好友关系中的下一名玩家。
2. 仍按“可采摘优先，其次浇水”的真实入口规则。
3. 日末多数好友田地已重新播种或处于成长中，因此这一轮主要覆盖浇水缩短时间。
4. 社交助力使用模拟时钟写入日期键与助力记录时间。

### 10. 掠夺 3 次

1. 每名机器人每天执行 3 次掠夺尝试。
2. 目标必须跨阵营。
3. 目标优先选择当前可掠夺、非保护期对象。
4. 不能同阵营互掠。
5. 不能打自己。

### 11. 日终快照

1. 汇总当天动作结果。
2. 记录资源变化。
3. 记录卡点与硬错误。
4. 写入当日快照表。

## 掠夺口径

掠夺是单日流程里最需要小心的一段。

### 进攻与损失

1. A 掠夺 B 时，A 是进攻方，B 是防守方。
2. 战斗结果默认由双方灵宠面板和词条决定；阵营优势仅在专项规则集开启时参与。
3. A 成功后获得掠夺收益。
4. B 会损失可掠夺金币池中的金币。

### 扣金口径

这里的“扣金币”优先定义为：

1. 扣除 B 的可掠夺金币池。
2. 主要来源可来自田地当前暴露金币。
3. 若后续业务层把金库也纳入可掠夺对象，需要在统计里单独拆分来源。

### 结果记录

每次掠夺都要记录：

1. 进攻方是否胜利。
2. 进攻方获得多少金币。
3. 防守方损失多少金币。
4. 是否触发阵营战斗加成。
5. 是否因保护期失败。
6. 是否因目标池为空失败。

### 目标选择

1. 只能跨阵营选目标。
2. 优先选可结算目标。
3. 保护期内目标只能记为当日不可打，不应算规则错误。

## 当日统计口径

### 收入

1. 农田金币收入。
2. 阵营俸禄收入。
3. 掠夺收入。
4. 好友采摘收入。
5. 灵宠/其他奖励收入。

### 支出

1. 喂养灵宠的灵根、灵髓、灵玉、金币等。
2. 播种相关成本。
3. 掠夺造成的金币损失。

### 卡点

以下情况优先记为卡点：

1. 没有成熟田可收。
2. 没有空地可种。
3. 灵根不足，不能喂养。
4. 突破材料不足。
5. 掠夺目标处于保护期。
6. 当天俸禄已领取。

没有可社交助力对象或好友没有可助力田地记为 `no-op`，不算卡点。

### 硬错误

以下情况记为硬错误：

1. 接口报错。
2. 状态版本冲突。
3. 规则互相矛盾。
4. 状态已坏到无法继续推进。

## 实现边界

1. 单日 runner 只负责“跑一天”。
2. 28 天循环只负责重复调用单日 runner。
3. 数据分析只读快照，不倒推业务规则。
4. 后续如需更细粒度的资源分解，再扩展字段，不推翻单日契约。

## 待决项

1. 掠夺损失是否只扣可掠夺金币池，还是同时扣金库。
2. 单日内掠夺 3 次是否固定按顺序执行，还是允许目标重选。
3. 灵宠喂养是否在单日内允许多次连续执行。
4. 单日快照是否按“每玩家 1 条”还是“每玩家每阶段 1 条”记录。

## 结论

先把单日逻辑定义清楚，再做 28 天循环。

这样最利于调平衡，也最利于后面定位问题。
