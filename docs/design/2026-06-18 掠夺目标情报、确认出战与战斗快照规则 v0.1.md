# 2026-06-18 掠夺目标情报、确认出战与战斗快照规则 v0.1

## 1. 文档目的

本文用于统一掠夺目标从“看到信息”到“正式开打”的全链路规则，解决以下核心问题：

1. 目标详情和真实结算之间是否允许不一致。
2. 深度侦查拿到的精确信息是否有时效，何时失效。
3. 主宠切换、挂机升级、回血、突破、洗词条等成长变化，是否影响尚未发起的战斗。
4. 确认出战、创建订单、扣挑战次数、锁定快照之间的事务边界在哪里。
5. 如何防止因为信息过时、并发变化、重复请求和竞态条件产生严重 bug。

本文适用于当前 TrinityWar 的异步掠夺战斗模型。

## 2. 核心原则

当前版本必须坚持以下七条原则：

1. 目标详情是情报，不是结算承诺。
2. 深度侦查是精确快照，但不是永久锁定。
3. 确认出战是唯一正式事务入口。
4. 所有影响战斗结果的数据，都以确认出战时的服务端快照为准。
5. 订单一旦创建成功，本场战斗只认订单快照。
6. 主宠切换必须有冷却，避免零责任反复换宠。
7. 任何扣挑战次数、消耗资源、写订单和锁快照的动作，都必须在一个原子事务里完成。

一句话总结：

> 免费信息可以过时，付费精确信息可以失效，但真正结算永远只认确认出战时的原子快照。

## 3. 名词定义

### 3.1 目标详情

指玩家点击目标后看到的免费详情页信息。

特点：

1. 免费。
2. 可反复打开和关闭。
3. 不消耗挑战次数。
4. 不锁定目标。
5. 不保证实时。

### 3.2 深度侦查

指消耗侦查资源后获得的一次精确情报快照。

特点：

1. 消耗侦查次数、天机符或等价资源。
2. 返回精确信息。
3. 绑定 `intelToken`。
4. 绑定 `targetBattleVersion`。
5. 在目标战斗版本变化后可能失效。

### 3.3 battleVersion

指目标当前“战斗状态版本号”。

只要发生任何会影响战斗结果的变化，`battleVersion` 就必须递增。

### 3.4 确认出战

指玩家正式点击“确认出战”的动作。

这是唯一允许：

1. 校验目标状态。
2. 比对侦查版本。
3. 锁定双方快照。
4. 扣挑战次数。
5. 创建战斗订单。

的正式事务入口。

### 3.5 快照锁定

指订单创建成功后，把本场战斗所需的攻守双方战斗数据写入订单快照。

后续任何实时变化都不再影响本场。

## 4. 标准流程

### 4.1 阶段一：目标详情

目标详情页的定位是“免费读模型”。

允许行为：

1. 查看基础目标信息。
2. 查看模糊主宠信息。
3. 退出并重新进入。
4. 切换自己的主宠后再回来继续看。

目标详情页不应提供以下承诺：

1. 不承诺目标此刻主宠一定不变。
2. 不承诺目标等级在正式出战前不会变化。
3. 不承诺目标五行、词条、血量或状态不会变化。

目标详情页必须展示提示文案：

> 情报仅供参考，实际守备以确认出战时的目标快照为准。

### 4.2 阶段二：深度侦查

深度侦查的定位是“付费精确信息”。

深度侦查成功后，服务端必须返回：

1. `intelToken`
2. `observedAt`
3. `targetId`
4. `targetBattleVersion`
5. `snapshot`
6. `freeRefreshRemaining`
7. `expiresAt`

建议数据结构：

```ts
type RaidDeepIntelResult = {
  intelToken: string;
  targetId: string;
  observedAt: string;
  targetBattleVersion: number;
  freeRefreshRemaining: number;
  expiresAt: string;
  snapshot: {
    defenderMainSpiritId: string | null;
    spiritName: string | null;
    level: number | null;
    element: 'metal' | 'wood' | 'water' | 'fire' | 'earth' | null;
    rarity: string | null;
    traits: Array<{
      code: string;
      value: number;
    }>;
    hpState: 'normal' | 'low' | 'injured' | 'down';
  };
};
```

当前建议：

1. 每次深度侦查默认提供 `1` 次免费刷新纠偏额度。
2. 免费刷新只在“确认出战时发现版本变化”时触发。
3. 免费刷新不是无限实时追踪。

### 4.3 阶段三：确认出战

确认出战是整个系统最关键的原子时刻。

客户端提交建议：

```ts
type ConfirmRaidBattleRequest = {
  targetId: string;
  selectedAttackerSpiritSlotId: string;
  requestIdempotencyKey: string;
  intelToken?: string;
  expectedTargetBattleVersion?: number;
};
```

服务端必须在单个事务内完成以下动作：

1. 校验请求幂等性。
2. 重新读取攻击方当前可出战主宠。
3. 重新读取防守方当前守备主宠。
4. 重新读取双方所有影响战斗的关键状态。
5. 校验挑战次数、保护状态、阵营关系、冷却等规则。
6. 若提交了 `intelToken`，比对 `expectedTargetBattleVersion` 与当前版本。
7. 若校验通过，写入攻守双方快照。
8. 扣除挑战次数。
9. 创建战斗订单。
10. 返回订单编号和后续结算信息。

这一阶段是唯一 authoritative snapshot。

### 4.4 阶段四：订单创建后

只要订单创建成功，后续本场战斗只认订单快照。

以下变化均不得再影响本场：

1. B 切换主宠。
2. B 主宠升级。
3. B 主宠突破。
4. B 主宠洗词条。
5. B 主宠回血或掉血。
6. A 切换自己的主宠。
7. A 自己成长变化。

规则口径：

> 确认前的变化会影响本场；确认后的变化不会影响本场。

## 5. battleVersion 规则

### 5.1 必须递增的场景

以下任何变化发生时，`battleVersion` 必须递增：

1. 主宠切换。
2. 主宠等级变化。
3. 主宠突破。
4. 主宠五行变化。
5. 主宠词条变化。
6. 主宠当前 HP 变化。
7. 主宠当前健康状态变化。
8. 任何会影响 raid 战斗面板的阵营加成变化。
9. 任何会影响攻击、防御、血量、克制关系、状态系数的系统配置变化。

### 5.2 不应递增的场景

以下变化不应影响 `battleVersion`：

1. 玩家昵称变化。
2. 头像变化。
3. 留言板变化。
4. 非主宠的成长变化。
5. 与本场战斗无关的资源变化。

### 5.3 版本字段建议

建议在读取目标时始终可拿到：

```ts
type RaidBattleVersionInfo = {
  battleVersion: number;
  mainSpiritSlotId: string | null;
  updatedAt: string;
};
```

避免把版本变化判断散落在多个模块里。

## 6. 免费刷新规则

### 6.1 设计目的

免费刷新不是福利，而是纠偏机制。

它用于处理：

1. A 已经付费深度侦查。
2. B 在 A 正式确认前发生了有效战斗变化。
3. 如果系统不做纠偏，A 会因为过时精确情报被动吃亏。

### 6.2 推荐规则

每次深度侦查默认携带：

1. `freeRefreshRemaining = 1`

确认出战时若检测到：

```text
currentBattleVersion != expectedTargetBattleVersion
```

则：

1. 不扣挑战次数。
2. 不创建订单。
3. 不消耗本次正式出战。
4. 返回 `TARGET_STATE_CHANGED`。

若 `freeRefreshRemaining > 0`：

1. 服务端免费返回最新侦查快照。
2. `freeRefreshRemaining` 置为 `0`。
3. 前端提示玩家重新确认。

若 `freeRefreshRemaining == 0`：

1. 本次侦查正式失效。
2. 需要重新深度侦查，或放弃本次挑战。

### 6.3 为什么只能免费一次

如果无限免费刷新，会带来两个漏洞：

1. 玩家把深度侦查变成低成本实时透视工具。
2. 玩家故意长期挂着侦查结果监控高价值目标。

因此推荐：

> 一次侦查，一次免费纠偏；第二次变化后侦查失效。

## 7. 主宠切换冷却

### 7.1 设计目的

主宠切换冷却用于防止：

1. B 被侦查后无责任秒换克制宠。
2. B 高频切换主宠导致 A 的情报持续作废。
3. 全体玩家把守备主宠切换当成零成本应对动作。

### 7.2 推荐规则

建议守备主宠切换采用以下规则之一：

1. 每次切换后 `30` 分钟冷却。
2. 每次切换后 `60` 分钟冷却。
3. 每日最多切换 `3` 次，同时仍有单次冷却。

推荐当前版本先采用：

```text
守备主宠切换冷却 = 30 分钟
每日最多切换 = 3 次
```

### 7.3 冷却期内限制

冷却期内：

1. 不可再次切换守备主宠。
2. 可正常被挑战。
3. 已被侦查的结果依然可能因为其他成长变化而失效。

### 7.4 客户端文案

切换后建议显示：

1. `守备主宠已切换，30 分钟内不可再次更换。`
2. `本日剩余切换次数：2 次`

## 9. 标准判定案例

### 9.1 免费详情后目标换主宠

场景：

1. A 查看详情，看到 B 的主宠是苍狼。
2. B 之后切换为灵鹿。
3. A 此时才点确认出战。

结果：

1. A 最终打的是灵鹿。
2. 因为免费详情不是结算承诺。

### 9.2 深度侦查后目标换主宠

场景：

1. A 深度侦查，看到 B 的主宠是苍狼，版本为 `v18`。
2. B 切换为灵鹿，版本变为 `v19`。
3. A 点确认出战。

结果：

1. 系统发现版本不一致。
2. 不扣挑战次数。
3. 不创建订单。
4. 若还有免费刷新次数，则返回灵鹿的最新快照。

### 9.3 免费详情后目标升级

场景：

1. A 查看详情时，B 为 18 级。
2. B 在这之后挂机升到 19 级。
3. A 点确认出战。

结果：

1. 最终按 19 级开打。
2. 因为详情页不是锁定快照。

### 9.4 深度侦查后目标升级

场景：

1. A 深度侦查时，B 为 18 级，版本为 `v21`。
2. B 升到 19 级，版本变为 `v22`。
3. A 点确认出战。

结果：

1. 系统发现版本变化。
2. 触发免费刷新或侦查失效逻辑。

### 9.5 订单创建后目标再变化

场景：

1. A 成功确认出战，订单已创建。
2. B 再换主宠、再升级、再回血。

结果：

1. 本场战斗完全不受影响。
2. 只按订单中的防守快照结算。

## 10. 防 bug 总要求

### 10.1 单一事实来源

必须明确三类数据的边界：

1. `详情页数据`
   - 参考情报
   - 可过时

2. `深度侦查数据`
   - 精确情报
   - 可能失效

3. `订单快照数据`
   - 唯一结算依据
   - 创建后不可变

任何模块都不能混用这三类口径。

### 10.2 事务完整性

以下动作必须在同一个事务里完成：

1. 幂等校验。
2. 当前目标状态读取。
3. 版本对比。
4. 订单快照写入。
5. 挑战次数扣除。
6. 订单创建。

不允许出现：

1. 扣了次数但没建单。
2. 建了单但没锁快照。
3. 快照写了一半。

### 10.3 幂等性

确认出战接口必须强制要求：

1. `requestIdempotencyKey`

同一个 key 重复提交时：

1. 若订单已创建，直接返回已有订单。
2. 若上次因版本变化失败，不应重复扣挑战次数。
3. 若上次事务未完成，应有明确重试或失败恢复策略。

### 10.4 回放一致性

回放必须优先使用订单结算时保存的 `battleReplayJson` 或等价快照产物。

不允许在用户事后查看回放时，再按当前实时主宠重建战斗对象。

否则会出现：

1. 当时打的是苍狼，回放里变成灵鹿。
2. 当时打的是 18 级，回放里显示成 19 级。

### 10.5 battleVersion 漏算防线

必须有一份明确的“版本递增触发清单”，并由测试覆盖。

任何新功能若会影响战斗结果，都必须回答：

1. 它会不会影响 `battleVersion`
2. 它会不会影响侦查失效
3. 它会不会影响快照内容

## 11. 高风险 bug 清单

### 11.1 看到苍狼，打到灵鹿，但没有提示

原因：

1. 深度侦查后版本变化未被检测。
2. 免费详情和深度侦查共用同一套无版本数据。

防线：

1. 深度侦查必须绑定 `targetBattleVersion`。
2. 确认出战时强制比对版本。

### 11.2 看到 18 级，打到 19 级，但侦查没有失效

原因：

1. 升级没有触发 `battleVersion` 递增。

防线：

1. 等级变化必须递增 `battleVersion`。
2. 升级接口测试里必须补 battleVersion 断言。

### 11.3 扣了挑战次数，但订单没创建

原因：

1. 扣次数和建单不在同一事务。

防线：

1. 扣次数和建单必须一个事务。
2. 接口失败时统一回滚。

### 11.4 创建了订单，但快照缺失或不完整

原因：

1. 订单字段写入和快照字段写入拆成多步。

防线：

1. 创建订单时同步写完整快照。
2. 快照校验失败时整单回滚。

### 11.5 主宠切换冷却被并发绕过

原因：

1. 两次切换请求同时到达。

防线：

1. 切换接口在事务里校验冷却。
2. 使用行锁或版本号防并发覆盖。
3. 更新成功后立刻写入新的 `lastSwitchAt` 和 `battleVersion`。

### 11.6 免费刷新被无限刷

原因：

1. 免费刷新次数没有绑定侦查结果。
2. 每次确认失败都再次赠送纠偏。

防线：

1. `freeRefreshRemaining` 必须写入侦查上下文。
2. 只允许每次深度侦查提供一次免费刷新。

### 11.7 回放对象和真实结算对象不一致

原因：

1. 回放读取时重建了当前目标状态，而不是订单快照。

防线：

1. 回放永远优先读取订单保存结果。
2. 回放接口不再依赖当前主宠实时状态。

## 12. 测试清单

### 12.1 功能测试

1. 免费详情后目标换主宠，确认出战应打最新主宠。
2. 深度侦查后目标换主宠，确认出战应触发版本变化提示。
3. 免费详情后目标升级，确认出战应按最新等级开打。
4. 深度侦查后目标升级，确认出战应触发版本变化提示。
5. 订单创建后目标变化，不影响本场战斗。
6. 主宠切换后进入冷却，冷却内不可再次切换。
7. 深度侦查第一次失效时可免费刷新一次。
8. 深度侦查第二次失效时不再免费刷新。

### 12.2 并发测试

1. A 确认出战与 B 切主宠并发，验证最终只按事务先后结果之一成立。
2. A 确认出战与 B 升级并发，验证版本和快照一致。
3. B 连续两次切主宠，验证冷却不被绕过。
4. A 重复点击确认出战，验证只生成一个订单。

### 12.3 一致性测试

1. 订单中的 defenderSnapshot 与实际结算对象一致。
2. battleReplay 显示对象与订单快照一致。
3. 战报、回放、结算结果中的主宠名称、等级、五行一致。

## 14. 最终收口

当前版本推荐采用如下最终口径：

1. 免费详情不锁定目标，也不消耗挑战次数。
2. 深度侦查提供一次精确快照，并绑定 battleVersion。
3. battleVersion 变化后，侦查可能失效。
4. 每次深度侦查仅提供一次免费刷新纠偏。
5. 确认出战是唯一原子快照时刻。
6. 订单创建后，本场战斗完全只认订单快照。
