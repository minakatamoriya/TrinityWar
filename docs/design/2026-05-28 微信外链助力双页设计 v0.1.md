# 2026-05-28 微信外链助力双页设计 v0.1

## 1. 定位

微信外链助力不是站内社交页的替代品，而是玩家从微信分享链接进入后看到的独立轻页面。

本阶段只设计并验证两个助力场景：

1. 帮忙浇水。
2. 助力一起掠夺。

两者都不要求助力者完整进入正式游戏流程。页面只需要确认“这个人是否帮忙成功”，后续再由系统为发起者和助力者发放福利。

## 2. 首版开发边界

本次先做客户端模拟，不接微信 JS-SDK、不接真实分享链接、不接正式经济结算。

模块隔离硬规则见：

`docs/dev/2026-05-28 微信外链助力模块隔离规则 v0.1.md`

后续所有微信外链助力开发都必须遵守这条边界：测试入口、外链页、认证建档、正式助力结算、站内社交和通知模块分开实现，不能为了省事把测试按钮扩成正式流程。

首页增加四个模拟入口：

1. 新用户助力浇水流程。
2. 新用户助力掠夺流程。
3. 老用户助力浇水流程。
4. 老用户助力掠夺流程。

点击后进入独立助力页面，完成一次确认动作，并展示助力成功结果。

首版目标是验证：

1. 页面信息是否足够清楚。
2. 助力按钮是否足够明确。
3. 成功态是否能让助力者理解“我已经帮到了对方”。
4. 两类助力页面是否能作为后续微信外链承载页。
5. 新用户和老用户后续入口是否能清晰分流。

## 2.1 新老用户后续流程

当前模拟阶段不接真实微信身份，只在入口按钮上显式选择“新用户测试”或“老用户测试”。

当前四个测试入口中的“新用户/老用户”指助力者身份。被助力人固定模拟为“已注册用户”，用于验证玩家 A 已存在、玩家 B 从微信链接进入并完成助力的路径。

正式微信小游戏接入时建议：

1. 如果助力者已经有有效登录态，按老用户处理。
2. 如果没有登录态，服务端用微信 code 换取 openid/unionid。
3. 如果 openid/unionid 已绑定玩家，按老用户处理。
4. 如果 openid/unionid 未绑定玩家，先记录助力身份，再进入新手流程。
5. 新用户完成建档后再绑定待领奖励。
6. 老用户助力成功后可以直接登录并领取道具或待领奖励。

页面后续按钮建议：

| 入口类型 | 助力成功后按钮 | 后续行为 |
| --- | --- | --- |
| 新用户助力浇水 | 开始新手流程 | 进入新手建档，保留浇水助力奖励 |
| 新用户助力掠夺 | 开始新手流程 | 进入新手建档，保留助战参与奖励 |
| 老用户助力浇水 | 登录并领取奖励 | 进入游戏，展示待领取浇水奖励 |
| 老用户助力掠夺 | 登录并领取奖励 | 进入游戏，展示待领取助战奖励 |

外链助力确认本身不应该依赖助力者是否已完成完整新手教程。助力成功是一个独立事件，后续只决定奖励归属和进入路径。

## 3. 页面一：帮忙浇水

### 3.1 触发来源

玩家 A 在游戏内分享“帮我浇水”的微信链接。

玩家 B 从微信点进来后，进入独立页面：

`/share/water/:campaignId`

当前开发阶段用首页按钮模拟该入口。

### 3.2 页面信息

页面需要传达：

1. 谁需要帮忙。
2. 对方正在做什么。
3. 我点按钮会带来什么效果。
4. 我是否已经帮忙成功。

推荐文案结构：

```text
好友正在培育灵田

帮 TA 浇一次水，可以大幅提升本轮收取效率。
你完成助力后，双方都会获得一份奖励。

[帮 TA 浇水]
```

成功态：

```text
浇水成功

你已经帮 TA 完成一次浇水助力。
TA 上线后会看到你的帮助，并获得本轮收取效率提升。

[进入游戏]
```

### 3.3 设计效果

帮忙浇水的正式效果建议：

1. 显著缩短目标田地当前阶段剩余时间。
2. 或增加本轮收取效率，例如本轮成熟收益额外提升。
3. 发起者不损失任何资源。
4. 助力者获得小额奖励或待认领奖励。

首版建议先用“效率提升”作为展示口径，比具体分钟数更适合外链页。

### 3.4 限制建议

正式接入时建议：

1. 同一助力者对同一链接只能帮一次。
2. 同一链接最多接受 3 到 5 次浇水。
3. 同一发起者每天最多创建 3 个浇水链接。
4. 同一助力者每天最多帮别人 5 次。
5. 链接 24 小时过期。

## 4. 页面二：助力一起掠夺

### 4.1 触发来源

玩家 A 在掠夺目标页或战报页发起“找人助力一起掠夺”的微信链接。

玩家 B 从微信点进来后，进入独立页面：

`/share/raid/:campaignId`

当前开发阶段用首页按钮模拟该入口。

### 4.2 页面信息

助力掠夺比浇水更敏感，因为它涉及攻击他人。页面必须清楚表达：

1. 谁发起了邀请。
2. 目标是一次协同挑战，而不是直接扣助力者大量资源。
3. 助力者确认后，会为发起者提供一次战力助力。
4. 具体战斗结算以后在正式流程中完成。

推荐文案结构：

```text
好友邀请你助力掠夺

TA 正准备挑战一个较强的目标。
你确认助力后，会提供一部分协同战力，帮助 TA 提高胜率。

[确认助力]
```

成功态：

```text
助力成功

你已经加入这次协同掠夺。
正式结算时，系统会为双方发放参与奖励。

[进入游戏]
```

### 4.3 设计效果

正式效果建议：

1. 助力者不直接进入实时战斗。
2. 助力者提供一定比例战力快照，例如 50% 到 70%。
3. 发起者获得主要收益。
4. 助力者获得参与奖励，不承担过重损失。
5. 防守方仍保留保护、战报和防守奖励。

### 4.4 流程待定点

助力一起掠夺还需要继续讨论：

1. 助力者是否消耗今日助战次数。
2. 助力者是否承担灵宠血量损耗。
3. 助力后是否立即结算，还是等待发起者确认开战。
4. 分享链接是绑定目标，还是只绑定“协同邀请”。
5. 失败时助力者能拿多少安慰奖励。

首版页面不解决这些问题，只确认助力意愿。

## 5. 通用状态

两个页面都需要支持以下状态：

1. 待助力。
2. 助力成功。
3. 已经助力过。
4. 链接过期。
5. 助力次数已满。

本次客户端模拟先实现：

1. 待助力。
2. 助力成功。
3. 已经助力过。

## 7. 后续接口

游戏内创建链接：

```text
POST /api/client/share-assist/campaigns
```

外链页读取：

```text
GET /api/public/share-assist/campaigns/:id
```

确认助力：

```text
POST /api/public/share-assist/campaigns/:id/assist
```

玩家领取奖励：

```text
GET /api/client/share-assist/rewards
POST /api/client/share-assist/rewards/:id/claim
```

## 8. 验收标准

本次前端模拟完成标准：

1. 首页出现四个模拟微信入口：新用户助力浇水、新用户助力掠夺、老用户助力浇水、老用户助力掠夺。
2. 点击新用户浇水入口进入独立浇水助力页，并展示新用户测试提示。
3. 点击新用户掠夺入口进入独立掠夺助力页，并展示新用户测试提示。
4. 点击老用户浇水入口进入独立浇水助力页，并展示老用户测试提示。
5. 点击老用户掠夺入口进入独立掠夺助力页，并展示老用户测试提示。
6. 两类页面都有明确主按钮。
7. 点击主按钮后进入成功态。
8. 新用户成功态按钮展示“开始新手流程”。
9. 老用户成功态按钮展示“登录并领取奖励”。
10. 返回游戏后不影响正式首页、任务、农场、掠夺等流程。
