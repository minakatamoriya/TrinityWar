# 2026-05-30 灵宠互撞对战与数值模拟设计 v0.1

## 1. 文档目标

本文用于沉淀当前灵宠对战的第一版设计口径，并作为后续实现“模拟对战工具”的依据。

当前方向不追求复杂战斗系统，而是先做一套足够简单、可解释、可录像、可模拟的自动对战：

1. 双方灵宠只有攻击和血量两个核心战斗维度。
2. 词条负责改变流派和战斗表现。
3. 战斗采用双方同时互撞、同时扣血。
4. 单场最多 10 回合。
5. 战斗结果影响偷菜收益，但不做复杂的玩家操作档位。
6. 数值曲线不先拍脑袋定死，而是通过模拟器批量验证。

这份文档的重点不是最终平衡数值，而是先定义一套稳定的战斗规则、结算字段和模拟指标。

## 2. 设计原则

### 2.1 战斗规则必须容易理解

玩家看到录像时，应能直观看懂：

1. 双方灵宠对撞。
2. 每回合双方同时掉血。
3. 谁先倒下，谁输。
4. 打掉对方血越多，偷菜收益越高。

第一版不引入先后手、技能释放顺序、手动操作、贪婪档位、连招或复杂状态机。

### 2.2 胜负服务于偷菜，但不完全决定偷菜

灵宠对战不应该变成“赢了全拿，输了没有”。更稳定的设计是：

1. 攻方即使失败，也能获得少量基础偷取。
2. 攻方打掉守方血量越多，偷取比例越高。
3. 攻方获胜时获得额外奖励。
4. 守方胜利时减少损失，并获得守护反馈。

这样可以避免强者滚雪球，也能让弱势玩家在打得接近时仍然有正反馈。

### 2.3 先实现可配置规则，再做正式曲线

攻击、血量、等级成长、稀有度差距、词条权重，都应该从模拟结果反推。

在没有模拟器之前，不建议精细设计正式曲线。否则容易出现纸面合理、实战失衡的问题，例如：

1. 大量战斗拖满 10 回合。
2. 攻击型宠物 2 回合秒杀。
3. 血量型宠物互撞没有结论。
4. 传说宠物无脑碾压普通和稀有。
5. 等级差过大导致词条和克制失去意义。

## 3. 第一版战斗规则

### 3.1 入场状态

第一版建议每场偷菜战斗都使用完整战斗快照：

1. 攻方灵宠以本次出战时的攻击和最大血量入场。
2. 守方灵宠以守护时的攻击和最大血量入场。
3. 第一版可以先按双方满血处理，降低防守连续受击的实现复杂度。
4. 后续如需支持守方残血、防守耐久或自动回血，可在战斗快照中增加 `currentHp`。

建议第一版先采用：

```text
attackerHp = attackerMaxHp
defenderHp = defenderMaxHp
```

### 3.2 回合流程

每回合双方同时互撞：

```text
attackerDamage = calculateDamage(attacker, defender)
defenderDamage = calculateDamage(defender, attacker)

attackerHp -= defenderDamage
defenderHp -= attackerDamage
```

回合结束后判断死亡。

### 3.3 最大回合数

单场战斗最多 10 回合：

```text
maxRounds = 10
```

设计目标：

1. 优势局 3-5 回合结束。
2. 均势局 6-8 回合结束。
3. 极端血量局最多 10 回合强制结算。

### 3.4 死亡与胜负

每回合结束后判断：

| 状态 | 结果 |
|---|---|
| 攻方存活，守方死亡 | 攻方胜利 |
| 守方存活，攻方死亡 | 守方胜利 |
| 双方同回合死亡 | 守方胜利 |
| 10 回合后双方都存活 | 按剩余血量比例判定 |

10 回合未分出生死时：

```text
attackerHpRatio = attackerHp / attackerMaxHp
defenderHpRatio = defenderHp / defenderMaxHp
```

判定：

1. 攻方剩余血量比例更高，攻方小胜。
2. 守方剩余血量比例更高，守方守护成功。
3. 比例相同，守方守护成功。

这里使用血量比例，而不是绝对血量，避免血牛在超时判定中天然占优。

## 4. 偷菜收益结算

### 4.1 推荐第一版公式

偷菜收益直接和“守方损失血量比例”绑定：

```text
defenderLostHpRatio = (defenderMaxHp - max(defenderHp, 0)) / defenderMaxHp

stealRatio = baseStealRatio
           + defenderLostHpRatio * defenderLostHpStealFactor
           + attackerWinBonus
```

推荐初始配置：

```text
baseStealRatio = 20%
defenderLostHpStealFactor = 40%
attackerWinBonus = 10%
minStealRatio = 20%
maxStealRatio = 70%
```

其中 `attackerWinBonus` 只在攻方胜利时生效。

最终结果需要 clamp：

```text
finalStealRatio = clamp(stealRatio, minStealRatio, maxStealRatio)
```

### 4.2 示例

攻方击败守方：

```text
守方损失血量 = 100%
偷取比例 = 20% + 100% * 40% + 10% = 70%
```

攻方失败，但打掉守方 75% 血量：

```text
偷取比例 = 20% + 75% * 40% = 50%
```

攻方被碾压，只打掉守方 20% 血量：

```text
偷取比例 = 20% + 20% * 40% = 28%
```

### 4.3 这个公式的好处

1. 不需要玩家理解额外操作档位。
2. 战斗表现和收益连续挂钩。
3. 输了也不一定体验很差。
4. 打得越接近，收益越高，录像有意义。
5. 后续可以很容易通过配置调整偷菜强度。

## 5. 防守保护口径

第一版不要把防守回血和连续围攻做复杂。

建议先采用最小保护规则：

1. 每场防守战斗开始时，守方灵宠按满血快照进入战斗。
2. 单块菜田被偷后进入保护时间。
3. 单块菜田有累计损失上限。
4. 同一攻方对同一守方有偷菜冷却。

推荐初始配置：

```text
fieldProtectionMinutesAfterRaid = 30
fieldMaxLossRatioPerMatureCycle = 50%
sameAttackerTargetCooldownHours = 12
```

这套规则的目的：

1. 离线玩家不会被一群人无限围殴。
2. 守方不需要复杂残血恢复逻辑。
3. 攻方仍然可以通过打得更好获得更高收益。

后续如果要增加防守连续性，再考虑：

1. 守方当前血量持久化。
2. 防守后按结果回血。
3. 守护耐久。
4. 守护宠疲劳。

## 6. 数值曲线的临时基准

正式曲线应在模拟器完成后再定。为了先实现系统和模拟器，可以使用一套临时曲线。

### 6.1 同级基准

希望同等级、同稀有度、均衡模板互撞时，约 6 回合结束。

因此先设：

```text
standardHp = standardAttack * 6
```

例如：

```text
standardAttack = 100
standardHp = 600
```

### 6.2 等级成长临时值

先用温和成长：

```text
levelGrowthRate = 3%
standardAttack(level) = baseAttack * (1 + (level - 1) * levelGrowthRate)
standardHp(level) = standardAttack(level) * 6
```

这不是最终公式，只是模拟器第一版输入。

后续需要通过模拟验证：

1. 1-3 级差是否仍然可挑战。
2. 4-6 级差是否优势明显。
3. 7 级以上是否接近压制但不绝对无解。

### 6.3 稀有度预算临时值

第一版临时预算：

| 稀有度 | 同级预算倍率 | 设计意图 |
|---|---:|---|
| 普通 | 100% | 成型快、低成本、可作为有效战力 |
| 稀有 | 110% | 稳定主力 |
| 传说 | 120% | 上限更高，但后续应依赖机制和成本平衡 |

这只是模拟起点。正式目标是不让传说成为无脑最优。

### 6.4 攻血模板临时值

在同一预算下，宠物可以有不同模板：

| 模板 | 攻击倍率 | 血量倍率 | 说明 |
|---|---:|---:|---|
| 均衡型 | 100% | 100% | 标准对照组 |
| 攻击型 | 120% | 80% | 更快结束战斗，但更容易同归于尽 |
| 血量型 | 80% | 120% | 更稳定，但可能拖到回合上限 |
| 极攻型 | 135% | 65% | 用于测试秒杀风险 |
| 极血型 | 65% | 135% | 用于测试拖局风险 |

这些模板主要用于模拟器压测，不一定全部作为正式宠物模板。

## 7. 反拖局规则

如果模拟器显示血量型互撞经常拖满 10 回合，可以增加最低伤害比例。

推荐先作为可配置项，不一定默认开启：

```text
minDamageByTargetMaxHpRatio = 8%
finalDamage = max(rawDamage, targetMaxHp * minDamageByTargetMaxHpRatio)
```

设计含义：

1. 攻击太低时仍然有最低破防能力。
2. 血量仍然有价值，但不能让战斗没有结论。
3. 极端局更容易在 10 回合内出现明确结果。

需要注意：最低伤害比例会削弱血牛价值，所以必须通过模拟器判断是否开启，以及取 6%、8% 还是 10%。

## 8. 词条第一版边界

当前战斗只有攻击和血量两个核心维度，词条需要负责流派差异。

第一版词条可以先只支持纯数值和少量概率事件：

| 类型 | 示例 | 作用 |
|---|---|---|
| 攻击加成 | 攻击 +5% | 缩短战斗 |
| 血量加成 | 血量 +5% | 提高容错 |
| 暴击率 | 10% 概率造成 150% 伤害 | 增加爆发和录像波动 |
| 减伤 | 受到伤害 -5% | 克制稳定输出 |
| 吸血 | 造成伤害后回复 5% | 强化持久战 |
| 反伤 | 受到伤害时返还 5% | 克制高攻 |

第一版建议先少做词条，优先跑通事件日志和模拟器：

1. 攻击加成。
2. 血量加成。
3. 暴击。

等基础曲线稳定后，再加入吸血、反伤、护盾、斩杀等复杂词条。

## 9. 战斗日志与录像

录像不记录画面，而是记录服务端结算出的事件日志。

### 9.1 快照

战斗开始时记录双方快照：

```json
{
  "battleId": "battle_001",
  "configVersion": "spirit-collision-v0.1",
  "attacker": {
    "playerId": "player_a",
    "spiritId": "spirit_a",
    "level": 12,
    "rarity": "rare",
    "attack": 120,
    "maxHp": 720
  },
  "defender": {
    "playerId": "player_b",
    "spiritId": "spirit_b",
    "level": 12,
    "rarity": "common",
    "attack": 105,
    "maxHp": 650
  }
}
```

### 9.2 回合日志

每回合记录事件和回合后血量：

```json
{
  "round": 1,
  "events": [
    {
      "type": "hit",
      "source": "attacker",
      "target": "defender",
      "value": 120
    },
    {
      "type": "crit",
      "source": "defender",
      "target": "attacker",
      "value": 158
    }
  ],
  "after": {
    "attackerHp": 562,
    "defenderHp": 530
  }
}
```

概率事件只记录已经发生的结果。客户端播放录像时不重新随机、不重新计算胜负。

### 9.3 结算结果

```json
{
  "winner": "attacker",
  "reason": "defender_dead",
  "rounds": 6,
  "attackerHp": 108,
  "defenderHp": 0,
  "defenderLostHpRatio": 1,
  "stealRatio": 0.7
}
```

## 10. 模拟器目标

模拟器的目标不是做漂亮界面，而是批量回答平衡问题。

### 10.1 必须输出的指标

每组对战至少输出：

1. 攻方胜率。
2. 守方胜率。
3. 双方同归于尽比例。
4. 10 回合超时比例。
5. 平均回合数。
6. 平均偷取比例。
7. 偷取比例分布。
8. 攻方平均剩余血量比例。
9. 守方平均剩余血量比例。

### 10.2 必须覆盖的对照组

第一版至少跑：

1. 同级普通 vs 普通。
2. 同级普通 vs 稀有。
3. 同级稀有 vs 传说。
4. 低 3 级传说 vs 高 3 级稀有。
5. 低 5 级传说 vs 高 5 级普通。
6. 攻击型 vs 均衡型。
7. 血量型 vs 均衡型。
8. 血量型 vs 血量型。
9. 极攻型 vs 极血型。
10. 带暴击词条 vs 纯攻击词条。

### 10.3 需要重点观察的问题

1. 血量型是否经常拖满 10 回合。
2. 攻击型是否过多 2-3 回合秒杀。
3. 传说是否在低等级时仍然过强。
4. 稀有是否有足够主力价值。
5. 普通是否仍然有可用窗口。
6. 偷取比例是否过度集中。
7. 失败局是否仍然给了过高收益。
8. 双方同归于尽比例是否过高。

## 11. 推荐实现顺序

### 11.1 第一阶段：纯函数战斗核心

实现一个不依赖数据库的纯函数：

```text
simulateSpiritBattle(input, config) -> battleResult
```

输入：

1. 攻方快照。
2. 守方快照。
3. 战斗配置。
4. 随机种子或随机接口。

输出：

1. 回合事件日志。
2. 胜负。
3. 偷取比例。
4. 关键统计字段。

### 11.2 第二阶段：命令行模拟器

实现一个开发用脚本：

```text
npm run simulate:spirit-battle --workspace @trinitywar/game-server
```

脚本读取本地配置，批量跑固定对照组，输出表格。

### 11.3 第三阶段：接入偷菜结算

等模拟器验证基础曲线后，再把战斗核心接入真实偷菜结算。

第一版接入时只需要：

1. 根据双方灵宠生成战斗快照。
2. 调用战斗核心。
3. 根据 `stealRatio` 计算偷取资源。
4. 保存战斗日志。
5. 前端读取日志播放简易录像。

## 13. 当前结论

第一版可以先只做双方互撞 10 回合，不需要贪婪档位，也不需要复杂回血机制。

推荐核心口径：

1. 战斗简单：双方同时扣血，最多 10 回合。
2. 胜负清楚：死亡判胜，超时看剩余血量比例。
3. 收益连续：打掉守方血越多，偷得越多。
4. 防守安全：菜田保护和损失上限防止离线被围殴。
5. 数值后置：先实现可配置战斗核心和模拟器，再正式调攻击、血量、等级、稀有度和词条。
