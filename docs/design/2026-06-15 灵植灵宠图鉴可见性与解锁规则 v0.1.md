# 2026-06-15 灵植灵宠图鉴可见性与解锁规则 v0.1

## 1. 文档目的

本文用于收口灵植、灵宠在图鉴、社交、掠夺、战斗和回放中的“名称是否可见”与“图鉴是否点亮”口径，避免同一资源在不同界面出现互相冲突的规则。

本版重点解决 3 个问题：

1. 场景中看见名字，是否等于图鉴永久解锁。
2. 灵宠精魄并非必掉时，战斗与回放是否应免费暴露真名。
3. 灵植与灵宠是否需要完全共用同一套规则。

本版结论：

> “场景中显示名称”与“个人图鉴永久点亮”拆成两条规则线处理。  
> 灵宠名称的永久解锁只由精魄/拥有触发，不由单纯遇见、战斗、回放或窥视触发。  
> 好友灵田中的灵植名称可以公开显示，但不会因此点亮访问者自己的灵植图鉴。  
> 可见与否的规则判断以后端为准，前端只负责根据后端返回状态渲染展示。  
> 任何从“不可见”到“可见”或从“可见未解锁”到“已解锁”的状态提升，都必须即时给玩家明确提示。

## 2. 核心原则

### 2.1 拆分两种可见性

后续统一只区分两类可见性：

1. 场景可见性：当前界面是否直接显示真实名称。
2. 图鉴可见性：玩家自己的图鉴是否永久显示真实名称、进度与解锁条件。

规则要求：

1. 场景可见，不等于图鉴永久可见。
2. 图鉴永久可见后，允许所有依赖图鉴认知的界面稳定显示真实名称。
3. 回放只展示“战斗发生当时玩家被允许知道的信息”，不负责事后补发真实名称。

### 2.2 不追求“万物同一规则”，只追求“同一原则”

灵植与灵宠遵循同一原则，但场景不同：

1. 灵植是公开摆放在田里的经营对象，社交场景更强调识别与互助。
2. 灵宠是掠夺中的情报资产，战斗场景更强调侦查与收集价值。

因此：

1. 灵植允许在好友灵田中直接公开名称。
2. 灵宠默认不因“仅遇见一次”而公开名称。

### 2.3 前后端职责

为了方便多人协作与 AI 开发，后续统一按下面分工执行：

1. 后端负责可见性与解锁规则判断。
2. 后端负责返回权威状态与脱敏后的展示字段。
3. 前端负责根据后端状态渲染文案、按钮、角标、进度条和提示动画。
4. 前端不应再基于真实名称字段自行猜测“现在该不该显示真名”。

建议后端后续统一返回：

1. `codexState`: `hidden / visible-progress / unlocked`
2. `sceneVisibility`: `masked / named`
3. `displayName`: 已脱敏后的展示名称
4. `displayRarity`: 已脱敏后的稀有度展示字段
5. `unlockPromptType`: 本次是否需要弹出解锁/发现提示

## 3. 状态定义

### 3.1 图鉴状态

图鉴统一收口为 3 档：

| 状态 | 含义 | 名称 | 详情 | 进度/条件 |
|---|---|---|---|---|
| `hidden` | 玩家个人尚未拿到永久认知资格 | 隐藏 | 隐藏主体信息 | 可显示基础提示 |
| `visible-progress` | 玩家已知道该对象，但尚未完成最终解锁 | 可见 | 可见部分详情 | 展示进度与解锁条件 |
| `unlocked` | 玩家已完成该对象的正式解锁 | 可见 | 可见完整详情 | 展示已解锁状态 |

说明：

1. `visible-progress` 是“看得见名字，但还没完全拿到资格”的状态。
2. 灵植与灵宠都允许存在 `visible-progress`，但进入条件不同。

### 3.2 场景状态

场景只分两档：

| 状态 | 含义 |
|---|---|
| `masked` | 当前界面只显示“未知 / ？？” |
| `named` | 当前界面直接显示真实名称 |

## 4. 灵宠规则

### 4.1 灵宠图鉴永久可见的触发条件

灵宠图鉴从 `hidden` 进入 `visible-progress` 或 `unlocked`，只认下面几类事件：

1. 首次获得该灵宠的对应精魄。
2. 该灵宠达到待合成状态。
3. 当前拥有该灵宠。
4. 曾经拥有过该灵宠。

不作为永久点亮触发条件的事件：

1. 仅在掠夺列表中遇到该灵宠。
2. 仅在详情中看到该灵宠。
3. 仅发生一次战斗。
4. 仅观看一次战斗回放。
5. 仅执行一次窥视。

### 4.2 灵宠图鉴状态解释

| 图鉴状态 | 触发条件 |
|---|---|
| `hidden` | 没有精魄、未待合成、未拥有、未曾拥有 |
| `visible-progress` | 已获得精魄但未达到合成门槛 |
| `unlocked` | 已达到待合成，或已拥有/曾拥有 |

### 4.3 灵宠精魄门槛

当前实际采用分档门槛，不再是统一 100：

| 稀有度 | 合成所需精魄 |
|---|---|
| 普通 | 20 |
| 稀有 | 60 |
| 传说 | 100 |

当前服务端种子数据以此为准：

1. 普通：`COMMON_SPIRIT_SHARD_REQUIREMENT = 20`
2. 稀有：`RARE_SPIRIT_SHARD_REQUIREMENT = 60`
3. 传说：`LEGENDARY_SPIRIT_SHARD_REQUIREMENT = 100`

### 4.4 掠夺、战斗与回放中的名称显示

灵宠在掠夺相关场景中的名称显示统一如下：

| 场景 | 未永久解锁时 | 已永久解锁时 |
|---|---|---|
| 掠夺列表 | `？？` / `未知` | 显示真名 |
| 目标详情 | `？？` / `未知` | 显示真名 |
| 战斗中 | `？？` / `未知` | 显示真名 |
| 战斗回放 | `？？` / `未知` | 显示真名 |

本版裁决：

1. 如果对战后没有获得该灵宠精魄，则不解锁该灵宠真名。
2. 如果对战当时未被允许知道真名，则回放中依然显示未知。
3. 回放不是免费鉴定器，不能绕过精魄掉落概率直接获得永久名称认知。

### 4.5 窥视规则

本版为降低复杂度，先采用最简单口径：

1. 窥视不负责永久点亮灵宠图鉴。
2. 窥视也不负责临时公开灵宠真名。
3. 窥视不识别“这只宠物到底是哪一种”，界面仍保持 `？？` / `未知`。
4. 窥视只负责展示本次目标的属性情报，例如五行、攻击评级、防御评级、当前状态等。

后续如果需要提升窥视价值，可以单独扩展“本场临时名称可见”机制，但不在本版范围内。

## 5. 灵植规则

### 5.1 灵植图鉴永久可见的触发条件

灵植图鉴沿用个人研究/解锁链路：

1. 满足基础开放条件的灵植，按既有新手与基础流程进入可见。
2. 玩家自己完成研究发现后，图鉴进入 `visible-progress`。
3. 玩家自己完成正式解锁后，图鉴进入 `unlocked`。

### 5.2 好友灵田中的名称显示

好友灵田采用公开展示口径：

1. 只要好友田里已经种下某种灵植，拜访者可直接看到其真实名称。
2. 该显示只服务于社交互助、采摘与复活判断。
3. 看到好友田里的灵植名称，不会自动点亮拜访者自己的灵植图鉴。
4. 拜访好友灵田，也不会直接推进拜访者自己的灵植研究进度。

原因：

1. 灵植是田面公开资产，好友互助需要快速识别作物。
2. 如果好友田全部显示未知灵植，会明显损害社交操作体验。

### 5.3 灵植图鉴状态解释

| 图鉴状态 | 触发条件 |
|---|---|
| `hidden` | 玩家个人尚未发现该灵植 |
| `visible-progress` | 玩家个人已发现但尚未正式解锁 |
| `unlocked` | 玩家个人已正式解锁，可进入正常种植链路 |

## 6. 统一判断矩阵

| 对象 | 场景 | 是否显示真名 | 是否点亮个人图鉴 |
|---|---|---|---|
| 灵宠 | 掠夺中首次遇见，但未掉精魄 | 否 | 否 |
| 灵宠 | 战斗结束后掉到该灵宠精魄 | 是 | 是 |
| 灵宠 | 看战斗回放，但这场没掉精魄且原本未解锁 | 否 | 否 |
| 灵宠 | 之前已获得过该灵宠精魄 | 是 | 是 |
| 灵植 | 自己图鉴中尚未研究到 | 否 | 否 |
| 灵植 | 自己已研究发现但未解锁 | 是 | 是 |
| 灵植 | 拜访好友灵田看到对方种植的灵植 | 是 | 否 |

## 7. 提示与反馈

### 7.1 提示原则

任何状态升级都必须即时提示，不能只在图鉴里悄悄变化。

需要提示的升级包括：

1. `hidden -> visible-progress`
2. `visible-progress -> unlocked`
3. `hidden -> unlocked`

### 7.2 灵宠提示

灵宠至少需要覆盖下面几类提示：

1. 首次获得某灵宠精魄时，提示：`获得 XXX 精魄，灵宠图鉴已可见。`
2. 某灵宠精魄达到合成门槛时，提示：`XXX 已达到合成条件。`
3. 某灵宠正式合成入栏时，提示：`XXX 已入栏。`

说明：

1. 首次获得精魄对应的是“图鉴可见解锁”，不是“正式拥有解锁”。
2. 文案应明确区分“图鉴已可见”“可合成”“已拥有”三种状态，避免都叫“已解锁”。

### 7.3 灵植提示

灵植至少需要覆盖下面几类提示：

1. 灵植正式进入可研究状态时，可提示：`发现新灵植：XXX。`
2. 灵植正式解锁时，提示：`灵植 XXX 已解锁。`
3. 灵植解锁后首次可播种时，可在操作提示中补充：`已可播种。`

## 8. 实现建议

### 8.1 数据层建议

为了避免逻辑继续发散，后续实现应尽量收口为以下判断：

#### 灵宠

1. `codexVisible = shardCount > 0 || readyToCompose || ownedCurrent || ownedEver`
2. `sceneVisible = codexVisible`
3. `replayVisible = battleTimeVisible`
4. `intelVisible = true` 但 `nameVisible = false`

#### 灵植

1. `codexVisible = unlocked || discovered`
2. `friendFieldSceneVisible = true`

### 8.2 当前需要同步修正的旧口径

本版会覆盖以下旧口径：

1. “灵宠通过窥视即可永久解锁 unknown”的旧约定。
2. “灵宠普通、稀有、传说统一 100 精魄”的旧约定。
3. 所有仍写着“精魄满 100 后可合成”的前端或文档文案。
4. 前端自行根据真实名称字段判断是否隐藏名称的旧实现。

## 9. 本版裁决总结

1. 灵宠真名不是“见过就解锁”，而是“拿到精魄或拥有过才解锁”。
2. 灵宠回放不应事后暴露玩家当时无权知道的真名。
3. 窥视只看属性，不识别灵宠种类，不公开名称。
4. 灵植好友田允许直接公开名称，但不点亮拜访者图鉴。
5. 可见与否由后端权威判断，前端只负责渲染。
6. 任何状态升级都必须即时提示玩家。
7. 统一原则是“场景显示”与“图鉴永久点亮”拆开处理。
8. 当前灵宠精魄门槛为普通 20、稀有 60、传说 100。
