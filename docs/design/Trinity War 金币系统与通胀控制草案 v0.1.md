# Trinity War 金币系统与通胀控制草案 v0.1

## 一、文档目标

这份文档只解决六个问题：

1. 金币在首发版本里到底从哪里来。
2. 金币主要花到哪里去。
3. 哪些金币应当安全，哪些金币应当暴露。
4. 服务器如何保证金币结算安全、稳定、不可重复扣发。
5. 阵营上缴和每周三界争霸如何接入金币循环。
6. 如何避免金币总量爆炸式增长。

这份草案的核心原则只有一句话：

金币系统必须同时满足三件事：玩家永远有得玩、暴露资产永远有风险、总量永远不能失控膨胀。

## 二、首发金币系统的核心结构

首发继续只保留一种核心货币：金币。

但金币不能被视为一个单一数字，而必须被拆成四种不同状态：

1. 金库金币。
2. 外场投入金币。
3. 余额金币。
4. 阵营公库金币。

这四种状态的意义不是为了复杂，而是为了把“安全资产”“暴露资产”“公共资产”分开处理。

## 三、金币来源设计

首发建议把金币来源严格控制在下面五类：

1. 主城基础税收。
2. 阵营小时分红。
3. 外场成熟收取。
4. 掠夺外场收益。
5. 掠夺余额金币。

其中主城基础税收建议明确采用下面这套：

1. 按主城等级给固定小时收入。
2. 先累计为待领取税收，不直接进入金库。
3. 玩家上线后可主动领取，再进入金库。
4. 不与金库余额挂钩。
5. 不做利息型滚雪球收益。
6. 待领取收益最多保留 24 小时，超出后暂停新增并提示玩家处理。

这条规则必须写死，因为如果主城税收按金库余额计息，会同时带来三个问题：

1. 富玩家越富，通胀加快。
2. 玩家更倾向长期囤积金库，不愿暴露金币。
3. 安全金币会反过来压过外场和掠夺的意义。

此外可以保留两个轻量补充来源：

1. 日常活跃奖励。
2. 首发轻付费礼包中的少量金币补贴。

这里必须守住一条底线：

首发不要同时开放太多系统性产金来源。

如果后续再叠加市场套利、自动挂机产金、活动补贴和公共工程返利，金币总量会很快失控。

## 四、金币去向设计

如果金币来源决定玩家为什么开心，金币去向决定经济系统会不会爆掉。

首发建议至少保留下面八个核心花钱点：

1. 外场投入。
2. 主城升级。
3. 金库升级。
4. 外场位升级。
5. 训练新兵。
6. 伤兵恢复。
7. 防守建筑与保护升级。
8. 金币上缴阵营。

另外建议保留两个辅助回收点：

1. 高频便利操作的小额手续费。
2. 改阵营手续费。

这里建议明确一条：

首发不建议把“余额回存自家金库”设计成收费行为。

更合理的限制方式是：

1. 回存冷却。
2. 遭受攻击后的冻结窗口。
3. 回存频率限制。

## 五、金币安全分层

金币系统稳定的关键，不是“让玩家存不到钱”，而是让不同状态的金币承担不同风险。

### 1. 金库金币

定位：安全资产。

规则建议：

1. 已成功领取入账的金币默认进入金库。
2. 金库金币不进入日常可掠夺池。
3. 金库容量有限，逼迫玩家扩仓与调度。
4. 金库金币主要承担长期建设和保底恢复功能。

补充说明：

税收和分红在玩家点击领取前，属于待领取收益记录，不算第五种金币状态，也不能被消费或掠夺。

### 1.1 金库满仓规则

首发建议金库满仓后采用最轻量的处理方式：

1. 主城税收和阵营分红先累计到待领取收益，不自动直充金库。
2. 玩家上线后，通过领取动作把待领取收益转入金库。
3. 如果金库空间不足，先弹出确认提示，让玩家选择“取消”或“确认领取”。
4. 玩家点击取消后，本次不领取，待领取收益继续保留，玩家可先去消费或扩仓。
5. 玩家点击确认后，按当前金库可承接部分入库，超出金库上限的部分直接损失。
6. 待领取收益达到 24 小时保留上限后暂停新增。
7. 用红点或提示明确告诉玩家“有收益可领取”或“待领取已满，继续离线会浪费收益”。

这样做能同时解决三件事：

1. 避免整点结算直接撞上金库上限，出现静默溢出。
2. 玩家会被推动尽快消费或扩仓。
3. 余额仍然保留“主动暴露资产”的独立语义。

建议交互示例：

1. 玩家离线 2 小时，阵营每小时分红 100。
2. 玩家上线后收到提示：“你有 200 阵营分红可领取”。
3. 玩家点击领取后，这笔金币才正式进入金库。
4. 如果当前金库只剩 120 空间，则弹窗提示“本次可入库 120，剩余 80 将损失，是否确认领取？”。

### 1.2 默认扣费顺序

首发建议默认花费规则如下：

1. 建设、外场投入、上缴等普通操作默认从金库扣费。
2. 训练、补兵、强化等战备类操作，必须先由玩家主动把金币转入余额，再从余额扣费。
3. 余额中的金币不会被普通建设自动消耗。
4. 待领取收益不会被任何花费自动抵扣，必须先领取入库。

这套顺序建议继续保留，原因如下：

1. 普通建设属于安全成长，应主要消耗安全资金。
2. 战备类行为本身就是一次主动暴露决策，应该明确走余额。
3. 如果普通建设也自动吃余额，玩家会失去对风险暴露的掌控感。

### 2. 外场投入金币

定位：高收益暴露资产。

规则建议：

1. 投入外场后，金币立刻从安全态变成暴露态。
2. 外场成熟前不算真正收益。
3. 外场成熟后可被掠夺，但不建议被一次性全部打空。
4. 外场是首发最主要的风险收益博弈场。

如果按玩家理解来讲，外场可以直接视为“种植/培育收益位”。

只是机制上不强绑定具体题材，所以文档统一使用“外场物资”这个中性表达。

### 3. 余额金币

定位：高流动暴露资产。

规则建议：

1. 训练、补兵和部分高阶操作前，必须先把金币转入余额。
2. 余额金币可被掠夺。
3. 余额回存金库应有冷却或冻结窗口限制，不建议额外收费。
4. 余额不应成为可无限秒进秒出的避风港。

补充规则：

余额只应由玩家主动从金库转入，不应由被动收入自动灌入。

### 4. 阵营公库金币

定位：公共资产。

规则建议：

1. 玩家上缴金币后，资金进入阵营公库。
2. 阵营公库不被日常个人掠夺直接命中。
3. 阵营公库主要用于小时分红、阵营周战和公共增益。
4. 阵营公库应是个人金币系统的抽水池，而不是额外印钞机。

## 六、服务器安全结算原则

金币系统要安全稳定，后端必须先守住结算边界。

### 1. 所有改金币请求都走原子事务

下面这些请求必须统一走“读取最新状态 -> 校验 -> 原子写入 -> 失败回滚”的模型：

1. 收取外场。
2. 领取税收与分红。
3. 金库转入余额。
4. 余额回存金库。
5. 金币上缴阵营。
6. 建筑升级。
7. 训练与补兵。
8. 掠夺结算。
9. 分红结算。

### 2. 版本号与幂等必须同时做

建议至少保留下面两层保护：

1. 账户版本号校验，防止旧界面重复提交旧请求。
2. 请求幂等 ID，防止同一请求因为重试被重复扣款或重复发钱。

### 3. 金币日志必须可追踪

每一笔金币变化，建议至少记录下面字段：

1. 玩家 ID。
2. 变动前金额。
3. 变动后金额。
4. 变动类型。
5. 关联事件 ID。
6. 来源模块。
7. 时间戳。

这不仅是查 bug 用，也是后面做通胀监控和反作弊的基础。

## 七、上缴与阵营周战如何接入金币循环

你提到“每周上缴的资金或者上缴战力会不会影响战局”，答案是应该影响，而且必须影响。

否则上缴就会退化成“点一下换贡献”的空按钮。

### 1. 首发上缴的直接作用

首发建议金币上缴至少影响三件事：

1. 提高个人贡献值。
2. 提高个人小时分红权重。
3. 提高阵营公库规模。

首发不建议把“日常部队上缴”直接纳入基础循环。

更稳的原因有三点：

1. 玩家对部队的损失厌恶明显高于金币。
2. 部队已经承担进攻、防守和战损成本，再要求日常上缴会加重挫败感。
3. 日常上缴金币已经足够建立阵营贡献循环。

如果后续做每周三界争霸，可以再开放“周度军备支援”作为第二阶段内容：

1. 玩家可上缴一小部分部队或军备额度。
2. 这些支援只影响周战公共军备、周战恢复或周积分。
3. 这类支援不应成为玩家每天必须做的动作。

### 2. 每周三界争霸的接入方式

首发可以做“每周三界争霸”，但不建议做成重实时大战。

更稳的做法是：

1. 以一周为统计周期。
2. 统计全周上缴金币、阵营掠夺成绩、守家成绩和阵营活跃度。
3. 汇总成阵营周积分。
4. 周末统一结算三界争霸胜负。

### 3. 周战结果的作用范围

周战胜负建议主要影响下面三类内容：

1. 荣誉展示奖励。
2. 下周阵营基础分红的小幅加成。
3. 阵营公库的周度加成或周度保护值。

这里要特别注意：

周战奖励可以有，但不应重到足以让胜方下周继续碾压。

更适合的方向是：

1. 胜方有荣誉和轻优势。
2. 败方有保底和追赶空间。

## 八、阵营福利是否应该有差异

可以有差异，但不建议做成“强阵营一直更富、弱阵营一直更穷”。

更稳的结构是两层福利：

### 1. 固定阵营福利

首发建议按阵营定位给固定福利：

1. 仙界：防守型福利更强，例如被掠损失略低。
2. 魔界：进攻型福利更强，例如掠夺收益略高。
3. 人界：上缴与分红型福利更强，例如贡献转化效率略高。

### 2. 动态局势福利

建议系统根据每周阵营强弱，给予动态修正：

1. 弱势阵营基础分红底薪略高。
2. 弱势阵营阵营公库修复或保护成本更低。
3. 强势阵营周度统治红利变小，统治成本变高。

这样才不会把阵营差异做成长期滚雪球。

## 九、如何避免金币爆炸式增长

金币通胀不会只靠砍一个产出点解决，而要靠一整套阀门同时工作。

首发建议至少守住下面六个阀门：

1. 金库容量上限。
2. 外场持续投入成本。
3. 战损后的补兵与恢复消耗。
4. 建筑升级成本曲线。
5. 金币上缴抽水。
6. 赛季软重置回收边际增长。

另外建议再加两个轻阀门：

1. 高频便利操作的小额手续费。
2. 金库满仓后的被动收益暂停累积。

## 十、首发推荐的反通胀监控指标

如果后端想尽早发现失衡，建议至少每天看下面六项指标：

1. 全服金币总存量。
2. 全服日均产金量。
3. 全服日均金币消耗量。
4. 金库金币占比。
5. 余额金币占比。
6. 人均上缴金币与人均分红金币。

如果出现下面几种现象，就说明系统开始失衡：

1. 总存量持续上升，但消耗增长跟不上。
2. 玩家大部分金币都长时间停留在金库，不愿意暴露。
3. 分红发放量长期高于上缴抽水量。
4. 训练和补兵不再构成真实消耗。

## 十一、首发推荐的最小可用金币模型

如果首发要做最小可用版本，建议直接采用下面这套：

1. 金币只有四种状态：金库、外场、余额、阵营公库。
1. 金币只有四种状态：金库、外场、余额、阵营公库。
2. 金币主要来源压缩到税收、分红、收获、掠夺四类。
3. 金币主要消耗压缩到投入、升级、补兵、防守、上缴五类。
4. 所有改金币请求都走原子事务、版本号和幂等校验。
5. 每周三界争霸只做周度汇总结算，不做重实时大战。
6. 赛季软重置负责回收边际膨胀。

只要先守住这六条，金币系统就能同时满足三件事：

1. 玩家不会因为失败而彻底没法玩。
2. 暴露资产仍然会持续产生风险与冲突。
3. 全服金币总量不会在几周内直接失控。

## 十二、后续可继续细化的部分

后续如果继续展开，建议优先细化下面四项：

1. 主城、金库、外场位的金币消耗曲线。
2. 小时分红与上缴抽水之间的比例关系。
3. 每周三界争霸的积分计算公式。
4. 金币通胀预警阈值和后台监控报表。
