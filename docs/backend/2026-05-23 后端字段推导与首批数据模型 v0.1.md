# TrinityWar 后端字段推导与首批数据模型 v0.1

## 一、目标

这份文档解决一个最实际的问题：

后端如何根据设计文档确定字段，而不是凭感觉建表。

结论先写在前面：

字段不是从界面上猜出来的，而是从规则、状态、约束、结算时点和查询需求一起推出来的。

## 二、字段推导的统一方法

### 1. 先找名词，确定实体

设计文档里反复出现的稳定名词，通常对应实体或聚合。

例子：

1. 玩家。
2. 金库。
3. 余额。
4. 待领取收益。
5. 外场培育位。
6. 建筑。
7. 士兵。
8. 掠夺事件。
9. 战报。
10. 阵营公库。

这些对象大概率至少要在数据库里有表，或者在同一个聚合中有明确字段。

### 2. 再找状态，确定枚举、时间点和版本号

设计文档里所有“阶段”“状态”“窗口”“冷却”“成熟”“过熟”这类词，都不该只写在注释里。

它们通常会落成：

1. status 字段。
2. stage 字段。
3. start_at、mature_at、overripe_at 之类的时间字段。
4. version 字段。
5. cooldown_until 字段。

### 3. 再找动作，确定命令接口和日志

设计文档里的每个核心动作，至少要映射出下面两类东西：

1. 一条命令型接口。
2. 一类结果记录或流水。

例子：

1. 领取收益。
2. 金库转入余额。
3. 余额回存金库。
4. 外场收获。
5. 发起掠夺。
6. 掠夺结算。
7. 上缴阵营。

如果一个动作会影响资产、安全性或战斗结果，通常就不该只有最终值，没有日志。

### 4. 再找约束，确定字段精度与校验规则

设计文档里的“最多保留 24 小时”“容量上限”“保护比例”“只能主动转入”“不能自动灌入”这类语句，都会影响字段设计。

常见落点是：

1. capacity。
2. protected_ratio。
3. max_retention_seconds。
4. source_type。
5. is_manual_only。

### 5. 最后找查询需求，确定索引和冗余字段

前端和后台一定会查的列表，不要完全等接口写到一半才补字段。

常见查询包括：

1. 主城首页摘要。
2. 外场位列表。
3. 战报列表。
4. 可掠夺目标列表。
5. 阵营排行。
6. 后台按玩家检索状态。

因此某些摘要字段可以适度冗余，但规则来源必须单一。

## 三、从设计文档到字段的具体推导例子

### 1. 从“金币的四种状态”推导资产字段

来源：

1. docs/design/2026-05-18 简版游戏设计文档 v0.3.md

设计规则：

1. 金库金币安全，不可被日常掠夺直接带走。
2. 余额金币是暴露资产，可被掠夺。
3. 待领取收益最多保留 24 小时。
4. 阵营公库独立于个人日常掠夺。

推导结果：

1. 不能只做一个 player.gold 字段。
2. 至少要拆出 vault_gold、wallet_gold、pending_income、faction_treasury 这几个状态层。
3. pending_income 需要 updated_at 或 last_accrual_at 来判断 24 小时封顶。
4. vault 需要 capacity 字段，否则无法实现“领取时金库空间不足”的规则。

### 2. 从“外场成长阶段”推导外场字段

来源：

1. docs/design/2026-05-18 简版游戏设计文档 v0.3.md
2. docs/design/2026-05-18 田地时间状态机草案 v0.1.md

设计规则：

1. 外场有播种、成长、成熟、丰熟、过熟阶段。
2. 被掠不会整块消失，只减少当前可收价值。
3. 只有关键时点才真正落库。

推导结果：

1. 外场位需要 stage 或可计算阶段的时间字段。
2. 需要 invested_gold、current_claimable_gold、raided_gold_total、harvested_gold_total。
3. 需要 seed_at、mature_at、full_mature_at、overripe_at。
4. 需要 status_version，避免并发收获与掠夺覆盖旧值。

### 3. 从“发起后延迟结算的掠夺结构”推导战斗字段

来源：

1. docs/design/2026-05-18 异步结算与并发规则草案 v0.1.md

设计规则：

1. 发起掠夺时冻结兵力。
2. 到结算时间读取防守方真实状态。
3. 结算写结果、战损、战报、复仇资格。

推导结果：

1. raid_order 表要有 settle_at、status、attacker_snapshot、transport_capacity。
2. 需要 frozen_army_count 或快照结构，不可只存 attacker_id。
3. 需要 raid_settlement 或在同表保存 settlement_result、loot_gold、loss_attacker、loss_defender。
4. 需要 report_id 或单独 battle_report 记录用于前端和后台查询。

## 四、首发阶段建议的核心实体

### 1. player

用途：

1. 玩家基础身份。
2. 阵营归属。
3. 主城级别与总览入口。

关键字段建议：

1. id
2. open_id
3. nickname
4. avatar_url
5. faction_id
6. castle_level
7. last_login_at
8. created_at
9. updated_at
10. state_version

### 2. player_wallet

用途：

1. 承载玩家个人资产状态。

关键字段建议：

1. player_id
2. vault_gold
3. vault_capacity
4. wallet_gold
5. wallet_capacity
6. wallet_protected_ratio
7. pending_income_gold
8. pending_income_capped_at
9. updated_at
10. balance_version

### 3. player_field_slot

用途：

1. 承载单个外场培育位状态。

关键字段建议：

1. id
2. player_id
3. slot_index
4. status
5. invested_gold
6. current_claimable_gold
7. seed_at
8. mature_at
9. full_mature_at
10. overripe_at
11. harvested_gold_total
12. raided_gold_total
13. slot_version
14. updated_at

### 4. player_building

用途：

1. 存放主城、金库、外场位、防守建筑等成长值。

关键字段建议：

1. player_id
2. castle_level
3. vault_level
4. field_level
5. defense_level
6. updated_at
7. building_version

### 5. player_army

用途：

1. 存放兵力、驻守、冻结兵力。

关键字段建议：

1. player_id
2. total_soldier_count
3. garrison_soldier_count
4. available_soldier_count
5. frozen_soldier_count
6. wounded_soldier_count
7. updated_at
8. army_version

### 6. faction

用途：

1. 阵营基础信息和公共资产。

关键字段建议：

1. id
2. name
3. treasury_gold
4. hourly_dividend_rate
5. contribution_score
6. updated_at

### 7. raid_order

用途：

1. 存放发起后的待结算掠夺事件。

关键字段建议：

1. id
2. attacker_player_id
3. defender_player_id
4. target_field_slot_id
5. status
6. dispatched_at
7. settle_at
8. attacker_soldier_snapshot
9. attacker_transport_capacity
10. request_idempotency_key
11. settlement_version

### 8. battle_report

用途：

1. 给客户端战报页和后台查询使用。

关键字段建议：

1. id
2. raid_order_id
3. attacker_player_id
4. defender_player_id
5. result
6. loot_gold
7. attacker_loss
8. defender_loss

10. created_at

### 9. player_asset_log

用途：

1. 记录所有关键金币变化。

关键字段建议：

1. id
2. player_id

6. before_gold

## 五、后端落字段时的三条硬规则

### 1. 不要为了省表把不同语义的资产揉成一个字段

金库、余额、待领取收益、外场投入、阵营公库语义不同，不能只靠一列 gold 配业务代码硬分。

### 2. 任何会影响结算结果的状态，都要有时间或版本边界

没有版本号、时间点或幂等键，就无法稳定处理并发。

### 3. 先按业务规则定字段，再按界面需求补摘要

字段的第一来源是规则，不是页面长什么样。

## 六、当前结论

当前设计文档已经足够推导首发阶段的核心字段。

后端接下来不该再停留在“要不要建这些字段”的讨论，而应该进入：

1. 定首批表结构。
2. 定 client API。
3. 定 admin API。
4. 定 job 清单。
