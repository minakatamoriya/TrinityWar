## 二、当前本地启动口径

后端服务：

- 地址：`http://127.0.0.1:4000`
- 健康检查：`http://127.0.0.1:4000/api/health`
- 环境文件：`services/game-server/.env`

前端开发地址：

- 玩家端：`http://127.0.0.1:6001`
- 后台端：`http://127.0.0.1:6002`

Redis：

- 使用 Docker Desktop 运行 Redis 容器。
- 容器名：`trinitywar-redis`
- 地址：`redis://localhost:6379`
- 当前机器 Docker Desktop 未配置为开机自动启动；如果重启后 Redis 不可用，需要先打开 Docker Desktop，再执行 `docker start trinitywar-redis`。
- 验证命令：`docker exec trinitywar-redis redis-cli ping`
- 预期输出：`PONG`

常用启动命令：

```powershell
npm run dev
npm run worker:raid-settlement --workspace @trinitywar/game-server
npm run worker:raid-settlement:sweep --workspace @trinitywar/game-server
```

## 四、当前后台接口口径

只读接口仍然放在现有后台模块下，例如：

- 玩家查询、玩家详情、资源、钱包流水、任务、订单、灵宠和种田状态。
- 系统状态、功能开关、开放模块。
- 赛季、排行榜、奖励历史和后台概览。

写接口已经存在但必须按危险操作处理，例如：

- `POST /api/admin/players/:playerId/resources/adjust`
- `POST /api/admin/notifications/player`
- `DELETE /api/admin/players/:playerId`
- `DELETE /api/admin/configs/:category/:key`

审计查询：

- `GET /api/admin/audit-logs`
- 支持按 `action`、`targetId`、分页查询。

因此，后续文档不应再表述为“后台只有只读操作”。更准确的说法是：后台以只读排障为主，少量高风险写操作必须带原因、确认文本、写入审计，并受写权限 key 控制。
