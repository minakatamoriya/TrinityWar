# Trinity War 微信小游戏首发技术方案 v0.1

## 一、方案目标

这份方案不是面向完整版商业产品，而是面向首发玩法验证版。

目标很明确：

1. 用最少的工程复杂度，把三阵营经营策略战争的核心闭环跑起来。
2. 保证所有关键结算以后端为准。
3. 保证微信小游戏端能快速迭代和低成本测试。
4. 不在首发阶段为未来可能用不到的扩展性付出过高成本。

核心判断：

首发版本优先选择“单体后端 + 微信小游戏 2D 客户端 + PostgreSQL + Redis + 定时任务”的组合。

## 二、推荐技术栈

## 1. 客户端技术栈

推荐：

1. Cocos Creator 3.x
2. TypeScript
3. 微信小游戏原生适配

推荐理由：

1. 对微信小游戏支持成熟。
2. TypeScript 开发体验稳定，适合快速迭代。
3. 2D UI、页面切换、资源管理和状态展示足够成熟。
4. 即便你不强调画面，Cocos 也比纯原生小游戏 API 更适合做一套可维护的游戏界面。

首发阶段的使用原则：

1. 只做 2D。
2. 不做复杂骨骼动画。
3. 不做重战斗表现。
4. 把它当作一个游戏壳层和交互容器，而不是画面卖点引擎。

不推荐首发直接用的方案：

1. Unity：包体、构建链和小游戏适配成本偏高，对你当前验证目标不划算。
2. 原生小游戏 API 手搓整套 UI：技术上可行，但会显著降低界面迭代效率。
3. 小程序框架硬改成小游戏：交互模型和渲染体系都不合适。

## 2. 后端技术栈

推荐：

1. Node.js
2. NestJS
3. TypeScript
4. Prisma

推荐理由：

1. 开发速度快，适合小团队和首发验证。
2. TypeScript 可以和前端共享部分数据结构定义。
3. NestJS 模块化足够清晰，后续即使拆服务也不至于太痛苦。
4. 你的游戏逻辑以状态管理、规则结算、定时任务为主，不是极端高并发实时战斗，Node.js 完全够用。

后端定位：

1. 账户服务。
2. 游戏状态服务。
3. 定时任务服务。
4. 排行榜与缓存服务。

但首发阶段建议仍然放在一个单体应用里，不要拆成多服务部署。

## 3. 数据库与缓存

推荐：

1. PostgreSQL
2. Redis

PostgreSQL 用途：

1. 存玩家核心状态。
2. 存资源、人口、建筑、兵力。
3. 存战报、分红记录、贡献记录。
4. 存排行榜快照与任务状态。

Redis 用途：

1. 登录态缓存。
2. 目标池缓存。
3. 排行榜热点缓存。
4. 分布式锁。
5. 队列与定时任务。

首发不建议只用 MySQL 不配缓存，也不建议直接上 MongoDB。

原因：

1. 你的核心数据是强结构化状态，不适合文档型优先。
2. PostgreSQL 对事务、约束和复杂查询更稳。
3. Redis 对你的目标刷新、分红结算、战报写入削峰会很有帮助。

## 4. 队列与定时任务

推荐：

1. BullMQ
2. Redis 作为 job backend

原因：

1. 小时分红、目标刷新、护盾到期、恢复结算都适合异步任务模型。
2. 比单纯依赖 cron 更容易控重试、失败补偿和任务状态。
3. 后续扩展活动结算也能继续沿用。

首发阶段至少需要以下任务：

1. 小时分红结算任务。
2. 目标池刷新任务。
3. 人口或兵力恢复任务。
4. 排行榜刷新任务。
5. 保护状态过期任务。

## 三、推荐整体架构

首发架构建议如下：

1. 微信小游戏客户端。
2. 单体 NestJS 后端。
3. PostgreSQL。
4. Redis。
5. BullMQ worker。
6. 管理后台。

逻辑关系：

1. 客户端只负责展示和发起操作。
2. 后端 API 接收请求并校验状态。
3. 所有资源、人口、掠夺、分红都由后端计算。
4. 需要延后执行的逻辑交给队列和 worker。
5. 管理后台只给内部使用，不进入客户端。

## 四、模块划分建议

## 1. 后端模块

建议按下面方式拆 NestJS 模块：

1. auth 模块
2. player 模块
3. resource 模块
4. population 模块
5. building 模块
6. army 模块
7. raid 模块
8. report 模块
9. dividend 模块
10. contribution 模块
11. leaderboard 模块
12. job 模块
13. admin 模块

这样的拆法已经足够清晰，而且后续如果要拆服务，也能沿模块边界演进。

## 2. 客户端页面模块

建议按页面拆，而不是按复杂功能树拆：

1. 登录页
2. 阵营选择页
3. 主城页
4. 建筑页
5. 人口与部队页
6. 掠夺页
7. 战报页
8. 阵营分红页
9. 排行榜页
10. 设置页

首发阶段不要做太深的嵌套交互，优先保证每个页面信息清楚、跳转短链路。

## 五、管理后台建议

虽然首发是小游戏，但你一定要有一个内部后台。

推荐：

1. Next.js
2. TypeScript
3. Ant Design

后台职责：

1. 玩家状态查看。
2. 资源与人口修正。
3. 分红任务状态查看。
4. 战报查询。
5. 简单公告与配置。
6. 开关活动或调整参数。

首发阶段后台不需要复杂权限体系，但一定要能帮你排查问题和调数值。

## 六、数据库设计原则

首发阶段数据库不要过度范式化，也不要过度抽象。

原则：

1. 玩家主状态尽量直观可查。
2. 高频读写数据与日志数据适度分离。
3. 战报、分红、贡献日志独立建表。
4. 所有关键数值变动都尽量留日志。

推荐重点表：

1. player
2. player_resource
3. player_population
4. player_building
5. player_army
6. raid_target_pool
7. raid_report
8. dividend_log
9. contribution_log
10. leaderboard_snapshot
11. system_job_log

## 七、接口设计原则

接口要围绕页面动作设计，不要一开始追求特别通用。

首发阶段高优先接口：

1. 登录与建档接口。
2. 获取主界面状态接口。
3. 升级建筑接口。
4. 训练掠夺兵接口。
5. 获取掠夺目标列表接口。
6. 发起掠夺接口。
7. 获取战报列表接口。
8. 领取分红接口。
9. 上缴资源接口。
10. 获取排行榜接口。

核心原则：

1. 每个接口都要有幂等意识。
2. 每个扣资源接口都要能避免重复提交。
3. 每个结算接口都要能记录日志。

## 八、部署方案建议

首发验证版推荐部署到一台云服务器即可。

推荐结构：

1. 1 台 Linux 云服务器
2. Docker Compose 部署
3. Nginx
4. NestJS API 容器
5. BullMQ worker 容器
6. PostgreSQL 容器或托管数据库
7. Redis 容器或托管 Redis
8. 后台管理容器

推荐原因：

1. 足够快开工。
2. 运维成本低。
3. 便于你快速验证和调试。
4. 后续如果用户规模起来，再拆基础设施也来得及。

如果预算允许，推荐：

1. PostgreSQL 用托管服务。
2. Redis 用托管服务。

这样你只维护应用层容器即可。

## 九、环境划分建议

首发至少保留三套环境：

1. 本地开发环境。
2. 测试环境。
3. 线上环境。

测试环境职责：

1. 联调小游戏登录。
2. 验证分红定时任务。
3. 验证掠夺结算。
4. 验证数据库迁移。

线上环境职责：

1. 给真实测试用户使用。
2. 记录埋点。
3. 做灰度发布。

## 十、埋点与日志方案

你这个游戏首发是否成立，很大程度取决于数据而不是感觉。

至少要做的埋点：

1. 登录次数。
2. 分红领取次数。
3. 掠夺发起次数。
4. 掠夺成功率。
5. 被掠夺后回流率。
6. 主城升级卡点分布。
7. 上缴行为频率。
8. 三阵营人数分布。

至少要做的日志：

1. 资源变动日志。
2. 人口变动日志。
3. 掠夺结算日志。
4. 分红结算日志。
5. 系统任务执行日志。

推荐工具：

1. 应用日志用 Pino。
2. 错误监控用 Sentry。
3. 埋点数据首发可先落库或进 ClickHouse 替代方案后置。

## 十一、版本控制与发布建议

推荐：

1. GitHub 管理代码。
2. GitHub Actions 做基础 CI。
3. 自动运行 lint 和基础测试。
4. 自动构建后台和后端镜像。

小游戏发布流程建议：

1. 开发版自测。
2. 体验版小范围测试。
3. 灰度验证。
4. 正式提审。

## 十二、首发阶段不建议投入的技术点

1. 微服务拆分。
2. Kubernetes。
3. 复杂事件总线。
4. 实时 PvP 同步框架。
5. 高规格 3D 引擎表现。
6. 复杂反作弊体系。

不是这些东西没用，而是对你当前阶段，它们的性价比太低。

## 十三、推荐开发节奏

### 第 1 周到第 2 周

1. 搭项目骨架。
2. 接微信登录。
3. 建玩家主状态表。
4. 完成主界面资源展示。

### 第 3 周到第 4 周

1. 完成资源和人口系统。
2. 完成主城、住房、资源建筑升级。
3. 完成单兵种训练。

### 第 5 周到第 6 周

1. 完成掠夺目标池。
2. 完成掠夺结算。
3. 完成战报。
4. 完成保护仓和基础防守。

### 第 7 周到第 8 周

1. 完成小时分红。
2. 完成上缴和贡献。
3. 完成三张榜。
4. 接入埋点和后台。

### 第 9 周

1. 内部压测。
2. 调整主城外露资源门槛。
3. 调整分红频率和收益感。
4. 调整掠夺风险收益比。

### 第 10 周

1. 小范围体验测试。
2. 观察留存和回流。
3. 做第一轮数值修正。

## 十四、结论

如果你按微信小游戏做首发验证版，最稳妥的技术路线就是：

1. 前端用 Cocos Creator + TypeScript。
2. 后端用 NestJS + TypeScript 单体服务。
3. 数据层用 PostgreSQL + Redis。
4. 定时与异步任务用 BullMQ。
5. 部署先用单机 Docker Compose。
6. 后台用 Next.js 做内部工具。

这套方案的优点不是“最先进”，而是：

它足够稳、足够快、足够适合你现在要验证的玩法。
